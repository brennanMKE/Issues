# 0106 — Async attachment fetch on the viewer

| | |
|---|---|

## Description

Phase 5 of the remote-access plan (#0076), and the resolution of `RemoteAccess.md` Open Question 4. Today the inline thumbnail path uses `NSImage(contentsOf:)` synchronously — fine for a local file, broken for a remote attachment that hasn't arrived yet. Remote attachments need an async fetch with a placeholder, plus an actual streaming download for large media (the 4.5 MB `.mov` in #0072 is the realistic upper bound).

## Where to look

- **`Issues/Views/AttachmentThumbnailView.swift`** (or wherever inline thumbnails render — find via `grep -rn "NSImage(contentsOf:" Issues/`).
- **`Issues/Models/Issue.swift`** — `attachments: [String]` already exists. The remote source returns relative names; the local source returns absolute paths. The view layer needs a uniform way to load bytes from either.
- **New: a small `AttachmentLoader` actor** that takes `(IssueSource, attachmentName)` and returns `Data` (or an `Image`). Local source loads from disk; remote source uses `URLSession.bytes(for:)` against `/v1/folders/{id}/issues/{iid}/attachments/{name}` (#0081).

## Behavior

### Loading

```swift
actor AttachmentLoader {
    func load(name: String, from source: IssueSource) async throws -> Data
    func loadImage(name: String, from source: IssueSource) async throws -> NSImage
    var cache: [CacheKey: Data]   // small in-memory cache
}
```

- **Local source**: read the file off disk. No change from today's behavior except the call moves off the main thread.
- **Remote source**: stream via `URLSession.bytes(for:)`. Honor the `Content-Length` header to update progress. Decode the image once after bytes complete.
- **Cache**: keep decoded images in memory keyed by `(folderId, issueId, name)`. Evict on memory pressure (`NSCache` natively, or a manual LRU). Never persist to disk — the viewer is read-only and disk-cache-free per `RemoteAccess.md` non-goals.

### Quick Look path

For the existing Quick Look integration (referenced in `RemoteAccess.md` Open Q4 from #0073), the same `AttachmentLoader` materializes a temp file when Quick Look needs a URL. Write to `NSTemporaryDirectory()`, hand Quick Look the URL, delete on dismiss.

### UI

- Thumbnail view starts as a small placeholder (a generic file icon plus the filename), shows a determinate progress bar if `Content-Length > 256 KB`, replaces with the decoded image on success.
- On failure (404, network error), show an inline error glyph with a tap-to-retry; don't blank the row.
- Loading is per-issue; opening a different issue triggers a fresh load (the cache short-circuits if the bytes are already in memory).

## Verification

- Open a remote issue with a `.png` attachment → placeholder briefly, image renders within ~200 ms on LAN.
- Open #0072's 4.5 MB `.mov` over remote → progress bar shows, video preview becomes available within a couple of seconds on LAN, longer over Tailscale.
- Open the same attachment again on the next issue switch — cache hit, no network call.
- Toggle Wi-Fi off mid-load — error glyph appears with retry.
- Quick Look on a remote attachment → preview opens; closing it removes the temp file.

## Notes

- `URLSession.bytes(for:)` is the right primitive for streaming; `URLSession.shared.dataTask` would buffer the entire response into memory.
- Avoid concurrent fetches for the same `(folderId, issueId, name)` tuple — collapse to a single in-flight task with multiple awaiters. `AsyncSemaphore`-style or a simple `[CacheKey: Task<Data, Error>]` map.
- Don't move the local-attachment path off `NSImage(contentsOf:)` if it's already working — only the remote path needs the streaming fetch. Both can flow through the same `AttachmentLoader` interface, with different impls.

## Scope (out)

- Range requests / partial downloads on the server — not in #0081, not added here.
- On-disk caching — explicitly excluded by the read-only viewer model.
- Thumbnail size variants — the server doesn't generate them; the viewer scales after decode.
