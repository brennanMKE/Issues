# 0031 — Replace `DispatchQueue.main.async` in `KeyboardShortcutRecorder` with Swift Concurrency

| | |
|---|---|
| **Status** | in-progress |
| **Module** | Views |
| **Platform** | macOS |
| **First seen** | 2026-05-04 |

## Description

`Issues/Views/Settings/KeyboardShortcutRecorder.swift:28` uses `DispatchQueue.main.async` to hop out of the AppKit event handler before mutating the SwiftUI binding. Project-wide convention is modern Swift Concurrency — GCD calls aren't used anywhere else in the app.

```swift
// KeyboardShortcutRecorder.swift, line 26-32
view.onCommit = { newBinding in
    // Hop out of the AppKit event path before mutating SwiftUI state.
    DispatchQueue.main.async {
        self.binding = newBinding
    }
}
```

## Suggested change

Replace with `Task { @MainActor in ... }`. The `onCommit` closure already fires on the main thread (it's wired from `RecorderView.keyDown(with:)`), so the dispatch is purely there to defer state writes a tick — `Task { @MainActor in ... }` accomplishes the same with structured concurrency.

```swift
view.onCommit = { newBinding in
    Task { @MainActor in
        self.binding = newBinding
    }
}
```

## Notes

- The recorder's body otherwise looks fine; only the GCD call needs replacing.
- Confirm interactively that the binding still updates after a key is pressed (the visual state should return to idle and the new shortcut should appear).
