## Description

Phase 5 of the remote-access plan (#0076). Today (after #0099) the viewer reopens its remote tabs on launch but each one needs an actual `(host, port)` to connect to. This ticket wires the auto-resolve-and-reconnect flow: look up the cached token, resolve the host (Bonjour browse or persisted manual entry), and start the source — with a short timeout so a missing host doesn't block the UI.

## Where to look

- **`Issues/State/IssueStore.swift`** — owns the per-tab source lifecycle.
- **`Issues/Remote/RemoteServiceBrowser.swift`** (from #0089) — used to discover hosts by `hostId`.
- **`Issues/Remote/ViewerTokenStore.swift`** (from #0095) — fetches the token by `hostId`.
- **`Issues/State/RemoteSubscriptionStore.swift`** (from #0098) — knows which `(hostId, folderId)` pairs to reopen.

## Behavior

On launch, for each persisted remote tab:

1. Fetch the token via `ViewerTokenStore.token(forHost: hostId)`. If missing, mark the tab as "needs token" and surface the paste UI on activation (#0096).
2. Resolve the host:
   - **Bonjour-discovered hosts** — start a `RemoteServiceBrowser` and wait up to **2 seconds** for a record matching `hostId` to appear. If found, take its `endpoint`.
   - **Manually-entered hosts** — read the persisted `(host, port)` from #0091.
3. Construct `RemoteHostIssueSource` (#0094) with the resolved triple.
4. Call `start()`. The WS wrapper (#0102) takes care of any subsequent disconnects and reconnects via its own backoff schedule.

### Timeouts and fallback

- The 2-second Bonjour wait is intentionally short. If discovery is slow on this network, the tab shows "Disconnected" and the WS reconnect backoff (#0102) keeps trying. The user doesn't sit through a long blocking spinner on launch.
- If the source's first `/v1/folders/{folderId}` fetch returns `404 not_found`, the `folderId` has changed (folder moved, bookmark renewed). Fall back to **find-by-name**:
  - Fetch `/v1/folders` (still authorized).
  - If exactly one folder has a matching `project.json` `name`, silently rebind the tab to that new `folderId` and update the persisted record.
  - If zero or multiple matches, emit `.folderUnavailable` and surface "this folder moved — pick again" in #0098's manage sheet.

### Resolving multiple tabs efficiently

- Tabs that share a `hostId` share one `RemoteHostConnection` (a small object that owns the Bonjour-resolved endpoint and the WS, with multiple subscriptions). The client side already supports a single WS with multiple folder subscriptions (#0102 / #0100). Spawn one connection per host, attach all that host's tabs to it.

## Verification

- Quit and relaunch with two remote tabs from the same host → both reconnect through one WS within a couple of seconds.
- Quit, turn the host off, relaunch → tabs show "Disconnected" within 2 seconds. Turn the host back on → tabs reconnect within the WS reconnect backoff window (~10–30 seconds).
- Quit, change the host's bookmark for a folder (re-locate it via "Locate folder…"), relaunch → the viewer find-by-name rebinds and the tab loads its new `folderId`. Persisted record updates.
- Quit, revoke a token, relaunch → the affected tabs show "Token revoked" via #0104; other tabs unaffected.
- Forget a host while disconnected — its tabs close on next reconnect attempt.

## Notes

- Don't synchronously block scene construction on Bonjour. Construct tabs first, kick off resolution asynchronously.
- The "find-by-name" fallback is opportunistic. If the name is shared by two folders on the host, refusing to auto-pick is the right call — the user disambiguates manually.
- The shared-connection-per-host design makes connected-viewer rows on the host side (#0092) match user expectations: one row per WS, listing all subscribed folders.
