## Description

The app has no in-app help. Add a Help UI that renders bundled markdown so users can learn what the app does, how tabs work, what the keyboard shortcuts are, and how the IssuesSkill / AI-agent integration is intended. Reuses Textual's `StructuredText` (already a dependency from #0001) for rendering — same look as the issue markdown sheet.

## Two design options

The user explicitly flagged both; pick one during implementation.

### Option A — Single bundled markdown file

`Issues/Help/Help.md` (or `Resources/Help.md`) added to the bundle. The Help command opens a sheet (or window) containing one big `ScrollView { StructuredText(...) }` reading that file. User scrolls. Headings provide visual section breaks but no jump targets.

- **Pros**: trivial. ~20 lines of Swift. Easy to edit content — one file.
- **Cons**: long pages are hard to navigate without jump-to-section. No table-of-contents UX.

### Option B — Multiple bundled markdown files in a fixed-tab layout

A short, hardcoded list of help sections. Each is a separate `.md` file in the bundle (e.g. `Help/getting-started.md`, `Help/tabs.md`, `Help/shortcuts.md`, `Help/issues-format.md`, `Help/ai-integration.md`). Render with SwiftUI's built-in `TabView` (or a sidebar) so the user can jump between sections.

- **Pros**: organized; easy to scan; future sections drop in as new files.
- **Cons**: more setup; tabs are statically defined in code (or by enumerating bundled files at launch).

**Important constraint** (user-stated): the Help tab interface is **NOT user-reorderable**. This is *not* a clone of the main app's tab bar (which has the Safari-style reorder from #0021). Use the system `TabView` or a plain sidebar — not the custom `TabBarView`. Reordering or persistence here would just be confusing alongside the real tab bar.

**Recommendation for v1**: start with Option A (single file). If the file grows past ~3 screens of content, split into Option B. The content authoring is the bulk of the work; the UI shell is small.

## Content to cover

- **Overview** — what Issues.app is, the IssuesSkill flow, how the markdown files are the source of truth.
- **Tabs** — open a folder via the `+` button, drag to reorder, right-click for the context menu (#0010), Cmd+1…9 to jump, Cmd+Shift+arrow to step.
- **Keyboard shortcuts** — full list from #0008 (and #0024 once that lands; the help text should pull from the live binding at render time if possible, otherwise note the defaults).
- **Searching and filtering** — Cmd+F, status pills, module/platform pickers.
- **Viewing an issue** — single-click to select + see the detail panel; double-click to open the Textual sheet (#0001, #0005). Scroll the detail panel for the full markdown body (#0018).
- **Lint findings** — what the amber capsule means and what each lint rule checks (#0019).
- **AI agent integration** — the IssuesSkill writes/edits files; the app watches and updates within ~150 ms; do not edit `issues/Issues.md` by hand.
- **Privacy / sandbox** — folders are user-selected; security-scoped bookmarks persist across launches.

Aim for ~600–1000 words total. Plain prose, occasional pipe-tables for shortcut lists. Inline screenshots optional but require fresh captures kept in `Help/Images/`.

## UI shell

- **Entry point**: replace the standard Help menu via `CommandGroup(replacing: .help) { Button("Issues Help") { …open help… } }` in `IssuesApp.swift`'s `.commands` block. Bind `Cmd+?` per macOS convention (already the system default for Help).
- **Presentation**: a separate window (preferred) so the user can keep help open alongside the main window. Use the `Window` scene in `IssuesApp` (macOS 13+) with a fixed `defaultSize(width: 720, height: 800)` and `.windowResizability(.contentMinSize)`.
- **Shell view**: `HelpView` lives in `Issues/Views/HelpView.swift`. For Option A: `ScrollView { StructuredText(markdown: bundledText, baseURL: bundleURL) }`. For Option B: `NavigationSplitView` (sidebar listing sections, detail rendering the selected file) OR a `TabView` with a tab per section.
- **baseURL**: set to the bundled `Help/` directory so relative image paths in the markdown resolve.

## Notes

- **Reading bundled markdown**: `Bundle.main.url(forResource: "Help", withExtension: "md")` then `String(contentsOf:encoding:.utf8)`. Add files to the bundle via Xcode's standard "Resources" mechanism — the file-system-synchronized group should pick up `.md` files in the right folder, but verify the build settings include them in the target's Copy Bundle Resources phase.
- **Theme**: the help window honors the system color scheme post-#0020. Confirm Textual's defaults read OK against both light and dark — adjust the help view's background to `Color.appBackground` if the contrast is off.
- **Linkable shortcuts inside the help text**: out of scope. Pure prose.
- **Localization**: out of scope; English only for v1.
- **Search within help**: out of scope.
- **Migrating help content over time**: keep `Help.md` (or `Help/*.md`) in version control and update it whenever a feature changes — same discipline as PRD.md.
