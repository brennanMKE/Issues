## Description

Phase 3 of the remote-access plan (#0076). The viewer needs a small Keychain layer to securely store the bearer token per host so the user pastes it once and reconnects without re-entering it. This is the viewer counterpart to the host-side `AccessToken` model in #0078 — much simpler because the viewer just stores plaintext per host.

## Where to look

- **New: `Issues/Remote/ViewerTokenStore.swift`** — Keychain helpers. Cross-platform; iOS will reuse it as-is when #0108 lands.
- **`Issues/Remote/RemoteClient.swift`** (from #0094) — calls `ViewerTokenStore.token(forHost:)` to fetch the bearer.

## API

```swift
struct ViewerTokenStore {
    static func store(token: String, forHost hostId: String) throws
    static func token(forHost hostId: String) throws -> String?
    static func remove(forHost hostId: String) throws
    static func allHosts() throws -> [String]    // for "forget all hosts" Settings entry
}
```

- `hostId` is a stable string identifying the host. For Bonjour-discovered hosts, the Bonjour service name. For manually-entered hosts, a hash of `host:port`. The ID is set by the picker (#0096) and persisted alongside the tab state (#0099).

## Persistence

- One Keychain item per host (unlike the host side which uses a single blob — viewer-side per-host items make remove-on-revoke a clean atomic delete).
- Item attributes:
  - `kSecClass = kSecClassGenericPassword`
  - `kSecAttrService = "co.sstools.Issues.RemoteAccess.ViewerTokens"`
  - `kSecAttrAccount = hostId`
  - `kSecAttrSynchronizable = false` — token doesn't need to roam.

## Behavior

- Storing replaces any existing token for the same host (the user re-pastes a new one if the old one was revoked).
- Reading a missing item returns `nil`, not an error. Errors are reserved for actual Keychain failures (rare).
- Removal on token revocation by the host is implicit: when `RemoteClient` gets `401 invalid_token` (#0094), the viewer surfaces the expired/revoked UI (#0104) and the user can either re-paste or remove the token via "Forget this host."

## Verification

- Unit tests in `IssuesTests/`:
  - `store` then `token(forHost:)` round-trips.
  - `remove` then `token(forHost:)` returns `nil`.
  - `allHosts` lists every host that has a stored token.
  - Storing a second token for the same host replaces the first (no duplicates).
- Manual: paste a token for a host, quit and relaunch the app, reconnect → no re-prompt. Forget the host, relaunch → re-prompt.

## Notes

- Don't sync these via iCloud. The user can paste the same token onto a different Mac if needed; per `RemoteAccess.md` the user said they may copy-paste via Notes. iCloud-synced viewer tokens add a category of "did the wrong Mac just connect?" surprises.
- Tokens are plaintext `iat_…` strings. The Keychain is encrypted at rest by the OS; we don't need to layer additional encryption.

## Scope (out)

- Token paste UI — #0096.
- "Forget this host" UI — small surface in the picker / manage subscriptions sheet (#0098); not its own ticket.
- iCloud-synced token store — explicitly excluded from v1.

## Root cause

Greenfield. The viewer side's bearer storage is symmetrical with #0078 but intentionally simpler — no JSON blob, no expiry metadata, just one Keychain item per host.

## Fix

New `Issues/Remote/ViewerTokenStore.swift`. Gated `#if os(macOS) || os(iOS)` so the iOS target (#0108) can reuse it as-is.

- One generic-password item per host under `kSecAttrService = "co.sstools.Issues.RemoteAccess.ViewerTokens"`, with the `hostId` as `kSecAttrAccount`. Per-host items mean revoking is `SecItemDelete` against a single account — no read-modify-write of a shared blob.
- `store(token:forHost:)` mirrors the AccessToken pattern: `SecItemUpdate` first, fall back to `SecItemAdd` on `errSecItemNotFound`. Replaces rather than duplicates when the user re-pastes for the same host.
- `token(forHost:)` returns `nil` for missing items; `ViewerTokenError.keychain(OSStatus)` is reserved for genuine Keychain failures. A `.malformed` case is reserved for non-UTF-8 blobs (shouldn't happen — only by external tampering).
- `allHosts()` is the read for the eventual "Forget all hosts" entry. Returns the sorted list of `hostId`s under this service.
- `deleteAll(service:)` exists for tests and for the future bulk-forget affordance. Service name parameterization with the `defaultService` constant lets tests use a unique service per test and clean up in `defer`.

## Files changed

- `Issues/Remote/ViewerTokenStore.swift` — new. The five public methods + `deleteAll`.
- `IssuesTests/ViewerTokenStoreTests.swift` — new. 6 tests: round-trip, nil-on-missing, replace-not-duplicate, single-host remove leaves siblings, idempotent remove, `allHosts` listing.

## Gotchas

- **`kSecAttrSynchronizable = false`.** Token sync via iCloud is intentionally off; the spec excludes it for v1 and the host side does the same. Don't quietly flip this to `true` for "convenience" — it changes the security model (a token entered on Mac A now lives in iCloud Keychain).
- **`allHosts()` returns `[]` on `errSecItemNotFound`**, not an empty array from a successful query. Keychain's behavior when the predicate matches zero items varies by macOS release — defensive treating "not found" as "no items" keeps the result type clean.
- **No metadata on each item.** Unlike the host side which carries name/createdAt/lastUsed in the blob, viewer items are just `(hostId, plaintext)`. The viewer doesn't need the metadata: the host knows when it was last used (#0078 tracks it), and the viewer-side "name" is whatever label the picker (#0096 / #0097) renders from the Bonjour record or manual entry.
