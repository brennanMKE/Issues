# 0122 — Adopt `@concurrent` for I/O paths that should run off MainActor

| | |
|---|---|

| **Module** | Services |
| **Platform** | macOS |
| **First seen** | 2026-05-12 |

## Description

After the Swift 6 migration, several places in the codebase use `Task.detached { … }` to push I/O work off whatever actor the caller is on. That works, but Swift 6 has a more direct way: the **`@concurrent`** attribute on `async` functions, which tells the compiler "this function always runs on the concurrent executor, not the caller's actor."

The relevant rule from [SE-0461 / SE-0462](https://github.com/swiftlang/swift-evolution) and the project's `swift-guidance/references/nonisolated-concurrent.md`:

- Under Swift 6 + `NonisolatedNonsendingByDefault`, a `nonisolated async` function **inherits the caller's actor**. Calling it from MainActor stays on MainActor — no implicit hop.
- To explicitly leave isolation, mark the function `@concurrent`. The compiler then forces it onto the concurrent executor every time it's called.

`Task.detached { … }` is the pre-Swift-6 way to express the same thing. It still works, but it's noisier (always wraps the body in a closure, returns a `Task` that has to be `.value`-awaited, can't be used from a non-async context without ceremony) and it forces every caller to know the work runs off-actor. `@concurrent` on the function declaration is the modern, declarative form.

This issue is the migration: identify each `Task.detached` site, decide whether `@concurrent` is the right substitute, and convert the appropriate ones.

## Where to look

Three call sites today (`grep -rn "Task\\.detached" Issues/`):

1. **`Issues/Remote/AttachmentLoader.swift:101`** — synchronous disk read inside an actor method:

   ```swift
   return try await Task.detached(priority: .utility) {
       try Data(contentsOf: url, options: [.mappedIfSafe])
   }.value
   ```

   The detached task runs off `AttachmentLoader`'s actor so the actor's mailbox isn't blocked on the disk read. Replace with:

   ```swift
   return try await Self.readData(at: url)
   ```

   where `readData(at:)` is a new `@concurrent` static func:

   ```swift
   @concurrent
   nonisolated private static func readData(at url: URL) async throws -> Data {
       try Data(contentsOf: url, options: [.mappedIfSafe])
   }
   ```

   Same off-actor execution, no `.value` awaiting a wrapper Task, signature reads exactly what the function does.

2. **`Issues/Remote/RemoteServer.swift:643` and `:694`** — Keychain bookkeeping after a request resolves:

   ```swift
   Task.detached {
       do {
           try AccessToken.touch(hash: record.hash, from: remote)
       } catch {
           logger.warning("token touch failed: \(error.localizedDescription, privacy: .public)")
       }
   }
   ```

   `AccessToken.touch` is currently a `nonisolated static func … throws` (synchronous). To migrate, give it an async `@concurrent` overload:

   ```swift
   @concurrent
   nonisolated static func touch(
       hash: Data,
       from peer: String?,
       service: String = defaultService
   ) async throws {
       // existing Keychain read-modify-write body
   }
   ```

   And the call site becomes:

   ```swift
   Task {
       do {
           try await AccessToken.touch(hash: record.hash, from: remote)
       } catch {
           logger.warning("token touch failed: \(error.localizedDescription, privacy: .public)")
       }
   }
   ```

   The outer `Task` is now just for fire-and-forget scheduling; the actual off-actor hop is enforced by `@concurrent` on the function. No more `.detached` ceremony, and the function's signature *documents* that it always runs off-actor.

   **Tension to resolve before doing this:** there are still synchronous call sites for `touch` (e.g. tests). Either keep the sync version too (giving `AccessToken` both `touch(...) throws` and `touch(...) async throws @concurrent`), or migrate the sync callers to use `await Task { try await … }.value` for the test cases. Two overloads with the same name and different async-ness compile cleanly; it's the simplest path.

3. **A possible third site** — `AccessToken.load` and `AccessToken.save` are called from both the MainActor settings UI (`generate`, `revoke`, `list`) and from `Task.detached { touch(...) }`. After the `touch` migration, only `touch` needs an `@concurrent` overload — the underlying `load` / `save` stay `nonisolated` synchronous because they're called from both isolation contexts. **Do not** flip `load` / `save` to `@concurrent`; that would force every MainActor caller (settings sheet) to be `async`, which propagates outward.

## Where `@concurrent` is *not* appropriate

To keep the migration tight, here's where to leave existing patterns alone:

- **`Task { @MainActor in … }` in `NotificationService`** (#0121). The work *requires* MainActor isolation (touches `NSApp`, `NSWindow`, etc.). `@concurrent` would forbid the access.
- **Synchronous `nonisolated` reads** like `IssueSource.folderURL`. Already nonisolated; calling them from any actor is a free, in-isolation read. No async needed, so `@concurrent` doesn't apply.
- **Closures inside SwiftUI views.** SwiftUI's body builder is `@MainActor`; introducing `@concurrent` async funcs called from views requires `Task { await … }` ceremony around each call. Not worth it for UI-bound operations.
- **`URLSession.bytes(for:)` and the WebSocket receive loop.** These already manage their own executor; adding `@concurrent` would be a no-op at best.

## Behavior

After migration:

- `AttachmentLoader.loadRaw` reads files off-actor exactly as it does today; the only difference is the call site doesn't allocate a wrapper `Task`.
- `AccessToken.touch` (async overload) runs off MainActor when called from MainActor contexts, off-actor when called from actor contexts — `@concurrent` guarantees neither caller hops onto MainActor for the Keychain write.
- The synchronous `AccessToken.touch` overload stays callable from tests (which run on MainActor and don't need to await).
- `Task.detached` count in the codebase drops from 3 to 0 (or close to it, depending on whether tests still need it).

## Verification

- **Build clean** on Swift 6 with no warnings.
- **Tests pass** — including any test that exercises `AccessToken.touch` via the new async overload.
- **Manual smoke**: open a remote folder (real or curl-equivalent), trigger a request, confirm `lastUsedAt` updates on the host's token list. (`#0087` smoke CLI exercises the path.)
- **Manual smoke 2**: open an issue with a screenshot attachment on a *local* folder; thumbnail renders. Open an issue with a screenshot on a remote folder; thumbnail renders. No new lag or blocking on the main thread.
- **Activity Monitor / Instruments**: the disk read inside `AttachmentLoader.loadRaw` shouldn't show up on a MainActor backtrace; it should appear on a concurrent-executor thread instead. (Confirmation rather than measurement — the difference vs. today should be zero in practice.)

## Notes

- This is a small refactor, not a behavior change. The reason to do it is *readability* and *self-documentation*: `@concurrent` on the function says "I always run off-actor" right there in the declaration. `Task.detached` says nothing at the call site about where the work runs.
- The `swift-guidance/references/nonisolated-concurrent.md` reference walks through the SE-0461 model in detail. Worth re-reading before the implementation PR opens.
- `@concurrent` is **Swift 6 only**. The current project is on `SWIFT_VERSION = 6.0`, so this is unblocked.
- The `nonisolated` keyword on the function declaration is **redundant** when `@concurrent` is also present (`@concurrent implies nonisolated`), but writing both is clearer. The references file calls this out explicitly.
- Don't try to combine `@concurrent` with `@MainActor` or any actor isolation — the compiler rejects it (rule 3 in `nonisolated-concurrent.md`).
- Don't access actor-isolated state from `@concurrent` functions — also a compile error (rule 4). For `AttachmentLoader.readData(at:)` and `AccessToken.touch`, the bodies touch only the URL and Keychain APIs respectively, so this is fine.

## Scope (out)

- Audit of every `await` in the codebase for unnecessary actor hops. Out of scope for v1 of this ticket; just convert the three known `Task.detached` sites. Wider audit is a follow-up if it ever pays off.
- Migrating the host-settings UI's synchronous `AccessToken.generate` / `revoke` / `list` calls to async. They're already on MainActor and the operations are fast (Keychain calls are typically sub-millisecond); no benefit to making the UI await them.
- Changing `Task { @MainActor in … }` patterns. Those are correct as-is; `@concurrent` doesn't apply to MainActor-bound work.
- Profiling the Keychain write performance. Today it's fire-and-forget after a request; the user doesn't feel it either way.

## Dependencies

- **Depends on the Swift 6 migration being committed** (the staged xcconfig + edits). `@concurrent` is a Swift 6 attribute; trying to use it on Swift 5 won't compile.
