## Description

Second step of the TLS migration (#0111). With the host generating and persisting a cert identity in #0112, this ticket extends the token transfer flow so the cert's fingerprint travels with the bearer. The viewer pins both at paste time and uses the fingerprint to verify TLS connections later (#0114).

## Wire format

The displayed token string carries two parts joined with a `.`:

```
iat_<43 base64url chars>.<64 hex sha256 chars>
```

Total length: **108 characters**. Still a single line, still copy/pasteable, still QR-safe for #0108's iPhone path.

- Left of `.` — the bearer plaintext, unchanged from #0078.
- Right of `.` — the host's current cert fingerprint at the moment the token was generated. Lowercase hex.

Token *records* on the host don't store the fingerprint — the fingerprint is a property of the identity (#0112), not of the token. The host always returns its current cert's fingerprint via TLS; the pinned value just needs to match.

When the host's cert rotates (#0115), all existing tokens still look valid as plaintext but their **embedded fingerprint no longer matches**, so the viewer rejects the TLS handshake. Effectively, cert rotation invalidates all viewers until they re-paste — which is the intended UX.

## Where to look

- **`Issues/Remote/AccessToken.swift`** (from #0078) — `generate(name:expiresAt:)` returns `(plaintext: String, record: TokenRecord)`. Extend to `(combined: String, plaintext: String, fingerprint: String, record: TokenRecord)`. `combined` is what the user sees; the others are for tests / future internals.
- **`Issues/State/RemoteHostController.swift`** — passes the loaded identity (#0112) to `AccessToken.generate(name:expiresAt:identity:)` so the fingerprint comes from the live identity rather than being recomputed every call.
- **`Issues/Views/GenerateTokenSheet.swift`** (from #0084) — the one-time reveal sheet displays the **combined** string. The Copy button copies the combined value.
- **`Issues/Views/AccessTokenListView.swift`** (from #0084) — no per-row change; the list still shows token name + created + last used + expires.
- **`Issues/Views/RemoteTokenPasteView.swift`** (from #0096) — the input field accepts the combined form, parses the two parts, validates each. "Continue" stays disabled until both parts look well-formed.
- **`Issues/Remote/ViewerTokenStore.swift`** (from #0095) — stores `(token, fingerprint)` per host, not just the token.

## API changes

### Host

```swift
struct AccessToken {
    static func generate(
        name: String,
        expiresAt: Date?,
        identity: RemoteServerIdentity
    ) throws -> Generated

    struct Generated {
        let combined: String          // "iat_…<43>.<64-hex>"
        let plaintext: String         // "iat_…<43>"
        let fingerprint: String       // "<64-hex>"
        let record: TokenRecord
    }
}
```

### Viewer

```swift
struct ViewerTokenStore {
    static func store(token: String, fingerprint: String, forHost hostId: String) throws
    static func entry(forHost hostId: String) throws -> Entry?       // replaces token(forHost:)
    static func remove(forHost hostId: String) throws
    static func allHosts() throws -> [String]

    struct Entry: Equatable {
        let token: String
        let fingerprint: String
    }
}
```

Keychain item keyed by `hostId` now stores a small JSON blob `{"token": "...", "fingerprint": "..."}` instead of just the token string. This is a one-key-per-host setup; the migration on first-run-after-update is "if the stored value parses as JSON with both fields, use it; otherwise treat as a legacy bare token and surface a re-paste prompt." Single-user, two devices — re-pasting once is fine.

## Paste-side parsing

`RemoteTokenPasteView` accepts:

- The combined `iat_<43>.<64-hex>` form (the canonical case).
- A whitespace-tolerant variant (trim leading/trailing whitespace, no other normalization).

Reject:

- Bare `iat_<43>` with no `.<fingerprint>` — surface "this token is from an older host; ask the host to regenerate it" inline. Don't try to negotiate or fall back.
- Anything else — generic "this doesn't look like a token" inline error.

On Continue:

1. Parse into `(token, fingerprint)`.
2. Store via `ViewerTokenStore.store(token:fingerprint:forHost:)`.
3. Hand off to the validation step (#0094) — which in #0114 will actually perform the TLS handshake with the pin.

## Behavior

- The Generate sheet (#0084) shows the combined string and ONLY the combined string. No separate "your fingerprint is …" line — that splits the user's attention. Tooltip on the field can explain "this includes the host's TLS fingerprint so your other device can recognize the host".
- The Copy button copies the combined value to the clipboard.
- The QR-code variant (mentioned for #0108) encodes the combined value — single scan, full credential.
- The host's token list (#0084) does *not* display fingerprints. The fingerprint is a property of the host's cert, not the token; showing it per-row is noise. (One host setting could expose the current cert's fingerprint for debugging — that's part of #0115's surface.)

## Verification

- Generate a token. The displayed string is 108 characters: `iat_` + 43 + `.` + 64 hex. Copy works.
- Token list shows the new token with `Last used —`. No per-row fingerprint column.
- Unit tests in `IssuesTests/`:
  - `AccessToken.generate(...)` returns a `Generated` whose `combined` equals `plaintext + "." + fingerprint`.
  - `fingerprint` matches the identity's `fingerprintHex`.
  - `record.hash` is the SHA-256 of the **plaintext only**, not the combined string.
- Paste UI tests:
  - Combined string → parses, both halves stored.
  - Bare `iat_<43>` → inline error suggesting regenerate.
  - Garbage → inline error.
  - Leading/trailing whitespace → trimmed.
- `ViewerTokenStore` tests:
  - Round-trip stores and reads both fields.
  - Reading a host with only a token (legacy) returns `nil` (signals "needs re-paste").
  - `allHosts` lists every entry regardless of legacy/new format.

## Notes

- The `.` separator is intentional. `iat_` and `.` are both invalid base64url, so the boundary is unambiguous. Future formats could carry more parts (`iat_<tok>.<fp>.<v2hint>`); parsing stays trivial.
- We don't encrypt or wrap the combined string further — it's already exactly as sensitive as the bearer token alone (compromise of either is equivalent to compromise of both for a given host).
- The host's `lastUsedAt` tracking continues to update on incoming bearer use; the fingerprint isn't a per-request thing.

## Scope (out)

- Viewer's actual cert-pinning during the TLS handshake — #0114.

## Root cause

#0112 generated and persisted the host cert. Without the fingerprint reaching the viewer, the cert can't be pinned at TLS handshake time. The token transfer was the natural channel — it already moves a host-bound secret one direction.

## Fix

Three layers updated to carry the fingerprint end-to-end:

- **`AccessToken.swift`** — new `Generated` struct with `combined: String`, `plaintext: String`, `fingerprint: String`, `record: TokenRecord`. New identity-aware overload `generate(name:expiresAt:identity:service:) throws -> Generated` that pulls the fingerprint from the live `RemoteServerIdentity` (#0112). The pre-#0113 no-identity overload stays for tests that don't care about the fingerprint. `parseCombined(_:) throws -> (plaintext, fingerprint)` is the inverse — whitespace-tolerant, rejects bare tokens with `.malformed`, validates the 64-char lowercase-hex fingerprint shape.
- **`ViewerTokenStore.swift`** — new `Entry: Codable` shape carries both fields. `store(token:fingerprint:forHost:)` writes a JSON blob; `entry(forHost:)` reads it. Legacy bare-UTF-8 bearer blobs decode as `Entry(token: "iat_…", fingerprint: "")` so a viewer upgrading from pre-#0113 storage signals "needs re-paste" through the empty fingerprint string. The pre-#0113 single-arg `store(token:forHost:)` + `token(forHost:)` keep working as thin wrappers.
- **`RemoteHostController.generateToken(name:expiresAt:)`** — the settings-sheet entry point. Threads the loaded identity into `AccessToken.generate` so the combined string is automatically bound to the cert the listener is serving. Throws `RemoteHostControllerError.noIdentity` if no identity is loaded (shouldn't happen post-#0112).
- **`GenerateTokenSheet.swift`** — the Generate action routes through the controller; the reveal phase displays the combined `iat_<43>.<64-hex>` (108 chars). The Copy button still copies the full revealed string — no UI changes beyond the longer value.
- **`RemoteFolderPickerView.swift`** — the token-paste step now parses combined first. Bare tokens (`iat_<43>` with no `.`) surface the inline "this token is from an older host; ask the host to regenerate it" message per spec. Garbage is generic "this doesn't look like a token". Whitespace is trimmed.

## Gotchas

- **The `.` separator** is unambiguous because `iat_` and `.` are both invalid in base64url. Future schemas can extend (`iat_<tok>.<fp>.<v2hint>`) without breaking parsing — `split(maxSplits: 1)` keeps the right side intact.
- **The host's token list (`AccessTokenListView`) does NOT show fingerprints.** The fingerprint is a host-level property of the identity, not a token-level property; surfacing it per-row would split the user's attention and the spec calls it out. #0115's cert section is where the host's current fingerprint will be displayed.
- **Legacy blob fallback in `ViewerTokenStore`.** A pre-#0113 build wrote raw UTF-8 of the bearer. Decoded fallback is `Entry(token: <legacy>, fingerprint: "")` so the empty fingerprint signals "re-paste needed" once #0114 starts pinning. New writes always go through JSON.
- **`Generated.combined` is the only string the user should ever see.** Don't be tempted to render `plaintext` separately in the reveal sheet — the user has to copy the whole thing for pinning to work.
- **`parseCombined`'s fingerprint check** matches `[0-9a-f]` strictly (lowercase per the spec). Uppercase hex is rejected; this matches the convention `openssl x509 -fingerprint -sha256 | tr A-Z a-z` produces.
