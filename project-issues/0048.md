# 0048 — Preview sheet is too narrow and inconsistent with the detail-panel rendering

| | |
|---|---|
| **Status** | in-progress |
| **Module** | Views |
| **Platform** | macOS |
| **First seen** | 2026-05-05 |

## Description

The markdown preview sheet (`IssueMarkdownSheet`, opened via Enter / spacebar / right-click → Preview Markdown / detail-panel link) feels too narrow for a long-form markdown read, and its content doesn't match what the detail-panel sidebar already renders. Two specific gaps:

1. **Width.** The sheet currently caps at `minWidth: 600, idealWidth: 900`. On a typical 14"+ display the user's window is wider than 900pt, so the sheet sits cramped in the middle. For comparison, the detail panel renders the same markdown body at 360pt and feels acceptable because that's its full assigned width — but on a sheet the user expects "open me wide for reading."

2. **Inconsistent render path.** `IssueMarkdownSheet` uses `IssueMarkdownSheetContentView` to read the **whole** markdown file from disk via `String(contentsOf: issue.fileURL)` and pass it raw to `StructuredText`. `DetailPanelDescriptionView` is more careful: it crops above the H1 + metadata table (so the rendered output starts at the first `## …` heading), and falls back through several render paths if the body is missing or the file is unreadable. The two views show different things from the same file, which is confusing — the sheet shows the H1 and metadata table again even though the sheet's own header already has the issue id and title.

## Proposed direction

Reuse the existing `DetailPanelDescriptionView` (or its underlying logic) inside the sheet. The detail panel's render path is the better one — it already strips the redundant front-matter and uses Textual's `StructuredText` with the same baseURL semantics. Two reasonable shapes:

- **Embed `DetailPanelDescriptionView` directly** in the sheet's content area. The sheet's existing header (id + filename + close button) stays; the body becomes whatever the detail panel would have shown, just at a wider frame. This is the smaller change.
- **Promote a shared `IssueBodyView`** that both `DetailPanelDescriptionView` and the sheet wrap. Cleaner long-term but introduces an extra type for what's currently a small body.

The second option is probably the right one if we want both call sites to evolve together (e.g. hover-to-reveal raw markdown, attachment-rendering work down the line). The first is what to ship if we want this resolved in one pass.

Either way, the sheet should:
- Render the same body content as the detail panel.
- Keep its own header (the wider sheet header is fine; preserve the `xmark` close button + `.keyboardShortcut(.cancelAction)` + `.onKeyPress(.space)` from #0045).
- Be **wider**: bump `minWidth` to ~720pt and `idealWidth` to ~1000–1100pt so reading flows comfortably. The vertical sizing is already generous (`minHeight: 400, idealHeight: 800`).

## Edge cases to keep working

- Cold-start path: opening the sheet for an issue whose file was just deleted should fall through the existing `bodyMarkdown()` failure branch in `DetailPanelDescriptionView` rather than crashing or showing an empty pane.
- The `.id(issue.id)` modifier from #0041 (force-rebuild on selection change) must remain on whatever wraps the description view, otherwise switching previewed issues mid-sheet would re-trigger the Textual stale-content bug.
- The Quick Look toggle (`.focusable()` + `.onKeyPress(.space)` from #0045) stays on the sheet's outer `VStack`.

## Out of scope

- Changing the detail panel's behavior. This issue is about the sheet matching the panel, not the other way around.
- Inline attachment rendering. PRD lists that as v1-out; revisit separately.
- Keyboard shortcut to toggle the sheet's width (Cmd+plus / Cmd+minus). Defer.

## Verification

After the fix:
- Open the sheet via Enter or spacebar on a selected issue.
- The sheet opens visibly wider than today (subjectively: lines of body text wrap at 70–80 chars, not 50–60).
- The first thing in the sheet body is the issue's first H2 (e.g. "## Description"), not the H1 line and metadata table that the sheet header already shows.
- Toggle preview off (space) and on (space) for several different issues — each shows its own body, no stale Textual rendering.
- The detail-panel side bar and the sheet show the same markdown content for the same issue.
