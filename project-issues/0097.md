## Description

Phase 3 of the remote-access plan (#0076). Once a viewer has a valid token (#0096), it fetches the host's folder list and lets the user pick one, several, or all in a single click each. This is the user-decided "1 / subset / all" flow described in #0076.

## Where to look

- **New: `Issues/Views/RemoteFolderPickerView.swift`** — the picker UI. Cross-platform; iOS reuses it in #0108.
- **`Issues/Remote/RemoteClient.swift`** (#0094) — `GET /v1/folders` returns the `[FolderInfo]` displayed.
- **`Issues/State/IssueStore.swift`** + tab system — selecting folders creates remote tabs (#0099).

- One row per folder in `/v1/folders`.
- Title: `project.json` `name` if present, else parent folder name.
- Subtitle: `description` from `project.json`, or the parent path if no `project.json`.
- Repo link: small "↗ host" line that opens the URL in the default browser when clicked. Right-click / long-press copies the URL. (Per `RemoteAccess.md` Open Q8 — single click opens, no popover menu.)
- Checkbox at left. Single click anywhere on the row toggles it.
- "Select all" and "Select none" persistently visible at the top.
- Confirm button at bottom shows the count: "Connect to N folder(s)". Disabled when N=0.

## Behavior

- The picker fetches `/v1/folders` once on entry. While fetching, show a small inline progress indicator and a Cancel.
- Selection is local to the picker until Confirm. Cancelling the sheet doesn't subscribe to anything.
- On Confirm, for each selected folder the picker creates a remote tab (#0099) and seeds it with `(host, folderId)`. The token is already in the Keychain (#0095).
- The picker remembers the previous selection per host (#0098). On reopen for the same host, the previously-selected folders are pre-checked; new folders default to unchecked; folders no longer present are silently dropped.
- If two folders share a `name`, the picker disambiguates by also showing the parent path under the title.
- Folders without `project.json` show no description and no repo link; the parent path stands in.

## Verification

- Picker opens with all rows from `/v1/folders`.
- Click "Select all" → every checkbox checks. Confirm count updates.
- Click "Select none" → all uncheck. Confirm disabled.
- Click individual rows → checkboxes toggle.
- With no folders selected, Confirm is disabled.
- Click an "↗ github.com/…" link → opens in default browser.
- Reopen the picker for the same host → previously-selected folders are pre-checked.

## Notes

- Drag-to-select / shift-click range-select are out of scope. The lists are short enough (1–10 folders typically) that a click each is fine.
- Avoid showing folder paths by default — they're noisy. The parent path is a fallback only when there's no `project.json`.
- Cross-platform: same view file works on iOS with a `List` shape; the only iOS-specific bit is the "tap" affordance and slightly larger touch targets.

## Root cause

Greenfield. The viewer-side IssueSource (#0094) was in place but had no UI affordance to point at a host. This issue is the user-facing entry point for the entire remote-access feature.

## Fix

Built in `worktree-agent-B` (`4635e4b`), merged onto `main` as `acb3876`. The picker is intentionally a three-phase modal that mirrors how a user actually thinks about connecting:

- **Phase A — pick host.** "Discovered hosts" section (empty state today; scaffolded for #0089 once Apple approves multicast — see #0093), "Recent" section listing `RemoteHostRecents.list()` with right-click → "Forget this host", and "or enter manually" with `Host:` / `Port:` `TextField`s. Continue disabled until both validate; Continue calls `URLSessionRemoteHostProbe.fetchHost(host:port:token:nil)` with a 3 s timeout. A 401 response is the success signal — "host reachable, awaiting token" → advance to Phase B. Other failures map to inline errors via the `RemoteHostProbeError` cases (`transport`, `httpStatus`, `invalidURL`, `decoding`).
- **Phase B — paste token.** `TextEditor` (monospace, multi-line) for the `iat_…` bearer plus Back / Cancel / Continue. Continue validates by re-calling `fetchHost(token:)`. On 200, store via `ViewerTokenStore.store(token:forHost:)` and advance. 401 → inline "Token rejected"; other failures route to the same error catch as Phase A.
- **Phase C — pick folders.** Calls `fetchFolders(token:)`, renders a row per `FolderInfo` with a checkbox, repo-link button (via `folder.url` — the renamed field from the #0080 spec amendment), and a parent-path fallback when there's no `project.json` name. "Select all" / "Select none" buttons plus an instance-count Continue ("Connect to 3 folders").

Plumbing for the four bundled issues:

- **#0091 manual host:port** — Phase A's "or enter manually" inputs with explicit port-range validation.
- **#0096 token paste** — Phase B; the bearer never gets logged and is only echoed in the TextEditor while the user is still typing.
- **#0097 RemoteFolderPickerView** — the picker itself + `RemoteFolderPickerModel` (`@Observable @MainActor`).
- **#0099 remote tab type** — `TabsModel.openRemoteTab(host:port:token:folderId:displayName:)` builds an `IssueStore(source: RemoteHostIssueSource(...))`, persistence under sibling `openRemoteTabs` UserDefaults key. `TabChipView` renders an `antenna.radiowaves.left.and.right` glyph when `store.folderURL.scheme == RemoteHostIssueSource.urlScheme`. Restore looks the bearer back up from `ViewerTokenStore` so tabs survive launches.

### Merge resolution notes

The agent's worktree forked from `833e619` (before my #0094 work) and reimplemented `RemoteClient` + `RemoteHostIssueSource` from scratch. The reimplementations were ADD/ADD conflicts on merge; resolution:

- Kept my source-side `RemoteClient` (host-bound, per-session) as `Issues/Remote/RemoteClient.swift` and my `RemoteHostIssueSource` unchanged.
- Lifted the agent's probe-style client into a new file `Issues/Remote/RemoteHostProbe.swift` with renamed types: `RemoteHostProbe` (protocol), `RemoteHostProbeError` (enum), `URLSessionRemoteHostProbe` (class). The picker uses these per-call; the source uses the session client. The two coexist.
- Added an optional `displayName:` seed to my `RemoteHostIssueSource.init` so `TabsModel.openRemoteTab` can prefill the chip label from the wire `FolderInfo`.
- Exposed `RemoteHostIssueSource.urlScheme = "issues-remote"` as a static so `TabChipView` and other consumers detect a remote tab without casting through the existential.
- Updated the picker's `folder.repository` reference to `folder.url` per the user's #0080 spec amendment.
- Broke `TabChipView`'s body into `folderGlyph` / `remoteIndicator` helpers — the inline expression tipped SwiftUI's type checker into a timeout.

## Gotchas

- **Visual verification still needed** for all three phases of the picker, the recent-row context menu, the folder-row checkbox states and repo links, and the remote-tab indicator on the chip. The unit tests only cover validation rules and the URL builder.
- **Token reaches the Keychain only after Phase C confirms.** A user who cancels in Phase B never persists; the typed bytes go away with the view.
- **`fetchHost(token: nil)` returning 401 is the success signal.** Treating it as an error would force the user back to Phase A — but the host *is* reachable, the token's just missing. The `RemoteHostProbeError.unauthorized` catch in Phase A is the branch point.
- **Restore drops remote tabs without a stored token** silently with a log line. That handles the case where the user revoked the token or this is a borrowed Mac without the same Keychain access group.
- **Bonjour discovery (#0089) is blocked on Apple's multicast-entitlement approval** (request `P8372QL98M`, see #0093). The "Discovered hosts" section is the scaffolded slot; it renders empty today.
- **`RemoteHostProbe` is the picker's API, `RemoteClient` is the source's.** Don't conflate them — the picker probes many hosts before one connects; the source is bound to one `(host, port, token)` for its lifetime.
