## Description

Translate every key in `Localizable.xcstrings` to Spanish. Depends on #0059 being complete (the catalog and the `es` slot must exist before content can land).

## Translation guidance

- Use neutral / international Spanish (avoid Latin-American-only or Spain-only idioms unless an obvious better choice exists).
- Use formal address (usted) sparingly — for app chrome, the typical macOS convention is impersonal/imperative ("Abrir carpeta…", not "Abre la carpeta…").
- Match Apple's Spanish localization conventions where possible: "Cerrar" for "Close", "Recargar" for "Reload", "Ajustes" for "Settings" (matches macOS Sonoma+ naming).
- Status display names: "Open" → "Abierto", "In Progress" → "En curso", "Resolved" → "Resuelto", "Closed" → "Cerrado", "Won't Fix" → "No se corregirá".
- View modes: "Swimlanes" → "Carriles", "Timeline" → "Cronología", "List" → "Lista", "Recent" → "Reciente".
- Plural-aware strings (e.g. `^[\(count) issue](inflect: true)`) need the corresponding inflection rules in xcstrings — Xcode's catalog editor surfaces plural variants for `one`/`other`.

## Process

1. Open `Issues/Localizable.xcstrings` in Xcode.
2. For each key, fill in the `es` slot.
3. Build the app with the system language set to Spanish (`AppleLanguages = ("es")` in the scheme's environment, or change the system preference) and walk every screen, fixing any awkward truncations or capitalization mismatches.
4. Pay attention to controls with fixed widths (status pills, view-mode capsule, the `+` button) — Spanish text is typically 15–20% longer than English. May need `.fixedSize(horizontal: true, vertical: false)` adjustments or wider capsule padding.

## Verification

- Run with system language `es` → every visible English string renders in Spanish.
- No truncation in the toolbar, stats bar, settings panes.
- Pluralization renders correctly (1 issue / 2 problemas — confirm the inflection rules are in place).
- Switching back to English flips cleanly (no caching of the previous locale).

## Out of scope

- Translating issue markdown bodies (user content).
- Translating the IssuesSkill format spec.
- Adapting date formats — Apple's `DateFormatter` already adapts via the active locale; just confirm dates render Spanish-style (`5 de mayo de 2026`).

## Root cause

Content work, not a bug. With #0059 scaffolding the catalog, the Spanish slots were populated in the same pass.

## Fix

Filled the `es` localization for every key in `Issues/Localizable.xcstrings` following the impersonal/imperative macOS convention noted in the spec. Status names use the mapping from the spec ("Abierto", "En curso", "Resuelto", "Cerrado", "No se corregirá"); view modes match ("Carriles", "Cronología", "Lista", "Reciente"); attachment filters and tooltip-style strings landed verbatim through the rest of the catalog. Inverted opening punctuation (¿…?) used where dialog titles ask a question (e.g. "¿Rotar el certificado?"). The catalog ships translated alongside #0061 and #0062 because all three issues share the same JSON file.

## Files changed

- `Issues/Localizable.xcstrings` — `es` slot populated for every key (one JSON file edit, ~70 strings).

## Gotchas

- Spanish runs ~15-20% longer than English on average — toolbar pills and the stats bar are tight but currently fit. If a future contributor adds a long string and it overflows, the fix is `.fixedSize(horizontal: true, vertical: false)` on the affected label rather than abbreviating the translation.
- Spanish convention for question marks is the matched pair `¿…?` — easy to forget the opening `¿`. The dialog title got it; ad-hoc strings added later should too.
