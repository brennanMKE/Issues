# 0126 — Formalise the IssueSource protocol; add a remote source that mirrors a web-server folder into a client-side temp directory

| | |
|---|---|
| **Status** | open |

| **Platform** | macOS |

## Description

Builds on the #0124 pivot (Mac mini as the canonical host; web-server + WebSocket transport instead of Bonjour-discovered peer-to-peer). The host serves the canonical folder of `NNNN.md` files; the viewer Mac runs a **remote `IssueSource`** that mirrors those files into a **temporary directory on the client** and re-uses the existing parsing / watching machinery against that mirror.

The name `IssueSource` stays — adapters all just *source* the same data, they don't write back. The host's folder is read-only from the client's perspective; nothing on the source Mac is ever mutated by the viewer.

## What the remote source does

1. **On connect (initial sync)**: viewer requests the full state from the host. The host responds with the list of issues (or it streams a "snapshot" message). For each issue, the viewer fetches the markdown by id and writes `NNNN.md` into its own temp directory. `project.json`, if present, is fetched and written alongside. After the snapshot lands, the existing parser + lint runner read the temp dir exactly as if it were a local folder.

2. **On change events (live updates)**: the WebSocket session emits a typed event per change — created / updated / deleted / attachment-changed / project-json-changed, each carrying the issue id (or a sentinel for `project.json`). The viewer reacts by:
   - **Updated / created**: `GET /issues/NNNN.md` and overwrite the temp-dir copy. The file watcher picks up the change and triggers the same reload path local folders use.
   - **Deleted**: remove `NNNN.md` from the temp dir.
   - **Attachment changed**: fetch and write into `NNNN/<name>` (existing relative-path convention).
   - **`project.json` changed**: refetch the JSON, overwrite the temp-dir copy.

3. **On disconnect**: delete the temp-dir mirror entirely. The tab keeps its tab-state (filters, view mode) but the issue list goes empty until reconnect. On reconnect the source resyncs from scratch via a fresh snapshot — same code path as the very first connect, no special "resume" branch.

## Architecture: compose, don't fork (decision locked)

The remote source materialises files; the *parsing and watching* don't change. So the shape is **composition**:

```
RemoteWebServerSource  (replaces today's RemoteHostIssueSource)
├── owns a temp directory at FileManager.default.temporaryDirectory/Issues/<tabId>/
├── owns a WebSocket session to the host
├── on connect:    snapshot fetch → writes files
├── on event:      GET file → writes file
├── on disconnect: delete the temp directory
└── delegates parsing/watching to an internal LocalFolderIssueSource
    pointed at the temp directory
```

`IssueStore` keeps consuming the `IssueSource` protocol. The internal `LocalFolderIssueSource`'s `onUpdate` becomes `RemoteWebServerSource`'s `onUpdate`, transparently. The watcher in the inner source observes the temp-dir writes the outer source performs.

This means the existing `MarkdownIssueParser`, `LintRunner`, `FolderWatcher`, and all of `IssueStore`'s reload semantics stay untouched. The new code is purely the **sync engine**: WebSocket connection lifecycle, snapshot pull, per-event fetch, temp-dir management.

The latency cost of going through FSEvents (WS event → file write → ~150 ms debounce → parse) is accepted in exchange for code reuse — the viewer's `IssueStore` doesn't know it's looking at a remote source.

## Wire shape (sketch — not load-bearing)

```
GET  /v1/folder/snapshot                 → JSON manifest of all issue ids + project.json hash
GET  /v1/folder/issue/NNNN               → raw markdown body
GET  /v1/folder/issue/NNNN/<attachment>  → raw attachment bytes
GET  /v1/folder/project.json             → raw JSON
WS   /v1/folder/events                   → typed events as listed above

Auth: bearer token (existing #0078 model) over wss:// with cert-fingerprint pinning (#0114).
```

The host already speaks something close to this (`#0079`/`#0100`/`#0101`); the new schema is whatever the post-#0124 web server looks like. Reuse where possible.

## Temp directory lifecycle

- **Location**: `FileManager.default.temporaryDirectory/Issues/<tabId>/`. `tabId` is stable per tab and survives across launches (same identifier we already use for tab persistence).
- **Lifetime**: created on connect, deleted on disconnect. No offline-cached "last known state" mode — disconnect means the viewer empties out, and reconnect resyncs from scratch via the snapshot endpoint.
- **No partial cleanup states**: deleting on disconnect keeps the disk story simple. The OS reclaims `temporaryDirectory` if the app crashes mid-session; we don't try to be defensive there.
- **No sandbox prompts**: `temporaryDirectory` lives inside the app's sandbox container, so no `NSOpenPanel` round-trip and no security-scoped bookmark.

## Acceptance criteria

- `IssueSource` keeps its current name and signature; the doc comment is updated to formalise the monitoring contract and to describe the temp-dir mirror pattern for remote sources.
- `RemoteHostIssueSource` is **replaced** by a new `RemoteWebServerSource`. The old file is removed; any callers (TabsModel, the picker flow, persistence) point at the new type.
- `RemoteWebServerSource` opens a WS session, performs initial snapshot sync, writes files into its tab-scoped temp directory, fetches per change event, and deletes the temp directory on disconnect.
- The inner `LocalFolderIssueSource` composition is what produces the `issues` / `lintFindings` / `loadError` / `folderInvalidated` outputs. No special-casing in `IssueStore`.
- `project.json` events flow through the same path (overwrite the temp-dir copy → watcher fires → store re-reads).
- Folder picker can launch the new remote source via "Connect to a host" with host + port + token (existing UI from #0125).
- Reconnect after disconnect: temp dir is recreated, full snapshot fetched, viewer's `issues` list repopulates cleanly.
- Build clean; existing tests pass; one new test exercises the snapshot-then-event sequence against an in-process fake server.

## Decisions locked

- **Replace, don't keep alongside.** `RemoteWebServerSource` replaces `RemoteHostIssueSource`. Single remote conformer.
- **Full snapshot on every reconnect.** The host doesn't track per-viewer cursors; disconnect deletes the temp dir; reconnect rebuilds via the snapshot endpoint. No delta-since-N protocol surface.
- **Composition over forked code path.** The remote source materialises files; an internal `LocalFolderIssueSource` does the parsing and watching. The viewer's `IssueStore` doesn't know its source is remote.

## Future scaling (not part of this ticket)

The snapshot grows linearly with the issue count, which is fine today (~100 issues) but won't be at 1000+. A follow-up will **filter closed issues out of the snapshot after they age past 30 or 90 days** — they stay readable on the host, but the viewer doesn't pull them. The cutoff applies to the snapshot endpoint and is server-side, so the viewer doesn't have to know.

## Out of scope (v1)

- **Writes of any kind.** The host's folder is read-only from the viewer's perspective; the viewer's temp dir is a read-only mirror managed by the source. The user edits issues on the host Mac (or via IssuesSkill there); the viewer just observes.
- **Editing the temp-dir copy manually.** Possible to type-trick a user into editing the mirror; it'd just get overwritten on the next event. Document it; don't try to prevent it.
- **Encryption of files at rest** in the temp directory. macOS sandbox + FileVault is the boundary; if the host's data is sensitive, the user picks where to run the viewer.
- **Multiple viewers of the same tab**, multi-host per-tab, or any kind of merge. Each tab is one source.
- **A third-party-backend REST flavour.** Earlier draft of this ticket flirted with a generic REST adapter; this round narrows scope to the web-server/WebSocket case the user actually has in flight. A third-party flavour can be filed separately if needed.

## Notes

- This is the natural shape after #0124: the Mac mini *is* the web server, the viewer is a thin client that mirrors what the server is showing. Treating the viewer's local copy as a temp-dir mirror means we don't need a parallel "remote data model" — the existing models, parsers, and watchers do their normal job against files that happen to live in `temporaryDirectory`.
- The transport-layer pieces from the previous remote-access cluster (`AccessToken`, `PinnedHostSessionDelegate`, `RemoteWebSocket`) are reusable. The pivot doesn't throw away that code; it reshapes how the *client* uses it.
- The file-watcher does the heavy lifting on the viewer side. Writing a file into the temp dir triggers FSEvents; the inner `LocalFolderIssueSource` debounces (150 ms today) and re-parses. Result: the viewer's `IssueStore` doesn't know the source is remote.
