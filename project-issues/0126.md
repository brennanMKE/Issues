# 0126 — Generalise IssueSource into an adapter protocol with pluggable monitoring backends

| | |
|---|---|
| **Status** | open |
| **Module** | Services |
| **Platform** | macOS |
| **First seen** | 2026-05-12 |

## Description

Today the source of an `IssueStore`'s data is abstracted behind `IssueSource` (#0077): two conformers exist — `LocalFolderIssueSource` (FSEvents-watched folder of `.md` files) and `RemoteHostIssueSource` (HTTP + WebSocket to a remote Issues.app host). The protocol works for read-only consumption but is shaped specifically around "I observe a folder or a host and emit one `onUpdate` callback when something changes."

This ticket reframes `IssueSource` as a deliberate **adapter** abstraction:

- A clearly-named **adapter** concept with a documented contract for monitoring, reading, and (newly) writing issues + attachments.
- The current `LocalFolderIssueSource` becomes the **default adapter** — its behaviour doesn't change.
- A new **REST adapter** sits alongside the existing `RemoteHostIssueSource`, talking HTTP/WebSocket to a backend that the user chooses (could be the existing Issues.app host, could be a third-party service in the future).
- The user picks the adapter per tab when opening a folder: "Local folder…", "Connect to remote host…" (existing), and "Connect to REST endpoint…" (new).

The motivation is flexibility: the markdown-file-on-disk model is great for solo use, but the user wants the option to back a tab with something else — a hosted service, a shared team store, a remote dev box, etc. — without rewriting `IssueStore` each time.

## What changes vs. what's already there

**Already in place** (#0077):
- `IssueSource` protocol with `start()` / `stop()` / `reload()` / `onUpdate` / observable `issues`, `lintFindings`, `loadError`, `folderInvalidated`.
- `LocalFolderIssueSource` (default, file-watcher-backed).
- `RemoteHostIssueSource` (HTTP + WebSocket to another Issues.app instance).
- `IssueStore` consumes either via the protocol; tab persistence is shaped around the two existing types.

**What this ticket adds**:
- Rename / re-document `IssueSource` as the adapter protocol. Update the doc comment to lead with the adapter framing.
- Document the **monitoring contract** explicitly: how an adapter signals "something changed", what counts as a change event (issue created / updated / deleted / attachment changed), and whether the adapter pushes (FSEvents, WebSocket) or polls (REST polling).
- Add **write support** to the protocol — currently the app is read-only; adapters that talk to a backend will need to send new / updated markdown back. Keep the default file-watcher adapter write-capable too (writes through to disk so external editors stay the source of truth).
- Add a **REST adapter** as the third conformer. Reuses the existing `RemoteClient` / `RemoteWebSocket` pieces where they apply; the difference vs. `RemoteHostIssueSource` is that the REST adapter targets a generic REST schema (versioned, documented separately) rather than the proprietary Issues.app host protocol — so the same wire format could be implemented by a non-Apple backend.
- Surface adapter choice in the **folder picker** flow. Local stays the one-click default; remote-host and REST endpoints sit alongside it with their own configuration UIs.

## Where to look

- `Issues/Services/IssueSource.swift` — the existing protocol. Likely the file that grows the adapter framing + write methods, or gets renamed.
- `Issues/Services/LocalFolderIssueSource.swift` — default adapter; add the write path here so external editors and the app share the same on-disk behaviour.
- `Issues/Remote/RemoteHostIssueSource.swift` — existing remote adapter; keep it as the "Issues.app host" specialisation. May share base types with the new REST adapter.
- `Issues/State/IssueStore.swift` — consumer; should not need behavioural changes once the protocol supports writes through a clearly-defined API.
- `Issues/State/TabsModel.swift` — `openTab(url:)` + `openRemoteTab(...)` today. Will need an `openRestTab(...)` (or a generalised `open(adapter:)`) once the third conformer lands.
- `Issues/Views/FolderPickerSceneView.swift` and `Issues/Views/RemoteFolderPickerView.swift` — picker UIs. The Local + remote-host + REST choice surfaces here.
- `RemoteAccess.md` and `Issues/Remote/RemoteProtocol.swift` — existing wire types; the REST adapter's schema either reuses these or forks into a documented v1.

## Proposed protocol shape (sketch — not load-bearing)

```swift
protocol IssueAdapter: AnyObject, Sendable {
    // Identity + display (same as today)
    nonisolated var folderURL: URL { get }
    var displayName: String { get }
    var repoName: String { get }
    var bookmarkData: Data? { get }
    var projectMetadata: ProjectMetadata? { get }

    // Read state (same as today)
    var issues: [Issue] { get }
    var lintFindings: [LintFinding] { get }
    var loadError: String? { get }
    var folderInvalidated: Bool { get }

    // Monitoring contract (was implicit, now explicit)
    var onUpdate: ((any IssueAdapter) -> Void)? { get set }
    func start()
    func stop()
    func reload()

    // Write contract (new)
    /// Capability bit so views can hide write affordances on read-only
    /// adapters. The default `LocalFolderIssueSource` returns `true`;
    /// adapters whose backend forbids writes return `false`.
    var supportsWrites: Bool { get }

    /// Persist a new or updated issue's full markdown body. The adapter
    /// is responsible for the actual write (filesystem, HTTP PUT, etc.)
    /// and for surfacing the resulting change back through `onUpdate`.
    func write(issue: Issue, markdown: String) async throws

    /// Remove an issue from the backing store.
    func delete(issueId: String) async throws

    /// Upload an attachment to the issue's sibling folder.
    func writeAttachment(issueId: String, name: String, data: Data) async throws
}
```

The exact signatures will be refined when implementation starts; this is to anchor the discussion.

## Acceptance criteria

- The protocol carries a clear "adapter" framing in its doc comment, plus an `## Adapters` section in the `Services/` README (or equivalent inline doc).
- `LocalFolderIssueSource` implements the new write methods by writing to disk through the existing security-scoped resource.
- A new `RestApiIssueAdapter` (or similar) exists, even if its first cut is a stub that only supports `start` / `reload` / read. Write methods can land in a follow-up ticket.
- `IssueStore` consumes adapters uniformly; no special-casing in the view layer beyond the picker.
- The folder picker presents three options: **Open local folder**, **Connect to Issues.app host**, **Connect to REST endpoint**. The REST option may be hidden behind a feature flag for v1 if the REST schema isn't finalised.
- Build clean (`xcodebuild`); existing tests pass.

## Decisions to make

- **REST schema**: does the REST adapter reuse `RemoteProtocol.swift`'s types verbatim, or does it ship a separate documented v1 schema (with version negotiation)? Reusing is faster; forking lets a third-party backend implement without locking into Issues.app-specific shapes.
- **Write path conflict resolution**: if the user edits an `.md` file in another editor while the adapter is mid-write, who wins? Default-adapter answer is "filesystem wins; the watcher re-reads and `onUpdate` fires." REST answer is more nuanced (server-side timestamp comparison, conditional writes).
- **Adapter-per-tab vs. adapter-per-folder**: today a tab is a folder; if a future user wants the same folder mirrored across two adapters (e.g. local backup + REST server), is that one tab with two adapters, or two tabs? Probably two tabs for v1.
- **Naming**: stick with `IssueSource` (read-shaped) or rename to `IssueAdapter` (read-write-shaped)? Renaming ripples through the codebase but better matches the intent.

## Out of scope (v1)

- Implementing the REST backend itself. The adapter is the client; the backend is a separate project (or an existing third-party service the user points at).
- Two-way sync between adapters. Each tab is bound to one adapter.
- Conflict-resolution UI. The first cut surfaces "your change was overridden" as an error banner; richer merge UX is later.
- Encryption / signing of payloads in transit. The remote-host adapter already pins TLS fingerprints via `PinnedHostSessionDelegate` (#0114); a REST adapter against an arbitrary backend may need a different model.
- Adapter discovery (e.g. "list available REST endpoints on the network"). Manual configuration only.

## Notes

- The existing `RemoteHostIssueSource` already proves the "non-local adapter" path works end-to-end. This ticket formalises the contract and unlocks one more conformer without rewriting the consumer.
- Write support is the biggest open question. Today the file is the source of truth and external editors do the writing — the app is intentionally a viewer. Adding a write API doesn't force the app to expose write UI immediately; it just enables adapters whose backend requires HTTP PUT to participate.
- The REST adapter is positioned for "user chooses their backend" — Issues.app host, a self-hosted service, or eventually a third-party offering. The adapter doesn't care; the schema does.
- Keep `IssueStore`'s contract narrow. It should look at the adapter only through the protocol, never `as?`-downcast except where unavoidable (e.g. capturing the security-scoped bookmark for the local case).
