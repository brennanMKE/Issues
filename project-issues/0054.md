## Description

Issue files frequently mention sibling issues by number ("see #0042", "fixes the regression from #0007"). Today those references render as plain text. Make them clickable: tapping `#0042` selects issue 0042 in the active store and opens the markdown sheet (or scrolls the detail panel into view, depending on context).

This is a high-value navigation primitive — most issue threads reference their predecessors, and walking a chain manually right now means dismissing the sheet, finding the row in the list, and reopening.

## Behavior

- `#0042` (4 digits, no whitespace required after the hash) renders as a link.
- Click → `store.selectedIssueID = "0042"` then `showingMarkdownSheet = true`. If the user is in the detail panel, the panel switches; if the markdown sheet is already open, the sheet body swaps in place (working since #0049).
- If the referenced issue doesn't exist in the current store, the click no-ops (or flashes a brief error). Don't crash.
- Skip recognizing the pattern inside fenced code blocks — same exclusion the linter uses for image refs in `LintRunner.stripCodeSpans`.

## Implementation sketch

Two reasonable paths:

1. **Pre-process the markdown** before passing it to `StructuredText`: regex-replace `#(\d{4})` with a Markdown link `[#\1](issues://0042)` and register a custom URL scheme handler that hooks into the active `IssueStore`. Textual's `StructuredText` already accepts the `baseURL` parameter; a custom scheme would route through the SwiftUI environment via `Environment(\.openURL)` or a per-store routing closure threaded through the view chain.

2. **Use Textual's link delegate** if it exposes one. Cleaner if available; falls back to option 1 if not.

Pre-processing is the simpler v1. The replacement is straightforward; the routing closure can live on `DetailPanelDescriptionView` and `IssueMarkdownSheetContentView` as a `let onOpenIssue: (String) -> Void` parameter wired up to `MainContentAreaView.openMarkdown(_:)` (from #0049).

## Edge cases

- Five-or-more-digit numbers (`#12345`) shouldn't match; constrain regex to exactly four digits with a non-digit trailing boundary.
- Hash inside a URL (`https://example.com/path#0042`) — pre-processing should ideally not turn URL fragments into links. Run the substitution outside Markdown link syntax (i.e. only in plain text spans). Easiest: skip lines that already contain `[`/`](` link syntax around the match position, or use `replacingOccurrences(of:options:.regularExpression)` with a negative lookbehind.
- Hash mid-word (e.g. `code#0042`): same — ideally guarded by a leading word boundary.

A simple regex like `(?<![\w/])#(\d{4})(?!\d)` handles both edge cases above.

## Verification

- Body containing "see #0042" renders #0042 as a link in accent color.
- Clicking jumps to issue 0042 — preview swaps if open, list selection moves if not.
- A non-existent reference (`#9999`) clicks but no-ops cleanly.
- A code block containing `#0042` renders as plain code, not a link.
- A URL fragment like `https://example.com#0042` renders without injecting a stray link.

## Root cause

Issue bodies are rendered via Textual's `StructuredText`, which only links text the Markdown parser already recognizes as a link. Mentions like `#0042` in prose are plain text by Markdown's rules, so there was no anchor for the user to click — every cross-reference required dismissing the preview, scrolling the list, and reopening the new issue.

## Fix

Pre-process the markdown body in `DetailPanelDescriptionView` before handing it to `StructuredText`. A new `IssueCrossRef.rewrite(_:)` (in `Services/`) walks the text once, peeling out fenced code blocks (``` ``` ```) and inline code spans (`` ` ``) so matches inside code stay verbatim, then runs the regex `(?<![\w/])#(\d{4})(?!\d)` over the prose halves and substitutes each match with `[#NNNN](issue://NNNN)`. The negative lookbehind guards `code#0042` and `path/0042`; the negative lookahead guards `#00420`.

The view installs an `OpenURLAction` via `.environment(\.openURL, …)` that calls `IssueCrossRef.issueID(from:)` on each tapped URL. `issue://NNNN` returns the four-digit id and routes to a new `onOpenIssue: (String) -> Void` callback; everything else falls through (`.systemAction`) so file:// attachments and external `https://` links keep their normal behavior.

The callback is threaded through `DetailPanelView` and `IssueMarkdownSheetContentView` so both the side panel and the preview sheet route the same way:

- Side panel: `MainView` swaps `store.selectedIssueID = id`. The animated panel transition (#0049) handles the visual switch.
- Preview sheet: `IssueMarkdownSheet.openIssue(_:)` swaps `store.selectedIssueID`. The content view re-resolves `store.selectedIssue` and Textual rebuilds via the existing `.id(issue.id)` invalidation (#0041) so the body actually changes — same machinery #0049 relies on for arrow-key nav.

Both call sites guard on `store.issues.contains { $0.id == id }` so a click on a non-existent reference no-ops cleanly instead of blanking the panel.

## Files changed

- `Issues/Services/IssueCrossRef.swift` — new. Pre-processor + URL parser.
- `Issues/Views/DetailPanelDescriptionView.swift` — adds `onOpenIssue`, calls `IssueCrossRef.rewrite`, installs `OpenURLAction`.
- `Issues/Views/DetailPanelView.swift` — forwards `onOpenIssue`.
- `Issues/Views/IssueMarkdownSheetContentView.swift` — forwards `onOpenIssue`.
- `Issues/Views/IssueMarkdownSheet.swift` — supplies a `private func openIssue(_:)` wired through `store.selectedIssueID`.
- `Issues/Views/MainView.swift` — wires the side-panel `onOpenIssue` to `store.selectedIssueID`.

## Gotchas

- `OpenURLAction` returning `.systemAction` is the correct fall-through for unrecognized schemes; returning `.discarded` would silently break external links inside issue bodies.
- The pre-processor walks fenced blocks before inline spans (matching `LintRunner.stripCodeSpans`) so a triple-backtick block containing single backticks doesn't get mangled by the inline pass.
- `URL(string: "issue://0042")` may park the digits in either the host or the path component depending on the parser path Foundation takes; `IssueCrossRef.issueID(from:)` checks `host ?? lastPathComponent` so callers don't have to care.
- Textual caches its laid-out content per view identity, so `DetailPanelDescriptionView` keeps its existing `.id(issue.id)` to force a rebuild when selection changes via a cross-ref click — the same mechanism #0049 already relies on.
