# 0127 — Age out closed issues older than 90 days from the snapshot endpoint

| | |
|---|---|
| **Status** | open |
| **Module** | Remote / Services |
| **Platform** | macOS |
| **First seen** | 2026-05-12 |

## Description

Follow-up to #0126's "Future scaling" note. The host's `GET /v1/folder/snapshot` endpoint currently returns every issue in the folder, which grows linearly with the issue count. At ~100 issues today it's fine; past 1000 the snapshot becomes a real payload.

The fix is server-side: when building the snapshot, **exclude issues whose status is `closed` *and* whose `Closed` date is more than 90 days old**. They stay on disk and are still readable via the raw markdown files; they just don't ship to the viewer in the live mirror.

## Why 90 days and why only `closed`

- Closed issues are by definition "done." A user looking at the live dashboard wants the live state — open work, in-progress, recently-resolved.
- The user can always read the markdown file directly on the host Mac (or via the IssuesSkill, or any text editor) if they want to dig up a closed issue from the archive. The mirror is for the *dashboard view*, not a complete archive.
- `wontfix` and `resolved` are *not* filtered. `resolved` is a confirmation-pending state ("work landed, waiting on user") and should stay visible. `wontfix` is rare enough that 90-day filtering would surface the wrong omissions.
- 90 days is the round number that matches "a release cycle plus margin." If a closed issue is older than that, the user has already moved past it; if they need it back, the exact-id search from #0128 covers that case.

## What this ticket changes

- The host's snapshot builder filters out matching issues *before* serialising. The filter is one boolean per issue: `status == .closed && closedDate < (now - 90 days)`.
- Same filter applies to WebSocket *events* — the host doesn't broadcast updated/deleted events for filtered issues either. They're invisible to the viewer.
- The viewer doesn't change. It just sees a smaller snapshot.

The filter is part of the host's snapshot construction logic, not a separate endpoint or query parameter. It's the host's policy for what "the snapshot" means.

## Where to look

- `Issues/Remote/RemoteHandlers.swift` — request handler for the snapshot endpoint (and per-issue + attachment fetchers). The filter sits in the path that materialises the issue list.
- `Issues/Models/Issue.swift` — the `closed: Date?` field already exists from #0026; the parser populates it from the `**Closed**` row. If `closed` is `nil` on a `closed`-status issue (parsing miss), treat the issue as "include" — better to over-ship than to hide something unexpectedly.
- `Issues/State/IssueStore.swift` — no change required; the store consumes whatever the source emits.

## Acceptance criteria

- A host with N issues where M are `closed` and older than 90 days returns `N − M` issues from `/v1/folder/snapshot`.
- A `closed`-status issue with a missing or unparseable `Closed` date is **included** in the snapshot (fail-open).
- WebSocket events for filtered issues are not emitted to viewers.
- An issue that transitions from `closed` → still-`closed`-but-now-91-days-old over time silently drops off the next snapshot; viewers see it disappear on next reconnect (acceptable, matches the lifecycle).
- An issue that gets reopened (`closed` → `open` / `in-progress`) reappears in the snapshot immediately on the next event tick.
- Unit test: build a fixture folder with a mix of statuses + closed dates and assert the snapshot's `id` set is the expected subset.

## Out of scope

- Viewer-side filtering. The viewer takes what the snapshot gives it.
- A user-configurable age cutoff. 90 days is the policy.
- Filtering by other statuses (`wontfix`, `resolved`). Only `closed`.
- An "archive" endpoint that returns everything. The user opens the markdown directly if they want the archive.
- Filtering attachments. Closed-old issues stay on disk with their attachments; the filter just excludes them from snapshots/events.

## Notes

- This is purely a host-side change. No protocol version bump needed — the wire shape doesn't change, the *contents* just get smaller.
- The 90-day window slides: today's snapshot reflects "now − 90 days," tomorrow's reflects a slightly later cutoff. This is fine; it's exactly the same "drops off after enough time" behaviour you'd expect from a dashboard.
- If a future feature wants to *see* the filtered issues without disabling the filter app-wide, that's #0128 (search by exact id). The two tickets together are "live view stays slim; archive lookup stays possible."
