# 0112 — Host: generate self-signed cert and configure NWListener for TLS

|---|---|

## Description

First step of the TLS migration (#0111). The host generates a durable self-signed certificate on first hosting-enable, persists it in Keychain, and configures `RemoteServer` (#0079) to listen with TLS using that identity. This ticket covers the cert lifecycle and the listener changes; the token-format extension that exposes the fingerprint to the viewer is #0113, and the viewer-side trust check is #0114.

After this ticket lands, the host serves over TLS but no shipping viewer can connect (the URL scheme is still `http://` in the viewer code, and there's no pinning logic yet). That's intentional — the migration is staged so each ticket is small.

## Where to look

- **New: `Issues/Remote/RemoteServerIdentity.swift`** — cert/key generation, Keychain persistence, fingerprint computation. Pure `Security.framework`; no third-party deps.
- **`Issues/Remote/RemoteServer.swift`** (from #0079) — `start()` currently configures plain TCP via `NWListener`. Add a `NWProtocolTLS.Options` step that binds the listener's TLS to the persisted identity.
- **`Issues/State/RemoteHostController.swift`** — owns the listener lifecycle. Construct or load the identity on `start()`; pass it to `RemoteServer`.

## Identity shape

```swift
struct RemoteServerIdentity {
    let secIdentity: SecIdentity        // private key + cert pair
    let certificate: SecCertificate     // leaf cert, also reachable via secIdentity
    let fingerprintSHA256: Data         // 32 bytes
    var fingerprintHex: String          // lowercase 64-char hex
}
```

### Generation

- Algorithm: **ECDSA P-256**.
- Validity: **10 years from now**. We don't auto-renew; rotation is manual via #0115.
- Subject: `CN=Issues Host` (placeholder — pinning is by fingerprint, the subject is irrelevant).
- Subject Alternative Name: none. The viewer doesn't validate hostnames.
- Extended key usage: `serverAuth`.
- Self-signed: the cert's issuer is itself.

Use `SecKeyCreateRandomKey` for the private key, then build the cert with `Security.framework`'s X.509 helpers. The pbxproj already links `Security` implicitly via Foundation; no new framework needed.

### Persistence

- Stored as a single Keychain item:
  - `kSecClass = kSecClassIdentity`
  - `kSecAttrLabel = "Issues Remote Access Host Identity"`
  - `kSecAttrSynchronizable = false` (don't sync to iCloud — the cert is per-Mac unless explicitly transferred)
- Reload on `RemoteServerIdentity.load()` via `SecItemCopyMatching` keyed by label. Returns `nil` on first run; the caller then calls `generate()` and stores.
- Plaintext token records (#0078) and the identity live in different Keychain items by service/account; no entanglement.

### Fingerprint

- `SHA256.hash(data: SecCertificateCopyData(certificate) as Data)` — i.e., SHA-256 of the **DER-encoded leaf cert**. Match the convention used by `openssl x509 -fingerprint -sha256`.
- Exposed as lowercase 64-character hex for display and for embedding in tokens (#0113).

## TLS listener wiring

```swift
let tlsOptions = NWProtocolTLS.Options()
let secProtocolOptions = tlsOptions.securityProtocolOptions
sec_protocol_options_set_local_identity(secProtocolOptions, identity.secIdentity as! sec_identity_t)
sec_protocol_options_set_min_tls_protocol_version(secProtocolOptions, .TLSv12)
// No client-cert verification — viewers authenticate via bearer + fingerprint pin.

let parameters = NWParameters(tls: tlsOptions, tcp: .init())
let listener = try NWListener(using: parameters, on: .any)
```

- Listener still binds to all non-loopback interfaces by default (same as today).
- The HTTP route table sits on top of this TLS-enabled connection — same handlers, same auth middleware. No code change in `RemoteHandlers`.

## Behavior

- **First host-enable** (or first time the identity is missing from Keychain): generate, persist, then start the listener with the new identity.
- **Subsequent enables**: load from Keychain, start.
- **Manual rotation** (deferred to #0115): regenerate, replace the Keychain entry, restart the listener.
- **Keychain failure to load** (corrupted, denied) → log + fail with a clear error in host settings. Don't silently fall back to plain HTTP; the v2 path is TLS or nothing.

## Verification

- Toggle hosting on; on first run a new identity is created. `security find-identity -p ssl-server` reveals the new entry; the label matches.
- Toggle hosting off and on; the same identity is reused (no regeneration).
- `openssl s_client -connect <host>:<port>` against the listening port shows a TLS handshake completing with a self-signed P-256 cert. The cert's SHA-256 fingerprint via `openssl x509 -fingerprint -sha256` matches `RemoteServerIdentity.fingerprintHex`.
- New unit tests in `IssuesTests/`:
  - `generate()` produces a `SecIdentity` and a 64-hex fingerprint.
  - `load()` after `generate()` returns the same identity (same fingerprint).
  - `generate()` again replaces the persisted identity (new fingerprint).
- Smoke-test CLI (#0087) is **expected to fail** after this lands until the CLI itself learns TLS — that's the next ticket (#0114) and we'll update the CLI to pin a passed-in fingerprint.

## Notes

- We're not going through `OpenSSL` or any third-party X.509 builder. `Security.framework` has enough to build a self-signed cert directly (`SecCertificateCreateWithData` plus a hand-rolled DER, or via the lower-level `SecKeyCreatePair` + `SecCertificateRequestCreate` path). The implementation is ~80 lines.
- `sec_protocol_options_set_local_identity` requires the `sec_identity_t` form, which bridges from `SecIdentity` via `sec_identity_create`. Easy gotcha to miss.
- The Keychain label is human-readable on purpose — `security find-identity` output is the canonical "what cert am I serving" check during debugging.
