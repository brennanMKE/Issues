# 0038 — Cell tap gestures bypass `Button`, losing accessibility traits

| | |
|---|---|
| **Status** | in-progress |
| **Module** | Views |
| **Platform** | macOS |
| **First seen** | 2026-05-04 |

## Description

Several rows / cards / table cells use `.onTapGesture` for primary activation instead of wrapping the content in a `Button`. Per `swiftui-pro/references/accessibility.md`:

> Never use `onTapGesture()` unless you specifically need tap location or tap count. All other tappable elements should be a `Button`.
> If `onTapGesture()` must be used, make sure to add `.accessibilityAddTraits(.isButton)` or similar so it can be read by VoiceOver correctly.

The mixed single-tap-toggles-selection / double-tap-opens-markdown pattern is the *legitimate* exception (`onTapGesture(count: 2)` requires the count parameter, so a plain `Button` won't cover it). But:

- The single-tap path can still be a `Button` underneath, with the double-tap as a separate `.onTapGesture(count: 2)` modifier on top.
- Even when both gestures stay as `.onTapGesture`, `.accessibilityAddTraits(.isButton)` is currently missing on every site, so VoiceOver doesn't announce the row as actionable.

## Affected sites

- `Issues/Views/IssueCardView.swift:33-34` — single tap selects, double tap opens markdown.
- `Issues/Views/RecentView.swift:58-59` — same pattern, applied to the row container.
- `Issues/Views/ListView.swift:26-91` — every `TableColumn` cell wraps `.onTapGesture(count: 2)`. The single-tap (selection) is handled by `Table`'s built-in selection binding, so the issue here is the *double-tap*: those cells need `.accessibilityAddTraits(.isButton)` and a corresponding `.accessibilityAction` for the open-markdown affordance.
- `Issues/Views/TimelineView.swift:179-180` — bar receives both gestures.
- `Issues/Views/TabBarView.swift:220` — chip activates on tap; the chip is otherwise rendered as a labeled view, not a button. With per-chip `.contextMenu`, a `Button` wrapper would also pick up Cmd+click and the focus ring naturally.
- `Issues/Views/MainView.swift:73-74` — tap on `Color.clear` to deselect (this is a clickaway-to-dismiss; `.accessibilityHidden(true)` is appropriate here rather than `.isButton`).

## Suggested change

For row-style cells (cards, recent rows, table cells, timeline bars), prefer a `Button` for the primary single-tap action and layer a `.onTapGesture(count: 2)` for the open-markdown affordance:

```swift
Button(action: onTap) {
    /* row content */
}
.buttonStyle(.plain)
.simultaneousGesture(TapGesture(count: 2).onEnded { onOpenMarkdown(issue) })
.accessibilityAction(named: "Preview Markdown") { onOpenMarkdown(issue) }
```

If keeping `.onTapGesture`, add the trait:

```swift
.contentShape(Rectangle())
.onTapGesture(count: 2) { onOpenMarkdown(issue) }
.onTapGesture { store.toggleSelection(issue.id) }
.accessibilityAddTraits(.isButton)
.accessibilityAction(named: "Preview Markdown") { onOpenMarkdown(issue) }
```

For the click-away in `MainView.contentArea`:

```swift
Color.clear
    .contentShape(Rectangle())
    .onTapGesture { store.deselect() }
    .accessibilityHidden(true)
```

## Notes

- VoiceOver doesn't currently announce these rows as actionable, which makes the app effectively non-navigable for the screen-reader path even though all the same actions are reachable via menu shortcuts.
- The keyboard navigation in `MainView.contentArea` (`onKeyPress(.upArrow)` / `.downArrow` / `.return`) covers selection/open via keyboard — that's fine. This issue is specifically about VoiceOver semantics and the mouse-tap hit targets.
