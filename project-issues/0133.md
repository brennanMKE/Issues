## Description

Add a second command-line binary alongside `issues` that renders the contents of a `project-issues/` (or `issues/`) folder as a `top`-style terminal dashboard. The dashboard re-reads the folder on a periodic timer, parses each `NNNN.md`, and shows a table of issues that fills the terminal. Truncates rather than scrolls — terminal grows, more rows appear; terminal shrinks, rows fall off the bottom.

This is essentially the proven-out POC at `~/Developer/brennanMKE/IssuesDashboard/`, re-homed into the `Issues` Xcode project so it ships and updates alongside the app and the existing `issues` CLI.

## Design

### Binary name and target

- New Xcode target **`IssuesDashboard`** (`com.apple.product-type.tool`) producing a binary named **`issues-dashboard`** (set `PRODUCT_NAME` in `Configuration/Dashboard.xcconfig`).
- The existing `issues` binary stays a thin dispatcher; it does **not** absorb a `--tui` flag or subcommand. The user types `issues-dashboard` to get the table view. Keeping them separate avoids dragging SwiftTUI's runtime into the dispatch-only binary that's meant to launch the macOS app via `issues:///open?path=…`.
- Source folder `IssuesDashboard/` under the repo root, registered as a `PBXFileSystemSynchronizedRootGroup` so new `.swift` files auto-register (mirrors the existing `IssuesCommand/` pattern).

### Data source

CLAUDE.md is explicit: "The markdown files **are** the source of truth. There is no JSON, generated artifact, or index to keep in sync. Do not introduce one." Hence:

- The TUI reads `project-issues/NNNN.md` (or `issues/NNNN.md`) directly. **No `issues.json` intermediate.** The POC's JSON file was convenient for the standalone demo; we don't need it here.
- Path resolution mirrors the `issues` CLI: no arg → resolve `$PWD`, prefer `$PWD/project-issues` then `$PWD/issues`; one arg → absolute / relative / `~`-prefix accepted; fail with exit 1 if the target isn't a directory.
- A small `--json <path>` escape hatch is **out of scope for v1** — file a follow-up if a real use case appears.

### Source sharing with the macOS app

The parser and models are pure, MainActor-by-default value types and would compile cleanly into a second target. Three viable shapes:

1. **Add explicit Target Membership** in `project.pbxproj` for `MarkdownIssueParser.swift`, `Issue.swift`, `IssueStatus.swift` so they belong to both `Issues` and `IssuesDashboard`. Works but the `Issues/` folder is currently a `PBXFileSystemSynchronizedRootGroup` and mixing synchronized-group sources with explicit per-file membership is fragile.
2. **Duplicate the three files** into `IssuesDashboard/` initially. Low ceremony, ~250 lines of code copied, drift risk is real but contained (parser changes are rare and well-covered by `IssuesTests/MarkdownIssueParserTests.swift`). When/if drift starts to bite, escalate to option 3.
3. **Extract an internal SwiftPM package `IssuesCore/`** that both targets depend on. Correct long-term shape; significant up-front structural work. Worth doing once a third consumer appears (e.g. the iPhone target tracked under #0108).

**Recommendation: option 2 for v1.** Duplicate. Add a comment at the top of each copy pointing at the canonical file in `Issues/` and noting "keep in sync; tests live in `IssuesTests/MarkdownIssueParserTests.swift`". Revisit if a real drift bug appears.

### Layout (lifted from the POC, adjusted for our richer model)

`GeometryReader` measures the terminal area, the view subtracts 4 chrome rows, slices `store.issues.prefix(visibleCount)` so the table truncates rather than scrolls.

Columns:
- `#` — 6 chars, right-aligned (issue number, zero-padded matches the filename so widths stay stable up to 9999).
- `TITLE` — flexible width, truncated with `.prefix(titleWidth)`. The em-dash separator (U+2014) is *not* shown — we render just the title text, since the parser already strips the leading `NNNN — `.
- `MODULE` — 16 chars, left-aligned. `Issue.modules.joined(separator: ",")` (the model already splits on ` / `). Truncate to column width.
- `STATUS` — 11 chars (`"in-progress"` is the longest token), colour-coded per `IssueStatus`.
- `DATE` — 19 chars (`YYYY-MM-DD HH:MM:SS`). Use `First seen` (or `Closed` if present and the issue is non-open) — TBD; pick `First seen` for v1 since every issue has it.

Status colour mapping (terminal ANSI):

| Status        | Colour            |
| ------------- | ----------------- |
| `open`        | blue              |
| `in-progress` | yellow            |
| `resolved`    | green             |
| `closed`      | gray              |
| `wontfix`     | bright black/dim  |

Chrome (non-issue) lines, in order:
- Title bar: `Issues Dashboard — N items — updated HH:MM:SS — <folder display path>`
- Column header.
- Divider (`─` × terminal width).
- Issue rows (variable count).
- Footer: error message if `loadError != nil`, otherwise `press Ctrl-C to quit`.

### Polling vs. FSEvents

POC polls at 1 Hz with `Task.sleep`. The macOS app uses `FolderWatcher` (FSEvents, 150 ms debounce) and is dramatically more responsive.

For v1, **match the POC and poll**. Reasons:

- The CLI tool doesn't carry the same "feels live" expectation as a windowed app — 1 Hz is fine for a terminal dashboard.
- `FolderWatcher` lives in the Swift package `Watcher` that the app already depends on. Adding it to `IssuesDashboard` is feasible but expands the target's dependency surface for marginal v1 benefit.
- Polling is portable to Linux if SwiftTUI ever takes us there; FSEvents is macOS-only.

A follow-up issue can swap polling for FSEvents once polling proves to be the bottleneck.

### Status / filter / search / sort

**Out of scope for v1.** The dashboard renders everything in the folder, newest first (sort by `First seen` desc, tiebreak by number asc — same as POC). No keyboard input handling beyond Ctrl-C. Adding `?` for filter/sort toggles can come later under a separate ticket.

### Dependencies

- New `XCRemoteSwiftPackageReference` to `https://github.com/rensbreur/SwiftTUI.git`, branch `main` (mirror what the POC already uses; pin to a tag if/when SwiftTUI tags a release we want).
- No other third-party dependencies. Foundation is enough for the parser and JSON-less load loop.

## CLI shape

```
issues-dashboard                 # resolves $PWD (prefers project-issues/ or issues/)
issues-dashboard <path>          # absolute, relative, or ~-prefixed
issues-dashboard -h | --help     # usage, exits 0
issues-dashboard --version       # exits 0
```

Exit codes:
- `0` on normal Ctrl-C exit.
- `1` on argument or filesystem error (e.g. `<path>` is not a directory).
- `2` reserved for future use (parser refused everything, etc.); not currently emitted.

Env override `$ISSUES_DASHBOARD_INTERVAL` (float seconds, default `1.0`) for the poll interval. Useful for the eventual `--watch` flag and for tests.

## Verification

### Build

- `xcodebuild -project Issues.xcodeproj -scheme IssuesDashboard -configuration Debug -destination 'platform=macOS' build` → BUILD SUCCEEDED, no SourceKit diagnostics.
- The full `Issues` scheme still builds (the new SwiftTUI dep is target-scoped, not project-wide).

### Smoke

- `issues-dashboard --help` exits 0 with usage.
- `issues-dashboard --version` exits 0.
- `issues-dashboard /tmp/nope` exits 1, stderr `error: not a directory: …`.

### Functional

- `cd ~/Developer/brennanMKE/Issues && issues-dashboard` — the table renders, populated from `project-issues/`. Title bar shows the total count and an `HH:MM:SS` stamp that updates each second.
- Touch a markdown file (`touch project-issues/0001.md`) or add a new one — the count and `updated …` stamp tick within a second.
- Resize the terminal vertically — the visible row count grows/shrinks to match. Resize horizontally — the `TITLE` column flexes.
- Mark an issue `in-progress` in the underlying markdown — the status cell recolours on the next poll.
- Delete a file — the row disappears.

### Failure modes

- Point `issues-dashboard` at a folder with no `NNNN.md` files — table is empty, footer shows the quit hint, no error.
- Corrupt one markdown file (truncate the metadata table) — the corrupted file is silently skipped (parser already does this), the rest render, no footer error.
- Move/delete the folder while the dashboard is running — footer flips to the load error; previously-rendered issues stay on screen so the view doesn't blank.

## Out of scope

- Keyboard interaction beyond Ctrl-C (filter, sort, search, jump-to-issue). Track separately when needed.
- Scrolling. The "truncate, don't scroll" rule is a feature — keep it.
- Editing or writing markdown files from the TUI. Read-only view.
- Opening selected issues in Issues.app, $EDITOR, or a browser. Future enhancement.
- An `issues dashboard` subcommand on the existing `issues` binary. Two binaries is cleaner; revisit if subcommands become a pattern.
- A JSON intermediate / generated index. Explicitly forbidden by CLAUDE.md; parse markdown directly.
- iOS / iPhone equivalent. SwiftTUI is terminal-only; iPhone has no terminal. Out of scope, paired with #0108.
- FSEvents-based refresh. Track separately; polling is the v1 strategy.
- Sharing source between targets via a SwiftPM `IssuesCore` package. v1 duplicates the parser/models; revisit when a third consumer appears.
