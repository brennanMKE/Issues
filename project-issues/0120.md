## Description

Sparkle is now wired into the app (#0118) but inert — the menu item is disabled until `SUFeedURL` resolves to a real `appcast.xml`. This ticket builds the static site that will live at **https://issues.sstools.co/** to serve that appcast (and double as the project's public landing page).

Scope is the **website content**, not the DNS / hosting setup (the user will point the subdomain at GitHub Pages or equivalent once the site is ready). All files live in a new top-level `website/` directory in this repo so the source-of-truth for the site travels with the code.

## Acceptance criteria

- `website/index.html` — public landing page. Brief project description, download link to the latest DMG, screenshot, AppIcon prominently displayed. Purple-themed to match the AppIcon's identity.
- `website/appcast.xml` — Sparkle feed at `https://issues.sstools.co/appcast.xml`. Starts with zero `<item>` entries (or a placeholder pinned to the *current* shipped version so existing clients don't surface a phantom update); per-release additions are documented in `scripts/SPARKLE.md`.
- `website/releases/` — one HTML page per shipped release for the appcast's `<sparkle:releaseNotesLink>` (or for the landing page's "Changelog" entry). Format consistent across versions.
- `website/assets/` — `AppIcon.png` (copied from repo root), a CSS file, and any screenshots.
- `website/404.html` — minimal page so an unmatched path doesn't drop to the host's default 404. Same branding.
- Site is **pure static HTML+CSS**. No JavaScript needed, no build step. A future markdown→HTML helper script is fine but out of scope here.

## Where to look

- **`AppIcon.png`** (repo root) — the 1024×1024 PNG. Copy into `website/assets/icon.png` (and downsample for `<link rel="icon">` if you want a 32×32 favicon variant — `sips -z 32 32` or similar).
- **`scripts/SPARKLE.md`** (from #0118) — the existing user-side setup checklist already sketches the appcast format. The website work delivers the actual served file.
- **`scripts/release.sh`** — the existing release flow that produces signed/notarized DMGs. Doesn't change in this ticket; the per-release `sign_update` step in `SPARKLE.md` still runs manually after each release until the integration is fully wired.

## Branding

Pull from `AppIcon.png`:

- **Primary color**: the icon's purple. Eyeballing the artwork: roughly `#8B7DEC` to `#7064D8` (folder body), with darker shadows around `#5D52BD`. Pick one as the brand purple — `#7064D8` is a reasonable midpoint. Light-mode CSS uses this for accents (buttons, links, the H1 underline); dark-mode flips to a slightly desaturated variant so it doesn't glow.
- **Document accent**: white / `#FFFFFF` for the page background of cards (matches the document peeking out of the folder).
- **Text**: high-contrast neutral (e.g. `#1A1A1A` on light backgrounds, `#E8E8E8` on dark). Don't tint text purple — keep the purple as accents only.
- **Typography**: system stack (`-apple-system, BlinkMacSystemFont, "SF Pro Text", "Helvetica Neue", sans-serif`). No web fonts; keeps the page fast and the offline-build trivial.
- **Layout**: single-column, ~720px max-width, centered. Don't over-design — this is a download page, not a marketing site.

A short single-line tagline goes under the title. Suggestion: *"A native macOS viewer for project-issues markdown — fast, focused, file-based."*

### `appcast.xml`

Initial skeleton (no items until the first release goes through):

```xml
<?xml version="1.0" encoding="utf-8"?>
<rss version="2.0" xmlns:sparkle="http://www.andymatuschak.org/xml-namespaces/sparkle">
  <channel>
    <title>Issues</title>
    <link>https://issues.sstools.co/appcast.xml</link>
    <description>Updates for Issues — the native macOS viewer for project-issues markdown.</description>
    <language>en</language>
  </channel>
</rss>
```

The per-release `<item>` template is already in `scripts/SPARKLE.md`. Each new release appends an `<item>` to this file and commits.

### `releases/x.y.z.html`

One page per release. Template (consistent across versions so a future script can generate them):

```
┌──────────────────────────────────────────────────┐
│  Issues — Version x.y.z                          │
│  Released 2026-MM-DD                             │
│                                                  │
│  ## What's new                                   │
│  - …                                             │
│  - …                                             │
│                                                  │
│  ## Notes                                        │
│  - …                                             │
│                                                  │
│  ← Back to issues.sstools.co                     │
└──────────────────────────────────────────────────┘
```

The appcast `<item>` references this via `<sparkle:releaseNotesLink>https://issues.sstools.co/releases/x.y.z.html</sparkle:releaseNotesLink>`. Sparkle renders the linked page inside its update sheet, so keep it plain and reflow-friendly. No fixed-width tables, no inline images larger than the sheet width (~600px).

### `404.html`

One-screen "not found" page with the AppIcon and a link back to `/`. Same CSS as `index.html`; no extra surface.

## Deploy plan (out of scope, captured for the next step)

The user will:

1. Configure `issues.sstools.co` as a CNAME pointing at GitHub Pages (or another static host).
2. Push the `website/` directory's contents to a `gh-pages` branch (or configure Pages to serve from `/website` on `main`).
3. Add `SUFeedURL = https://issues.sstools.co/appcast.xml` and `SUPublicEDKey` to `Info.plist` per `scripts/SPARKLE.md`.

This issue lands only the *content*; the user wires up the hosting + plist when ready.

## Verification

### Content

- `website/index.html` renders cleanly in Safari (file:// preview is fine). AppIcon visible at the top; download button styled in the brand purple; screenshot fits the column.
- `website/appcast.xml` validates as XML (`xmllint --noout website/appcast.xml`). The RSS+Sparkle namespace declarations match Sparkle's spec.
- `website/releases/` exists with at least one example file (even an empty `0.0.0.html` placeholder) so the directory's CSS scope works.
- `website/404.html` renders the same look as `index.html` minus the page-specific content.

### Branding

- All foreground accents (links, buttons, dividers) use a single CSS variable for the brand purple. Changing the value in one place ripples across the whole site.
- The AppIcon is the only image with its specific colors; no other UI element repeats the purple gradient inside the icon. Keeps the icon visually distinct.
- Light-mode and dark-mode both legible. Test by toggling the system appearance with the page open.

### Sparkle round-trip (after the user wires the plist)

- `Info.plist`'s `SUFeedURL` points at `https://issues.sstools.co/appcast.xml`.
- App launches, Check for Updates… enables.
- Initial empty-feed case: clicking the menu item shows Sparkle's "You're up to date" sheet.
- After appending the first signed `<item>`: clicking the menu item shows the update prompt, the release-notes link renders inside the sheet, accepting installs the new build on relaunch.

## Notes

- **No JavaScript.** A static site is enough for v1, and JS-free pages load instantly and survive any host. If we ever need analytics or interactive bits, revisit then.
- **The screenshot can wait.** A clean placeholder image (e.g. a 720×450 PNG of the swimlane view with the project-issues folder loaded) is enough for v1; replace with a polished screenshot in a follow-up.
- **One CSS file, not multiple.** The whole site fits in well under 200 lines of CSS — splitting into per-page files just slows reviews.
- **Release-notes pages should not duplicate the appcast `<item>`'s description.** Sparkle's update sheet fetches the linked release-notes page; if the same content is also inside `<description><![CDATA[…]]></description>`, the user sees it twice in some Sparkle versions. Pick one (release-notes page recommended, since it's also reachable from the landing-page changelog).
- **Markdown source for release notes** is a possible follow-up — write `.md`, run a script to render `.html` at commit time. Out of scope here; for v1 hand-authoring `.html` is fine.
- **`AppIcon.png` is already at the repo root** (1024×1024, RGBA). Copy it into `website/assets/icon.png` rather than referencing the root file from inside `website/` — keeps the website directory self-contained for the eventual `gh-pages` deploy.

## Scope (out)

- DNS / GitHub Pages / Cloudflare setup. The user runs this once when the site's content is in place.
- Hosting the actual DMG files on `issues.sstools.co/downloads/`. v1 expects DMGs to live wherever they live today (GitHub Releases); the appcast's `<enclosure url="…">` can point anywhere. If we move DMG hosting onto the same domain later, add a `website/downloads/` step then.
- A blog / news / about page. The landing page is the surface; anything else is future scope.
- Localization. English only for v1.
- Open Graph / Twitter card metadata. Nice-to-have; can add when there's a reason to share the URL widely.
- A site-wide search. The site is tiny; search is overkill.

## Dependencies

- **None blocking.** This work can land alongside or before any Sparkle-side configuration. The site stands on its own; Sparkle just consumes one of its files.
- **Not gated on #0110** (the Info.plist explicit-file migration). The website lives entirely outside the Xcode project.
- **Once this lands**, the natural next steps are: (a) the user configures `issues.sstools.co` DNS + hosting, (b) the user runs `scripts/SPARKLE.md`'s `generate_keys` to produce the EdDSA pair, (c) `SUFeedURL` + `SUPublicEDKey` go into `Info.plist`, and (d) the first signed `<item>` lands in `appcast.xml` from the next release.

## Root cause

Not a bug — feature work. The Sparkle integration from #0118 needs a real `SUFeedURL` to function, and the project also wants a public landing page. Both surfaces live on the same domain and share branding, so they land together.

## Fix

Built the website as a self-contained `website/` directory in the repo. Pure static HTML + a single CSS file; no JavaScript, no build step, no external assets beyond a copy of `AppIcon.png`. Branding pulls from the AppIcon: brand purple `#7064D8` (light mode) / `#8B7DEC` (dark mode), document-white card backgrounds, and the system font stack so the page renders identically on every platform that might reach it. Light/dark themes adapt via `prefers-color-scheme`.

The `appcast.xml` ships with zero items in `<channel>` — Sparkle treats that as "you're up to date", which is the right behaviour until the first signed release lands and a per-release `<item>` is appended (process documented in `scripts/SPARKLE.md`). The 1.0 release-notes page is in place even though 1.0 isn't yet shipped, so the appcast template has a real `<sparkle:releaseNotesLink>` target as soon as the first item is added.

Screenshot is intentionally deferred to a follow-up — the spec's "screenshot can wait" note made it explicit, and the index.html flow stands on its own without one.

## Gotchas

- The download button currently links to `https://github.com/brennanMKE/Issues/releases/latest`. If the user later moves DMG hosting onto `issues.sstools.co/downloads/`, that's a one-line `href` change in `index.html` plus a new `downloads/` directory — but the appcast's `<enclosure url="…">` is the load-bearing path for auto-updates, not the landing-page button.
- `404.html` references assets via root-absolute paths (`/assets/...`) because GitHub Pages serves 404.html from the URL that was requested, not from `/404.html` — relative paths would break for nested URLs that hit the 404. The other pages can keep relative paths.
- `index.html`'s changelog version is hand-edited per release for v1. When this becomes a chore (probably around release 3 or 4), the natural next step is a Markdown→HTML script that regenerates the index.html changelog section + the appcast item from a single source. Out of scope here.
- The two-commit split (d70d9af for the status flip, 58f816e for the actual files) is a workflow artefact — the initial code commit was issued with `path/to/0120.md` as a positional arg, which silently overrode the staged `website/` files. If this happens again, the recovery is `git reset --soft HEAD~1` (denied by the auto-mode classifier in this session, hence the follow-up commit).
