## Description

Phase 0 of the remote-access plan (#0076). Today `IssueStore.reload()` lists a directory and reads markdown files directly off disk. To support a remote viewer in later phases, the store needs to read through a protocol so a future `RemoteHostIssueSource` can swap in without touching the store, the views, or the parser.

This is a pure refactor — no behavior change, no remote functionality yet. The single goal is that the local app keeps working bit-for-bit identically while the file-reading code moves behind an `IssueSource` abstraction.

## Where to look

- **`Issues/State/IssueStore.swift`** — owns folder loading. Today `reload()` does the directory walk and parser invocation inline. Pull that into a new `LocalFolderIssueSource` type that the store holds, and route `reload()` through it.
- **`Issues/Services/MarkdownIssueParser.swift`** — unchanged. The local source still uses the parser; the protocol shouldn't leak parser specifics.
- **`Issues/Services/FolderWatcher.swift`** — unchanged for v1; the local source owns the watcher, the remote source will own a WebSocket subscription instead. The store doesn't care which.
- **`Issues/Services/FolderBookmarkService.swift`** — unchanged. Bookmark resolution stays where it is; the local source receives a resolved `URL`.

## Protocol shape

```swift
protocol IssueSource: AnyObject {
    var displayName: String { get }
    var projectMetadata: ProjectMetadata? { get }
    func start() async throws
    func stop()
    func reload() async -> [Issue]
    var changes: AsyncStream<Void> { get }   // or a delegate; pick whichever fits SwiftConcurrency cleanly
}
```

- `start()` / `stop()` lifecycle parallels `IssueStore.start()` so the store doesn't grow new lifecycle methods.
- `reload()` returns the parsed list rather than mutating the store directly — the store sorts and applies filters as it does today.
- `changes` (or equivalent callback) is the source's "something changed, please reload" signal. The local source backs it with `FolderWatcher`; the remote source will back it with WebSocket events later (#0100).
- `displayName` and `projectMetadata` come from the source so the remote source can serve them from the host's `project.json` without re-reading the file from a path that doesn't exist on the viewer.

## Behavior

- **No visible change** for the user. The local app reads the same folder, parses the same files, and produces the same `[Issue]` array.
- **Tab title and metadata** still come from `project.json` per #0075, but now via `IssueSource.projectMetadata` rather than read directly inside the store.
- **Reload triggers** still fire from FSEvents debouncing in `FolderWatcher`; the local source wires the watcher to its `changes` stream.
- **Lifetime** — the store creates the local source on `start(folder:)` and tears it down on `stop()`. Security-scoped resource access stays paired exactly as today.

## Verification

- All existing `IssuesTests` pass without modification.
- Open this repo's `project-issues/` folder — the swimlane / list / timeline / recent views render identically to before the refactor.
- Add or delete a `.md` file in the watched folder — the views update within the existing ~150 ms debounce window.
- `log stream --predicate 'subsystem == "co.sstools.Issues"'` shows the same lifecycle messages it does today (any new logs from the source layer are fine, but nothing should be missing).
- Toggle a `project.json` `name` change — tab title updates, exactly as #0075 specifies.

## Notes

- Resist the urge to design the protocol around the eventual remote case. The remote source has different cancellation, latency, and error shapes; we'll add affordances when we get to #0094 rather than baking them in now.
- Keep `LocalFolderIssueSource` macOS-only (`#if os(macOS)`) since it relies on FSEvents. The protocol itself is platform-neutral so the iOS target can later link only the remote source.
- Don't introduce a second store, a new state container, or a "manager-of-sources". The store still owns one source at a time and calls into it.

## Scope (out)

- No `RemoteHostIssueSource` in this ticket. That's #0094.
- No view-layer changes beyond what the type plumbing requires.
- No new tests for the remote case yet — only ensure the existing tests keep passing.

## Root cause

Not a bug — the file-loading half of `IssueStore` (security-scoped resource access, FSEvents watcher wiring, `contentsOfDirectory` + parser + lint pass, `loadError`/`folderInvalidated` state) was tightly coupled to the directly-observable view layer. The remote viewer in `RemoteAccess.md` Phase 3 needs to drop in a different loader (HTTP+WebSocket against a host) without touching the views, the parser, or `TabsModel`. That swap requires a seam where today there is none.

## Fix

Introduced an `IssueSource` protocol that owns `issues`, `lintFindings`, `loadError`, `folderInvalidated`, `displayName`, `projectMetadata`, `folderURL`, `repoName`, plus `start()`/`stop()`/`reload()` and a single `onUpdate` callback. `LocalFolderIssueSource` is the only implementation in this issue and is bit-for-bit equivalent to the pre-refactor `IssueStore.start/stop/reload` — same logging category (`IssueStore`), same log lines, same selection-cleanup ordering, same "don't fire upstream onReload on failed reloads or folder invalidation" semantics.

`IssueStore` keeps every piece of view state it had before (`statusFilters`, `moduleFilter`, `platformFilter`, `attachmentFilter`, `searchQuery`, `viewMode`, `selectedIssueID`, sort, derived `filteredIssues`/`visibleIssueOrder`/`groupedByPrimaryModule`/`snapshot`/`statusCounts`, `persistedState`/`apply`, `selectNext`/`selectPrevious`). It now holds an `any IssueSource` and mirrors the source's state into its own `@Observable` storage on each `onUpdate` tick. `init(folderURL:)` becomes a convenience that wraps a `LocalFolderIssueSource`, so `TabsModel.openTab(url:)`, `TabsModel.restore`, `IssueStoreTests.makeStore`, and `PreviewSamples.makeStore` keep their call sites untouched.

`displayName` and `projectMetadata` are wired into the protocol now even though they aren't surfaced to the UI yet — `LocalFolderIssueSource` returns `nil`/parent-folder-name for them. That gives #0075 (project.json consumer) a clean home to land its `JSONDecoder` pass and gives #0094's remote source a place to inject the host's `project.json` without another protocol change.

The work spans two commits: `f37e309` (the main refactor) and `5e2c7d8` (a one-line `reload()` forwarding fix; see Gotchas).

### Deviations from the spec above

- **Sync, not async.** The protocol sketch above used `start() async throws` and `reload() async -> [Issue]`. The implementation kept both sync, matching today's `IssueStore` semantics. Reasoning: every existing call site relies on `start()` synchronously populating `issues` before returning (e.g. `TabsModel.openTab` reads `store.snapshot` and `store.persistedState()` immediately after `store.start()`). Going async would have rippled into `TabsModel.openTab` / `TabsModel.restore` and the SwiftUI handlers above them — a much larger surface than this issue called for. The protocol is intentionally not `@MainActor`-annotated, so a future remote source can pick its own isolation and add async overloads when it lands.
- **No `#if os(macOS)` gate on `LocalFolderIssueSource` yet.** The project is macOS-only today; adding the gate would protect nothing and complicate the diff. The gate goes in alongside the iOS target in Phase 5+ (#0103+), not here.
- **`changes: AsyncStream<Void>` not added.** The protocol uses a single `onUpdate` callback instead — closer to today's `IssueStore.onReload` ergonomics and lets the store do its success-vs-failure-vs-invalidation discrimination in one place. We can switch to `AsyncStream` if the remote source benefits from it (#0100); the seam is small.

## Files changed

- `Issues/Models/ProjectMetadata.swift` (new) — lenient `Decodable, Equatable` struct (`name: String?`, `url: URL?`). Type only; #0075 wires the actual file read.
- `Issues/Services/IssueSource.swift` (new) — protocol declaration. Intentionally not `@MainActor`-annotated.
- `Issues/Services/LocalFolderIssueSource.swift` (new) — owns folder access, FSEvents wiring, the directory walk + parser + lint pass, `loadError`/`folderInvalidated`. Logging stays on the `IssueStore` category. Failure path preserves prior `issues` so the host's snapshot baseline stays stable.
- `Issues/State/IssueStore.swift` — file-IO removed; holds an `any IssueSource`, mirrors its observable fields via `handleSourceUpdate(_:)`, fires its own success-only `onReload` (preserving the `TabsModel.handleReload` contract). `folderURL`/`repoName` become forwards onto the source. `init(folderURL:)` is now a convenience over `init(source:)`. `reload()` stays as a thin forwarder for `AppCommandsController.reloadActive`. DEBUG-only `setIssuesForPreview(_:)` continues to write the store's own observable storage so existing test fixtures bypass the source unchanged.

## Gotchas

- The `@MainActor` annotation on the protocol that the spec sketched would have broken `IssueStoreTests` (the suite isn't `@MainActor`-isolated, so it can't construct `@MainActor`-isolated types from non-MainActor test contexts, and the spec required tests to keep passing without modification). Solution: leave the protocol non-isolated, mirroring today's `IssueStore` pattern where main-actor isolation is enforced de facto by callers, not the type system.
- The source fires `onUpdate` on success, failure, and folder invalidation — but the store's upstream `onReload` is success-only (matches the pre-refactor contract that `TabsModel.handleReload` relies on for the unseen-changes indicator). The discrimination lives in `IssueStore.handleSourceUpdate`: `if src.loadError == nil && !src.folderInvalidated { onReload?(self) }`.
- Easy to miss: `AppCommandsController.reloadActive` calls `store.reload()` (the menu Reload command). The first commit `f37e309` dropped the public `reload()` from `IssueStore` (folded into the source) and broke the build at that one call site. `5e2c7d8` adds it back as `func reload() { source.reload() }`. Future protocol-extracting refactors should grep for `store.<name>` across the whole codebase, not just `Views/` and `State/`.
- `xcodebuild test` requires a host OS at or above the test target's deployment (macOS 26.4 here). On a macOS 26.3.x host, override at the command line: `MACOSX_DEPLOYMENT_TARGET=26.3 -only-testing:IssuesTests`. Don't commit that change to the project.
