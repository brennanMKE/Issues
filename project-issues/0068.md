## Description

When the user has a single status selected in the stats-bar status filter (e.g. only `Open` is active), an Option-click on that same chip should **invert** the selection: deselect the currently-active status and select all the others.

Concretely, if `Open` is the only active status and the user Option-clicks `Open`:

- Before: `[Open]`
- After: `[All, In Progress, Resolved, Closed, Won't Fix]` — every status except `Open`

This is a quick way to say "show me everything that isn't this." Clicking the chip without Option keeps today's behavior (toggle that status on/off as a single-status filter).

## Behavior

- Trigger: Option-click (NSEvent modifier flag `.option`) on a status chip.
- Precondition: exactly one status filter is currently active, and the chip being clicked is that one.
- Effect: the active status becomes the only deselected one; every other status becomes selected.
- If the precondition isn't met (zero active, multiple active, or Option-clicking a different chip), Option-click should fall back to plain-click behavior (or be a no-op — pick the less surprising of the two; plain-click fallback is probably more discoverable).

## Where to wire it up

The status chips live in the stats bar — likely `Issues/Views/StatsBarStatusFilterView.swift` (or wherever `IssueStore.statusFilter` is toggled from). SwiftUI's `Button` doesn't pass modifier flags, so the click needs to either:

- Use `.simultaneousGesture(TapGesture().modifiers(.option))` for the inverted variant alongside the regular tap, or
- Read `NSEvent.modifierFlags.contains(.option)` inside the existing tap handler.

The latter is simpler and matches how AppKit-flavored shortcuts already get handled elsewhere in the app.

## Verification

- Click `Open` so it's the only active status. Option-click `Open` → all other statuses become active, `Open` becomes inactive.
- Option-click any chip when zero or multiple statuses are active → behaves like a plain click.
- Without Option, click behavior is unchanged.

## Notes

Discoverability is a separate concern; this is a power-user shortcut. A future tooltip update could mention "⌥-click to invert" but that's not required for this issue.

## Root cause

The status chips referenced in the issue are `ToolbarStatusPillView` instances (the toolbar row, not the stats-bar count rows — the latter are not tappable). The view already read `NSEvent.modifierFlags.contains(.option)` and treated Option-click as "toggle membership in the active set". Toggling the only-active pill removed it, leaving an empty filter (i.e. "show all") rather than the inverse of the prior selection. There was no path that produced "every status except X".

## Fix

Inside `handleClick(isActive:)`, special-case the single-active branch when Option is held: when the pill being clicked is `isActive` *and* `store.statusFilters.count == 1`, replace the set with `Set(IssueStatus.displayOrder).subtracting([status])` instead of just removing the lone status. All other Option-click and plain-click branches are preserved. The tooltip on the pill was extended to mention the new shortcut so the behavior is documented in `.help(...)`.

## Files changed

- `Issues/Views/ToolbarStatusPillView.swift` — added the invert branch in `handleClick(isActive:)`, refreshed the type doc comment, and extended the `.help` tooltip to mention the Option-click invert shortcut.

## Gotchas

- `Set(IssueStatus.displayOrder)` is the source of truth for "every status the UI knows about". If a new `IssueStatus` case is added later, make sure it's appended to `displayOrder` so the invert produces the expected set.
- The clicked pill must be the **only** active filter; inverting from an empty set (no filters → "show all") would just exclude one status, which is what plain-click already does, so Option-click on an empty set still falls through to the existing add-to-set branch.
