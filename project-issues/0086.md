# 0086 — Host-side entitlements for the remote server

|---|---|
| **Status** | open |

## Description

Phase 1 of the remote-access plan (#0076). Without the right entitlements the sandboxed app can't bind a listening socket or accept incoming connections. This is a small but load-bearing change — the listener in #0079 won't start without it. Bonjour-specific entitlements live in #0093 alongside the viewer-side network entitlements, since both ends need them.

## Where to look

- **`Issues/Issues.entitlements`** — already includes `com.apple.security.app-sandbox` and the read-only file/bookmark keys. Add the network entries below.
- **No project.pbxproj edit needed** — entitlements are declared in the `.entitlements` file and referenced from build settings, which already point at `Issues/Issues.entitlements`.

## Entitlements to add (host)

```xml
<key>com.apple.security.network.server</key>
<true/>
<key>com.apple.security.network.client</key>
<true/>
```

- `network.server` allows `NWListener` to bind and accept incoming TCP connections.
- `network.client` is needed because the same Issues.app can also act as a viewer in another tab (a host-and-viewer instance, per `RemoteAccess.md` §Topology).

Bonjour multicast (`com.apple.developer.networking.multicast`) and `NSBonjourServices` Info.plist entries are deferred to #0093 to keep the changes paired with the Bonjour code that uses them.

## Verification

- Build a Debug build, launch, toggle hosting on (#0083). Server binds successfully; `lsof -nP -iTCP -sTCP:LISTEN | grep Issues` shows the listener.
- Try to bind without `network.server` (revert the entitlement, rebuild) — `NWListener` start should fail with a sandbox violation in the system log. This is the negative test, used once to confirm we got the right entitlement.
- Signed/notarized release build also accepts connections (matches the test rig described in #0076).

## Notes

- `network.server` does **not** bypass the firewall on first run — macOS still prompts the user with "Issues.app would like to accept incoming network connections" the first time. That's the correct user-visible behavior for a hosting app and shouldn't be suppressed.
- Don't add `network.client` to a future iOS target unless the iOS target also acts as a viewer (it will — see #0093 for the iOS-side entitlements).
- Resist the temptation to add `com.apple.security.network.outgoing` or other older keys; the modern sandbox uses `network.client` / `network.server`.

## Scope (out)

- Multicast / Bonjour entitlements → #0093 (paired with Bonjour code in #0088 / #0089).
- iOS entitlements → #0093 will mention them; the iOS target itself doesn't exist until #0108.
- Codesigning identity changes — the existing Developer ID flow already covers entitlement updates.
