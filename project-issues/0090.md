## Description

Phase 2 of the remote-access plan (#0076). Bonjour publishing (#0088) and browsing (#0089) are configured to "just work" on the same broadcast domain, but in practice mDNS traversal depends on the network. This is a verification ticket — not new code, but a documented test pass on the actual two-Mac rig described in #0076 (MacBook Air ↔ Mac mini, Wi-Fi LAN + Tailscale).

The result of this ticket goes in this issue's Notes section as a record of what was tested and what worked, so future changes have a baseline to compare against.

## What to test

For each network condition below, on the viewer Mac:

1. Toggle hosting off on Mac A, then on. The viewer's discovery list should populate within 2 seconds, then empty within 2 seconds when hosting toggles off.
2. With hosting on, connect from the viewer using the discovered host (Phase 3 flow once it lands; for now `dns-sd -L` is enough proof).
3. Stress-toggle: toggle hosting off/on three times, confirm the viewer re-discovers each cycle.

### Network conditions to cover

| Condition | Expected | Notes |
|---|---|---|
| Both Macs on same home Wi-Fi | Discovery works, sub-second | Baseline. |
| Both Macs wired Ethernet to same switch | Same as Wi-Fi | Wired vs wireless shouldn't matter. |
| MacBook Air on Wi-Fi, Mac mini wired to same router | Discovery works | Most common mixed case. |
| Both Macs on Tailscale, **MagicDNS enabled**, on different physical networks | Discovery works | Tailscale forwards mDNS when MagicDNS is on. |
| Both Macs on Tailscale, **MagicDNS off** | Discovery does **not** work | Expected limitation; manual `host:port` (#0091) is the path. |
| Coffee shop Wi-Fi (client isolation enabled) | Discovery does not work | Expected; #0105 will pause publishing on untrusted networks anyway. |

## Tools

- `dns-sd -B _issues._tcp` on the viewer side, run from a terminal — quickest probe of "is mDNS finding it".
- `dns-sd -L "<service name>" _issues._tcp` to resolve a discovered service to host:port + TXT.
- `dns-sd -G v4 <hostname>.local` to resolve the host's mDNS name to an IPv4. Useful when Bonjour resolves but the next hop fails.
- Tailscale: `tailscale status` to confirm both nodes are up; `tailscale netcheck` if there's a connectivity question; MagicDNS on/off via the admin console or `tailscale up --accept-dns=true/false`.

## What to record (in this issue's Notes)

For each condition above:

- Whether discovery succeeded.
- How long it took to appear / disappear.
- Anything weird in `log stream --predicate 'subsystem == "co.sstools.Issues"' --level debug` during the test.

## Verification

Verification *is* the test pass — this issue isn't done until the table above has been walked end-to-end on the rig. If a condition doesn't behave as expected, file a follow-up issue rather than working around it inline.

## Notes

- This isn't a one-time check. If `Network.framework` behavior changes in a future macOS release, or if Tailscale changes how it handles mDNS, we re-run this list. Linking it from #0076 makes that easy.
- The "expected does not work" rows aren't bugs — they're documented limits. The manual host:port path (#0091) covers them.

## Scope (out)

- Manual host:port fallback → #0091.
- Network-change pause behavior → #0105.
- Documenting the mDNS limits in user-facing UI — could be a small "Why isn't my host showing up?" tip in the picker; out of scope here.
