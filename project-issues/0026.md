# 0026 — Notification tap opens a new window instead of focusing the primary tabbed window

| | |
|---|---|
| **Status** | resolved |
| **Module** | App / State |
| **Platform** | macOS |
| **First seen** | 2026-05-04 |

## Description

Tapping a user notification (delivered by the work in #0013) opens a **new** Issues.app window instead of routing the user back to the existing primary window. The app's design has exactly one main window — the tabbed bar at the top, content below — and a notification tap should behave as a **deep link**: bring the existing window forward, switch to the tab whose folder owns the changed issue, and (when the notification is about a specific issue) select that issue so the detail panel and the markdown sheet path can pick it up.

## Expected behavior

1. App is in the background; user receives a notification (`#0042 — Reply button broken`, kind=`status` change).
2. User clicks the notification.
3. The existing primary window is activated and brought to the front. **No new window appears.**
4. `TabsModel.activeTabID` is set to the tab whose folder the notification came from.
5. `IssueStore.selectedIssueID` is set to `0042` so the detail panel reflects the targeted issue.
6. If the user is a power-user who has the markdown sheet open, the same routing logic should leave the sheet alone (don't auto-open it from a notification — only update the selection).

For rollup notifications (`5 issues changed in BucketKit`), the notification's `userInfo` carries `tabID` only and `kind=rollup`. The tap should activate the relevant tab and clear (or leave) the issue selection — no specific issue to focus.

## Actual behavior

A second window appears. The user has to manually switch to the original window or close the new one. Selection / tab routing inside the new window is undefined.

## Likely cause

The app uses SwiftUI's `WindowGroup` scene in `IssuesApp.swift`. By default, `WindowGroup` allows multiple instances and macOS will spawn a new one on activation under various conditions — especially if `NSApplication.shared.activate()` is called during a `terminated → launched` transition triggered by the notification.

Two paths to investigate:

- **Switch to a single-instance scene**: replace `WindowGroup` with `Window` (macOS 13+) when there's only ever one main window. `Window("Issues", id: "main")` guarantees a single instance and exposes `OpenWindowAction` for explicit re-show. Activating the app then re-uses the existing window.
- **Or keep `WindowGroup` and intercept the activation path**: `NotificationService.userNotificationCenter(_:didReceive:withCompletionHandler:)` (added in #0013) currently calls `NSApplication.shared.activate()` followed by `AppCommandsController.shared.tabs?.setActive(id:)`. If the app was background-but-running, this works. If the app was relaunching, the launch sequence may produce a fresh `WindowGroup` instance before the delegate method runs. Use `@Environment(\.openWindow)` + a single `Window` scene to make the routing deterministic.

## Notes

- The `NotificationService` already builds `userInfo` with `tabID` (always), `issueID` (when the notification is about a specific issue), and `kind` (`addition` / `removal` / `status` / `rollup`) — see `Issues/Services/NotificationService.swift`. The deep-link logic is mostly there; the bug is just the window-multiplication.
- `AppCommandsController` (from #0008) already holds weak references to the active `TabsModel` and the `[IssueStore]` — the routing path can read from there without re-resolving.
- If the app was completely terminated and a notification re-launches it, the persisted tab list is restored on init (see `TabsModel.restore()` from #0009/#0002). The notification's tap handler then needs to wait until that restore completes before calling `setActive`. A simple approach: stash the pending notification's `userInfo` on the controller and replay it once the tabs finish restoring.
- **Future-related**: #0025 (Help UI) proposes adding a separate `Window` scene for help. That scene is independent of the main window and is the *only* additional window the app should ever have. When implementing this fix, make sure the Help window stays opt-in (only opens via the Help menu) and isn't auto-spawned on activation.
- The existing `defaultSize(width: 1200, height: 800)` should carry over to whichever scene type is chosen.

## Fix outline (recommended path)

1. Change `IssuesApp.swift`'s scene from `WindowGroup { RootView() }` to `Window("Issues", id: "main") { RootView() }`. Keep `.defaultSize`, `.commands`, the `init()` notification-delegate wiring, etc. Verify Cmd+W still closes the window (it does, but won't spawn a second on activation).
2. Inject `@Environment(\.openWindow) private var openWindow` into the notification tap handler so it can explicitly request the `id: "main"` window if it ever isn't visible. `openWindow(id: "main")` re-shows the existing window when one is hidden, and is a no-op when it's already visible.
3. The tap handler then runs the existing routing logic against `AppCommandsController.shared`.
4. If the app is cold-launched by the tap, queue the routing on the controller until `TabsModel.restore()` finishes, then drain.

Verify by:
- Sending a notification with the app foregrounded (should be suppressed per #0013 — confirm).
- Sending with the app backgrounded; tap → primary window comes forward, tab switches, issue selected.
- Sending while the app is fully terminated; tap → app launches, primary window appears, tab + issue routed once tabs are restored.
- Confirm no second window appears in any path.

## Follow-up — second *process* spawning on notification tap

The Window-scene fix (commit `82c5be2`) addressed the second-window-in-same-process case, but the user reports notification taps now launch a **second OS process** of the app entirely. The Activity Monitor shows two `Issues.app` instances after a tap; `NSApp.activate(...)` only operates within the current process and can't merge them.

### Cause

By default macOS allows an app to have multiple running instances. LaunchServices, when handling a notification activation, may spawn a fresh process if it doesn't conclude that the existing process is the "right" target — most commonly during development when builds live in different DerivedData paths or are signed with rotating ad-hoc identities. The Window-scene change doesn't influence LaunchServices' choice.

### Fix

Set `LSMultipleInstancesProhibited = YES` in the bundle. With this key, macOS guarantees only one instance of the app can run; LaunchServices reuses the existing process and routes the activation (including the notification's `userInfo`) into it via the standard delegate path, instead of launching a new copy.

The project uses Xcode's synthesized Info.plist, so the equivalent build setting is `INFOPLIST_KEY_LSMultipleInstancesProhibited = YES` on the Issues target's Debug and Release configurations.

After this lands, all three notification paths from the verification list above (foregrounded / backgrounded / fully terminated) should never produce a second process. The deep-link routing already in place from commit `82c5be2` handles the post-activation work in the single existing process.
