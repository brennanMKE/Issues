# 0035 — Hardcoded `Color.white` in active view-mode tab text

## Description

`Issues/Views/StatsBarView.swift:96` uses a literal `Color.white` for the active view-mode segment's text:

```swift
.foregroundStyle(active ? Color.white : Color.appMuted)
```

The active background is `Color.appAccentDim` (purple at 60% opacity). White-on-purple reads OK in both modes today, but the rest of the palette is asset-catalog-driven (#0030) and adapts to system appearance — leaving a single literal `.white` in the active state is an inconsistency waiting to break the next time the accent color is changed in Xcode.

## Suggested change

Add a semantic color set to `Assets.xcassets` (e.g. `accentForeground` or `onAccent`) that pairs with the accent — typically white in both light and dark, but the asset catalog is the right place to express that. Reference it via the generated symbol just like the other palette colors:

```swift
.foregroundStyle(active ? Color.accentForeground : Color.appMuted)
```

If a new asset feels heavy for one site, at minimum use `.white` via `Color(.white)` (the `NSColor` flavor, which is guaranteed-white in both modes) so the codebase has a single literal-white pattern.

## Notes

- The view-mode capsule lives in `StatsBarView.viewModeSwitcher` (StatsBarView.swift:85-108).
- Same pattern is used in `FolderPickerView.swift:44` (`.foregroundStyle(Color.white)` on the "Add folder…" button text) — covered by the same fix.
- No user-visible change today; this is a maintenance/theming consistency fix.
