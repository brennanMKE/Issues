## Description

Phase 1 of the remote-access plan (#0076). The host can have many folders open as tabs; remote-access shouldn't force the user to publish all of them. This issue adds a per-folder hosting toggle in the host settings sheet, plus the multi-folder container the server reads from.

## Where to look

- **New: `Issues/State/MultiFolderStore.swift`** — a small `@Observable` container that owns the user's open `IssueStore`s and an `isShared: Bool` per folder. The existing tab UI already iterates a list of folders; this just gives that list an extra flag.
- **`Issues/State/IssueStore.swift`** — unchanged. The store doesn't know whether it's being shared; the multi-folder container does.
- **`Issues/Views/RemoteHostSettingsView.swift`** (from #0083) — add a "Folders being served" section.
- **`Issues/Remote/RemoteServer.swift`** (from #0079) — already takes a `MultiFolderStore`; it filters by `isShared` when answering `/v1/folders`.

## UI

```
┌─ Folders being served ──────────────────────────────┐
│                                                     │
│  [✓] Issues.app          ~/Developer/Issues         │
│  [✓] Bluesky Migration   ~/Developer/bluesky-mig    │
│  [ ] Scratch             ~/Documents/scratch        │
│                                                     │
│  [ Share all ]   [ Share none ]                     │
│                                                     │
└─────────────────────────────────────────────────────┘
```

- One row per open folder. Display name comes from `project.json` `name` (#0075) when present, otherwise the parent folder name. Subtitle is the abbreviated folder path.
- Per-row checkbox toggles `isShared`. Default for newly-added folders is **on** when remote hosting is enabled, **off** when remote hosting is disabled (so a fresh folder doesn't surprise-publish).
- "Share all" / "Share none" are convenience buttons.

## Behavior

- `isShared` persists per-folder in `UserDefaults` keyed by `folderId` (#0082) under `RemoteServer.sharedFolders.<folderId>`.
- When `isShared` flips off:
  - The folder disappears from `/v1/folders` immediately.
  - Any viewer currently subscribed to it via WebSocket (#0100) gets a `{ "type": "unsubscribed", "folderId": "<id>", "reason": "host_unshared" }` event and the host stops sending updates for it.
- When `isShared` flips on:
  - The folder appears in `/v1/folders` on the next request.
  - Existing connected viewers don't auto-subscribe — they have to refresh their picker (#0098).
- If hosting is globally disabled (#0083), per-folder toggles are visible but disabled; flipping the global switch back on uses the previously-saved per-folder values.

## Verification

- Enable hosting (#0083), open three folders, toggle two on and one off. `curl … /v1/folders` returns two `FolderInfo` entries.
- Toggle the third on. `curl … /v1/folders` returns three.
- "Share none" disables all three. `/v1/folders` returns `[]`.
- Quit and relaunch — same `isShared` values persist.
- Add a new folder via the existing tab picker — it appears in this list with the default-on or default-off behavior described above.

## Notes

- The `MultiFolderStore` is a small refactor: today the app conceptually has one store per tab, and the views walk a list. Formalizing that as `MultiFolderStore` gives `RemoteServer` a clean dependency without forcing the views to change their iteration shape.
- `RemoteServer` should not snapshot the folder list — it should ask the store on each request so toggle changes apply immediately.

## Scope (out)

- Folder-level access tokens (e.g. "this token can only see folder X"). Out of scope; the auth model is per-host.
- Per-folder rate limiting or read budgets. Single user, not needed.

## Root cause

#0079 introduced `MultiFolderStore` as a forward-declared protocol. #0080 grew it with the methods the REST handlers needed. Until #0085, no concrete type satisfied the protocol — every `RemoteServer` test used a stub. This issue lands the production implementation.

## Fix

New `Issues/State/HostFolderStore.swift` — `@MainActor @Observable` concrete `MultiFolderStore` for the host:

- **Source of truth.** `setStores(_:)` accepts the latest snapshot of open `IssueStore`s (driven by `TabsModel` once #0083 wires it up). The store itself doesn't own the lifecycle, just the per-folder share preference.
- **`isShared(folderId:)` / `setShared(folderId:_:)`.** Reads through to `UserDefaults` keyed `RemoteServer.sharedFolders.<folderId>` on first access; writes are write-through. An in-memory cache avoids hitting `UserDefaults` per request once the flag has been seen.
- **`isGlobalHostingEnabled`.** When `false`, `currentlyHostedFolders()` returns `[]` regardless of per-folder flags. Per the spec, the per-folder values stay set so flipping the global switch back on restores them.
- **`defaultIsSharedForNewFolders`.** First-time folder ids inherit this. Default is `false` — a freshly-opened folder doesn't auto-publish, even when global hosting is on. The settings UI in #0083 will expose a "share new folders by default" checkbox if the user wants the inverse.
- **`setSharedForAll(_:)`.** One-call form for the "Share all" / "Share none" buttons in the settings sheet.
- **`hostDisplayName`.** `Host.current().localizedName` (the user-visible computer name from System Settings → Sharing) with a `ProcessInfo.processInfo.hostName` fallback.
- **`HostedFolder` shape.** Built per request from each `IssueStore` so toggle changes apply immediately. `modifiedAt` is the max of the issues' `modifiedAt` so the wire response reflects "freshness" without needing a separate folder-mtime field.

### File-naming deviation

The issue spec named the file `Issues/State/MultiFolderStore.swift`. The protocol added in #0079 already owns the `MultiFolderStore` name; the concrete is `HostFolderStore` (file: `Issues/State/HostFolderStore.swift`) so the protocol/implementation distinction stays clear in imports. Functionality matches the spec.

### Out of scope (deferred to #0083)

- The settings sheet rows (per-folder checkbox, "Share all" / "Share none" buttons, abbreviated path subtitle).
- `TabsModel` integration (calling `setStores(_:)` whenever tabs change).
- WebSocket fanout for `unsubscribed` events on isShared → false (that's #0101).

These rely on UI scaffolding that #0083 lands; the data layer is in place and #0083 can wire `setStores` + the settings rows on top.

## Gotchas

- **`Host.current().localizedName` returns `nil` in some early-launch contexts.** The fallback to `ProcessInfo.processInfo.hostName` keeps the `/v1/host` response valid even before the system has fully published its name (rare in practice for a UI app).
- **`UserDefaults` keys are process-wide.** Tests use `UUID()`-prefixed folder ids and clean up in `defer` so concurrent runs don't see each other's flags. Don't reuse a fixed test id like `"foo"` — it'll persist between runs and surprise the next test.
- **The store is `@MainActor`** because it's `@Observable` and read by SwiftUI views in the host settings sheet (#0083). Network code reading through `RemoteServer` already hops to MainActor for state mutation; the request handlers run in `@MainActor` task contexts.
- **Don't snapshot the folder list inside `RemoteServer`.** Each request calls `currentlyHostedFolders()` fresh so an isShared toggle takes effect on the next request without restarting the listener — the spec calls this out and the protocol's `currentlyHostedFolders()` (returns a value, not a property) makes it explicit.
