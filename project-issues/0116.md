## Description

In Settings → Shortcuts, clicking a shortcut row's recorder field focuses it (the field tints, the placeholder changes to "Press a shortcut…"), but the first key combo the user presses **runs the corresponding app action** instead of being captured by the recorder. For example, focusing the New Tab recorder and pressing `⌘N` opens a new tab in the main window rather than re-binding the shortcut. The same happens for every combo that has a registered menu equivalent.

The recorder field's `keyDown(with:)` never fires for those combos because AppKit dispatches `⌘`-bearing events as **key equivalents** first. Key equivalents walk a separate path: window → first-responder's `performKeyEquivalent(with:)` → menu's `performKeyEquivalent(with:)`. The menu carries the app's `.commands { … .keyboardShortcut(…) }` entries from `IssuesApp.swift`, so it matches and fires the action before the responder chain ever sees the key.

## Steps to reproduce

1. `⌘,` to open Settings → Shortcuts tab.
2. Click any recorder field to start recording (the field tints, the placeholder updates).
3. Press a key combo that's already bound to a menu action (e.g. `⌘N` for New Tab, `⌘⇧→` for Show Next Tab, `⌘1` for Swimlanes — anything in `ShortcutAction`).

## Expected behavior

The recorder captures the key combo and updates the shortcut binding. The pressed action does **not** fire.

## Actual behavior

The bound app action fires. The recorder field stays in its "Press a shortcut…" state until the user clicks elsewhere or presses Esc.

## Where to look

- **`Issues/Views/Settings/RecorderView.swift`** — the AppKit `NSView` that backs the recorder. Currently overrides `keyDown(with:)` only. Needs `performKeyEquivalent(with:)` too.
- **`Issues/IssuesApp.swift`** — registers the menu shortcuts via `.commands { … .keyboardShortcut(store.keyboardShortcut(for: .…)) }`. No change needed here; the fix is in the recorder. (Disabling the commands globally while Settings is open is a heavier and less local solution; the responder-chain interception is the right scope.)
- **`Issues/State/ShortcutsStore.swift`** — unaffected.

## Fix

Override `performKeyEquivalent(with: NSEvent) -> Bool` on `RecorderView`. When the recorder is the first responder *and* `isRecording` is true, route the event through the existing `keyDown(with:)` translation logic and return `true` so AppKit treats the event as consumed and stops the dispatch — the menu never sees it.

Sketch:

```swift
override func performKeyEquivalent(with event: NSEvent) -> Bool {
    guard isRecording, window?.firstResponder === self else {
        return super.performKeyEquivalent(with: event)
    }
    keyDown(with: event)
    return true
}
```

That's enough. The two existing paths in `keyDown` (special-key by keyCode; plain character requiring at least one non-shift modifier) handle every combo; reusing them keeps the translation behaviour identical regardless of whether the event arrived as a key equivalent or a plain key down.

### Why this fix is local

- The recorder *only* eats events when it's both focused and recording. When the user is anywhere else in Settings (or in any other window), `performKeyEquivalent` falls through to `super` and the menu's normal key-equivalent matching takes over.
- No global state to toggle; no monitoring of which window is key; no edits to `IssuesApp.swift`'s `.commands` block. The recorder is the right scope because the recorder is the only context in which we want menu shortcuts not to fire.

## Verification

- Open Settings → Shortcuts. Click the New Tab row's recorder. Press `⌘N` → recorder captures `⌘N` as the binding (the field displays `⌘N`), no new tab opens in the main window.
- Click the Previous Tab row's recorder. Press `⌘⇧←` → captured. No tab navigation fires.
- Click the Reload row's recorder. Press `⌘R` → captured. Main-window reload does not run.
- Press Esc while recording → drops focus, binding unchanged (existing behavior preserved).
- Click a recorder, press a bare special key (e.g. `↓` with no modifiers) → still captured via the existing `keyDown` path because bare keys aren't dispatched as key equivalents.
- With **no recorder focused**: `⌘N` in any normal window still opens a new tab. Menu shortcuts work as today.

## Notes

- This is the kind of bug that's invisible until you actually try to use the feature — every keystroke "works" in the sense that the app does *something*, but it's the wrong thing. Worth a small unit test or at least a manual verification step on every code path through `keyDown` once we land this.
- `performKeyEquivalent` returning `true` while the recorder is active also blocks `⌘Q` from quitting the app mid-recording. That's a side effect but a benign one — the user can Esc out and then `⌘Q`, or just click outside the recorder. The reserved-shortcut warning row in `ShortcutRow` already handles the case where the user tries to bind ⌘Q.
- macOS dispatches Esc through `keyDown` rather than `performKeyEquivalent`, so the existing Esc-to-cancel branch in `keyDown(with:)` continues to work without any change.

## Scope (out)

- Visual changes to the recorder field. The fix is purely behavioural.
- Adding the same protection to any other custom NSView (there are none today; `RecorderView` is the only NSView that captures keystrokes).
- Changing the reserved-shortcut policy or expanding `ShortcutsStore.reservedShortcuts`.

## Root cause

`RecorderView` only overrode `keyDown(with:)`. AppKit dispatches modifier-bearing keystrokes (⌘-anything, ⌘⇧-anything) through a different path — `performKeyEquivalent(with:)` is called on the responder chain and then on the main menu *before* `keyDown:` ever fires on the first responder. The app's `.commands { … .keyboardShortcut(store.keyboardShortcut(for: …)) }` block in `IssuesApp.swift` registers those equivalents on the main menu, so the menu matched every recordable combo and ran the action. The recorder never saw the event because the dispatch was consumed upstream.

## Fix

Added a `performKeyEquivalent(with:)` override on `RecorderView`. While focused and recording, route the event through the existing `keyDown(with:)` translation and return `true` so AppKit treats the event as consumed — the menu's key-equivalent matcher never gets a chance to fire. When not recording (or not first responder), defer to `super.performKeyEquivalent(with:)` so menu shortcuts work as today.

The branch is intentionally narrow: only the recorder's *own* focus state suppresses menu shortcuts, and only for the duration of a recording session. Esc-to-cancel and bare-special-key handling continue to flow through `keyDown(with:)` because those events aren't dispatched as key equivalents — that path is unchanged.

## Files changed

- `Issues/Views/Settings/RecorderView.swift` — added the `performKeyEquivalent(with:)` override above the existing `keyDown(with:)`. Comment cites #0116 for the dispatch-order rationale.

## Gotchas

- **`performKeyEquivalent` returning `true` swallows the event for the whole app**, not just the recorder. That includes ⌘Q, ⌘W, etc. While recording, the user can't quit or close the window with the keyboard until Esc drops focus. This is the correct trade-off — the user is in a modal "press a shortcut" state and any keystroke is supposed to be a binding, not a side effect.
- **`window?.firstResponder === self` is the load-bearing guard.** Without it, `isRecording` could be stale (e.g. the view was hidden mid-record) and the override would silently absorb keystrokes from elsewhere in the Settings window.
- **Don't move the Esc handler from `keyDown` into `performKeyEquivalent`.** Bare Esc is not a key equivalent; it routes through `keyDown` directly, and the existing code handles it there.
- **The fix doesn't need to touch `IssuesApp.swift`.** A "disable all menu commands while Settings is open" approach was considered and rejected — it's app-wide global state for a problem that's local to one view. Responder-chain interception is the right scope.
