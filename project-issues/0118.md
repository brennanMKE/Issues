## Description

Wire [Sparkle](https://sparkle-project.org/) for auto-updates so users on direct DMG downloads can stay current without manual reinstalls. Today the release flow produces a signed/notarized DMG (`scripts/release.sh`) and users grab it from wherever it's hosted; there's no in-app prompt when a new version ships.

Modeled on the equivalent work done for Batty (its `issues/0038.md` is the reference). Same shape: add the SPM dependency, ship a guarded `UpdaterController` wrapper that no-ops until the appcast is configured, add a "Check for Updates…" menu item, document the user-side EdDSA key generation + hosting in a `scripts/SPARKLE.md`. The hosting URL for this project is **`https://issues.sstools.co/`** (appcast at `https://issues.sstools.co/appcast.xml`).

## Acceptance criteria

- Sparkle SPM dependency added to the Issues target.
- `SUFeedURL` configured in `Info.plist` pointing at `https://issues.sstools.co/appcast.xml`.
- "Check for Updates…" menu item under the **Issues** app menu (near "About Issues"), disabled when `SUFeedURL` isn't set.
- A signed appcast `<item>` served from `issues.sstools.co` can prompt users to update; the update applies on relaunch.
- EdDSA signing keys generated; the public key embedded in `Info.plist` as `SUPublicEDKey`; the private key lives outside the repo (Sparkle's `generate_keys` stores it in the macOS keychain under `https://sparkle-project.org`).

## UpdaterController sketch

```swift
import Foundation
import Sparkle
import os.log

final class UpdaterController {
    static let shared = UpdaterController()

    private let logger = Logger(subsystem: Logging.subsystem, category: "Updater")
    private let controller: SPUStandardUpdaterController

    private init() {
        // startingUpdater: true is safe — without SUFeedURL Sparkle initializes
        // but never polls, so no surprise network traffic on first launch.
        self.controller = SPUStandardUpdaterController(
            startingUpdater: true,
            updaterDelegate: nil,
            userDriverDelegate: nil
        )
    }

    /// True when SUFeedURL is a non-empty string in the running app's Info.plist.
    /// The Check for Updates menu item binds `.disabled` to `!isConfigured`.
    var isConfigured: Bool {
        let raw = Bundle.main.object(forInfoDictionaryKey: "SUFeedURL") as? String
        return raw?.isEmpty == false
    }

    func checkForUpdates() {
        guard isConfigured else {
            logger.notice("Check for Updates invoked but SUFeedURL is not configured; no-op.")
            return
        }
        controller.checkForUpdates(nil)
    }
}
```

## Menu wiring sketch

```swift
CommandGroup(replacing: .appInfo) {
    Button("About Issues") { /* existing about-window action, if any */ }
    Divider()
    Button("Check for Updates…") {
        UpdaterController.shared.checkForUpdates()
    }
    .disabled(!UpdaterController.shared.isConfigured)
}
```

If the project doesn't customize About today, leaving the default About behaviour intact via `CommandGroup(after: .appInfo)` instead of `.replacing` works fine — the Batty pattern replaces because Batty also customized About. Pick whichever is least invasive here.

## Notes

- **Hosting plan**: the URL is locked to `https://issues.sstools.co/`. The simplest path is **GitHub Pages** pointing the `issues.sstools.co` subdomain at a `gh-pages` branch under the `brennanMKE/Issues` repo — free, fits the project's existing GitHub presence, and adds nothing new to operate. S3 or any static host also works; the DMG enclosure URL inside the appcast doesn't have to live on the same host as the appcast itself.
- **Info.plist interaction with #0110**: the original Batty issue's gotcha was that `GENERATE_INFOPLIST_FILE = YES` makes adding arbitrary plist keys awkward. Issues.app has the same setup today. **#0110 (ATS) recommends flipping to an explicit `Info.plist`** for the same reason — the Sparkle work is much smaller if #0110 lands first, because by then `Issues/Info.plist` exists as a normal file and Sparkle keys are a one-line addition. Order to land: **#0110 → #0118 → #0093** (Bonjour, also wants the explicit plist when its multicast approval comes through). All three benefit from the same migration.
- **EdDSA private key never enters the repo.** Backed up by the user separately (e.g. encrypted note) so a developer-machine wipe doesn't break the ability to sign future updates. Losing the key means existing clients reject every new update; recovery is "ship a new public key and accept that existing users have to redownload manually."
- **`SUEnableAutomaticChecks` is not shipped in this ticket.** Sparkle defaults to manual-only checks; flip it once the appcast is live and you want the app to poll on its own. A reasonable default once configured: `SUScheduledCheckInterval = 86400` (daily).
- **Per-release `sign_update` stays in `SPARKLE.md`, not in `scripts/release.sh`**, for the same reason as the Batty work: don't surprise the user with a Sparkle signature attempt when Sparkle isn't even configured. Fold it into the release pipeline as a follow-up after the first successful update round-trip.

## Scope (out)

- Configuring `SUFeedURL` / `SUPublicEDKey` in the repo. That requires the user-side EdDSA key generation and the appcast host being set up first. This ticket lands the scaffolding; the user wires the values when ready.
- Setting up the actual `issues.sstools.co` host (DNS, GitHub Pages config, the appcast.xml structure). Documented in `SPARKLE.md`; physical setup is a one-time user task.
- Automating `sign_update` in `scripts/release.sh`. Follow-up after the first successful update.
- Delta updates / multi-channel betas. Sparkle supports them; v1 ships full updates only.

## Root cause

Greenfield. The release flow produces signed/notarized DMGs but the app had no mechanism to discover new releases; users on direct downloads had to manually check the project's distribution page and reinstall. Sparkle is the standard solution for this exact shape — the only reason it wasn't shipped earlier is that the appcast host and EdDSA keys are user-side artifacts that didn't exist yet. The fix is to land the in-app scaffolding so flipping those user-side switches activates Sparkle with zero further code changes.

## Gotchas

- **The menu item is `.disabled` until `SUFeedURL` exists in the *built* Info.plist** — not just the source. With `GENERATE_INFOPLIST_FILE = YES` (today's setup) that means adding the key via `INFOPLIST_KEY_SUFeedURL` build settings on the Issues target, or by switching to an explicit `Issues/Info.plist` (which #0110 will land). The Sparkle code already handles both paths because it reads from `Bundle.main` at runtime — whichever way the key gets there, it works.
- **`SPUStandardUpdaterController(startingUpdater: true, …)` constructs at `UpdaterController.shared` first-access**, which happens at the first menu-item interaction (the `.disabled` binding reads `isConfigured`, which constructs the singleton, which constructs the underlying controller). With no `SUFeedURL`, Sparkle initializes but never polls — no surprise network traffic on Debug builds.
- **The new pbxproj IDs use a fresh prefix (`2FB05118`)** so they're easy to find in `git blame`. Don't collapse them into the existing `2FA81FCC` block — leaving them visually distinct makes the next Xcode-generated edits less likely to step on them.
- **`Package.resolved` is committed.** Sparkle's pin is part of the repo state so a fresh checkout resolves to the same version. Don't add it to `.gitignore`.
- **`SUEnableAutomaticChecks` is intentionally not set.** Sparkle's default is manual-only — the user picks Check for Updates… themselves — and that's the right v1 default. Flip it (`true`) and pair with `SUScheduledCheckInterval = 86400` once the appcast is live and tested.
- **`sign_update` is intentionally not wired into `scripts/release.sh`**. Once the first appcast round-trip works, fold it into the release pipeline as a follow-up; until then keeping the Sparkle signing step manual avoids the script complaining about a missing EdDSA key or a non-existent appcast destination.
