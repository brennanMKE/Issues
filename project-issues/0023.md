# 0023 — App ignores the AccentColor asset and uses a hardcoded blue

| | |
|---|---|
| **Status** | resolved |
| **Module** | Theme |
| **Platform** | macOS |
| **First seen** | 2026-05-04 |
| **Closed** | 2026-05-04 |

## Description

The asset catalog's `AccentColor` is set to purple, but the app's accent UI (selected tab chip border, selected view-mode capsule, the link in the detail panel, the focused search field border, etc.) is rendering in blue. The app is reading from a hardcoded hex constant instead of the asset.

## Root cause

`Issues/Theme/Theme.swift:38` defines:

```swift
static let appAccent = Color(hex: "#0085ff")
static let appAccentDim = Color(hex: "#0059b3")
```

These are baked-in blues, set up before the `AccentColor` asset existed in the catalog. Every site that uses `Color.appAccent` is therefore stuck on blue regardless of what the asset says.

## Fix

Point `appAccent` at the catalog asset:

```swift
static let appAccent = Color.accentColor          // reads from AccentColor.colorset
static let appAccentDim = Color.accentColor.opacity(0.6)  // or a derived shade
```

`Color.accentColor` is the SwiftUI binding to the asset catalog's accent. Anything that was using `appAccent` will pick up purple automatically, and changing the asset later won't require a code edit.

## Notes

- `appAccentDim` is currently a deeper blue (`#0059b3`). Re-derive it from the live accent — `accentColor.opacity(0.6)` or a `mix(.black, by: 0.3)` on macOS 15+. Pick whichever reads correctly in dark mode.
- Don't remove `Color(hex: …)` entirely — it's still used for the rest of the palette (status colors, backgrounds). Just stop using it for the accent.
- Verify in: `TabBarView` (active chip border + drag preview), the view-mode capsule in `StatsBarView`, `HeaderView`'s focused-search border, `IssueCardView`'s selected fill, `DetailPanelView`'s "NNNN.md ↗" link.
- The PRD's palette table mentions `appAccent #0085ff // also used for AccentColor.colorset` — the comment is stale. Update or drop it after this lands.
