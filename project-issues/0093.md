# 0093 — Bonjour entitlements and Info.plist for browse and publish

| | |
|---|---|
| **Status** | open |
| **Module** | Build |
| **Platform** | macOS |
| **First seen** | 2026-05-08 |

## Description

Phase 2 of the remote-access plan (#0076). Bonjour publish (#0088) and browse (#0089) require multicast entitlements and an `NSBonjourServices` Info.plist entry on every target that uses them. This is split out from #0086 because it's paired with the Bonjour code rather than the HTTP listener.

## Where to look

- **`Issues/Issues.entitlements`** — already has the network keys from #0086. Add the multicast key.
- **`Issues/Info.plist`** — add `NSBonjourServices`. (And `NSLocalNetworkUsageDescription` later for iOS, see Notes.)

## Entitlements to add

```xml
<key>com.apple.developer.networking.multicast</key>
<true/>
```

Without this, `NWBrowser` and `NWListener.service` silently don't propagate beyond the loopback (or fail outright depending on the macOS release). The multicast entitlement is the load-bearing piece for both publish and browse on the same network.

## Info.plist to add

```xml
<key>NSBonjourServices</key>
<array>
    <string>_issues._tcp</string>
</array>
```

This declares the service types the app may browse for. Without it, `NWBrowser` returns no results even with the multicast entitlement.

## iOS additions (forward-looking)

When the iOS target lands in #0108 it will need:

```xml
<!-- Info.plist -->
<key>NSBonjourServices</key>
<array>
    <string>_issues._tcp</string>
</array>
<key>NSLocalNetworkUsageDescription</key>
<string>Issues uses your local network to find Mac hosts that publish issues folders.</string>
```

```xml
<!-- entitlements -->
<key>com.apple.security.network.client</key>
<true/>
<key>com.apple.developer.networking.multicast</key>
<true/>
<key>keychain-access-groups</key>
<array>
    <string>$(AppIdentifierPrefix)co.sstools.Issues</string>
</array>
```

The `keychain-access-groups` matches the macOS target so a token entered on iOS is visible to the same app group on a paired Mac (relevant if/when we do iCloud-shared tokens; not v1).

## Verification

- Install a build with the entitlement and Info.plist entry on the test Mac. Bonjour publish (#0088) and browse (#0089) work as described in #0090's test pass.
- Remove the multicast entitlement and rebuild — `NWBrowser` returns empty results, `NWListener.service` publishes only on loopback. This is the negative test, used once.
- The first browse on a fresh install does **not** prompt the user on macOS (the multicast entitlement covers it). On iOS, it does — the prompt uses `NSLocalNetworkUsageDescription`.

## Notes

- The multicast entitlement is a *capability*, not a request — Apple grants it on entitlement profile generation. Make sure the team's provisioning profile is regenerated after adding the key, or the signed build won't include it. The release script (`scripts/release.sh`) doesn't need changes; it's an Xcode/profile thing.
- `NSBonjourServices` accepts an array, so adding more service types later (highly unlikely for this app) is additive.
- The `keychain-access-groups` line on iOS is forward-looking only — relevant when the iOS target exists. Listing it here means we don't have to revisit when #0108 starts.

## Scope (out)

- Network/server entitlements (host) → #0086.
- iOS target itself → #0108.
- Provisioning profile mechanics — handled by Xcode signing, not by this ticket beyond a one-time profile regen.

## Blocked (2026-05-11)

Tried to land the changes during an autonomous run; both pieces are mechanically trivial but adding `com.apple.developer.networking.multicast` triggers a Mac Development provisioning-profile regen that requires the current Mac to be registered in the developer account first. Build output:

```
error: Device "brennan-mac-mini-m4" isn't registered in your developer account.
error: No profiles for 'co.sstools.Issues' were found.
```

`-allowProvisioningUpdates` doesn't help — Xcode needs the device on file in [developer.apple.com → Devices](https://developer.apple.com/account/resources/devices) before it can generate a profile that includes the multicast capability. Ad-hoc signing (`CODE_SIGN_IDENTITY="-" CODE_SIGNING_REQUIRED=NO`) is also rejected because multicast is a development-signed-only capability.

### Unblock action (revised 2026-05-11)

**The earlier "register the device in your dev account" guidance was wrong.** Tried building with the entitlement + Info.plist additions on the MacBook (which is registered); Xcode failed with a different error:

```
error: Provisioning profile "Mac Team Provisioning Profile: co.sstools.Issues"
       doesn't include the Multicast Networking capability. Multicast Networking
       capability needs to be assigned to your team and bundle identifier by
       Apple in order to be included in a profile.
error: Entitlement com.apple.developer.networking.multicast requires approval
       from Apple to include in a profile. Please request access to the
       associated capability.
```

`com.apple.developer.networking.multicast` is one of Apple's **restricted entitlements** — it's not granted automatically with a paid Developer Program membership. The team has to request it explicitly per bundle identifier, and Apple has to approve it before any profile (development or distribution) can carry it.

The real unblock:

1. Sign in at [developer.apple.com](https://developer.apple.com/) and submit the request at [Apple Developer Contact → Multicast Networking Entitlement Request](https://developer.apple.com/contact/request/networking-multicast/). Provide:
   - Bundle ID: `co.sstools.Issues`
   - Use case: "On-device Bonjour publish (`_issues._tcp`) and browse so paired Macs on the same LAN or Tailscale overlay can discover one another's Issues.app instances for read-only remote folder viewing. No multicast traffic leaves the local network; no third-party distribution; single-user developer tooling."
2. Wait for approval (typically days; sometimes longer).
3. Once approved, re-apply the two file changes below and re-run `xcodebuild -allowProvisioningUpdates build` — Xcode regenerates the profile with the now-granted capability.

Until approval, neither development nor Developer-ID-signed builds can include the multicast entitlement. The two file changes were applied + reverted during this attempt; the build is back to its prior working state.

### Workaround while waiting for approval

Bonjour publish (#0088) and browse (#0089) can be deferred without blocking the rest of Phase 2/3 progress. The manual `host:port` field (#0091) is the user's intended fallback when discovery doesn't traverse the network anyway, and it works with no restricted entitlements. Phases 2+ that don't need mDNS specifically can land first; #0088 / #0089 / #0090 get gated on this approval coming through.

The two file changes ready to apply once the device is registered:

1. `Issues/Issues.entitlements` — append
   ```xml
   <key>com.apple.developer.networking.multicast</key>
   <true/>
   ```
   after the existing `network.client` entry.

2. `Issues.xcodeproj/project.pbxproj` — add to both Debug and Release configs of the Issues target (next to `INFOPLIST_KEY_NSHumanReadableCopyright`):
   ```
   INFOPLIST_KEY_NSBonjourServices = "_issues._tcp";
   ```

Both were applied and rolled back so subsequent issues could keep building; the diff is small enough to re-apply by hand. After registering `brennan-mac-mini-m4` in the developer account, re-apply, run `xcodebuild … -allowProvisioningUpdates build`, and verify via `codesign -d --entitlements -` (per #0086's verification step).
