# 0029 — Folder picker as a separate Window scene; tab-bar `+` opens it

| | |
|---|---|

| **Module** | App / Views |
| **Platform** | macOS |
| **First seen** | 2026-05-04 |

## Description

Today the first-launch flow hijacks the main window: `RootView` swaps `FolderPickerView` in when `tabs.tabs.isEmpty`, then swaps `MainView` in once the user picks a folder. That conflates two different surfaces — onboarding/picker vs. live tabbed app — into one window that has to keep changing what it is.

Promote the picker to its own SwiftUI `Window` scene. The tab bar's `+` button (currently calls `bookmarks.presentOpenPanel()` directly, bypassing the remembered-folders list) opens the same scene. First-launch with no remembered folders also opens it.

## Expected behavior

- **Main window** always shows the tabbed UI (`MainView`). When no tabs are open, it shows a simple empty-state placeholder ("No folders open. Use ⌘+ or click the + button to open one."). It never hosts the picker.
- **Folder Picker** is its own window: title "Open Folder", id `"folderPicker"`. Default size ~480×520. Single-instance.
- **`+` in the tab bar** calls `openWindow(id: "folderPicker")`. If the picker window is already open, this just brings it forward.
- **First launch with no remembered folders** auto-opens the picker once, on `RootView.onAppear`, when `tabs.tabs.isEmpty == true`.
- **Picking a folder** in the picker calls `tabs.openTab(url:)` on the shared `TabsModel`, then dismisses the picker window (via `@Environment(\.dismissWindow)`). If the main window happens to be closed too, the picker also calls `openWindow(id: "main")` so the new tab shows up somewhere.

## SwiftUI implementation

It's roughly four edits, all small:

### 1. Add the scene

In `IssuesApp.swift`'s `body`, alongside the existing `Window("Issues", id: "main")`, the help window, and the `Settings` scene:

```swift
Window("Open Folder", id: "folderPicker") {
    FolderPickerView()
}
.defaultSize(width: 480, height: 520)
.windowResizability(.contentMinSize)
```

`Window` (vs. `WindowGroup`) makes it single-instance; `openWindow(id:)` on a Window that's already visible is a no-op (just brings it forward). That's the behavior we want for the `+` button.

### 2. Strip the picker hijack from `RootView`

`RootView` currently does roughly `if tabs.tabs.isEmpty { FolderPickerView() } else { MainView() }`. Replace with always-`MainView`, and trigger picker auto-open from `onAppear`:

```swift
struct RootView: View {
    @Environment(\.openWindow) private var openWindow
    // ... existing state ...

    var body: some View {
        MainView(...)
            .onAppear {
                if tabs.tabs.isEmpty {
                    openWindow(id: "folderPicker")
                }
            }
    }
}
```

`MainView`'s content area should render an empty-state view when `tabs.activeTabID == nil` (or when there's no active store). One short message + a button that opens the picker. The tab bar with the `+` is still visible at the top, so the user already has a path to add tabs without the empty-state button — but the explicit hint helps on first launch.

### 3. Wire the `+` button to `openWindow`

`TabBarView`'s `addButton` currently runs:

```swift
Button {
    if let url = bookmarks.presentOpenPanel() {
        tabs.openTab(url: url)
    }
}
```

Replace with:

```swift
@Environment(\.openWindow) private var openWindow

Button {
    openWindow(id: "folderPicker")
}
```

The `bookmarks` reference can stay if the chip-row still uses it elsewhere, otherwise drop it. The picker window is now the only entry point for adding tabs.

### 4. Make `FolderPickerView` self-dismissing

The picker view needs `@Environment(\.dismissWindow)` and `@Environment(\.openWindow)`. Its existing "Add folder…" path already calls `bookmarks.presentOpenPanel()` to surface an `NSOpenPanel` sheet on the picker window itself — that part is unchanged. After the user picks a remembered folder OR the panel returns a URL:

```swift
@Environment(\.dismissWindow) private var dismissWindow
@Environment(\.openWindow) private var openWindow

func selectFolder(_ url: URL) {
    AppCommandsController.shared.tabs?.openTab(url: url)
    openWindow(id: "main")  // no-op if main is already showing
    dismissWindow(id: "folderPicker")
}
```

Reach the `TabsModel` via the shared `AppCommandsController` so the picker doesn't have to thread a binding from the main window. The picker is its own scene with its own environment; the controller singleton is the cross-window channel.

## Notes

- **`@Environment(\.dismissWindow)` requires macOS 14+** — fine, the project targets 15.6.
- **Single-instance behavior**: `Window("Open Folder", id: "folderPicker")` won't spawn duplicates if the user mashes the `+` button.
- **First-launch race**: `RootView.onAppear` fires synchronously when the main window is created. Calling `openWindow(id: "folderPicker")` immediately is fine — SwiftUI queues the open until the scene system is ready.
- **Empty state**: don't bake too much into it. A plain `Text` + a "Open a folder…" button (running the same `openWindow` action) is enough.
- **Cmd+T**: the existing `New Tab` shortcut from #0008 / #0024 currently calls `commands.newTab()` which calls `bookmarks.presentOpenPanel()`. Change it to `openWindow(id: "folderPicker")` too — same UX as clicking `+`.
- **No-tab persistence**: if the user closes all tabs, the main window stays open with the empty state. Don't auto-open the picker on every empty state — only on the first launch when there are no remembered folders. (Or: auto-open whenever `tabs.tabs.isEmpty` and the remembered list is also empty, so users never get stuck looking at an empty window.)

## Out of scope

- Making the picker show a more elaborate UI (recent paths, drag-and-drop, search). Just promoting the existing view to its own scene.
- Closing the main window when the picker opens; both stay independent.
- A "welcome" / onboarding screen separate from the picker.
