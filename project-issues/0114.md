## Description

Third step of the TLS migration (#0111). With the host serving TLS (#0112) and the token carrying the cert fingerprint (#0113), the viewer now actually uses TLS: it flips URLs from `http://` / `ws://` to `https://` / `wss://` and installs a `URLSessionDelegate` that overrides server-trust evaluation by comparing the presented cert's SHA-256 against the pinned fingerprint. With this in place, the broad ATS exception added in #0110 is removed in the same PR.

This is the migration's breaking change: post-#0114 viewers can only talk to TLS-capable hosts (post-#0112). The two PRs ship as a coordinated pair from the user's perspective — there's no plain-HTTP fallback.

## Where to look

- **`Issues/Remote/RemoteClient.swift`** (from #0094) — uses `URLSession.shared` today. Switch to a per-source `URLSession` with a delegate that handles `serverTrust` challenges.
- **`Issues/Remote/RemoteWebSocket.swift`** (from #0102) — currently builds a `URLSessionWebSocketTask` from `URLSession.shared`. Same switch: use the per-source session so WS gets the same delegate.
- **`Issues/Remote/RemoteHostIssueSource.swift`** (from #0094) — owns the session; passes the pinned fingerprint into both `RemoteClient` and `RemoteWebSocket` so the delegate has something to check against.
- **`Issues.xcodeproj/project.pbxproj`** — remove the `NSAppTransportSecurity` entry added in #0110. The `Info.plist` migration from #0110 stays (the explicit plist is still the canonical place for `NSBonjourServices` once #0093 lands).
- **`Tools/IssuesRemoteSmoke/`** (from #0087) — the smoke CLI uses plain HTTP today. Update to TLS + accept a `--fingerprint <hex>` flag for pinning. Without the flag it fails to connect; with the right flag it passes the existing checks against `https://`.

## URL changes

- `http://<host>:<port>/v1/…` → `https://<host>:<port>/v1/…`.
- `ws://<host>:<port>/v1/events` → `wss://<host>:<port>/v1/events`.

`URLSession` and `URLSessionWebSocketTask` both negotiate TLS automatically for these schemes; the delegate is what makes self-signed certs acceptable.

## Pinning delegate

```swift
final class PinnedHostSessionDelegate: NSObject, URLSessionDelegate, URLSessionTaskDelegate {
    let expectedFingerprint: String  // lowercase 64-hex

    func urlSession(_ session: URLSession,
                    didReceive challenge: URLAuthenticationChallenge,
                    completionHandler: @escaping (URLSession.AuthChallengeDisposition,
                                                  URLCredential?) -> Void) {
        guard challenge.protectionSpace.authenticationMethod == NSURLAuthenticationMethodServerTrust,
              let trust = challenge.protectionSpace.serverTrust else {
            completionHandler(.performDefaultHandling, nil)
            return
        }

        guard let chain = SecTrustCopyCertificateChain(trust) as? [SecCertificate],
              let leaf = chain.first else {
            completionHandler(.cancelAuthenticationChallenge, nil)
            return
        }

        let der = SecCertificateCopyData(leaf) as Data
        let presented = SHA256.hash(data: der).map { String(format: "%02x", $0) }.joined()

        if presented == expectedFingerprint {
            completionHandler(.useCredential, URLCredential(trust: trust))
        } else {
            completionHandler(.cancelAuthenticationChallenge, nil)
        }
    }
}
```

- The delegate is per `URLSession`, per host. One delegate's lifetime equals one `RemoteHostIssueSource`.
- It also implements `URLSessionTaskDelegate` so WebSocket task challenges route through the same handler (some routes use the task delegate, not the session delegate).
- A non-`serverTrust` challenge (Basic, NTLM, etc.) shouldn't happen for our endpoints; `.performDefaultHandling` keeps the door open without crashing.

## Behavior

- **Happy path**: viewer paste step (#0113) stored token + fingerprint; reconnect builds a session with the delegate; first request triggers a `serverTrust` challenge; fingerprint matches → connection succeeds; bearer auth happens as today.
- **Fingerprint mismatch**: handshake fails. `RemoteClient` surfaces this as a new `RemoteClientError.fingerprintMismatch`. `RemoteHostIssueSource` emits `.fingerprintMismatch` on its event stream; the existing #0094 / #0102 state plumbing routes it to the connection-state field. UI side is in #0115.
- **WS mismatch**: same path. WS open fails; the source emits `.fingerprintMismatch`.
- **TLS configuration**: TLS 1.2 minimum (matches the host's #0112 config). No HTTP/2 considerations — we're on HTTP/1.1 + WS.

## ATS

The `NSAppTransportSecurity` exception from #0110 is **removed in this PR**. Reasoning:

- `https://` satisfies ATS's protocol requirement.
- TLS 1.2 + ECDHE on a P-256 self-signed cert satisfies ATS's TLS version + forward-secrecy requirements.
- Self-signed certs fail ATS's chain validation. But `URLCredential(trust:)` from a delegate override bypasses ATS's chain check — the app becomes the trust authority.

Net result: ATS is **on** for the app, with no exceptions, and the remote-access path works via delegate-overridden trust. That's the cleanest possible posture for a sandboxed Mac app.

Keep `Info.plist` itself (it's still the home for `NSBonjourServices` once #0093 lands); just remove the `NSAppTransportSecurity` key.

## Verification

- Build with the change. App launches; ATS posture in the `.app`'s `Info.plist` is bare (no `NSAppTransportSecurity` key).
- Pair the viewer with a TLS host (#0112 + #0113):
  - Paste a combined token. Validate succeeds. Folder picker advances.
  - Reload after restart. Auto-reconnect (#0103) succeeds without re-paste.
  - WebSocket events arrive (`reload`, `update`, `delete`).
- Pair with a *mismatched* fingerprint (paste the right token but a corrupted fingerprint):
  - Connect fails. Viewer surfaces `.fingerprintMismatch` (banner UI lives in #0115; here we verify the state propagates and the connection state is correct).
- Pair with a plain-HTTP host (legacy v1 — only possible during a partial-migration accident):
  - Connect fails immediately. Inline error mentions TLS or unreachable; no ATS dance.
- Smoke CLI:
  - `swift run issues-remote-smoke --host <ip>:<port> --token <token> --fingerprint <fp>` — passes all checks over `https://`.
  - Without `--fingerprint`, exits 1 with a clear "fingerprint required" message.
- Unit tests in `IssuesTests/`:
  - `PinnedHostSessionDelegate` returns `.useCredential` for matching fingerprint, `.cancelAuthenticationChallenge` for mismatch.
  - URL builder rewrites `http://` → `https://` and `ws://` → `wss://` everywhere they appear in code.

## Notes

- `URLSession.shared` is a singleton with no delegate; it stays in use for *other* parts of the app (if any) but the remote-access path uses its own session per source. Memory cost is negligible.
- `URLSessionWebSocketTask`'s delegate dispatch went through some changes across macOS releases. Test on macOS 15 (project minimum) explicitly; the `URLSessionTaskDelegate.urlSession(_:task:didReceive:completionHandler:)` path is the one to verify because some WS implementations dispatch challenges to the task delegate rather than the session delegate.
- Don't keep the v1 plain-HTTP path "just in case." Dual mode is a security footgun and a maintenance tax for a single-user app.
