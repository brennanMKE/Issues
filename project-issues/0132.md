## Description

Commit `4631653` (filed against #0119) shipped the `issues` CLI as a standalone SwiftPM package at `Tools/IssuesCLI/` — a directory with its own `Package.swift` and `Sources/IssuesCLI/main.swift` that is **not part of the Xcode project**. The existing `IssuesCommand` Xcode target (created in #0123, source file `IssuesCommand/IssuesCommand.swift`) was left holding its 10-line placeholder body that just prints `CommandLine.arguments`.

This contradicts the explicit "Update 2026-05-12" note in #0119, which said:

> Step 1 of the next pass is already done — the target exists, the `Command.xcconfig` overlay is wired up, the binary builds and runs. A fresh subagent should start at step 2 below (replace the placeholder body), not step 1.

Net effect: two parallel CLI sources, one of which is unused by the actual product. The Xcode target builds a no-op binary; the SwiftPM tool builds a working one that lives outside the project and won't be picked up by any future Copy Files build phase.

## What's actually wrong

- `IssuesCommand/IssuesCommand.swift` still contains the placeholder `print("Arguments:", CommandLine.arguments)` body.
- `Tools/IssuesCLI/Package.swift` and `Tools/IssuesCLI/Sources/IssuesCLI/main.swift` exist on disk and were committed, but the Xcode project has no reference to either file. `xcodebuild` against the `Issues` workspace never builds them.
- The app-side work from `4631653` (URL scheme in `Info.plist`, `TabsModel.openOrActivate`, `RootView.onOpenURL`) is in the right place and should be kept as-is.

## Where to look

- `IssuesCommand/IssuesCommand.swift` — the placeholder body to replace.
- `Tools/IssuesCLI/Sources/IssuesCLI/main.swift` — the working CLI source to migrate from.
- `Tools/IssuesCLI/Package.swift` — to be deleted with the rest of `Tools/IssuesCLI/`.
- `Configuration/Command.xcconfig` — the existing xcconfig overlay for the `IssuesCommand` target; confirm it already pulls in Foundation/AppKit, otherwise wire whatever the migrated body needs.
- `Issues.xcodeproj/project.pbxproj` — read-only reference; the `IssuesCommand` target is already registered. No structural edits needed for this issue.

## Fix

1. Port the body of `Tools/IssuesCLI/Sources/IssuesCLI/main.swift` into `IssuesCommand/IssuesCommand.swift`, adapting it to the `@main struct IssuesCommand { static func main() async { … } }` shape that's already there. Keep the behaviour identical:
   - `issues` with no args resolves `$PWD`, prefers `project-issues/` then `issues/` subdirectories.
   - `issues <path>` accepts absolute, relative, `~`-prefixed.
   - `--help`, `--version`, exit-code contract (0 dispatch / 1 arg-or-fs / 2 LaunchServices).
   - 5 s LaunchServices ack timeout so missing-app surfaces a non-zero exit.
2. Delete `Tools/IssuesCLI/` entirely (the `Package.swift` and the `Sources/` tree). It has no consumers — verified via `grep -r IssuesCLI` across the repo before deleting.
3. Verify `xcodebuild -project Issues.xcodeproj -scheme IssuesCommand -configuration Debug -destination 'platform=macOS' build` produces a working binary, and that `./<built-products>/issues --help` exits 0.
4. Update #0119: note that the actual CLI body now lives in the right place, the `Tools/IssuesCLI/` mis-step is gone, and the remaining follow-ups (Copy Files build phase into `Contents/SharedSupport/`, in-app "Install Command Line Tool…" menu item, folder-picker pre-targeting on `.needsGrant`) are still pending.

## Verification

- `grep -r IssuesCLI .` returns no matches after the cleanup.
- `IssuesCommand/IssuesCommand.swift` no longer contains the placeholder body.
- Building the `IssuesCommand` scheme produces an `issues` binary that:
  - `issues --help` prints usage and exits 0.
  - `issues --version` prints the embedded version.
  - `issues /tmp/nope` exits 1.
  - `issues ~/Developer/brennanMKE/Issues/project-issues` brings Issues.app forward and either activates or opens the folder.
- The app-side URL handling (already in place from `4631653`) still routes `issues:///open?path=…` to `TabsModel.openOrActivate(folderPath:bookmarks:)`.

## Root cause

Agent error in the session that produced commit `4631653`. The #0119 spec gave two viable shapes (in-bundle Xcode target vs. standalone SwiftPM tool), and #0119's "Update 2026-05-12" note explicitly chose the first by pointing at the already-created `IssuesCommand` target. The agent ignored that and went with the second shape anyway — modelling on the `IssuesRemoteSmoke` precedent that #0119 had already moved past.
