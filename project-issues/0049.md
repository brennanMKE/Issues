# 0049 — Arrow keys navigate between issues while the preview sheet is open

| | |
|---|---|
| **Status** | resolved |
| **Module** | Views |
| **Platform** | macOS |
| **First seen** | 2026-05-05 |

## Description

Today the markdown preview sheet (`IssueMarkdownSheet`) shows a single issue and the only keyboard interaction is space/Esc to dismiss. Once the sheet is open, the user has to dismiss → arrow-key in the list → re-open the sheet to read the next issue. macOS apps that show preview-style sheets (Mail, Photos, Music) let the user keep the preview open and walk the underlying list with ↑/↓; the preview swaps in place.

Add the same to Issues.app: while the sheet is presented, ↑/↓ should call `store.selectPrevious()` / `store.selectNext()` and the sheet should re-render with the newly-selected issue.

## Why this needs a small refactor

The sheet currently presents via `.sheet(item: $markdownSheetIssue)` with `IssueMarkdownSheet` taking a `let issue: Issue`. Mutating `markdownSheetIssue` to a new value with a different `id` causes SwiftUI to dismiss-then-present (because `.sheet(item:)` keys identity off the bound value's id). That re-present animation isn't what we want for an in-place swap.

The fix is to flip the presentation to `.sheet(isPresented: $showingMarkdownSheet)` and have the sheet read `store.selectedIssue` via an `@Bindable var store: IssueStore`. Arrow key handlers in the sheet mutate `store.selectedIssueID`; SwiftUI updates the sheet's body in place. The existing `.id(issue.id)` from #0041 on `DetailPanelDescriptionView` already invalidates Textual's stale cache when the displayed issue changes — that's the load-bearing piece that makes the in-place swap work.

## Changes

1. **`IssueMarkdownSheet`** — take `@Bindable var store: IssueStore` instead of `let issue: Issue`. Body unwraps `store.selectedIssue`; if `nil`, render an empty `Color.clear` (or call `dismiss()` defensively). Add two new modifiers on the outer `VStack`:

   ```swift
   .onKeyPress(.upArrow) {
       store.selectPrevious()
       return .handled
   }
   .onKeyPress(.downArrow) {
       store.selectNext()
       return .handled
   }
   ```

   Existing `.focusable()`, `.focusEffectDisabled()`, `.onKeyPress(.space) { dismiss() }` from #0045 stay.

2. **`IssueMarkdownSheetHeaderView` + `IssueMarkdownSheetContentView`** — keep their current `let issue: Issue` shapes (no churn). The outer sheet passes `store.selectedIssue` (force-unwrapped only inside the `if let` branch, so it's safe).

3. **`MainView`** — replace `@State var markdownSheetIssue: Issue? = nil` with `@State var showingMarkdownSheet: Bool = false`. Replace `.sheet(item: $markdownSheetIssue) { ... IssueMarkdownSheet(issue: issue) ... }` with `.sheet(isPresented: $showingMarkdownSheet) { IssueMarkdownSheet(store: store) }`.

4. **`MainContentAreaView`** — change the binding from `@Binding var markdownSheetIssue: Issue?` to `@Binding var showingMarkdownSheet: Bool`. The four sub-view callbacks (`onOpenMarkdown: { markdownSheetIssue = $0 }`) become `onOpenMarkdown: { issue in store.selectedIssueID = issue.id; showingMarkdownSheet = true }`. The Enter/space `.onKeyPress` handlers set `showingMarkdownSheet = true` (selection is already correct since the user navigated to it with ↑/↓).

5. **`AppCommandsController.openMarkdown`** — closure stays `(Issue) -> Void`. Its registration in `MainView.registerCommandHandlers()` updates to do the two-step (set selection + flip the bool).

The leaf views (`IssueCardView`, `RecentRowView`, `TimelineBarView`, the Table contextMenu) do not change — they still take `onOpenMarkdown: (Issue) -> Void`. Only the closures threading through `MainContentAreaView` evolve.

## Edge cases

- **No selection / empty filter:** `store.selectPrevious()` / `selectNext()` early-return when `visibleIssueOrder` is empty. The sheet just doesn't move. No crash.
- **Wrap-around:** `selectNext()` already wraps via `(idx + 1) % order.count` and `selectPrevious()` wraps backward. Matches Mail's behavior.
- **Sheet open with no selected issue:** shouldn't happen if callers always set selection before flipping the bool, but the sheet's body should defensively show an empty surface and not crash. (`if let issue = store.selectedIssue { … } else { Color.clear }`)
- **Quick Look toggle:** spacebar still dismisses (#0045). Arrow keys are additive.
- **`.id(issue.id)` from #0041:** preserved in `DetailPanelDescriptionView`. As `store.selectedIssueID` changes, the wrapped subtree rebuilds and Textual refreshes. ✓

## Verification

- Open the sheet via Enter or space on a selected row.
- Press ↓ → the list highlight moves to the next issue, the sheet re-renders with that issue's metadata + body.
- Press ↑ → moves backward.
- Hold ↓ to scrub through several issues quickly — every step re-renders the body cleanly (no Textual stale cache).
- Press space → sheet dismisses; selection stays on whichever issue you ended on (detail panel sidebar shows it).
- Repeat in Recent, Swimlane, Timeline view modes — all four `visibleIssueOrder` arrangements should walk correctly.

## Out of scope

- Cmd+arrow / page-up/down to jump multiple issues at a time.
- Smooth cross-fade between previewed issues — `.id(...)`-driven rebuild is fine for v1.
- Persisting which issue was previewed last across launches.
