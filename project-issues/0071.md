## Description

When triaging the issue list it's useful to focus on issues that ship attachments (screenshots, crash logs, sample data) — they tend to be visual / regression bugs that benefit from a different review pass than text-only feature requests.

There's no current way to filter the list by attachment presence. Add a filter affordance on the existing filter bar (between the status pills and the module/platform dropdowns, or as another toggle in that row) so the user can scope to "issues with attachments" with a single click.

## Attachments

![Filter bar where the new selector should go](0071/filter-bar.png)

## Steps to reproduce

1. Open Issues.app.
2. Look for a way to filter the list to only issues that have a populated attachment folder.
3. Observe there's none.

## Expected behavior

A filter control on the filter bar with three states (or a tri-state cycle):

- **All issues** (default — no filtering by attachment).
- **With attachments** — show only issues whose `NNNN/` folder exists and contains at least one file.
- **Without attachments** — the inverse, useful for surfacing text-only issues that might benefit from a screenshot.

Concrete shapes that fit the existing bar (any one is acceptable):

- A pill toggle styled like the status pills, e.g. **Has Attachments** (orange dot, tappable on/off).
- A dropdown styled like "All Modules" / "All Platforms", e.g. **All Attachments ▼** with options "All / With attachments / Without attachments".
- A small paperclip icon button that cycles through the three states with a tooltip.

Any of these compose with the existing status / module / platform filters using AND semantics — picking "Open" + "With attachments" shows only open issues that have attachments.

## Actual behavior

No filter control. Reviewers have to click into each issue (or scroll past hundreds) to find the ones with screenshots.

## Implementation notes

- Source of truth: `<issuesRoot>/NNNN/` directory existence + non-empty content. An issue with `NNNN/` present but empty (like the `0060/` folder in the current Bluesky-Migration repo) should count as "without attachments" — the filter should test for *files* inside, not just folder existence.
- Cheap to compute: walk the existing model that already maps each `NNNN.md` to its sibling folder.
- Persist the selected filter in the same place the existing filters persist (UserDefaults keyed off the issues-root path, presumably) so reload doesn't reset it.
- The image in the existing `## Attachments` section of an issue counts even when the file is missing on disk — but for the *filter*, prefer "folder exists and contains at least one regular file" rather than parsing the markdown for `![](X)` references. Filesystem signal is faster and more reliable.
