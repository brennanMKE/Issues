## Description

`swiftui-pro/references/views.md`: "Each type (struct, class, enum) should be in its own Swift file. Flag files containing multiple type definitions." Several files in this project bundle two or more top-level types together.

| File | Types |
|---|---|
| `Issues/IssuesApp.swift` | `IssuesApp`, `HelpMenuButton` |
| `Issues/Views/MainView.swift` | `MainView`, `EmptyMainView`, `FolderPickerSceneView` |
| `Issues/Views/TabBarView.swift` | `TabBarView`, `DisplayItem`, `TabChipView` |
| `Issues/Views/StatsBarView.swift` | `StatsBarView`, `StatsBarSearchField` |
| `Issues/Views/Settings/GeneralSettingsView.swift` | `GeneralSettingsView`, `KnownPathsList` |
| `Issues/Views/Settings/ShortcutsSettingsView.swift` | `ShortcutsSettingsView`, `ShortcutRow` |
| `Issues/Views/Settings/KeyboardShortcutRecorder.swift` | `KeyboardShortcutRecorder`, `RecorderView` |
| `Issues/Views/TimelineView.swift` | `TimelineGeometry`, `TimelineView` |
| `Issues/State/IssueStore.swift` | `IssueSnapshot`, `IssueStore` |
| `Issues/Views/HelpView.swift` | `HelpView`, `HelpLoadError` |
| `Issues/Views/LintSheetView.swift` (none — single type) | — |
| `Issues/Services/FolderBookmarkService.swift` | `FolderBookmarkError`, `FolderBookmarkService` |
| `Issues/Models/HelpSection.swift` | `HelpSection`, `HelpCatalog` |

(Nested types like `IssueStore.ViewMode`, `LintFinding.Kind`, `ShortcutBinding.SpecialKey`, `AppCommandsController.DeepLink` are conventional and don't need extraction — the rule is about *top-level* type definitions sharing a file.)

## Suggested change

Split each file so it contains exactly one top-level type. The `Issues/` source group is a `PBXFileSystemSynchronizedRootGroup`, so dropping a new `.swift` file in the appropriate subfolder auto-registers it with the target — no `project.pbxproj` edits needed.

Naming: keep the existing type's filename, move the secondary type to its own filename. For example:

- `MainView.swift` keeps `MainView`; create `EmptyMainView.swift` and `FolderPickerSceneView.swift`.
- `TabBarView.swift` keeps `TabBarView`; create `TabChipView.swift` and `DisplayItem.swift` (or move `DisplayItem` to nest inside `TabBarView` if it's only used there).
- `IssueStore.swift` keeps `IssueStore`; move `IssueSnapshot` to `IssueSnapshot.swift`.
- `HelpView.swift` keeps `HelpView`; move `HelpLoadError` to `HelpLoadError.swift`.
- `FolderBookmarkService.swift` keeps the service; move `FolderBookmarkError` to its own file.
- `HelpSection.swift` keeps `HelpSection`; move `HelpCatalog` to `HelpCatalog.swift`.

## Notes

- This is mechanical but cumulative — the value compounds with #0036, which extracts subview computed properties into `View` structs that need their own files anyway.
- `private struct ...` types within a view file (like `StatsBarSearchField`, `TabChipView`, `KnownPathsList`, `ShortcutRow`) are an intermediate stop on the way to extraction. Promoting them to internal types in their own files unlocks `#Preview` for each and makes them reachable from tests.
- Opt-in: split files when they are touched for other reasons rather than as a single sweep, to avoid one giant churn-only diff.
