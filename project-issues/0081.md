## Description

Phase 1 of the remote-access plan (#0076). The viewer needs to fetch attachment bytes (PNGs, MOVs, crash logs) referenced by an issue. This is a separate endpoint from the JSON metadata in #0080 because attachments can be MBs to tens of MBs and the response is binary, not JSON.

## Where to look

- **`Issues/Remote/RemoteServer.swift`** (from #0079) — add the route.
- **New: a small `FileResponseWriter` helper** alongside the server, to stream a file in chunks rather than loading the whole thing into memory. Existing `NWConnection.send` chunking is sufficient; no third-party dependency.
- **`Issues/Models/`** — no changes; the path components are the existing `(folderId, issueId, attachmentName)` triple.

## Endpoint

| Method | Path | Returns |
|---|---|---|
| `GET` | `/v1/folders/{folderId}/issues/{id}/attachments/{name}` | binary stream |

### Response

- `200 OK` with:
  - `Content-Type` from filename extension. Cover the common ones used in this project: `.png`, `.jpg`/`.jpeg`, `.gif`, `.heic`, `.mov`, `.mp4`, `.log`, `.txt`. Unknown extensions → `application/octet-stream`.
  - `Content-Length` set explicitly. The viewer needs it for the progress UI in #0106.
  - `Last-Modified` from the file's mtime.
  - Body is the file bytes streamed in ~64 KB chunks via `NWConnection.send`.
- `404 not_found` if the folder, issue, or filename doesn't resolve.
- `400 bad_request` if `name` contains `..` or any path separator. Filenames must be a single path component.

### Path safety

- Resolve `<folder root>/<issueId>/<name>` and verify the canonical absolute path is **inside** `<folder root>/<issueId>/`. Reject anything that escapes via `..`, symlinks, or absolute paths. This is the only reason a bearer-authenticated viewer could read arbitrary host files, so it's the security check that matters most for this endpoint.

### Range requests

- Not supported in v1. If `Range` is present, ignore it and return the full body. Add range support only when an actual user need shows up (large MOVs that need seek).

## Behavior

- **Streaming**, not buffered. A 4.5 MB `.mov` (cf. #0072) shouldn't allocate 4.5 MB in `NWConnection`'s send queue at once.
- **No caching headers** beyond `Last-Modified`. The viewer's async fetch in #0106 keeps decoded thumbnails in memory; that's the right layer for caching.
- **Auth** uses the same middleware as every other endpoint (#0079). No special-case for binary.

## Verification

- `curl -H "Authorization: Bearer iat_<valid>" http://<ip>:<port>/v1/folders/<fid>/issues/0072/attachments/recording.mov -o recording.mov` → file downloads, byte-for-byte identical to the host's copy (`shasum -a 256` matches on both sides).
- `Content-Type` is `video/quicktime` for `.mov`, `image/png` for `.png`, etc.
- Path-traversal probe: `curl … /v1/folders/<fid>/issues/0072/attachments/..%2F..%2F..%2F..%2Fetc%2Fpasswd` → `400 bad_request`.
- Unknown attachment → `404 not_found`.
- Quitting the app mid-download closes the connection cleanly; no zombie file handles.

## Notes

- `URLSession.shared.dataTask` on the viewer side will buffer the whole response into memory. The viewer should use `URLSession.bytes(for:)` for streaming, especially for large videos. That's the viewer's problem — see #0106.
- We could later put attachments behind signed short-lived URLs to avoid sending the bearer on every byte, but for v1 the bearer is fine — the local network is trusted (or it's a Tailscale tunnel).

## Scope (out)

- Range requests / partial downloads.
- Server-side image resizing or thumbnail generation. The viewer can render at the size it wants, or use Quick Look.
- Async client-side fetch UX is #0106.

## Root cause

#0080 covered every JSON endpoint but explicitly deferred attachment bytes — they need a different shape (binary, streamed, content-type per filename) and a different security check (path-traversal). #0081 closes that gap.

## Fix

`HTTPResponse` grew an opt-in streaming variant: `streamingFile: URL?` plus an explicit `contentLength: Int?`. When `streamingFile` is set, `serialized()` writes headers only and the server's send path streams the file in 64 KB `FileHandle.read(upToCount:)` chunks via `NWConnection.send`. `body` is unused in the streaming case. The `.file(url:contentType:contentLength:lastModified:)` factory builds the response with `Last-Modified` already formatted as RFC 7231 IMF-fixdate.

The new handler at `/v1/folders/{folderId}/issues/{id}/attachments/{name}` does three things in order:

1. **Resolve** the folder + issue from `MultiFolderStore`. Unknown ids → `404`.
2. **Validate the name** via `isSafeAttachmentName(_:)` — rejects empty, `.`, `..`, and anything containing `/`, `\\`, or NUL. Returns `400 invalid_name`.
3. **Canonicalize** both the attachment directory and the candidate file via `standardizedFileURL.resolvingSymlinksInPath()`, and require the resolved file path to start with the resolved directory path + `/`. This catches symlinks pointing outside the attachment folder — the cheap pre-check from step 2 can't see those.

After validation, `attributesOfItem(atPath:)` confirms the entry is a `typeRegular` file (not a directory we somehow let through, and not a symlink whose target slipped past the prefix check). `Content-Length` and `Last-Modified` come from those attributes.

`contentType(for:)` covers the formats this project actually uses (png/jpg/jpeg/gif/heic/webp/mov/mp4/log/txt/json/md); everything else gets `application/octet-stream`.

## Files changed

- `Issues/Remote/RemoteServer.swift` — `HTTPResponse` gained the streaming fields and `.file(...)`. The server's `send` was split into `sendBytes` + `streamFile`; streaming uses 64 KB chunks via `FileHandle`.
- `Issues/Remote/RemoteHandlers.swift` — new `attachment(store:captures:)` handler, plus `isSafeAttachmentName(_:)` and `contentType(for:)` helpers exposed for tests. `routes(store:)` registers the new path.
- `IssuesTests/RemoteHandlersTests.swift` — 7 new tests: happy-path streaming response shape, path-traversal name rejection, unknown-file 404, unknown-folder/issue 404, symlink-escape 400, content-type mapping, and headers-only `serialized()` for the streaming case.

## Gotchas

- **Two-stage path safety, not one.** A name-shape filter alone misses symlinks: the user could create `<folder>/0001/escape.png -> /etc/hosts` (a single path component, valid name) and read the link target. `standardizedFileURL.resolvingSymlinksInPath()` on both sides plus a prefix check is what closes it. Don't skip the prefix check just because the name looks safe.
- **`FileHandle.read(upToCount:)` returns nil at EOF in modern Swift**, but in practice macOS returns `Data()` instead. The streaming loop treats both `nil` and empty as end-of-stream so we don't deadlock waiting for more bytes.
- **`Content-Length` must be set on the wire** even for streaming responses; the viewer's `URLSession` and the progress UI in #0106 depend on it. We compute it from `attributesOfItem(.size)` once before sending headers — never read the file twice.
- **No range support in v1.** A `Range` header is silently ignored; the spec calls this out and the viewer just gets the whole body. If a future scrubbable-MOV use case lands, we add `206 Partial Content` then.
