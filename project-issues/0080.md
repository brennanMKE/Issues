## Description

Phase 1 of the remote-access plan (#0076). With the listener and auth middleware in place (#0079), wire the actual JSON endpoints a viewer uses to enumerate folders and read issues. This ticket covers everything except attachment streaming (#0081) and live updates (#0100).

## Where to look

- **`Issues/Remote/RemoteServer.swift`** (from #0079) — register handlers in the route table here.
- **New: `Issues/Remote/RemoteProtocol.swift`** — shared `Codable` types used in responses (`HostInfo`, `FolderInfo`, `IssueMetadata`, etc.). Cross-platform; the viewer (#0094) will link the same file.
- **`Issues/Models/Issue.swift`** — reuse the existing `Issue` shape. `IssueMetadata` is `Issue` minus the body, plus `hasAttachments: Bool`.
- **`Issues/State/IssueStore.swift`** — the server reads through the same store(s); folder enumeration comes from whatever multi-folder container we land on in #0085.
- **`Issues/Models/ProjectMetadata.swift`** (from #0075) — `FolderInfo` carries `name`, `repository`, and `description` from `project.json` when present.

## Endpoints

| Method | Path | Returns |
|---|---|---|
| `GET` | `/v1/host` | `{ displayName, version: 1, folderCount }` |
| `GET` | `/v1/folders` | `[FolderInfo]` |
| `GET` | `/v1/folders/{folderId}` | `FolderInfo` (single) |
| `GET` | `/v1/folders/{folderId}/issues` | `[IssueMetadata]` |
| `GET` | `/v1/folders/{folderId}/issues/{id}` | `{ metadata: IssueMetadata, body: String, attachments: [String] }` |

### Response shapes

```swift
struct HostInfo: Codable {
    let displayName: String       // e.g. "Brennan's MacBook Air"
    let version: Int              // 1
    let folderCount: Int
}

struct FolderInfo: Codable {
    let id: String                // 16-hex (from #0082)
    let name: String              // project.json name, or parent folder name when project.json absent
    let url: URL?                 // optional, from project.json's `url` field
    let description: String?      // optional; reserved for a planned IssuesSkill extension (not in today's schema)
    let parentPath: String        // for picker disambiguation when names collide
    let issueCount: Int
    let modifiedAt: Date          // ISO8601 in JSON
}

struct IssueMetadata: Codable {
    let id: String                // "0042"
    let title: String
    let status: String            // raw, lowercased, e.g. "in-progress"
    let modules: [String]
    let platform: String
    let firstSeen: Date?
    let closedAt: Date?
    let hasAttachments: Bool
    let modifiedAt: Date
}

struct IssueDetail: Codable {
    let metadata: IssueMetadata
    let body: String              // raw markdown body (Description, Steps, etc.)
    let attachments: [String]     // relative paths under <folder>/<id>/
}
```

### Errors

- Unknown `folderId` or `id` → `404 not_found`.
- Body too large for an inline read (e.g. > 5 MB markdown — extremely unlikely) → `413 payload_too_large` and ask the client to retry the attachment endpoint instead. Defensive only; not expected in practice.
- Filesystem error → `500 internal_error` (logged on host).

### Encoding

- All responses are `application/json; charset=utf-8`.
- Dates are ISO8601 with fractional seconds.
- Use a single shared `JSONEncoder` / `JSONDecoder` with `.iso8601` date strategy. Don't mix conventions per endpoint.

## Behavior

- **Source of truth** stays the markdown files on disk. The server reads through whatever in-memory store the host already has (no parallel cache, no JSON index).
- **Issue body** is the raw markdown after the title line and the field-row table — the same body the existing `MarkdownIssueParser` extracts. The viewer renders it the same way the local detail panel does today.
- **Attachments listing** is a directory listing of `<folder>/<issueId>/` if it exists, returning relative filenames only. The actual file bytes come from #0081.
- **`folderId`** is computed via #0082 and stable across server restarts as long as the security-scoped bookmark resolves to the same path.

## Verification

- `curl -H "Authorization: Bearer iat_<valid>" http://<ip>:<port>/v1/host` → JSON with the expected fields.
- `curl … /v1/folders` → array of `FolderInfo` for every folder the host is currently serving (per the #0085 toggles).
- `curl … /v1/folders/<id>/issues` → array of `IssueMetadata` with the same count as `ls *.md | wc -l` in the host folder.
- `curl … /v1/folders/<id>/issues/0042` → JSON with `metadata`, `body`, and `attachments` fields. Body matches `cat 0042.md` minus the title and field-row table.
- Bad folder id → `404`.
- The smoke-test CLI in #0087 exercises every endpoint at once.

## Notes

- The wire format is fully decided by `RemoteProtocol.swift`. Keep that file the schema authority — viewer code reads through it, not through string keys.
- `version: 1` lets a future host signal protocol bumps without breaking older viewers. We'll deal with negotiation if and when v2 happens.
- No pagination on `/v1/folders/{folderId}/issues`. Folders are small (60–200 issues typically); paginating now is premature.

## Scope (out)

- Attachment binary streaming → #0081.
- WebSocket events → #0100.
- Anything client-side → #0094.

## Root cause

#0079 left the route table seeded with two stub handlers (`/v1/host` returning `{"ok":true}` and `/v1/folders` returning `{"folders":[…ids…]}`). Real wire shapes were intentionally deferred until #0080 to keep the listener PR small and the wire schema centralized.

## Fix

Three new files plus a small parser helper:

- **`Issues/Remote/RemoteProtocol.swift`** — Codable wire types `HostInfo`, `FolderInfo`, `IssueMetadata`, `IssueDetail`, plus `RemoteProtocol.encoder` / `.decoder` and `RemoteProtocol.version = 1`. The encoder uses a custom ISO8601 strategy with `.withFractionalSeconds` so timestamps round-trip exactly; the decoder accepts both fractional and plain ISO8601 to tolerate prototypes. JSON keys match Swift property names verbatim (no `CodingKeys` overrides) so the schema is the file.
- **`Issues/Remote/RemoteHandlers.swift`** — handler functions plus `routes(store:)` that builds the `[Route]` the server installs. Each handler is a static func taking the `MultiFolderStore` and the captured route placeholders, returning a fully-formed `HTTPResponse`. Unknown `folderId` / issue `id` map to `404 not_found`; encode failures throw and become `500 internal_error` via the server's catch-all.
- **`Issues/Remote/RemoteServer.swift`** — `MultiFolderStore` grew the three methods #0080 needs (`hostDisplayName`, `currentlyHostedFolders()`, `currentlyHostedFolder(forId:)`). The placeholder `hostedFolderIds` is now a default-implemented derivation so older callers stay valid. A new `HostedFolder` adapter struct lets tests build wire input directly without spinning up an `IssueStore`. `installDefaultRoutes` now hands the table off to `RemoteHandlers.routes(store:)` so the listener doesn't know which paths exist.
- **`Issues/Services/MarkdownIssueParser.swift`** — new `body(from:)` static func extracts the raw markdown body after the H1 title and the pipe-table metadata block. Used by `/v1/folders/{folderId}/issues/{id}` to send the exact text the viewer renders.

`folder.repository` comes straight from `project.json`'s `url`; `description` is reserved for a future `project.json` field and currently sent as `null`. `parentPath` is the folder containing the watched issues directory (`/Users/x/Code/MyRepo` for `/Users/x/Code/MyRepo/issues`) — matches the spec's "for picker disambiguation when names collide."

## Files changed

- `Issues/Remote/RemoteProtocol.swift` — new. Wire types + shared encoder/decoder.
- `Issues/Remote/RemoteHandlers.swift` — new. The 5 endpoints + the `routes(store:)` builder.
- `Issues/Remote/RemoteServer.swift` — extended `MultiFolderStore`; added `HostedFolder`; `installDefaultRoutes` now delegates to `RemoteHandlers`.
- `Issues/Services/MarkdownIssueParser.swift` — added `body(from:)`.
- `IssuesTests/RemoteHandlersTests.swift` — new. 11 tests covering each endpoint shape (success + 404), `FolderInfo` derivation (parentPath, project.json url, issueCount), `body(from:)` for both with-table and bare-title cases, and `IssueMetadata` encoder/decoder round-trip.

## Gotchas

- **`Issue` is ambiguous in test files** — Swift Testing exposes its own `Issue` (used as `Issue.record(...)`), which collides with the project's `Issue` model when both are imported. Use `Issues.Issue` explicitly in test fixtures, matching `IssueStoreTests.swift`.
- **Don't introduce `CodingKeys` overrides on the wire types.** The viewer (#0094) and any future tool that links `RemoteProtocol.swift` rely on Swift property names being the canonical wire keys. Adding overrides quietly grows the surface where the host and viewer can disagree.
- **Listing attachments degrades, not errors.** `RemoteHandlers.listAttachments` returns an empty list on any filesystem failure rather than propagating, so a missing/unreadable `<id>/` directory yields `attachments: []` instead of a `500`. The viewer's #0081 fetch handles "no such file" cleanly anyway.
- **Body extractor is best-effort.** Files that don't follow the `# NNNN — Title` + pipe-table convention will still get a usable response; we strip the H1 if present and return the rest. The lint pane (#0019) is the right place to surface schema violations, not the wire response.
