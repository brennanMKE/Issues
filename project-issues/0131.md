## Description

Loading https://issues.sstools.co/ in a browser doesn't show the Issues icon in the tab — most browsers fall back to a generic placeholder. The 32×32 PNG at `assets/favicon.png` is served correctly (HTTP 200, `image/png`, 1537 bytes, valid 32×32 RGBA), but the page's `<head>` only declares that single icon link with no `sizes` hint, no `apple-touch-icon`, and there's no `/favicon.ico` at the site root.

## Steps to reproduce

1. Open https://issues.sstools.co/ in any browser (Safari, Chrome, Firefox).
2. Look at the browser tab.
3. **Actual**: generic globe / page icon. Tab title is visible; favicon is not.
4. **Expected**: the purple Issues folder icon appears at the tab.

## What's actually wrong

The deployed `<head>` is:

```html
<head>
    <meta charset="utf-8">
    <meta name="viewport" content="width=device-width, initial-scale=1">
    <title>Issues — A native macOS viewer for project-issues markdown</title>
    <meta name="description" content="…">
    <link rel="icon" type="image/png" href="assets/favicon.png">
    <link rel="stylesheet" href="assets/styles.css">
</head>
```

Two gaps that affect modern browsers:

1. **No `apple-touch-icon` link.** Safari (macOS and iOS) treats `apple-touch-icon` as the canonical tab/bookmark icon source. Without it, Safari often falls through to the generic icon even when `<link rel="icon">` is present.
2. **No `sizes` attribute on the icon link, no high-DPI variant.** Retina tabs render the favicon at effectively 64×64. The browser treats the unannotated 32×32 PNG as low-resolution and may fall back to "no icon" rather than blur it.

(Legacy `/favicon.ico` at the site root is deliberately *not* part of this fix. Modern browsers honor the `<link rel="icon">` tags and don't require the root `.ico` lookup; adding one is unnecessary overhead.)

The asset itself is fine — it's the declarations that are missing.

## Where to look

- **`website/index.html` `<head>`** (around line 8) — add the new `<link>` tags.
- **`website/assets/`** — add `apple-touch-icon.png` (180×180), `icon-192.png` (192×192), and `/favicon.ico` (multi-size: 16, 32, 48). The source `icon.png` (1024×1024) is already in the assets directory; downsamples are produced from it.
- **`website/404.html`** — same `<head>` block needs the same additions so a broken-link landing also gets the icon.

## Fix

### Generate the missing assets

From the existing `website/assets/icon.png` (1024×1024 RGBA):

```sh
cd website/assets

# Apple touch icon — Safari macOS, iOS home screen, iOS tab.
sips -z 180 180 icon.png --out apple-touch-icon.png

# High-DPI tab icon — Retina browsers want this.
sips -z 192 192 icon.png --out icon-192.png
```

Resulting layout:

```
website/
├── index.html             ← head updated
├── 404.html               ← head updated
└── assets/
    ├── icon.png           ← existing source (1024×1024)
    ├── favicon.png        ← existing (32×32) — keep
    ├── apple-touch-icon.png ← new (180×180)
    ├── icon-192.png       ← new (192×192)
    └── styles.css
```

### Updated `<head>`

```html
<head>
    <meta charset="utf-8">
    <meta name="viewport" content="width=device-width, initial-scale=1">
    <title>Issues — A native macOS viewer for project-issues markdown</title>
    <meta name="description" content="…">
    <link rel="icon" type="image/png" sizes="32x32" href="assets/favicon.png">
    <link rel="icon" type="image/png" sizes="192x192" href="assets/icon-192.png">
    <link rel="apple-touch-icon" href="assets/apple-touch-icon.png">
    <link rel="stylesheet" href="assets/styles.css">
</head>
```

Browsers pick the `<link rel="icon">` whose `sizes` best matches their target. Standard-DPI tabs use the 32×32; Retina tabs use the 192×192; Safari and "Add to Home Screen" use the `apple-touch-icon`. No `/favicon.ico` needed — modern browsers honor the link tags.

### Apply to both pages

`website/index.html` and `website/404.html` both need the same `<head>` icon block. The CSS link can stay as-is.

## Verification

After committing and redeploying:

- **Chrome / Edge / Firefox**: hard reload (`Cmd+Shift+R`) → tab icon appears.
- **Safari**: hard reload → tab icon appears. Add the page to Bookmarks; the bookmark uses the `apple-touch-icon`.
- **`curl -I https://issues.sstools.co/assets/apple-touch-icon.png`** returns `200 OK`, `Content-Type: image/png`.
- **`curl -I https://issues.sstools.co/assets/icon-192.png`** returns `200 OK`.
- View `https://issues.sstools.co/some-fake-url` (which hits `404.html`) → tab icon also appears.
- On iOS, "Add to Home Screen" produces a properly-rendered Issues icon, not a screenshot.

## Notes

- The browser tab is the visible symptom; the underlying fix also makes "Add to Home Screen" (iOS / iPadOS), "Pin to Taskbar" (Edge), and macOS Safari Reading List previews work. They all draw from the same `<link>` hints.
- We don't need a full PWA manifest (`manifest.json`) for v1; that's a separate concern (install-as-app, splash screen, theme color). The current change is purely about *seeing the icon in a tab*. A manifest is a reasonable follow-up if we ever want the site installable.
- The 1024×1024 `icon.png` already in `website/assets/` is the canonical source. All downsamples should regenerate from it if we ever change the artwork. Worth adding a tiny `website/scripts/regenerate-icons.sh` if this recurs — but for v1, one manual `sips` pass is fine.
- The `Last-Modified` header on the deployed favicon shows `Wed, 13 May 2026 07:03:36 GMT`, so the asset is fresh. The issue is purely the HTML declaration set.

## Scope (out)

- Full PWA manifest (`manifest.json`). Not needed for tab favicon visibility.
- Dark-mode-specific icon variants (Safari supports `<link rel="mask-icon" color="...">` for monochrome pinned-tab icons). Out of v1.
- Generating a `.svg` favicon for unlimited scaling. The PNG ladder covers every common DPI; SVG support varies enough that it's not worth the additional surface for v1.
- Updating the in-app `AppIcon.png`. The in-app icon set is separate from website assets and is already correctly configured for Issues.app.

## Dependencies

- **None.** Self-contained content change to `website/`. Doesn't touch the app or any other ticket.
- **Composes with `#0120`** (the website itself, resolved). Follow-on cleanup that the website ticket didn't catch because favicon UX is browser-specific and easy to miss in a "rendered correctly in my local preview" test.

## Root cause

The site's `<head>` declared a single small icon link (`<link rel="icon" type="image/png" href="assets/favicon.png">` pointing at a 32×32 PNG) with no `sizes` attribute, no `apple-touch-icon`, and no high-DPI variant. Modern browsers — particularly Safari and any Retina display — treated the lone 32×32 as too low-resolution to render in the tab and fell back to the generic placeholder icon. The asset existed and was served correctly; the *declarations* were the problem.

## Fix

Two new icon variants generated from the existing 1024×1024 `website/assets/icon.png` via `sips`:

- `website/assets/apple-touch-icon.png` — 180×180. Safari (macOS / iOS) and "Add to Home Screen" pick this one.
- `website/assets/icon-192.png` — 192×192. Retina tabs render the tab favicon at ~64×64, so a 192×192 source gives ~3× density which the browser scales cleanly.

Updated `<head>` in both `index.html` and `404.html`:

```html
<link rel="icon" type="image/png" sizes="32x32" href="assets/favicon.png">
<link rel="icon" type="image/png" sizes="192x192" href="assets/icon-192.png">
<link rel="apple-touch-icon" href="assets/apple-touch-icon.png">
```

`/favicon.ico` was considered and rejected — modern browsers honor the `<link rel="icon">` declarations directly and don't require the legacy root-`.ico` lookup. A confirmed working sibling site from the same author runs without one; adding one to this site would be unnecessary overhead.

## Files changed

- `website/index.html` — three icon links replacing the single old one.
- `website/404.html` — same change for the not-found page so it shares the brand.
- `website/assets/apple-touch-icon.png` — new, 180×180.
- `website/assets/icon-192.png` — new, 192×192.

## Gotchas

- **Browser cache is aggressive about favicons.** After redeploying, a hard reload (Cmd+Shift+R) is required to see the change. Some browsers cache "no favicon found" per-origin across normal reloads; closing and reopening the tab is sometimes needed too.
- **`/favicon.ico` is intentionally absent.** If a future tool or audit complains about it, the answer is "modern browsers don't need it; sister site `<x>` runs the same way." Adding one is one `Pillow` invocation away if it ever turns out a specific browser version actually needs it.
- **The 1024×1024 `icon.png` stays as the canonical source.** All downsamples (32×32, 180×180, 192×192) regenerate from it via `sips -z`. If the artwork ever changes, run the three `sips` commands from `assets/`.
- **Both pages need the same icon block.** `index.html` and `404.html` are the two HTML files in the website today. Any future HTML page added under `website/` should inherit the same block (consider extracting to a server-side include if more pages get added).
- **Path style differs between the two files.** `index.html` uses relative `assets/...` paths; `404.html` uses absolute `/assets/...`. Preserved the existing style in each. Both resolve correctly; consistency could be a future cleanup if the site grows.
