## Description

#0050 added a Finder folder drop zone on `FolderPickerView` and `MainView` / `EmptyMainView`, and the drop itself works. But the UI gives no hint that the surface accepts drops — the accent border only appears once a drag is already hovering over the window. A first-time user has no way to discover the feature.

Add a passive visual affordance so the user can tell at a glance that a folder can be dragged in.

## Where the affordance should live

- **`EmptyMainView`** — the empty-state surface shown when no folder is open is the highest-value spot. Today it's mostly blank; an icon + caption ("Drag an issues folder here, or use ⌘N") inside a dashed-border drop region makes the feature obvious.
- **`FolderPickerView`** — the picker also gains a small dashed-border drop zone near the "Add folder…" button, with a single-line caption ("…or drop a folder here").

## Suggested visual

```
┌───────────────────────────────┐
│            📁                 │
│    Drag an issues folder      │
│       here to open it         │
└───────────────────────────────┘   ← dashed RoundedRectangle, .secondary stroke
```

The dashed stroke should be subtle — `Color.appBorder` or `.secondary.opacity(0.4)` — so it sits as a hint rather than a primary control. While a drag is hovering, the existing accent-border behavior from #0050 still kicks in and supersedes the dashed style.

## Edge cases

- Once a folder is loaded, the affordance is gone (the main content area replaces it). Drop still works on the surrounding window — that's a power-user fallback, no visual hint needed there.
- Don't double up: when the dashed hint is visible, the active drag-hover overlay should replace it, not stack on top.

## Verification

- Launch the app with no remembered folders → empty state shows the dashed drop region with caption and icon.
- Drag a folder over it → dashed border switches to solid accent border (existing behavior from #0050).
- Drop the folder → tab opens, dashed region disappears with the empty state.

## Notes

This is the discoverability follow-up to #0050; close coordination with that view tree is expected.

## Root cause

`#0050` wired up the folder drop target, but the only feedback it produced was the accent border drawn during `isTargeted`. There was no passive marker on the empty surfaces, so a first-time user had no way to discover the drop affordance.

## Fix

Added a new reusable `FolderDropHintView` — a passive dashed `RoundedRectangle` with an SF Symbol and caption. It is purely decorative; the existing `folderDropTarget` modifier still owns the drop handling and continues to draw the accent border on hover, which sits on top of the dashed hint and visually supersedes it.

- `EmptyMainView`: replaced the static "Click the + button or press \u{2318}T" caption with the tall hint variant ("Drag an issues folder here to open it" + secondary line referencing the + button / \u{2318}T). The "Open Folder\u{2026}" button stays beneath the hint.
- `FolderPickerView`: added the compact variant ("\u{2026}or drop a folder here") below the Add folder button so the picker scene gains the same affordance.

The dashed stroke uses `Color.appBorder` per the issue's suggestion, with a 1 pt line and a `[5, 4]` dash pattern. Corner radius matches the 8 pt corner used by the drag-hover accent overlay so the two visuals align.

## Files changed

- `Issues/Views/FolderDropHintView.swift` — new reusable dashed-border hint view with tall and compact variants and DEBUG previews.
- `Issues/Views/EmptyMainView.swift` — empty-state surface now embeds the tall `FolderDropHintView` between the heading and the Open Folder button.
- `Issues/Views/FolderPickerView.swift` — picker now embeds the compact `FolderDropHintView` beneath the Add folder button.

## Gotchas

- The hint is intentionally not part of the `folderDropTarget` modifier itself — keeping it a plain decorative view avoids coupling the visual affordance to the drop lifecycle and means existing call sites get no behavior change beyond what they opt into.
- The dashed border sits inside the drop region, so the accent border (drawn by the modifier on the outer view) wraps around it without any z-order coordination — the modifier's overlay is rendered on top of the entire view, including the hint.
