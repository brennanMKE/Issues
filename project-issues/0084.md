## Description

Phase 1 of the remote-access plan (#0076). The host UI for issuing, naming, and revoking access tokens. Modeled directly on GitHub's PAT settings page. This is the only place tokens are managed — no automatic pairing, no QR-from-host in v1 (QR support is part of the iPhone work in #0108).

Builds on `AccessToken` (#0078) and lives inside the same settings sheet as #0083.

## Where to look

- **`Issues/Views/RemoteHostSettingsView.swift`** (created in #0083) — add an "Access tokens" section.
- **New: `Issues/Views/AccessTokenListView.swift`** — the section's contents.
- **New: `Issues/Views/GenerateTokenSheet.swift`** — the modal sheet that creates a new token and reveals the plaintext once.
- **`Issues/Remote/AccessToken.swift`** (from #0078) — `generate(name:expiresAt:)`, `revoke(hash:)`, `list()`.

## UI

### Token list section

```
┌─ Access tokens ────────────────────────────────────┐
│                                                    │
│  Name              Created      Last used   Expires │
│  ────────────────  ───────────  ──────────  ─────── │
│  MacBook Air       2026-05-08   2026-05-08   never  [ Revoke ]
│  Spare Mac mini    2026-05-08   —            2026-08-08  [ Revoke ]
│                                                    │
│                                  [ + Generate token ]
└────────────────────────────────────────────────────┘
```

- One row per token. Columns: `name`, `createdAt`, `lastUsedAt` (em-dash if never), expiration (`never` or absolute date).
- Last-used updates automatically as viewers connect; the view re-reads the token list when the sheet is open and the WS receives a connection event, or on a 5-second poll while the sheet is open. No background polling.

```
│             ( ) On a specific date  [ 2026-08-08 ] │
```

After Generate, the same sheet morphs into a one-time reveal:


### Revoke

- Clicking Revoke on a row prompts a small inline confirm ("Revoke 'MacBook Air'? Connected viewers will be disconnected.") — not a modal alert, an inline confirm to keep the flow tight.
- On confirm, calls `AccessToken.revoke(hash:)` and any open WebSocket using that token gets closed by the server (#0079 / #0100).

## Behavior

- The Generate sheet's defaults are: name empty (focus there), expiration "Never". The user picks a useful name and clicks Generate.
- "30 days" and "90 days" are computed from `Date.now`; the specific-date picker uses today as the minimum.
- The plaintext reveal shows once. Closing the sheet — even with the close button or `Esc` — discards the plaintext and the user has to revoke and regenerate to recover.
- All Keychain operations happen on a background actor; the UI shows a brief progress indicator if the operation takes longer than ~250 ms (rare).
- The list is sorted by `lastUsedAt` desc, with never-used tokens at the bottom by `createdAt` desc.

## Verification

- Generate a token. Plaintext is shown once and copyable. After clicking Done, the row appears in the list with `Last used —`.
- Use the token from another Mac via `curl -H "Authorization: Bearer iat_…" /v1/host` — the row's `Last used` updates within a few seconds.
- Generate a 30-day token. Row shows the right expiration date.
- Revoke a token. `curl` with that token returns 401. Connected WS is closed by the server.
- Quit and relaunch — token list survives (it's in the Keychain, not in memory).

## Notes

- The default-expires-never choice is intentional: the user is single-user and the inconvenience of an expired token mid-session outweighs the small leak-mitigation win for a token that lives on the user's own devices. The 30 / 90 buttons are there for the rare case the user wants to grant access to a temporary device.
- "Show QR code" is **not** on the reveal sheet in v1. It's part of the iPhone work (#0108) when scanning makes sense; for two-Mac use the user pastes via Notes/clipboard.
- Don't add a "regenerate" button. Revoke + generate is two clicks and is clearer about the security model.

## Root cause

#0083 brought the host settings sheet to life but left the token list as a spec gap. Without it the user had no way to mint a bearer, even though `AccessToken` (#0078) had been ready since Phase 1.

## Fix

Two new view files plus a one-line insert into `RemoteHostSettingsView`:

- **`AccessTokenListView`** — section content. Reads the Keychain via `AccessToken.list()` on appear and every 5 s while visible; the polling cadence matches the issue spec and stops in `.onDisappear`. Sort matches the spec: `lastUsedAt` descending with never-used by `createdAt` descending below them. Revoke uses an in-row inline confirm — when the user clicks Revoke the cell swaps to a `[Cancel] [Confirm]` pair, so the destructive action is two clicks but never opens a modal alert (matches the spec's "inline confirm to keep the flow tight"). Confirm calls `AccessToken.revoke(hash:)` and reloads. Empty state shows the "no access tokens yet" caption with the Generate button still visible so the path forward is one click.
- **`GenerateTokenSheet`** — two-phase modal. Phase 1 is a `Form`-style layout with a name `TextField` (`@FocusState` so the cursor lands there on appear), a radio-group `Picker` for expiration (`.never`, `.thirtyDays`, `.ninetyDays`, `.specific`), and a `DatePicker` that only appears when `.specific` is selected. Cancel and Generate are wired to `.cancelAction` and `.defaultAction` so Esc and Return work as expected. Phase 2 is the one-time plaintext reveal with a monospace block (selectable + `Copy` button) and the "Save this now — it won't be shown again." caption. Done dismisses; closing the sheet by any path discards the plaintext.

`RemoteHostSettingsView` grew an `Access tokens` section that just hosts `AccessTokenListView()`. The list talks to the static `AccessToken` API directly — no new state types or DI — so the existing 11 `AccessTokenTests` already cover the data path.

## Files changed

- `Issues/Views/AccessTokenListView.swift` — new. Token list + inline confirm + 5s polling.
- `Issues/Views/GenerateTokenSheet.swift` — new. Two-phase modal.
- `Issues/Views/RemoteHostSettingsView.swift` — added the "Access tokens" section that hosts `AccessTokenListView`.

## Gotchas

- **Inline confirm, not `.confirmationDialog`.** SwiftUI's confirmation dialog is a modal alert — the spec calls explicitly for an in-row swap. The current pattern (Cancel + Confirm replacing Revoke in the same row) keeps the rest of the list visible and clickable.
- **Timer.scheduledTimer fires on `.common` mode by default**, which means it pauses while the user is dragging the window or scrolling. That's fine for the 5s "have any viewers connected" refresh — if responsiveness becomes an issue, switch to `RunLoop.main.add(timer, forMode: .common)` plus `.tracking`. For now, idle-pause behavior is desired.
- **`AccessToken.generate` throws on Keychain failure.** Surfaced as the form's inline error label rather than a separate sheet — generation rarely fails on a healthy machine. If the user sees it, the underlying state (Keychain locked, profile mismatch) is the real diagnostic, not the wording.
- **Plaintext is only in memory for the lifetime of the reveal view.** Closing the sheet drops `revealedPlaintext` along with the view; SwiftUI doesn't persist `@State` across sheet dismissals. The spec calls for this explicitly — the user must save the value while the reveal is on screen.
- **Visual verification still needed.** Inline confirm hover/press feel, DatePicker layout when `.specific` is selected, sheet sizing on first appearance, and the Copy button's brief feedback are not unit-tested.
