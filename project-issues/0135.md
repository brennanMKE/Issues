## Description

Revise the `issues-dashboard` layout to drop the three labelled section headers (`ACTIVE` / `RECENT` / `NEXT UP`) and instead lean on **status colour** as the primary signal of an issue's state вҖ” mirroring how the Mac app renders status. The dashboard becomes a flatter, denser table where the row's status colour does the work of telling the user "this is in-progress" without consuming three rows of chrome.

User motivation, verbatim: "With the Mac app the status has a color. Instead of headers for the 3 sections we can simply show colors for the status."

Concretely:

- Remove the three `в”Җ ACTIVE в”Җв”Җв”Җв”Җв”Җв”Җ`, `в”Җ RECENT в”Җв”Җв”Җв”Җв”Җв”Җ`, `в”Җ NEXT UP в”Җв”Җв”Җв”Җв”Җ` header rows.
- Render the `STATUS` cell in the same colour palette the macOS app uses вҖ” see `Issues/Theme/StatusColor.swift` (`statusOpen`, `statusInProgress`, `statusResolved`, `statusClosed`, `statusWontfix`).
- Keep the title bar, the column header row, the hairline divider under it, and the footer. Net chrome drops from ~5 to ~3 rows, freeing 2 rows for actual data on every terminal size.
- 4-column layout (`#  STATUS  TITLE  MODIFIED`), 1 Hz polling, truncate-don't-scroll all stay.

## Ranking вҖ” the load-bearing design question

Dropping the band headers takes away the visual cue that says *why* an issue is on screen ("this is here because it's active" vs "this is here because it's queued"). The colour palette covers the first half (you can spot in-progress at a glance), but you still need a sort order that puts the right issues near the top. Two viable shapes:

### Option A вҖ” Flat sort by recency, status implied by colour

Sort the whole visible list by `modifiedAt` descending. Tiebreak by issue number ascending. The natural result:

- In-progress issues (yellow) cluster at the top because they're usually the most recently touched.
- Recently-resolved issues (green) and freshly-edited open issues sit immediately below.
- Older issues fall off the bottom.

Pros: simplest implementation, no model-side complication, no special-casing.

Cons: **loses the "next-up open queue" entirely.** An open issue with a high number that hasn't been touched recently never shows up. That's a real regression вҖ” the prior design surfaced these explicitly via the `NEXT UP` band so the user could see what was coming up next.

### Option B вҖ” Hybrid: recency-sorted top, queue-sorted tail

Two implicit bands, no labels:

- Top **70 %** of the visible row budget: all issues sorted by `modifiedAt` desc. Same as Option A but capped.
- Bottom **30 %**: open issues sorted by number ascending, with any issues already shown in the top band filtered out.

No divider line, no header. The transition is signalled only by colour pattern вҖ” the top band has a mix of yellow/green/blue/gray; the bottom band is uniformly blue (open). Once the user has scanned the dashboard a few times the pattern is internalized: "the blue block at the bottom is the queue."

Pros: preserves the original signal (active work + queue preview) without the chrome cost.

Cons: implicit structure is harder to discover than labelled bands. A new user staring at the screen for the first time has to figure out why the blue rows are grouped at the bottom.

### Option C вҖ” Pure recency, with a sentinel row when the queue would be cut off

Like Option A, but if any open issue has been excluded (because it's not recent enough to fit), append one final row: `вҖҰ N more open issues. issues-dashboard --queue to see them.` That row is dimmed (gray) and not a real issue.

Pros: communicates "there's more queued work" without rendering it inline.

Cons: introduces a second command (`--queue`) we'd then need to design. Probably overkill for the dashboard's monitoring role.

**Recommendation: Option B.** Mirrors the original design intent at a fraction of the chrome cost; the implicit-bands pattern is the same one `top` uses (top of `top` is "running processes," bottom is "everything else" вҖ” no labels). Revisit if user testing shows the implicit grouping isn't obvious.

## Notes

The original tri-band design (in #0133) was right at the time вҖ” the dashboard had to communicate "why each row is on screen" before the user had internalized any conventions. After a few days of use the section headers are noise; the colour palette now does the work. This issue is a deliberate simplification of the prior design, not a course correction.

## Root cause

The tri-band header rows (`в”Җ ACTIVE в”Җв”Җв”Җв”Җв”Җв”Җ`, `в”Җ RECENT в”Җв”Җв”Җв”Җв”Җв”Җ`, `в”Җ NEXT UP в”Җв”Җв”Җв”Җв”Җ`) were using 3 rows of chrome to signal grouping intent that status colour alone can convey. Each header also required a layout allocator that computed ipShow/recentShow/nextShow budgets with dedup logic, adding complexity without user-visible benefit once the colour palette was familiar.

## Fix

Replaced `DashboardSnapshot` with a two-array shape (`recency` sorted by modifiedAt desc; `openQueue` sorted by id asc) and `totalCount`. `ContentView` now uses an Option B hybrid layout: recency-sorted top (~70% of the data-row budget) followed by the open queue deduped against visible top rows (~30%), no dividers. A bold column header row and hairline were added so the column meanings are always on screen. Unused queue budget donates back to the top band so a short queue doesn't waste rows.

## Files changed

- `IssuesKit/Sources/IssuesDashboardCLI/DashboardModel.swift` вҖ” replaced three-array snapshot with `recency`/`openQueue`/`totalCount` shape; simplified model function.
- `IssuesKit/Sources/IssuesDashboardCLI/ContentView.swift` вҖ” dropped section-header rendering and band-budget allocator; added `columnHeader`, `hairline`, and `hybridLayout`; title bar now shows `totalCount`.
- `IssuesKit/Sources/IssuesDashboardCLI/IssueStore.swift` вҖ” updated error-path preserve to use new snapshot fields.
