## Description

Dropping a `.png` (or other image) onto an issue's card or detail panel should: copy the file into the issue's `NNNN/` sibling folder, append an `![filename](NNNN/filename.png)` line to the issue's `## Attachments` section, and save.

## Notes

- Drop targets: `IssueCardView`, `DetailPanelView`. Use `.dropDestination(for: URL.self)`.
- Reuse the markdown writer service from #0015 (atomic file rewrite). Inserting under `## Attachments` requires a small parser tweak: if the section is missing, append it.
- Image references must use the `NNNN/` prefix, per the project memory — bare filenames don't render in the Textual sheet.
- Filename collisions: append a `-1`, `-2` suffix.
- Sandbox: dropping a file from outside the app's scope works because `dropDestination` provides a temporarily-readable URL; copy it into the issue folder while that scope is active.
- macOS narrow-no-break-space gotcha for screenshot filenames is handled by `FileManager.copyItem` (it operates on URL bytes, not a globbed path), so no special handling needed at this layer.

## Resolution — won't fix

Issues.app is intentionally a **read-only** viewer of the markdown files. Only AI agents (via the IssuesSkill) write or edit issue files; the app is the live dashboard. Adding a write path — drag-and-drop attachments, status edits, anything that mutates the markdown — would expand the sandbox entitlements and split the source-of-truth between the agent and the GUI. Closing as `wontfix`. The same reasoning applies to #0015 (right-click status change) — flagging here so it gets the same treatment if the user agrees.
