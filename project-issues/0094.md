## Description

Phase 3 of the remote-access plan (#0076). With the `IssueSource` protocol in place (#0077), add a second implementation that talks to a remote host over HTTP (and WS once #0102 lands). To `IssueStore` and the views, a remote folder looks identical to a local one — same `[Issue]` array, same change signal, same lifecycle.

## Where to look

- **New: `Issues/Remote/RemoteClient.swift`** — `URLSession`-based wrapper for the v1 endpoints (#0080, #0081). Adds the `Authorization: Bearer …` header to every request.
- **New: `Issues/Remote/RemoteHostIssueSource.swift`** — implements `IssueSource` from #0077 by composing `RemoteClient` with a small in-memory issue array.
- **`Issues/Models/`** — uses `RemoteProtocol.swift` (from #0080) for wire types and the existing `Issue` for the in-memory shape.
- **`Issues/State/IssueStore.swift`** — already source-agnostic post #0077; nothing to change here.

## Behavior

- **Init**: takes a `(host, port, token)` triple plus a `folderId`.
- **`start()`**:
  - Calls `GET /v1/folders/{folderId}` to fetch `FolderInfo`. Populates `displayName` and `projectMetadata`.
  - Calls `GET /v1/folders/{folderId}/issues` for the metadata list.
  - Lazily fetches issue bodies — see "Body fetching" below.
  - Once #0102 lands, opens a WS to `/v1/events`, sends `subscribe` for this folder.
- **`reload()`**: re-fetches `/v1/folders/{folderId}/issues`, then materializes the diff.
- **`stop()`**: cancels in-flight requests, closes the WS, releases everything.
- **Errors**:
  - 401 → emit `.tokenInvalid` through the change stream and stop. The viewer surfaces "expired token" UI (#0104).
  - Connection failure → emit `.disconnected`. Reconnect logic is #0103.
  - 404 on the folder → emit `.folderUnavailable`. The viewer falls back to find-by-name (#0098).

### Body fetching

The metadata list arrives without bodies. The detail panel needs the body. Options considered:

- **Eager fetch all bodies up front** — fine for ~60 files; rejected because it's wasteful for a viewer that only opens a few.
- **Lazy fetch on selection, cache in memory** — the chosen design. When `IssueStore.selectedIssueID` changes, the source fetches `/v1/folders/{folderId}/issues/{id}` and stuffs the body into the in-memory `Issue`. Subsequent selections are instant.
- **Pre-fetch on hover** — too clever for v1.

Cache TTL is "until the source is destroyed". A `reload()` invalidates the cache for issues whose `modifiedAt` changed.

### Threading

- `RemoteClient` is an actor or `Sendable` value type with its own `URLSession`. Requests run off the main thread; the source publishes results back via `@MainActor`.
- The change stream uses `AsyncStream<RemoteEvent>` where `RemoteEvent` is an enum of `.reloaded`, `.issueChanged(id)`, `.disconnected`, `.tokenInvalid`, `.folderUnavailable`.
- Cross-platform: this file should compile on iOS unchanged for #0108.

## Verification

- New unit tests with a mocked `RemoteClient` (or against a real local host bound to 127.0.0.1):
  - `start()` populates `displayName` from the host.
  - `reload()` returns the same count as the host's `/v1/folders/{id}/issues`.
  - Selecting an issue triggers a body fetch; second selection of the same issue is a cache hit (no network call).
  - 401 surfaces `.tokenInvalid` and `start()` returns early.
- Manual: open a remote folder via the (still-to-come) viewer UI in #0099 → swimlane / list / timeline render exactly as for a local folder.

## Notes

- Don't introduce a separate viewer-side data model. Reuse `Issue` and feed it the same way the local source does. The whole point of the `IssueSource` refactor was to make remote source's output indistinguishable from local's.
- `URLSession.shared` is fine here. We don't need a custom session unless we want connection pooling; the request volume is low.
- Streaming for attachment bodies uses `URLSession.bytes(for:)` (per #0081 notes); that's #0106's problem, not this ticket's.

## Scope (out)

- WS event handling — depends on #0100 (server) and #0102 (viewer-side reconnect). For this ticket, leave a `// TODO(#0102)` marker where the WS wiring will go and ship the polling fallback for now (`reload()` is enough for visible parity).
- UI integration (token paste, picker, tab type) — #0095 / #0096 / #0097 / #0099.
- Async attachment fetch UI — #0106.

## Root cause

Greenfield. #0077 introduced the `IssueSource` protocol; #0080 and #0081 shipped the host endpoints; this issue finally creates the second concrete `IssueSource` so a remote folder can be plugged into the existing `IssueStore` machinery with no view changes.

## Fix

Two new files plus tests:

**`Issues/Remote/RemoteClient.swift`** — `RemoteClientProtocol` with three methods that map 1:1 to documented endpoints, plus the production `URLSessionRemoteClient` and a `RemoteClientError` enum that distinguishes:

- `.unauthorized` (401)
- `.folderNotFound` (404 on a folder path)
- `.notFound` (404 on an issue path)
- `.transport(message)` (`URLError` etc.)
- `.decode(message)` (JSON failure)
- `.unexpectedStatus(code)` (anything else)

Every request carries `Authorization: Bearer <token>`. Responses are decoded through `RemoteProtocol.decoder` so date round-trips stay exact. `#if os(macOS) || os(iOS)`-gated so the iOS viewer (#0108) reuses it verbatim.

**`Issues/Remote/RemoteHostIssueSource.swift`** — `@MainActor final class` conforming to `IssueSource`. The synthetic `folderURL` (`issues-remote://<host>:<port>/<folderId>`) lets `IssueStore.bookmarkData` / `folderId` produce stable hashes for the eventual remote tab type (#0099) without needing a security-scoped bookmark. `bookmarkData` is `Data("remote:\(host):\(port)|\(folderId)".utf8)` so two instances with the same identity hash identically.

- `start()` → `reload()` → async `performReload()` that fetches `FolderInfo` + `[IssueMetadata]`, maps each meta into a local `Issue` (with `module = modules.joined(separator: " / ")` so the existing splitter on `Issue.modules` round-trips), invalidates body cache entries whose `modifiedAt` moved, drops bodies for vanished ids, fires `onUpdate`, and yields `.reloaded` on `eventStream`.
- `fetchBody(for:)` is the lazy-cache entry point. Cache miss → `fetchIssueDetail`, stuff the body into the in-memory `Issue.description`, fire `onUpdate`, yield `.issueChanged(id)`. Cache hit → return immediately (no network). `hasCachedBody(for:)` is the matching predicate for the eventual UI.
- Error mapping fans into `eventStream` so the viewer UI (#0104) can react: `.tokenInvalid`, `.folderUnavailable`, `.disconnected`. `loadError` carries the user-visible string and `folderInvalidated` flips on `.folderUnavailable` so existing IssueStore handling (close the tab, prompt re-pick) works without changes.
- `// TODO(#0102)` marker in `start()` is the WS subscription seat — polling via `reload()` is the only refresh path today.

**`IssuesTests/RemoteHostIssueSourceTests.swift`** — 8 tests with a `StubRemoteClient`:

- `startPopulatesDisplayNameAndIssues` — happy path; `displayName` flips from `host:port` to the wire `name`; module string joins back to `Services / State`.
- `unauthorizedSurfacesTokenInvalidAndStopsLoading` — 401 → `.tokenInvalid` event + `loadError` set + `inFlightReloadTask` cleared.
- `folderNotFoundFlipsFolderInvalidatedAndYieldsEvent` — 404 → `folderInvalidated = true` + `.folderUnavailable`.
- `transportFailureYieldsDisconnected` — `.transport(...)` → `.disconnected`.
- `fetchBodyHitsTheNetworkOnceThenCaches` — first call goes to the wire, second is a cache hit (`stub.detailFetchCount["0001"] == 1`).
- `reloadInvalidatesCacheWhenMtimeMoves` — same id with a new `modifiedAt` clears the cached body.
- `reloadDropsCachedBodyForIssuesThatVanish` — issue id disappears from the metadata list → cache entry purged.
- `folderIdHashesRemoteIdentity` — bookmark bytes are deterministic on `(host, port, folderId)` and differ when any component changes.

## Gotchas

- **`@MainActor` source + async `URLSession`.** `performReload()` awaits the client off-main (the URLSession is its own actor), then state mutation lands on main because the surrounding `Task` inherits the source's isolation. Don't push the state mutation onto a background queue or you'll lose `@Observable` notifications.
- **`Issue.module` is a `String` joined with `" / "`.** The existing `Issue.modules` splits on `/` and trims whitespace — round-tripping through join preserves the contract that `Issue.modules` is the canonical accessor.
- **`fileURL` on remote issues is synthetic.** Local consumers like `IssueMarkdownSheet` that read `issue.fileURL` from disk will need a fork — they should check whether the URL has the `file://` scheme before stat'ing. That's #0099 / #0106 territory; the source intentionally doesn't try to fake a file.
- **Cancellation propagates through `Task.checkCancellation()`** after the second fetch. A `reload()` issued while a prior reload is in flight cancels the old one cleanly; the new run replaces it.
- **`AsyncStream.makeStream` would be cleaner** than the `unsafeBitCast`-equivalent continuation pattern, but it requires iOS 17/macOS 14. We target macOS 15.6+, so we could use it; the current capture-into-`var continuation` pattern is portable and equally correct. Leaving as-is for now.
