## Description

This issue is **two things bundled**, in this order:

1. **Add a native macOS Settings window** to the app. SwiftUI on macOS provides a built-in `Settings` scene that gives the standard "Issues → Settings…" menu item (Cmd+,) and a system-styled preferences window with toolbar-tabs across the top. The app currently has no Settings scene at all — adding one is a prerequisite for any user-facing preference, including this one. Use the SwiftUI `Settings` scene type (NOT a custom `Window` or a sheet); macOS users expect Settings windows to look and behave like every other Mac app's.
2. **Expose configurable keyboard shortcuts** as the first pane inside that Settings window. Future preferences (theme override, badge toggle, etc.) can live alongside it in additional panes; for v1 the Shortcuts pane is the only non-placeholder content.

### Why native Settings, not a custom window

- **Discoverability**: Cmd+, and the "Settings…" menu item are muscle memory for Mac users. Anything else is a friction point.
- **Visual consistency**: the SwiftUI `Settings` scene matches System Settings styling — toolbar-tabs across the top, fixed window size adapting to selected pane, no titlebar buttons beyond close. Reproducing this with `Window` would be hand-rolling the chrome.
- **Free behavior**: window position memory, single-instance behavior (Cmd+, brings the existing window forward, doesn't open a second), and proper restoration on app relaunch all come for free.

The shortcut customization itself is the user's example of a setting we need; the broader requirement is the Settings infrastructure.

### Configurable shortcuts (the first concrete preference)

The keyboard shortcuts shipped in #0008 are hardcoded. The Shortcuts pane lets the user rebind the ones with a reasonable choice of bindings, while keeping the numeric tab-selection shortcuts (Cmd+1…9) fixed since those have a long-established Safari/Chrome convention and are positional rather than mnemonic.

For example, **Cmd+Shift+Left / Cmd+Shift+Right** are currently the previous/next-tab shortcuts (matching Safari), but a user may prefer something simpler — e.g. Cmd+Option+Left / Right, Ctrl+Tab / Ctrl+Shift+Tab, or whatever fits their muscle memory.

## Stays fixed (not in the panel)

- **Cmd+1…9** — activate Nth tab. Positional, by-index, conventional. No reason to expose.
- **Up / Down arrows** — list selection navigation. Not a Cmd-shortcut and rebinding arrow keys creates more confusion than value.
- **Esc** — dismiss sheet. System convention.

## UI — the native `Settings` scene

Add a top-level `Settings { SettingsView() }` scene to `IssuesApp.swift`, alongside the existing `Window("Issues", id: "main")` and `Window("Issues Help", id: "help")` scenes (#0026, #0025). SwiftUI handles:

- The "Settings…" menu item under the app menu (with Cmd+, automatically wired).
- Single-instance behavior: Cmd+, brings the existing Settings window to the front; no duplicates.
- macOS-standard frameless preferences-style chrome.

`SettingsView` uses a SwiftUI `TabView` with `.tabViewStyle(.automatic)` (the default produces the macOS Settings toolbar look — icons across the top, content below). For v1:

- **General** (placeholder pane with a single label like "More preferences coming soon"). Icon: `gearshape`.
- **Shortcuts** — the meat of this issue. Icon: `keyboard`.

Each pane is its own SwiftUI view inside `Issues/Views/Settings/` (e.g. `GeneralSettingsView.swift`, `ShortcutsSettingsView.swift`). Decorate each with `.tabItem { Label("General", systemImage: "gearshape") }` etc. so the toolbar renders correctly.

The Shortcuts pane is a `Form` (not a custom layout) — it sits naturally inside `Settings` and inherits the right padding / row heights without manual tuning. Each row: action label on the leading edge, current shortcut on the trailing edge as a `KeyboardShortcutRecorder` (a small `NSViewRepresentable` wrapping a focusable control that captures `NSEvent.keyDown` and reports modifier flags + key code). A reset-to-default button per row and a "Reset all to defaults" button below the form.

There's no off-the-shelf SwiftUI shortcut recorder — the standard pattern is the `NSViewRepresentable` above. Rough sketch: focusable view that reads `NSEvent` modifier flags + key code on `keyDown`, formats them as `⌘⇧→`, returns the new binding. Existing implementations on GitHub (e.g. `KeyboardShortcuts` by sindresorhus) are MIT-licensed if we don't want to roll our own — worth a look during implementation, but a 60-line in-house recorder is also fine.

## Persistence

Store bindings in `UserDefaults` under a single dictionary key — `keyboardShortcuts: [ActionID: SerializedBinding]`. `ActionID` is an enum (`previousTab`, `nextTab`, `newTab`, …); `SerializedBinding` is a `{ key: String, modifiers: Int }` pair. `AppCommandsController` (added in #0008) loads on init and exposes `binding(for: ActionID) -> KeyboardShortcut`. The `IssuesApp.commands { ... }` block reads from the controller for every shortcut binding.

## Validation

- Detect collisions across actions and within macOS-reserved shortcuts (Cmd+Q, Cmd+H, Cmd+Tab, etc.). Show a warning row when the chosen shortcut is already used by another action; block the assignment if it's a system reserved shortcut (or accept it with a warning — TBD).
- Reset-to-default works per-row.

## Out of scope for v1

- Importing/exporting binding sets (e.g. shareable JSON).
- Multi-stroke bindings (Cmd+K then Cmd+S — too rare in macOS apps).
- Per-window or per-tab bindings.
- Cmd+1…9 customization (intentional — see "Stays fixed" above).

## Follow-up — recorder doesn't accept input + UX cleanup

After the first pass landed (commit `8ab808f`), the user reported three issues:

1. **The shortcut recorder doesn't capture input.** Clicking the recorder box has no visible effect — no focus ring, no "Press a shortcut…" prompt, no `keyDown:` interception. The user can't actually rebind anything.
2. **The explainer text at the top of the Shortcuts pane is unnecessary.** "Customize keyboard shortcuts. Cmd+1 through 9 (tab activation), arrow keys, and Esc are not customizable." — the pane title already says it's the Shortcuts pane; the customizable scope is self-evident from the rows shown. Remove it.
3. **The General pane currently has nothing in it.** Replace the "More preferences coming soon" placeholder with a useful **Known Paths** list — the issue folders the app has remembered (security-scoped bookmarks managed by `FolderBookmarkService`). Each row shows the folder's display path and last-used date, with a Remove button per row that drops the bookmark.

### Recorder fix

The custom `NSView` subclass needs three things to capture key events on macOS:

- `override var acceptsFirstResponder: Bool { true }` — without this the window won't even consider the view for first-responder.
- `override func mouseDown(with event: NSEvent)` that calls `window?.makeFirstResponder(self)` — clicking the box must explicitly transfer first-responder; SwiftUI doesn't do this for raw `NSView`s wrapped via `NSViewRepresentable`.
- `wantsLayer = true` set in `init` so focus-state redraws (the tinted ring while recording) work via Core Animation.

Also useful: override `becomeFirstResponder()` / `resignFirstResponder()` to update an `isRecording` flag that drives the visual state, and `override func draw(_:)` to render the focus-tinted background only while `isRecording == true`.

Verify in the running app: click a row → the box highlights, displays "Press a shortcut…" → press a chord → the binding updates, store persists, focus drops automatically.

### Explainer-text removal

Delete the leading `Text(...)` paragraph in `ShortcutsSettingsView.swift`. The form rows are self-explanatory.

### General pane → Known Paths

Rebuild `GeneralSettingsView` to show a List of remembered folders from `FolderBookmarkService`:

```swift
struct GeneralSettingsView: View {
    @Environment(FolderBookmarkService.self) private var bookmarks  // or reference shared instance

    var body: some View {
        Form {
            Section("Known Paths") {
                if bookmarks.rememberedFolders.isEmpty {
                    Text("No remembered folders. Open one from the tab bar's + button.")
                        .foregroundStyle(.secondary)
                } else {
                    ForEach(bookmarks.rememberedFolders) { folder in
                        HStack {
                            VStack(alignment: .leading) {
                                Text(folder.displayPath)
                                Text("Last used: \(folder.lastUsed, format: .dateTime.day().month().year().hour().minute())")
                                    .font(.caption)
                                    .foregroundStyle(.secondary)
                            }
                            Spacer()
                            Button("Remove", role: .destructive) {
                                bookmarks.forget(folder)
                            }
                            .buttonStyle(.borderless)
                        }
                    }
                }
            }
        }
    }
}
```

`FolderBookmarkService` already stores an array of remembered folders in `UserDefaults` under key `rememberedFolders` (see #0002 / #0009). Inspect the actual API — there's likely already a `forget(_:)` or similar method used by the right-click "Forget" context menu in the folder picker. If not, add one.

The Settings window and the main window share the same `FolderBookmarkService` instance, so removing a folder here updates both. Removing a folder that's currently open as a tab should NOT close the tab (the security-scoped resource for that tab's session stays valid until the tab closes); just drop it from the remembered-folders list. The next launch won't have it in the picker anymore.
