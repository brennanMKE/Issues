# 0024 — User-configurable keyboard shortcuts in Settings

## Description

The keyboard shortcuts shipped in #0008 are hardcoded. Add a Settings panel that lets the user rebind the shortcuts that have a reasonable choice of bindings, while keeping the numeric tab-selection shortcuts (Cmd+1…9) fixed since those have a long-established Safari/Chrome convention and are positional rather than mnemonic.

For example, **Cmd+Shift+Left / Cmd+Shift+Right** are currently the previous/next-tab shortcuts (matching Safari), but a user may prefer something simpler — e.g. Cmd+Option+Left / Right, Ctrl+Tab / Ctrl+Shift+Tab, or whatever fits their muscle memory.

## Configurable shortcuts

| Action | Default | Notes |
|---|---|---|
| Previous tab | Cmd+Shift+Left | configurable |
| Next tab | Cmd+Shift+Right | configurable |
| New tab | Cmd+T | configurable |
| Close active tab | Cmd+W | configurable |
| Reload active tab | Cmd+R | configurable |
| Focus search | Cmd+F | configurable |
| Swimlanes view | Cmd+Opt+1 | configurable |
| Timeline view | Cmd+Opt+2 | configurable |
| List view | Cmd+Opt+3 | configurable |
| Recent view | Cmd+Opt+4 | configurable |

## Stays fixed (not in the panel)

- **Cmd+1…9** — activate Nth tab. Positional, by-index, conventional. No reason to expose.
- **Up / Down arrows** — list selection navigation. Not a Cmd-shortcut and rebinding arrow keys creates more confusion than value.
- **Esc** — dismiss sheet. System convention.

## UI

`Settings` scene in `IssuesApp.swift` (currently absent — the app has no Settings command). Add a `SettingsView` with two tabs:

- **General** (placeholder for future preferences).
- **Shortcuts** — a list of action rows, each with the action name on the leading edge and a `KeyboardShortcutRecorder` view on the trailing edge that shows the current binding and lets the user click to record a new one. Reset-to-default per row and a "Reset all to defaults" button at the bottom.

There's no off-the-shelf SwiftUI shortcut recorder — the standard pattern is a small `NSViewRepresentable` wrapping an `NSTextField`-like control that captures key events. Rough sketch: a focusable view that reads `NSEvent` modifier flags + key code on `keyDown`, formats them as `⌘⇧→`, and returns the new binding. Existing implementations on GitHub (e.g. `KeyboardShortcuts` by sindresorhus) are MIT-licensed if we don't want to roll our own — out of scope for this issue, but worth a look during implementation.

## Persistence

Store bindings in `UserDefaults` under a single dictionary key — `keyboardShortcuts: [ActionID: SerializedBinding]`. `ActionID` is an enum (`previousTab`, `nextTab`, `newTab`, …); `SerializedBinding` is a `{ key: String, modifiers: Int }` pair. `AppCommandsController` (added in #0008) loads on init and exposes `binding(for: ActionID) -> KeyboardShortcut`. The `IssuesApp.commands { ... }` block reads from the controller for every shortcut binding.

## Validation

- Detect collisions across actions and within macOS-reserved shortcuts (Cmd+Q, Cmd+H, Cmd+Tab, etc.). Show a warning row when the chosen shortcut is already used by another action; block the assignment if it's a system reserved shortcut (or accept it with a warning — TBD).
- Reset-to-default works per-row.

## Out of scope for v1

- Importing/exporting binding sets (e.g. shareable JSON).
- Multi-stroke bindings (Cmd+K then Cmd+S — too rare in macOS apps).
- Per-window or per-tab bindings.
- Cmd+1…9 customization (intentional — see "Stays fixed" above).
