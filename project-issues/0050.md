# 0050 — Drop a folder from Finder onto the picker (or main window) to open it

## Description

Today, opening a new folder requires clicking the `+` button or `Add folder…`, which presents `NSOpenPanel`. macOS users expect to drag a folder out of Finder and drop it onto an app to open it. Add a drop zone:

- On `FolderPickerView` (entire VStack body) so dropping while the picker is shown opens that folder as a new tab.
- On `MainView` / `EmptyMainView` (the main window itself) so dropping anywhere on the main window also opens the folder, regardless of whether the picker is currently visible.

Highlight the drop zone with a subtle accent border while a drag is hovering, so the user gets standard macOS drop-target feedback.

## Implementation sketch

SwiftUI's `.onDrop(of:)` accepts `[.fileURL]`. Resolve the dropped item to a `URL`, then call the same path the picker's `Add folder…` button uses:

```swift
.onDrop(of: [.fileURL], isTargeted: $isDropTargeted) { providers in
    guard let provider = providers.first else { return false }
    _ = provider.loadObject(ofClass: URL.self) { url, _ in
        guard let url else { return }
        Task { @MainActor in
            do {
                try bookmarks.remember(url: url)
                onSelect(url)
            } catch {
                bookmarks.lastError = error.localizedDescription
            }
        }
    }
    return true
}
.overlay {
    if isDropTargeted {
        RoundedRectangle(cornerRadius: 8)
            .strokeBorder(Color.appAccent, lineWidth: 2)
    }
}
```

`@State private var isDropTargeted: Bool = false` lives on whichever view owns the drop zone.

## Edge cases

- Reject non-folder drops (file paths). Use `URL.resourceValues(forKeys: [.isDirectoryKey])` and bail if false.
- Reject duplicates: if the dropped folder already has an open tab, just activate that tab (`TabsModel.openTab(url:)` already does this).
- Multiple folders dropped at once → open each as a separate tab.
- Sandbox bookmark: the user-initiated drop should be treated identically to picker-driven selection — same security-scoped bookmark flow.

## Verification

- Drag a folder from Finder onto the empty `FolderPickerView` → folder opens, becomes the active tab.
- Drag a folder onto the main window when there's already one tab open → opens as a new tab.
- Drag a non-folder file → no-op (no tab added, no error banner).
- Drag while hovering over the drop zone → accent border appears; clears on drop or drag-out.

## Root cause

Not a bug — a missing affordance. Folder selection went exclusively through the picker buttons (`Add folder…`, remembered list) or the tab bar `+`; neither `FolderPickerView` nor `MainView` / `EmptyMainView` declared any `.onDrop` modifier, so dragging a folder out of Finder onto the app was a silent no-op.

## Fix

Introduced a reusable `FolderDropTargetModifier` (`Issues/Views/FolderDropTarget.swift`) that:

- Accepts `[.fileURL]` providers via `.onDrop`.
- Filters out non-directory drops by querying the filesystem (`FileManager.default.fileExists(atPath:isDirectory:)`); only true folders invoke the callback.
- Renders an accent-colored `RoundedRectangle.strokeBorder` overlay while a drag is hovering, gated on `isTargeted`.
- Calls back on the main actor once per dropped folder so multi-folder drops open as separate tabs.

Wired the modifier into the three drop surfaces:

- `FolderPickerView` — drop calls `bookmarks.remember(url:)`, then routes through the existing `onSelect(url)` so `FolderPickerSceneView` opens the tab and dismisses the picker, mirroring the picker buttons exactly.
- `MainView` — drop calls `bookmarks.remember(url:)` then `tabs.openTab(url:)`. `openTab` already dedupes by folder path so dropping an already-open folder just activates that tab.
- `EmptyMainView` — same flow as `MainView`, covering the case where the main window is up but no tab is active.

Bookmark-generation failures fall back to `bookmarks.lastError`, which the picker already surfaces and the main window logs.

## Files changed

- `Issues/Views/FolderDropTarget.swift` (new) — shared `FolderDropTargetModifier` + `View.folderDropTarget(onFolder:)` extension.
- `Issues/Views/FolderPickerView.swift` — applied `.folderDropTarget` to the picker body.
- `Issues/Views/MainView.swift` — applied `.folderDropTarget` to the main window body.
- `Issues/Views/EmptyMainView.swift` — applied `.folderDropTarget` to the empty-state body.

## Gotchas

- The drop overlay uses `.allowsHitTesting(false)` so the border ring doesn't intercept clicks while it's visible. (It's only on screen during a drag, but belt-and-suspenders.)
- Dropped URLs come with implicit security-scoped access via Powerbox, so we don't need to balance `startAccessingSecurityScopedResource()` here — `IssueStore.start()` does that for the bookmark we just persisted via `bookmarks.remember(url:)`.
- The picker route deliberately calls `onSelect` instead of `tabs.openTab` directly so `FolderPickerSceneView` still gets to dismiss the picker window after the drop, just like it does for button-driven selection.
