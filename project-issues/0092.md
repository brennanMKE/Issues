## Description

Phase 2 of the remote-access plan (#0076). The host should show a live list of currently-connected viewers — useful for confirming a viewer authenticated, spotting stale connections, and (eventually) revoking access mid-session. The data already exists on the server (`RemoteServer.connectedPeers` from #0079, refined when WS lands in #0100); this ticket wires it into the existing host settings sheet.

## Where to look

- **`Issues/Views/RemoteHostSettingsView.swift`** (from #0083) — add a "Connected viewers" section.
- **`Issues/Remote/RemoteServer.swift`** (from #0079) — already exposes `connectedPeers`. Make sure it's `@Observable` so the view re-renders on connect/disconnect.

## UI

```
┌─ Connected viewers ────────────────────────────────┐
│                                                    │
│  Token            Address          Subscribed       │
│  ────────────────  ───────────────  ─────────────── │
│  MacBook Air       100.74.12.5      Issues.app, …   │
│  Spare Mac mini    192.168.1.50     Bluesky Mig.    │
│                                                    │
│  No viewers connected.    ← when empty             │
└────────────────────────────────────────────────────┘
```

- One row per active connection (HTTP keep-alive *or* WebSocket once #0100 lands; for v1 mostly WS).
- Token column shows the **token name** (#0084), not any of the secret material. Resolved via the token's hash on connect.
- Address column shows the IPv4/IPv6 (or reverse-DNS if cheap to fetch — best-effort, no blocking).
- Subscribed column lists folder display names from the WS subscribe message (#0100). Truncates with a tooltip on hover when the list is long.

## Behavior

- The list updates in near-real-time as connections open and close. No manual refresh.
- HTTP-only requests (no WS attached) appear briefly and disappear when the response writes. To avoid flicker we include only connections that have either upgraded to WS or have an outstanding HTTP request older than 200 ms.
- Right-click on a row → "Revoke this token" (jumps to the access-token list with that token highlighted) and "Disconnect" (closes just this connection). Disconnect is a soft action — the viewer will reconnect immediately if its token is still valid; revoke is the permanent option.

## Verification

- Connect from another Mac via the smoke-test CLI (#0087) — a row appears.
- Stop the CLI — row disappears within a second.
- Open the Mac viewer (once #0099 lands) — row appears, Subscribed column shows the folder names.
- Right-click → Revoke → confirm in the token list. The viewer's WS closes within a second; row goes away.

## Notes

- Don't surface internal connection ids, request paths, or anything else that's not directly useful. The columns above are enough.
- Avoid auto-scrolling on update — if there are more viewers than fit, let the user scroll the list manually.

## Scope (out)

- A full audit log of past connections — out of v1; the per-token `lastUsedAt` / `lastUsedFrom` fields (#0078) cover the common need.
- "Kick + ban" (revoke + block this IP) — overkill for single-user.
- Per-folder fan-in counts ("3 viewers watching folder X") — interesting but not load-bearing.

## Root cause

#0079 stubbed `connectedPeers: [PeerInfo]` on `RemoteServer` and added an `accept(connection:)` insert, but `dropConnection(key:)` had a TODO no-op so the array only ever grew. The settings sheet (#0083) had a slot reserved for this list but nothing to bind to.

## Fix

Three small pieces:

- **Peer tracking.** `RemoteServer` now uses `peerInfo: [ObjectIdentifier: PeerInfo]` keyed by `ObjectIdentifier(connection)` and mirrors the values into the existing `connectedPeers: [PeerInfo]` array on every mutation. `accept` inserts; `dropConnection` removes; `stop` wipes both maps. This makes connect/disconnect/reconnect transitions clean and finally retires the placeholder.
- **Token-name threading.** `handleRequest` updates the peer entry with `record.name` after a successful auth, so the Token column shows the user-chosen name from #0084 (e.g. "MacBook Air") rather than the bearer hash or remote address.
- **Observability.** `RemoteServer` is now `@Observable` so SwiftUI re-renders when `connectedPeers` changes. `RemoteHostController.connectedPeers` is a passthrough getter.

**`ConnectedViewersListView`** is the new section content in `RemoteHostSettingsView`. Three columns:

- *Token* — `peer.tokenName ?? "—"` (em-dash before auth completes for the brief window).
- *Address* — `peer.remoteAddress`, monospace, truncated middle.
- *Subscribed* — placeholder `"—"` until `#0100` introduces WebSocket subscriptions. The spec accepts that trade-off rather than flickering on every short HTTP request.

The section is hidden when hosting is off so the sheet doesn't reserve empty real estate.

## Gotchas

- **`ObjectIdentifier(connection)` is the only stable key** for an `NWConnection` — the endpoint and remote address aren't unique enough (two viewers behind the same NAT). Don't try to be cleverer; the ObjectIdentifier survives the connection's lifetime exactly.
- **Token name is set after `accept`, not at upgrade time.** The brief window between accept and a successful auth shows `"—"` in the Token column. That's correct — the peer hasn't proved themselves yet.
- **SwiftUI re-renders depend on `@Observable @MainActor` propagation.** `RemoteHostController` is `@Observable`; reading `controller.connectedPeers` from a view registers a dependency on `controller`. But the underlying mutation is on `server.connectedPeers` (also `@Observable`). With both types `@Observable` and the chain on the main actor, SwiftUI's tracking does the right thing.
- **HTTP-only requests rarely show in the list** because they complete in single-digit milliseconds. With WS (#0100) most viewers will hold a long-lived connection and stay in the list for the session. That's the design.
- **Right-click context menu (Revoke / Disconnect) is not in scope here** — the spec called for it but it requires a per-connection `disconnect` API and tighter coupling to the token list. Follow-up work in #0084's UI evolution or a fresh issue.
