# 0064 — Generate a status-report markdown into `<folder>/reports/`

| | |
|---|---|
| **Status** | open |

| **Platform** | macOS |
| **First seen** | 2026-05-05 |

## Description

The app already has a comprehensive picture of the watched folder — parsed issues, status counts, lint findings, the unseen-changes diff machinery, and (when the folder is in a git repo) the surrounding commit history. Wrap that into a one-shot **Generate Report…** action that writes a self-contained markdown snapshot into a `reports/` subfolder of the active tab's watched directory.

The output file is a plain `report-<date>.md`. Because it lives inside the watched issues folder, the user's normal git workflow picks it up — if `reports/` isn't `.gitignore`d, the report becomes part of the project's history alongside the issue files themselves.

## Output location and filename

- Path: `<watched-folder>/reports/report-<localized-date>.md`. If `reports/` doesn't exist yet, create it.
- Filename example (US locale): `report-12-31-2025.md`. Spanish locale: `report-31-12-2025.md` (DD-MM-YYYY). Use `DateFormatter` with the user's current `Locale` and a numeric `.dateStyle = .short`-shaped pattern, then replace separator characters with `-` so the result is filesystem-safe.
- The "report" prefix stays English for v1 to keep filenames sortable across locales. Once #0059 lands, the prefix can be localized too — defer that decision until then.

## Report structure

Each generated `report-*.md` contains, in order:

```markdown
# <Repo name> — Status report

_Generated: <localized full date and time>_
_Folder: <absolute path>_

## Summary

- **Total issues:** N
- **Open:** N · **In progress:** N · **Resolved:** N · **Closed:** N · **Won't fix:** N
- **Lint findings:** N

## Open issues

| # | Title | Module | Platform | Filed |
|---|-------|--------|----------|-------|
| 0042 | … | Views | macOS | 2026-05-01 |
| … |
(rows for every issue currently in `.open` or `.inProgress`, sorted by id)

## Recent activity

Last 14 days, sorted newest first:

- **2026-05-05** — #0042 status changed: open → in-progress
- **2026-05-04** — #0040 closed
- **2026-05-04** — #0046 filed
- …

(Source: TabsModel snapshot diffs accumulated since the last report, OR if no
prior report, derived from current issue mtimes + status. See "Activity
source" below.)

## Recent commits

(only included when the watched folder lives inside a git repo)

- `b242ab9` — #0030 move Theme palette into the Asset catalog (2026-05-04)
- `96fe7d1` — #0029 promote folder picker to its own Window scene (2026-05-04)
- …

## Lint findings

(only included when `LintRunner.run(...)` returned at least one)

- `0007.md` references missing attachment screenshot.png
- `0012.md` has unknown status "in_progress"
- …
```

## Activity source

Two reasonable strategies:

1. **Snapshot diff against the previous report.** If `reports/` already contains an older `report-*.md`, parse its summary table out of the markdown to reconstruct the state at that time, then diff against the current state to get the activity list. Self-contained, doesn't need external state.
2. **Use mtimes + statuses directly.** Pull all issues whose `modifiedAt` is within the last 14 days, classify "filed" / "closed" / "status-changed" by inspecting whether `closedRaw` is set and whether the file's mtime differs from `firstSeen`.

Strategy 2 is simpler and is the v1 default. Strategy 1 can come later if reports are being run regularly enough that the user wants "since last report" precision.

## Trigger

Wire two entry points:

1. **File menu item** — `File → Generate Report… (⌘⇧R)`. Disabled when no tab is active. Shortcut configurable via `ShortcutsStore` once it's added to `ShortcutAction`.
2. **Toolbar button** in the stats bar (small icon, tooltip "Generate report"), so a mouse user has direct access without diving into the menu.

After generation, post a `UNUserNotification` (or a brief in-app toast — TBD; notification path is already wired) saying "Report saved to <path>". Clicking the notification reveals the file in Finder.

## Implementation sketch

A new `Issues/Services/ReportGenerator.swift`:

```swift
@MainActor
enum ReportGenerator {
    static func generate(for store: IssueStore) throws -> URL {
        let folder = store.folderURL
        let reportsDir = folder.appending(path: "reports")
        try FileManager.default.createDirectory(
            at: reportsDir,
            withIntermediateDirectories: true
        )
        let filename = "report-\(filenameDate()).md"
        let outURL = reportsDir.appending(path: filename)
        let body = buildBody(store: store, folder: folder)
        try body.write(to: outURL, atomically: true, encoding: .utf8)
        return outURL
    }

    private static func filenameDate() -> String {
        let formatter = DateFormatter()
        formatter.dateStyle = .short
        formatter.locale = .current
        return formatter
            .string(from: .now)
            .replacingOccurrences(of: "/", with: "-")
            .replacingOccurrences(of: ".", with: "-")
    }

    private static func buildBody(store: IssueStore, folder: URL) -> String { … }
}
```

For the git history block, run `git log --pretty=format:'%h %s (%ad)' --date=short -- <folder>` via `Process` and capture stdout. Bail silently if `git` returns non-zero (folder isn't a repo) — the section just doesn't render in that case. Don't add a hard dependency on git.

## Write-scope constraint

**The app must remain strictly read-only for the existing `NNNN.md` issue files.** Claude Code subagents read those files into context and write back changes — if Issues.app concurrently modifies an issue file, the subagent's later write would clobber edits made in the app, and vice versa. The conflict mode is silent (no merge, no warning), so the only safe rule is: Issues.app never touches the markdown sources.

Reports are a different story. They live in a fresh `reports/` subfolder that:
- Doesn't exist before the first report is generated.
- Is never read by the parser (filename gate is `^\d{4}\.md$`, which rejects `report-*.md`).
- Is never written by Claude subagents — they don't generate reports, the app does.

So write access can be scoped to *just* the `reports/` directory. Two implementation options:

1. **Bump the file-scope entitlement** from `com.apple.security.files.user-selected.read-only` to `read-write`, but in code only ever open files for writing under `<folder>/reports/`. The entitlement gives the app the *capability*; we restrict ourselves to the subdirectory by convention. Audit hook: a single `assert(url.path.contains("/reports/"))` check inside the writer guards against future regressions.
2. **Keep the read-only entitlement** and request a separate user-granted write bookmark for the reports folder when the user first triggers a report. macOS shows a save panel they accept once; the resulting bookmark persists alongside the existing folder bookmark. More plumbing but stronger sandbox boundary.

v1 default: option 1, with the assertion guard. The convention "we only ever write to `reports/`" is short, explicit, and easy to enforce in code review. If the security review later wants the tighter sandbox, swap to option 2 without changing the report shape.

## Edge cases

- Two reports generated on the same day: the filename collides. Either append a counter (`report-2026-05-05-2.md`) or include the time in the filename. v1: append a small suffix on collision.
- A repo with hundreds of issues: the markdown gets long but stays well under any practical limit. No pagination needed.
- The user's locale changes between reports: filenames will format differently. That's fine — the user's intentional choice.
- The user has `reports/` `.gitignore`d: report writes still succeed, they just don't end up in version control. That's the user's call.

## Verification

- Trigger from File menu → a `reports/report-<date>.md` appears in the watched folder. The file opens cleanly in any markdown viewer (and inside Issues.app's own preview if the user pastes the path in).
- Run `git status` in the watched repo → the new file is untracked (or staged-and-ready) per the user's git config.
- Generate again on the same day → filename suffix increments rather than overwriting.
- A folder outside a git repo → report still generates, the "Recent commits" section is omitted gracefully.
- A folder with zero issues → report renders with all zeros and a "No issues yet." note in place of the open-issues table.

## Out of scope

- Inline charts / images. Covered by #0065 (follow-up).
- HTML or PDF export of the report. Markdown only for now — the user's existing tooling can convert if needed.
- Scheduled / automatic report generation (e.g. weekly).
- Auto-commit. The user controls git, not the app.
