# 0039 — `UserInfoKey` enum inherits `@MainActor`, triggers warnings from nonisolated delegate

| | |
|---|---|

| **First seen** | 2026-05-04 |

## Description

`Issues/Services/NotificationService.swift` declares `UserInfoKey` as a private enum nested inside the `@MainActor`-isolated `NotificationService` class. Nested types inherit the enclosing class's actor isolation, so `UserInfoKey.tabID` / `.issueID` / `.kind` are themselves `MainActor`-isolated static properties.

The `userNotificationCenter(_:didReceive:withCompletionHandler:)` delegate method is intentionally `nonisolated` (it must satisfy the protocol's `nonisolated` requirement and run from whatever queue UN delivers from). Accessing the keys from there produces three Swift 6 strict-concurrency warnings:

```
NotificationService.swift:272:48  Main actor-isolated static property 'tabID' can not be referenced from a nonisolated context
NotificationService.swift:273:44  Main actor-isolated static property 'issueID' can not be referenced from a nonisolated context
NotificationService.swift:274:41  Main actor-isolated static property 'kind' can not be referenced from a nonisolated context
```

The keys are immutable `String` constants — they're trivially safe to access from any context. The MainActor isolation on them is spurious and forced by the nesting.

## Fix

Mark the enum `nonisolated` so its static properties shed the inherited isolation:

```swift
nonisolated private enum UserInfoKey {
    static let tabID = "tabID"
    static let issueID = "issueID"
    static let kind = "kind"
}
```

That's the entire change. No callers need updating — the keys are still accessed by the same names from both the nonisolated delegate method and the `@MainActor`-isolated `postIndividual` / `postRollup` posting helpers.

## Why `nonisolated` over `Sendable`

`Sendable` governs whether *values* of a type can cross actor boundaries. The warnings here are about *static-property access* from a nonisolated context — that's an isolation problem, not a value-transfer problem. `Sendable` wouldn't address the warning. `nonisolated` is the right tool: the namespace of string constants genuinely has no actor affinity.

Moving `UserInfoKey` to file scope would also work, but it's a larger churn (two callers update; loses the nice `NotificationService.UserInfoKey` namespacing) for no extra benefit. Keep it nested, just opt out of the inherited isolation.

## Verification

After the fix, `xcodebuild -project Issues.xcodeproj -scheme Issues -configuration Debug -destination 'platform=macOS' build` should complete without the three warnings. No runtime behavior change is expected.
