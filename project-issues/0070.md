## Description

When the user clicks a notification posted by `NotificationService` (#0013), the app brings the relevant tab to the front and selects the issue — but only if the current view mode and filters happen to leave the issue visible. If they don't, the click silently fails to reveal anything useful. Examples:

- View mode is `Recent`, status filter is `Open`. A notification fires for a status change to `Resolved`. The Recent view + Open-only filter excludes Resolved issues, so the row never appears.
- A module / platform filter excludes the issue's module.
- Search is active and the issue's title doesn't match the query.

Silently changing the user's selections to force the issue into view is jarring — the user lost their carefully-set filters and won't understand why. Instead, present an explicit confirmation prompt that summarizes the issue and offers an action to jump.

## Behavior

When a notification is tapped:

1. Resolve the issue from the payload (already wired up — payload carries `tabID` + `issueID`).
2. Activate the matching tab.
3. Check whether the current view mode + filters + search would render the issue. If yes, just select it (today's behavior).
4. If not, present a SwiftUI `confirmationDialog` modally:

   ```
   Reveal #0042 — <issue title>?

   <metadata grid: Status, Module, Platform, First seen, Closed?>

   <first paragraph of Description, ~3 lines max>

   [ Cancel ]   [ Reveal Issue ]
   ```

5. Tapping **Reveal Issue**:
   - Switches the tab's view mode to **List** (deterministic — every issue is reachable in List).
   - Clears any status / module / platform filter and search query that would hide the issue. Only the minimum needed; don't reset filters that aren't blocking.
   - Sets `selectedIssueID` to the target.
   - Optionally opens the markdown sheet (matches today's notification-arrival path).
6. Tapping **Cancel**: dismiss the dialog, leave selections untouched.

## Why a confirmation dialog

`confirmationDialog` is the right SwiftUI primitive — it's lightweight, pinned to the active scene, and the user clearly opts in to the context switch. A full sheet or custom modal would be overkill for a routine "go to this issue" prompt.

## Implementation sketch

- `NotificationService.handleTap(payload:)` already routes to a `MainView` callback. Extend the callback so it knows about the active store's view mode + filters.
- A new helper on `IssueStore`: `issueIsCurrentlyVisible(id:) -> Bool` that re-runs the existing filter pipeline against the issue's metadata, returning false if any of `searchQuery`, `statusFilter`, `moduleFilter`, `platformFilter`, or the active view mode hides the row. Recent / Swimlane / Timeline / List each have their own visibility rules — encode them in one place so the answer is consistent.
- A new `@State` on `MainView` like `pendingNotificationReveal: Issue?`. Setting it triggers the confirmation dialog.
- The dialog body uses the existing `DetailPanelMetadataView` snippet plus the issue's first description paragraph (already extracted today for the swimlane card preview).
- The "Reveal Issue" action calls a new `IssueStore.revealIssue(id:)` that flips view mode to `.list`, clears the offending filters, sets `selectedIssueID`, and optionally posts an `openMarkdownSheet` event.

## Verification

- Notification fires for a Resolved status change while filter is set to `Open`. Click the notification → confirmation dialog appears with the issue's metadata and first paragraph.
- Click **Reveal Issue** → view mode switches to List, status filter clears, the issue is selected.
- Click **Cancel** → no view changes; notification is dismissed.
- Notification fires for an issue that IS currently visible → no dialog; selection just changes (today's behavior preserved).
- Search query that excludes the issue → reveal clears the search query as well.

## Notes

- Don't auto-clear filters the user set when the issue IS visible — only when reveal would otherwise fail. The minimum-clearing principle keeps the surprise low.
- List view is the right default because it has no spatial filtering of its own. Picking Swimlane or Timeline could introduce its own visibility nuances.
- This is the user-experience layer on top of the notification infrastructure from #0013; the payload format and wiring don't need to change. The change is in `MainView` / `IssueStore` / a new helper for visibility checks.
- Related: #0055 (command palette) also needs a "jump to issue" primitive — they could share the same `revealIssue(id:)` plumbing.

## Root cause

`AppCommandsController.consumePendingDeepLinkIfPossible` set `store.selectedIssueID = issueID` unconditionally. When the user's filters / search / view mode hid the row, the row never appeared — the notification tap looked broken even though selection was technically set.

## Fix

New entry point on the store: `requestReveal(id:)` decides between today's silent-select (when the issue is in `filteredIssues`) and a queued reveal (when it isn't). The queue is `pendingReveal: Issue?` — a SwiftUI `confirmationDialog` in `MainView` watches it and presents the prompt.

- `issueIsCurrentlyVisible(id:)` reduces to "is the issue in `filteredIssues`?" since all four view modes draw from the same filtered set. No per-view-mode visibility rules needed.
- `revealIssue(_:)` flips `viewMode` to `.list` and clears only the filters that block the target. Non-blocking filters survive — if the user had a platform filter that already matches the target's platform, it stays. Search is cleared only when the trimmed query doesn't already match the issue's id/title/description.
- `cancelReveal()` drops `pendingReveal` without touching anything else.

`MainView`'s `confirmationDialog` reads `store.pendingReveal` via a tiny `Binding` wrapper, renders a title like `Reveal #0042 — <title>?`, and a message with the metadata line plus the first description paragraph (300-char cap). Reveal calls `revealIssue`; Cancel calls `cancelReveal`. `AppCommandsController.consumePendingDeepLinkIfPossible` now calls `requestReveal` so cold-launch and warm-app paths share the same logic.

## Files changed

- `Issues/State/IssueStore.swift` — `pendingReveal`, `issueIsCurrentlyVisible(id:)`, `requestReveal(id:)`, `revealIssue(_:)`, `cancelReveal()`.
- `Issues/State/AppCommandsController.swift` — `consumePendingDeepLinkIfPossible` routes through `requestReveal`.
- `Issues/Views/MainView.swift` — `confirmationDialog`, helper bindings + the message string.
- `IssuesTests/IssueStoreTests.swift` — 6 new tests: visible → select, hidden-by-status → queue, hidden-by-search → queue, `revealIssue` clears blocking + flips to list, `revealIssue` keeps non-blocking filters, `cancelReveal` leaves everything untouched.

## Gotchas

- **`confirmationDialog(_:isPresented:titleVisibility:presenting:)` requires the binding side to be in sync with the `presenting` value.** A `Binding` wrapper that maps `pendingReveal != nil` plus a setter that calls `cancelReveal` on `false` keeps the dialog reactive without leaking dismissal-without-action into stale state.
- **"All" platform short-circuit applies during reveal too.** A platform filter of `"macOS"` doesn't block an issue with `platform == "All"`, so the clearing condition matches `filteredIssues`'s logic.
- **Don't try to handle the empty-description case with a placeholder string.** The first-paragraph snippet is optional; the metadata line + the "Revealing switches..." caption already give the user enough to decide.
- **Visual verification still needed.** confirmationDialog layout on macOS varies slightly from the spec mock (it's the system's lightweight modal, not a custom sheet), but it's the right SwiftUI primitive per the spec.
