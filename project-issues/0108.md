## Description

Phase 6 of the remote-access plan (#0076). Tracking issue for the iPhone-side work — gated entirely behind the Mac-side phases (#0077–#0107) being stable in real use.

The Mac is the host (always); the iPhone is viewer-only. Most of the protocol/model/remote code is already cross-platform by design, so this phase is more about adding an iOS target, an iOS-shaped UI shell, and the QR-code transfer that makes pasting tokens onto a phone tolerable.

This is intentionally a **tracking issue** — the actual work splits into multiple smaller issues filed when we start. Don't try to plan it down to ticket-level detail today; the assumptions will shift between now and then.

## Prerequisites

Before any work starts on this:

- #0077–#0107 are filed. (They are.)
- Phases 1–5 of the Mac-side work have shipped and been used for at least a couple of weeks.
- The cross-platform files (`AccessToken` viewer-side, `RemoteClient`, `RemoteHostIssueSource`, `RemoteServiceBrowser`, `RemoteProtocol`, `ViewerTokenStore`, `RemoteFolderPickerView`) actually compile against an iOS SDK without surprises. The original design (`RemoteAccess.md`) anticipates this; reality will tell.

## Top-level work breakdown (will be re-broken when started)

1. **Add the iOS target** — new target in `Issues.xcodeproj`, deployment target chosen at the time, asset catalog adapted, app icon set, `Info.plist` with `NSBonjourServices` and `NSLocalNetworkUsageDescription`. Entitlements per #0093 (`network.client`, `multicast`, `keychain-access-groups`).
2. **Compile the cross-platform files** — gate macOS-only code (`LocalFolderIssueSource`, `RemoteServer`, FSEvents glue) behind `#if os(macOS)`. The shared files should already be platform-neutral; this step is verifying that.
3. **iOS app shell** — sidebar (iPad) or top-level segmented control / list (iPhone). Decide whether the iPhone follows one folder at a time or mirrors the Mac's "all open simultaneously". Probably one-at-a-time with a fast switcher, but defer the call until we have the device in hand. Doesn't change the protocol.
4. **iOS folder picker** — reuse `RemoteFolderPickerView` (#0097). Touch-sized affordances; minor layout adjustments.
5. **QR-code token transfer**:
   - **Host (Mac)**: render the token (or `iat_…@host:port` per `RemoteAccess.md` Open Q11) as a QR code in the generate sheet (#0084). `CIFilter.qrCodeGenerator()` covers it.
   - **iPhone**: scan via `AVCaptureSession` + `Vision`. No third-party libraries. Decode → call into `ViewerTokenStore.store(token:forHost:)` and the picker.
6. **iOS connect flow** — same Bonjour browse + manual `host:port` paths as the Mac. The local-network permission prompt fires the first time `NWBrowser.start()` runs.
7. **Adapt notification surface (if any)** — Issues.app on Mac has notifications (#0070); iOS likely doesn't get those in v1.
8. **Distribution** — TestFlight or ad-hoc; not figured out today.

## Verification

When this phase is done, on the iPhone:

- Browse to the test Mac in the discovery list (works on the same Wi-Fi LAN; works over Tailscale per #0090).
- Scan a QR code from the host's generate sheet → token stored, picker shows folders.
- Pick "all" / "subset" / "one" — same patterns as Mac. Tabs / sections / whatever the iPhone shell ends up being open accordingly.
- Live updates flow within a second of an edit on the Mac.
- Quit the app, relaunch — subscriptions reopen via #0103's logic.

## Notes

- The Mac-side polish phases (#0103–#0106) all assume the iPhone will hit the same code paths. Specifically: WS reconnect logic, async attachment fetch, network-change pause (probably equivalent on iOS via `NWPathMonitor`).
- Don't try to add iOS-only UI tricks before the basic flow lands. A boring port is the right v1.
- Expect 2–4 weeks of work when this starts, depending on how cleanly the cross-platform abstractions hold up.
