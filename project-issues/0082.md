# 0082 — Stable folderId derived from security-scoped bookmark

| | |
|---|---|
| **Status** | open |
| **Module** | Services |

## Description

Phase 1 of the remote-access plan (#0076). Folders need a wire-stable identifier that survives moves, renames, and process restarts as long as the underlying folder is the same. The proposal in `RemoteAccess.md` is `SHA-256(securityScopedBookmark)[..16hex]` — a 16-character hex string the host and viewer both use to refer to "the same folder" across requests.

## Where to look

- **`Issues/Services/FolderBookmarkService.swift`** — already produces and resolves bookmarks. Add a `folderId(for: Data) -> String` derived from the bookmark bytes.
- **`Issues/State/IssueStore.swift`** — exposes the active folder's `folderId` so the server (#0079, #0080) and the picker (#0097) can use the same value.
- **`Issues/Models/`** — no new model needed; `folderId` is just a `String`.

## Behavior

- `folderId = SHA256(bookmarkBytes)` truncated to the first **16 hex characters** (8 bytes of entropy). 16 characters is enough for tens of thousands of folders without realistic collision risk and short enough to read in logs and URLs.
- **Stable across launches** as long as the bookmark resolves. macOS may rewrite the bookmark over time (the resolved URL or staleness changes), but the bookmark *bytes* persisted in `UserDefaults` are what we hash. We keep hashing the persisted bytes; if the user re-locates a missing folder via "Locate folder…", a new bookmark is created and the `folderId` legitimately changes.
- **Display name** is unrelated. `project.json`'s `name` (#0075) is the user-visible label; the `folderId` is the wire identifier.
- **Collision handling**: theoretical only. If two folders ever produce the same 16-hex prefix (~1 in 2^64), extend to 20 hex on collision detection. We won't hit this; design a check, don't write retry logic.

## Wire usage

The server uses `folderId` in every folder-scoped path:

- `/v1/folders/{folderId}`
- `/v1/folders/{folderId}/issues`
- `/v1/folders/{folderId}/issues/{id}`
- `/v1/folders/{folderId}/issues/{id}/attachments/{name}`

The WebSocket events (#0100) carry it too:

```json
{ "type": "reload", "folderId": "a3f1e0c082b41d77" }
```

The picker (#0097) persists `(hostId, folderId)` per subscription so reconnect doesn't re-prompt.

## Behavior on bookmark invalidation

- If a bookmark resolves stale and the app re-creates it, the `folderId` changes (new bookmark bytes → new hash).
- The host serves the new id from `/v1/folders` on the next request; the old id disappears.
- The viewer sees the old id 404, falls back to **find-by-name** using `project.json`'s `name`. If a single folder with that name is in the new list, auto-rebind silently; if zero or more than one, surface "this folder moved — pick again" in the manage-subscriptions sheet (#0098).

## Verification

- Add a folder. `folderId` is 16 hex chars, lowercase.
- Restart the app. Same folder still produces the same `folderId`.
- Delete the bookmark and re-add the same folder — `folderId` may differ; that's the expected "user re-located" behavior.
- Hash collision: not testable in practice; covered by code review.

## Notes

- Don't hash the resolved URL string — it changes when the user moves the folder, which defeats the point.
- Don't hash the inode — sandbox can't reliably read it across volumes.
- Don't use the folder's `NSFileSystemFileNumber`. Same problem.

## Scope (out)

- Find-by-name fallback UX is split between #0097 (picker) and #0098 (manage subscriptions).
- The actual bookmark rotation logic is unchanged in this ticket; we're only deriving an id from existing bytes.
