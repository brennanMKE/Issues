## Description

Phase 1 of the remote-access plan (#0076). Folders need a wire-stable identifier that survives moves, renames, and process restarts as long as the underlying folder is the same. The proposal in `RemoteAccess.md` is `SHA-256(securityScopedBookmark)[..16hex]` тАФ a 16-character hex string the host and viewer both use to refer to "the same folder" across requests.

## Where to look

- **`Issues/Services/FolderBookmarkService.swift`** тАФ already produces and resolves bookmarks. Add a `folderId(for: Data) -> String` derived from the bookmark bytes.
- **`Issues/State/IssueStore.swift`** тАФ exposes the active folder's `folderId` so the server (#0079, #0080) and the picker (#0097) can use the same value.
- **`Issues/Models/`** тАФ no new model needed; `folderId` is just a `String`.

## Behavior

- `folderId = SHA256(bookmarkBytes)` truncated to the first **16 hex characters** (8 bytes of entropy). 16 characters is enough for tens of thousands of folders without realistic collision risk and short enough to read in logs and URLs.
- **Stable across launches** as long as the bookmark resolves. macOS may rewrite the bookmark over time (the resolved URL or staleness changes), but the bookmark *bytes* persisted in `UserDefaults` are what we hash. We keep hashing the persisted bytes; if the user re-locates a missing folder via "Locate folderтАж", a new bookmark is created and the `folderId` legitimately changes.
- **Display name** is unrelated. `project.json`'s `name` (#0075) is the user-visible label; the `folderId` is the wire identifier.
- **Collision handling**: theoretical only. If two folders ever produce the same 16-hex prefix (~1 in 2^64), extend to 20 hex on collision detection. We won't hit this; design a check, don't write retry logic.

## Wire usage

The server uses `folderId` in every folder-scoped path:

- `/v1/folders/{folderId}`
- `/v1/folders/{folderId}/issues`
- `/v1/folders/{folderId}/issues/{id}`
- `/v1/folders/{folderId}/issues/{id}/attachments/{name}`

The WebSocket events (#0100) carry it too:

```json
{ "type": "reload", "folderId": "a3f1e0c082b41d77" }
```

The picker (#0097) persists `(hostId, folderId)` per subscription so reconnect doesn't re-prompt.

## Behavior on bookmark invalidation

- If a bookmark resolves stale and the app re-creates it, the `folderId` changes (new bookmark bytes тЖТ new hash).
- The host serves the new id from `/v1/folders` on the next request; the old id disappears.
- The viewer sees the old id 404, falls back to **find-by-name** using `project.json`'s `name`. If a single folder with that name is in the new list, auto-rebind silently; if zero or more than one, surface "this folder moved тАФ pick again" in the manage-subscriptions sheet (#0098).

## Verification

- Add a folder. `folderId` is 16 hex chars, lowercase.
- Restart the app. Same folder still produces the same `folderId`.
- Delete the bookmark and re-add the same folder тАФ `folderId` may differ; that's the expected "user re-located" behavior.
- Hash collision: not testable in practice; covered by code review.

## Notes

- Don't hash the resolved URL string тАФ it changes when the user moves the folder, which defeats the point.
- Don't hash the inode тАФ sandbox can't reliably read it across volumes.
- Don't use the folder's `NSFileSystemFileNumber`. Same problem.

## Scope (out)

- Find-by-name fallback UX is split between #0097 (picker) and #0098 (manage subscriptions).
- The actual bookmark rotation logic is unchanged in this ticket; we're only deriving an id from existing bytes.

## Root cause

Greenfield. The store didn't have a wire identifier of any kind; tab persistence keyed off the standardized URL path, which is unsuitable for the wire (changes if the user moves the folder, leaks the user's filesystem layout, has no fixed length).

## Fix

`FolderBookmarkService.folderId(for:)` is the spec'd hash: SHA-256 of the bookmark bytes, first 8 bytes hex-encoded тЖТ 16 lowercase characters. Marked `nonisolated` because the project compiles with `-default-isolation=MainActor` and the hash is pure CryptoKit work тАФ without `nonisolated` the call site warns about crossing isolation domains.

`IssueSource` grew a `bookmarkData: Data?` requirement and `LocalFolderIssueSource` captures the bytes at construction time. The init signature gained an optional `bookmarkData:` parameter (default nil) so existing call sites and previews keep compiling. `IssueStore` exposes both `bookmarkData` and `folderId` as one-line passthroughs.

`TabsModel` is the producer of the bytes. `openTab(url:)` now mints a bookmark once and feeds it into the store; `restore()` passes the resolved blob through; `persist()` prefers `store.bookmarkData` over re-minting. The behavior change in `persist()` is small but load-bearing тАФ without it, every save would write a fresh bookmark blob whose bytes can differ from what was used to derive the id, which would let `folderId` drift across launches even though the underlying folder is the same.

`folderId` is a `String?` (nil when no bookmark) so previews, tests, and any future remote source that publishes its own id stay clean.

## Gotchas

- **`-default-isolation=MainActor` and pure helpers.** Static functions that don't touch shared state still inherit MainActor under that flag and warn at non-main call sites. `nonisolated` is the right marker тАФ it documents that the function has no isolation-relevant work and quiets the warning. The same pattern will apply to any other CryptoKit-style helper added under `Services/`.
- **`url.bookmarkData(...)` is not byte-stable.** Two calls on the same URL can produce different blobs (timestamps, resolution metadata embedded by the system). That's why the store captures bytes once and `persist()` uses them rather than re-minting; otherwise `folderId` would change every time the user toggles a filter.
- **`folderId` is `String?` rather than `String`.** Tests, previews, and future remote sources don't have local bookmarks; making it optional keeps them honest. Wire-side code should `guard let folderId = store.folderId else { тАж }` and treat the missing case as "this tab is not host-routable yet."
