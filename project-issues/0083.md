## Description

Phase 1 of the remote-access plan (#0076). The user needs a clear on/off control for the remote server, plus a visible indicator showing where viewers can reach it. This is the simplest piece of host UI; the access-token list (#0084) and per-folder toggles (#0085) get their own sheets.

## Where to look

- **New: `Issues/Views/RemoteHostSettingsView.swift`** — top-level container for host-side settings. This ticket adds only the enable toggle + the IP/port section; #0084 adds the token list, #0085 adds folders, #0092 adds connected viewers.
- **`Issues/Views/ToolbarView.swift`** (or wherever the toolbar is wired) — add an entry that opens the settings sheet. A gear icon next to the existing folder picker is fine.
- **`Issues/Remote/RemoteServer.swift`** (from #0079) — already exposes `listeningPort` and a `start()` / `stop()`. The view binds to `@Observable` server state.

## UI

```
┌─ Remote Access ─────────────────────────────┐
│                                             │
│  [✓] Enable hosting                         │
│                                             │
│  Reachable on this Mac at:                  │
│    192.168.1.42 : 51823    [Copy]           │
│    100.74.12.5  : 51823    [Copy]   ← Tailscale
│    fe80::abcd…  : 51823    [Copy]           │
│                                             │
│  Display name (shown to viewers):           │
│    [ Brennan's MacBook Air            ]     │
│                                             │
└─────────────────────────────────────────────┘
```

- **Enable hosting** toggle — when on, calls `RemoteServer.start()`. When off, `stop()` and tears down Bonjour publishing too (per #0088).
- **Reachable on this Mac at** — list every non-loopback IPv4/IPv6 address `Network.framework` reports for active interfaces, paired with the current listening port. Each row has a Copy button that puts `host:port` on the clipboard for paste-into-viewer.
- **Display name** — text field bound to a value persisted in `UserDefaults` under `RemoteServer.displayName`, defaulting to the system's `Host.current().localizedName`. This is what the host returns from `/v1/host` and what shows in the viewer's discovery list (#0089).

## Behavior

- The toggle's state persists in `UserDefaults` under `RemoteServer.enabled`. On launch, if `enabled = true`, the server starts automatically.
- Quitting the app stops the server (existing process-exit flow from #0074).
- IP list refreshes when the network changes — wire a small `NWPathMonitor` to refresh the list without restarting the server. (The server itself doesn't care — it's bound to all interfaces — but the *displayed* list should update.)
- If `RemoteServer.start()` throws (port in use, entitlement missing, etc.), surface a small inline error under the toggle and revert it to off. Don't crash, don't show a modal alert.

## Verification

- Toggle on. Server is reachable at every listed IP from another Mac on the same Wi-Fi network: `curl -H "Authorization: Bearer iat_<valid>" http://<ip>:<port>/v1/host` returns 200.
- Toggle off. `lsof -nP -iTCP -sTCP:LISTEN | grep Issues` shows nothing.
- Toggle on, change Wi-Fi networks (or connect Ethernet) — IP list updates within a couple of seconds.
- Edit the display name — `/v1/host` reports the new value on next request.
- Quit and relaunch with toggle on — server starts again automatically.

## Notes

- The IP list is informational. With Bonjour (#0088) and the manual `host:port` field (#0091), most users won't paste an IP by hand, but it's the most reliable fallback when discovery doesn't traverse a particular network.
- Avoid pre-resolving reverse DNS — slow, and the names are usually meaningless.
- Tailscale interfaces appear naturally because `Network.framework` reports them. We don't need to detect Tailscale specifically; just list everything.

## Scope (out)

- Access-token list → #0084.
- Per-folder hosting toggles → #0085.
- Connected-viewers list → #0092.
- Bonjour publishing — handled by #0088 but driven by this view's enable toggle (it starts/stops alongside the server).

## Root cause

The host server (#0079), the data layer (#0078 / #0085), and the entitlements (#0086) were all in place but nothing actually started the server — the user had no way to flip it on, the `HostFolderStore` had no tab data, and the smoke CLI (#0087) had nothing to hit.

## Fix

Three new files plus three small bridges:

- **`Issues/State/RemoteHostController.swift`** — `@MainActor @Observable` glue type. Owns one `RemoteServer` and one `HostFolderStore` for the app's lifetime. `setEnabled(_:)` is the only mutator the UI calls; it persists the new state under `RemoteServer.enabled`, calls `server.start()` / `stop()`, and on a thrown start error rolls back to disabled with the message surfaced via `lastStartError`. Auto-start on launch if the previous session left hosting on. `displayName` is a property with a `didSet` that persists under `RemoteServer.displayName` and writes through to `HostFolderStore.displayNameOverride` so `/v1/host` reflects the user's value. `NWPathMonitor` on a global queue triggers `refreshInterfaces()` whenever the network path changes.
- **`Issues/Services/NetworkInterfaceLister.swift`** — `getifaddrs(3)` wrapper that returns up, non-loopback IPv4 + IPv6 addresses with interface names. Tailscale interfaces show up naturally as `utun*` without special-casing. Sort: IPv4 before IPv6, then alphabetical by interface name so the order is stable across path changes.
- **`Issues/Views/RemoteHostSettingsView.swift`** — `Form { ... }`-based sheet. Sections: enable toggle (with inline error label under it), reachable-at IP list with per-row monospace `address : port` plus interface-name caption and a `Copy` button that drops `host:port` on `NSPasteboard.general`, display name `TextField` with the system computer name as the placeholder.

Wired through:

- **`HostFolderStore`** grew `displayNameOverride: String?`; `hostDisplayName` checks it first before the `Host.current().localizedName` fallback.
- **`AppCommandsController`** grew `hostController: RemoteHostController?` so the Settings scene can bind to the same instance the main window's `RootView` owns.
- **`RootView`** stands up the controller as `@State`, sets `AppCommandsController.shared.hostController` on appear, seeds `setStores(tabs.tabs)` once, and re-runs `setStores` on every `tabs.tabs.map(\.id)` change so `/v1/folders` is current.
- **`SettingsView`** gains a third tab "Remote Access" backed by a small `RemoteAccessSettingsTab` view that bridges through `AppCommandsController.shared.hostController` (the main window has to be open at least once for the controller to mount).

## Gotchas

- **Default-arg `HostFolderStore = HostFolderStore()` fails the build** because the default expression evaluates in a nonisolated context but `HostFolderStore` is `@MainActor`. Use `HostFolderStore? = nil` and unwrap inside the init, where `self.init`'s isolation applies. Same pattern will come up in #0094.
- **`Settings { ... }` lives in a separate window scene**, so `@State var hostController` on `RootView` isn't directly reachable from inside `Settings`. The `AppCommandsController` singleton is the bridge that exists for exactly this kind of cross-scene reference — don't duplicate the controller, share through the bridge.
- **`NetworkInterfaceLister` calls `getifaddrs(3)` synchronously** on the main actor when invoked from `refreshInterfaces()`. That's fine for a ~few-microsecond syscall, but if we ever need to broadcast over hundreds of interfaces, move the call to a background queue and hop the result back.
- **Visual verification still needed.** Form layout, hover states for the Copy button, and the inline-error look on a start failure are not unit-tested. Open Issues.app, Cmd+,, Remote Access tab, toggle on, and check.
