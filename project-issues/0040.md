# 0040 — List view row selection is unreliable; detail panel rarely appears on tap

| | |
|---|---|
| **Status** | in-progress |
| **Module** | Views |
| **Platform** | macOS |
| **First seen** | 2026-05-04 |

## Description

Single-clicking a row in List view rarely selects the issue or surfaces the detail side-panel. Recent view, which uses an explicit `onTapGesture { store.toggleSelection(issue.id) }`, behaves correctly — single click selects, second click deselects.

## Root cause

Every `TableColumn` cell in `Issues/Views/ListView.swift` chains `.onTapGesture(count: 2) { onOpenMarkdown(issue) }` (lines 26, 37, 51, 64, 75, 87) on top of the cell's `Text` plus `.contentShape(Rectangle())`. `Table` already binds row selection via `selection: $store.selectedIssueID`, but the cell-level `onTapGesture(count: 2)` competes with the Table's built-in single-click row-selection gesture:

- macOS must wait the system double-click interval to decide whether a tap is the first half of a double-tap or a standalone single-tap.
- During that window the cell's gesture recognizer is the active claimant. Sometimes the Table wins the race and selection sticks; usually the cell wins and the single-tap is consumed without ever propagating to the row.
- This is timing-dependent, which is why "sometimes a row can be selected but most of the time I cannot make that work" matches exactly.

## Fix

Replace `.onTapGesture(count: 2)` with `.simultaneousGesture(TapGesture(count: 2).onEnded { ... })` on every cell. `.simultaneousGesture` explicitly tells SwiftUI to fire alongside other recognizers rather than racing against them — single-tap row selection runs through Table's machinery as designed, and double-tap still opens the markdown sheet.

Pattern, applied to each of the 6 `TableColumn` cells:

```swift
// Before
Text("#\(issue.id)")
    .font(...)
    ...
    .frame(maxWidth: .infinity, alignment: .leading)
    .contentShape(Rectangle())
    .onTapGesture(count: 2) { onOpenMarkdown(issue) }
    .contextMenu { Button("Preview Markdown") { onOpenMarkdown(issue) } }
    .accessibilityAction(named: "Preview Markdown") { onOpenMarkdown(issue) }

// After
Text("#\(issue.id)")
    .font(...)
    ...
    .frame(maxWidth: .infinity, alignment: .leading)
    .contentShape(Rectangle())
    .simultaneousGesture(TapGesture(count: 2).onEnded { onOpenMarkdown(issue) })
    .contextMenu { Button("Preview Markdown") { onOpenMarkdown(issue) } }
    .accessibilityAction(named: "Preview Markdown") { onOpenMarkdown(issue) }
```

`.contentShape(Rectangle())` stays — it makes the hit-test extend across the full cell so right-click anywhere on the row produces the context menu rather than only on the text glyph.

## Out of scope

- IssueCardView, RecentView, TimelineView use `.onTapGesture(count: 2)` too, but those views are *not* inside a `Table` and don't have a competing selection recognizer — single tap there hits the explicit `.onTapGesture { store.toggleSelection(...) }` directly. They're behaving fine; leave them alone.
- This is unrelated to #0038's accessibility additions.

## Verification

After the fix, in the List view:
- Single-click on any row ⇒ row highlights, detail panel slides in on the right.
- Single-click the same selected row again ⇒ Table holds the selection; macOS Table convention is that single-click doesn't deselect (use Cmd+click). The detail panel stays open.
- Double-click any row ⇒ markdown sheet appears.
- Right-click any row ⇒ "Preview Markdown" context menu still works.
- Cmd+click a selected row to deselect; detail panel closes.

## Round 2 (2026-05-04)

The first attempt — swapping `.onTapGesture(count: 2)` for `.simultaneousGesture(TapGesture(count: 2)...)` — was insufficient. Single-click still rarely registered as row selection. The user reported that taps near the **top edge** of a row would occasionally select, but the body of a row almost never did.

Diagnosis: the per-cell `.contentShape(Rectangle())` extends the hit region across the whole cell frame and the cell-level recognizer (even with `.simultaneousGesture`) is still claiming the click event before NSTableView's row-selection machinery sees it. The thin gap between rows fell through to the Table itself, which is why the very top edge of a row sometimes worked.

### Round 2 fix

Strip **all** per-cell interaction modifiers from every `TableColumn` cell:
- Remove `.contentShape(Rectangle())`
- Remove the cell-level double-tap gesture
- Remove the per-cell `.contextMenu`
- Remove the per-cell `.accessibilityAction`

Move interaction to the `Table` itself:
- `.contextMenu(forSelectionType: String.self) { ids in ... }` — auto-selects the right-clicked row before showing the menu, surfaces "Preview Markdown" once instead of per cell.
- `.onTapGesture(count: 2) { ... }` on the Table — double-click opens the markdown sheet for `store.selectedIssueID`, which the single-click before it has already updated.

Cells become inert: just `Text` + `.foregroundStyle` + `.frame(maxWidth: .infinity, alignment: .leading)`. NSTableView's row-selection gesture is no longer competing with anything.

Enter-key open via `MainView.onKeyPress(.return)` continues to work unchanged.
