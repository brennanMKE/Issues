## Description

Search filters the issue list by matching `searchQuery` against `title` + `description` (case-insensitive substring), but the matched substring isn't visually highlighted in the rendered rows. The user trusts the filter but doesn't see *why* a row qualified — in a long title, the match might be near the end and feel like a false positive.

Render the matched substring with a soft accent-tinted background wherever the title or description text appears in the UI, so the eye can find the match instantly.

## Sites that need highlighting

- `IssueCardView` — the title `Text(issue.title)` (Swimlane view).
- `RecentRowView` — the title `Text(issue.title)` (Recent view).
- `ListView` — the Title column's `Text(issue.title)` cell.
- `DetailPanelHeaderView` — the issue title text.
- `IssueMarkdownSheetHeaderView` — `#NNNN — title` in the sheet's header.
- Optionally `DetailPanelDescriptionView` — but Textual renders the body; highlighting inside StructuredText is messier and may need to wait for a follow-up.

## Implementation sketch

Build an `AttributedString` helper that takes the body text and the active query and applies a `.backgroundColor` (mapped to a SwiftUI Color via `.foregroundStyle`-compatible attribute) on the matching range:

```swift
extension Text {
    init(_ text: String, highlighting query: String) {
        guard !query.isEmpty,
              let range = text.range(of: query, options: .caseInsensitive) else {
            self.init(text)
            return
        }
        var attr = AttributedString(text)
        if let attrRange = Range(range, in: attr) {
            attr[attrRange].backgroundColor = Color.appAccent.opacity(0.25)
        }
        self.init(attr)
    }
}
```

Each call site replaces `Text(issue.title)` with `Text(issue.title, highlighting: store.searchQuery)`. Trim the query first; empty/whitespace-only query falls through to the plain initializer.

## Edge cases

- Multiple matches: highlight all occurrences, not just the first. Loop with `range(of:options:range:)` to walk through.
- Diacritics / Unicode: use `.localizedStandardContains`-style options so the highlighted range matches what the filter accepts.
- Search box itself is empty: no highlight, no overhead.
- Long titles truncated by `.lineLimit(1)`: if the match falls past the truncation point, the user still sees nothing — that's an acceptable v1 limit; truncation strategy is a separate concern.

## Verification

- Type a query that matches several issues. Each result shows the matched chars with an accent-tinted background.
- Clear the query → highlights vanish.
- Mixed-case search ("BUTTON") highlights "button" / "Button" matches uniformly.
- Toggle to dark mode → highlight color still legible against the dark text.

## Root cause

`searchQuery` was only consumed by `IssueStore.filteredIssues` to drive list filtering — no view ever rendered the matched substring with any visual treatment. Once the filter narrowed the list down, the user had no signal pointing to *why* a row qualified, especially for matches deep inside long titles.

## Fix

Added a `SearchHighlight` helper plus a `Text(_:highlighting:)` initializer in `Issues/Views/SearchHighlightText.swift` that wraps text in an `AttributedString` and tints every case-insensitive occurrence of the trimmed query with `Color.appAccent.opacity(0.30)` as a background attribute. Empty / whitespace-only queries fall through to the plain `Text(_:)` initializer so call sites pay no extra cost when search is inactive. The matcher walks the string with `range(of:options:.caseInsensitive, range:)` and advances past `match.upperBound` after each hit so multiple matches are highlighted, not just the first.

Wired the helper through every site listed in the issue:
- `IssueCardView` (Swimlane) — title
- `RecentRowView` (Recent view) — title
- `ListView` — Title column
- `DetailPanelHeaderView` — issue title
- `IssueMarkdownSheetHeaderView` — `#NNNN — title` sheet header

Each consumer accepts an optional `searchQuery: String = ""` so existing previews and call sites stay source-compatible. The container views (`SwimlaneModuleSectionView`, `RecentView`, `MainView`'s `DetailPanelView` invocation, `IssueMarkdownSheet`) forward `store.searchQuery` from the active `IssueStore`. `ListView` already had `@Bindable var store`, so it reads `store.searchQuery` directly.

Description-body highlighting is intentionally deferred — the issue notes that StructuredText / Textual rendering needs a different approach and explicitly marks it optional / follow-up.

## Gotchas

- The match logic uses plain `.caseInsensitive` to mirror `IssueStore.filteredIssues` exactly. Switching to `.diacriticInsensitive` here would highlight things the filter doesn't accept, so the two stayed aligned.
- `AttributedString` ranges must be derived via `Range(stringRange, in: attr)`; constructing them by integer offsets is fragile when the source string contains non-BMP characters. The helper goes through `String.Index`-based search and converts each match to an `AttributedString` range, which preserves correctness for the U+2014 em-dash present in the sheet header (`#NNNN — title`).
- Each view exposes `searchQuery` as a defaulted property so older `#Preview` blocks (which don't pass a store-derived query) continue to compile and render unchanged.
