## Description

Fourth and final step of the TLS migration (#0111). With cert generation (#0112), token format (#0113), and viewer pinning (#0114) in place, the user-visible polish: a "Rotate host certificate" action in host settings, a clear surface for the current cert's fingerprint, and a viewer-side banner that explains "host identity changed" when a stale fingerprint trips the pin.

Without this, the user has to dig through Keychain or terminal to inspect the cert state, and a rotated host gives viewers a confusing "couldn't reach host" with no recovery hint.

## Where to look

- **`Issues/Views/RemoteHostSettingsView.swift`** (from #0083) — add a "Server certificate" section that shows the current fingerprint and offers Rotate.
- **`Issues/State/RemoteHostController.swift`** — `rotateCertificate()` regenerates `RemoteServerIdentity`, replaces the Keychain entry, restarts the listener, and bumps a `currentFingerprint` field for the view to bind to.
- **`Issues/Remote/RemoteHostIssueSource.swift`** (from #0094 / #0114) — already emits `.fingerprintMismatch` from the connection-state field once #0114 lands; this ticket adds the UI mapping.
- **`Issues/Views/RemoteConnectionBanner.swift`** (from #0104) — extend the banner state machine with a `.fingerprintMismatch` case that prompts "host identity changed — paste a new token".

## Host UI

In the host settings sheet, between the existing "Access tokens" and "Folders being served" sections:

```
┌─ Server certificate ───────────────────────────────┐
│                                                    │
│  Fingerprint (SHA-256):                            │
│   a3f1 e0c0 82b4 1d77 9b6e 4f3a c8d2 1e5b          │
│   7a0c 4d18 3b2f 6e91 5c0d 8e47 21fb 6a93          │
│                                                    │
│  [ Copy fingerprint ]   [ Rotate certificate… ]    │
│                                                    │
│  Rotation invalidates every existing access token. │
│  All viewers will need a fresh token.              │
│                                                    │
└────────────────────────────────────────────────────┘
```

- Fingerprint is rendered in two rows of 8 hex quads (32 chars each) for readability — same density used in `ssh-keygen` output and various wallet UIs.
- **Copy fingerprint** copies the raw 64-character lowercase hex (no spaces, no newlines) to the clipboard. Useful when the user wants to verify a viewer's pinned value matches by reading their Keychain.
- **Rotate certificate…** opens a confirm dialog before regenerating:

  > Rotating the certificate invalidates every access token issued against the current certificate. All connected viewers will be disconnected and must paste a new token before they can reconnect.
  >
  > [Cancel] [Rotate]

  On Rotate: regenerate identity, restart the listener, refresh the displayed fingerprint, and force-close every WebSocket session with code 4001-equivalent. The connected-viewers list (#0092) empties; existing tokens still parse but their embedded fingerprint no longer matches, so viewers will surface the mismatch banner described below.

## Viewer UI

`RemoteConnectionBanner` (#0104) already handles `.tokenInvalid`. Add a parallel `.fingerprintMismatch` case:

```
┌─────────────────────────────────────────────────────────┐
│  ⚠ The host's identity has changed                      │
│     The certificate fingerprint does not match the one  │
│     stored when you connected. The host may have        │
│     rotated its certificate, or someone may be          │
│     impersonating it.                                   │
│                                       [ Paste new token ]│
└─────────────────────────────────────────────────────────┘
```

- Yellow-leaning warning icon (not red — this is recoverable and usually benign).
- "Paste new token" opens the existing `RemoteTokenPasteView` (#0096) so the user pastes a fresh token (#0113's combined format includes the new fingerprint).
- The tab badge (#0104) gets a matching state — same yellow icon as `disconnected`, distinct tooltip: `Host identity changed — click to paste a new token`.

The banner replaces `.disconnected` / `.reconnecting` if a fingerprint mismatch is the actual cause — we don't pile both on the user. The state machine in `RemoteConnectionState` already supports this kind of replacement (#0104 set up the discriminant-keyed dismissal).

## Behavior

- **Rotate confirm dialog**: explicit confirmation prevents an accidental click from kicking every viewer off.
- **After rotation**, the connected-viewers list empties within a couple of seconds. Tokens-list rows stay (they're still valid as authentication credentials; only the embedded fingerprint is stale). The user can choose to revoke them in the same sheet.
- **Viewer's auto-reconnect** (#0103) hits the mismatch on its first retry and stops looping immediately — backoff applies to connection failures, not to fingerprint mismatches. The banner appears, retries stop, user takes action.
- **Re-paste flow**: new token from the rotated host carries the new fingerprint via #0113. `ViewerTokenStore.store(token:fingerprint:forHost:)` replaces the old entry. Connection retries succeed.

## Verification

### Host side

- Open settings → Remote Access. Server certificate section shows the current fingerprint formatted as two rows of 8 quads.
- Click Copy fingerprint → clipboard has 64 lowercase hex chars, no spaces.
- Click Rotate certificate… → confirm dialog appears. Cancel → nothing changes. Rotate → fingerprint display updates with the new value; connected viewers list empties.
- Cross-check via `openssl s_client -connect <host>:<port>` — the server now presents a cert with the new fingerprint.

### Viewer side

- Pair with the host pre-rotation. Tab is connected.
- Rotate the cert on the host.
- Within seconds the viewer tab shows the fingerprint-mismatch banner. Tab badge updates.
- Click "Paste new token" → token-paste sheet opens. Paste a fresh combined token (issued by the rotated host). Sheet closes; banner clears; tab reconnects.
- Confirm: pasting the **old** token (whose embedded fingerprint is stale) is rejected at paste time as token validation fails — the host correctly rejects authentication because the token plaintext was generated before rotation. (This is the auth-side guarantee in #0078: revoked-or-pre-rotation tokens fail with 401.)

### Cross-cutting

- Pair viewer A and viewer B (e.g. MacBook and Mac mini, both running the app) with the same host. Rotate. Both viewers surface the banner. Pasting a new token on viewer A doesn't affect viewer B; each viewer independently re-paires.

## Notes

- The "host identity changed" framing matches how SSH presents a similar event ("REMOTE HOST IDENTIFICATION HAS CHANGED!"). The wording is intentionally calm — for a single-user app, rotation is almost always intentional, not an attack.
- The fingerprint display in the host settings is also a useful debugging surface: the user can read it aloud over a phone call and compare to a viewer's pinned value if they ever wonder "is this the right host?".
- No automatic rotation schedule. Manual is fine for v2; if it becomes a chore, we add an "expire after N days" option later.

## Scope (out)

- A second fingerprint format alongside hex (e.g. base64 or `BLAKE3:…`). Hex is the lingua franca.
- "Recently used fingerprints" history on the host. One cert at a time; the previous one is gone after rotation.
- Per-token fingerprint visibility. Tokens stay name-only in the list; fingerprint is a host-level property.

## Fix

Two pieces; the viewer-side banner came in #0114 as part of the state-machine plumbing, so this issue owns the host-side rotation surface:

- **`RemoteHostController.rotateCertificate()`** — stops the listener if running, calls `RemoteServerIdentity.generate(label:)` (which `deleteAll`s the prior entry first per the agent's #0112 implementation), replaces the in-controller identity, rebuilds `RemoteServer` with the fresh identity (`sec_protocol_options`'s `local_identity` reference is stale, can't be swapped in place), and restarts the listener if it was on. The access-token database (#0078) is left alone — bearers are still valid; what's stale is the embedded fingerprint in any *combined* tokens (#0113) issued against the old cert, which viewers will refuse via the cert-pin check from #0114.
- **`Issues/Views/ServerCertificateSection.swift`** — the new section in `RemoteHostSettingsView`. Fingerprint rendered in two rows of eight 4-char hex quads (32 chars/row, ssh-keygen-style density). "Copy fingerprint" copies the raw 64-char lowercase hex; "Rotate certificate…" gates on a SwiftUI `confirmationDialog` with the spec's warning copy and a destructive `Rotate` action. Inline error label catches Keychain failures during the rotate (rare but possible).

The viewer-side banner already exists (#0114 added `RemoteConnectionState.fingerprintMismatch` + the matching banner case with the spec's "host identity changed" copy and a "Paste new token" action). Re-pasting a fresh combined token from the rotated host clears the banner; the new fingerprint replaces the old via `ViewerTokenStore.store(token:fingerprint:forHost:)`.

## Files changed

- `Issues/State/RemoteHostController.swift` — `identity` flipped to `private(set) var`, `server` to `var`, new `identityLabel` property, `rotateCertificate()` method.
- `Issues/Views/RemoteHostSettingsView.swift` — slots in the new section.
- `Issues/Views/ServerCertificateSection.swift` — new. Fingerprint display + Copy + Rotate.

## Gotchas

- **`RemoteServer` rebuilds, not in-place swap.** `sec_protocol_options_set_local_identity` is configured at parameter-construction time; reusing the same `RemoteServer` with a rotated identity would silently still serve the old cert. Constructing a fresh `RemoteServer` with the new identity is the only reliable path.
- **The token *database* is intentionally left alone on rotation.** Auth still works for the existing bearers; only their *embedded fingerprint* in the combined-token form is stale. The user re-issues tokens through the normal Generate sheet, which now carries the new fingerprint via #0113.
- **`confirmationDialog` for a destructive action** is the right primitive — modal, dismissable, explicit confirm, with a `.destructive` role on the action button so macOS renders it red. Spec called for an inline confirm but the destructive button + warning copy reads cleaner here.
- **Fingerprint formatted in two rows for readability.** Easier to read aloud over a phone call than a 64-char unbroken string; same density choice ssh-keygen and wallet UIs make. The Copy button still copies the raw form so pasted-into-the-viewer values don't carry the space-quad formatting.
- **Visual verification still needed**: fingerprint layout in light/dark mode, confirm dialog rendering, full pair-rotate-mismatch-paste round-trip on two Macs.
