## Description

Umbrella tracking issue for the v2 TLS migration of the remote-access feature (#0076). Today the host serves plain HTTP/WS and the viewer requires an ATS exception (#0110) to connect. The v2 design is **self-signed TLS with cert-fingerprint pinning**: the host generates a durable self-signed certificate once, the cert's SHA-256 fingerprint is bundled with the bearer token at issue time, and the viewer pins the fingerprint at paste time. From then on the viewer authenticates the host by matching the presented cert's fingerprint against the pinned value — hostname/IP is irrelevant.

This is the cleanest fit for the project's constraints: two single-user Macs (eventually +iPhone) with no static hostname, no public CA, no DNS. The cert acts as a stable host identity that survives address changes (LAN → Tailscale → different Wi-Fi), and identity travels alongside the bearer through the existing token-transfer flow.

## Phase plan

- **#0112 — Host: self-signed cert generation + TLS listener.** Generate a P-256 cert on first host-enable, persist in Keychain, configure `NWListener` with `NWProtocolTLS.Options` referencing the `SecIdentity`. Wire is TLS; tokens still work; viewers temporarily incompatible until #0114 lands.
- **#0113 — Token format: bundle cert fingerprint with bearer.** `AccessToken.generate` returns `(plaintext, fingerprint)`; host settings UI (#0084) shows both; the paste UI (#0096) parses the combined payload.
- **#0114 — Viewer: cert pinning + wire-scheme flip + ATS cleanup.** `URLSessionDelegate` on `RemoteClient` (#0094) and `RemoteWebSocket` (#0102) overrides server-trust evaluation by comparing fingerprints. URLs become `https://` / `wss://`. The `NSAllowsArbitraryLoads` exception from #0110 is removed in the same PR.
- **#0115 — Cert rotation UI + fingerprint-mismatch banner.** "Rotate host certificate" button in host settings; viewer-side `#0104` banner extended with a `.fingerprintMismatch` state surfacing "host identity changed — paste a new token".

## Decisions locked in this round

- **Self-signed cert with fingerprint pinning**, not a custom CA, not trust-on-first-use, not public certs. Rationale in the design discussion that produced this tracking issue: pinning composes with the existing token transfer, requires no extra prompt, and matches the single-user threat model.
- **P-256 ECDSA, 10-year validity, no subject binding.** The cert is purely an identity anchor — no hostname or IP in the SAN. SHA-256 is the fingerprint algorithm.
- **Combined token payload: `iat_<43 base64url>.<64 hex sha256>`.** ~108 chars total, still QR-safe for #0108's iPhone scan path.
- **Wire scheme flips entirely**: `http://` → `https://`, `ws://` → `wss://`. No mixed-mode support, no protocol fallback. v2 hosts and v2 viewers are not backward-compatible with v1 plain-HTTP peers.
- **`NSAllowsArbitraryLoads` exits in #0114.** With cert pinning via `URLSessionDelegate`, ATS's chain validation is overridden cleanly and the broad exception is no longer needed. (Self-signed cert + cert pinning is *more* secure than the v1 plain-HTTP state in terms of channel confidentiality, even though ATS would technically reject the cert chain without the delegate.)

## Sequencing notes

- **#0112 → #0113 → #0114** is the load-bearing chain: cert exists → fingerprint can be issued with tokens → viewer can pin it. #0114 is also where the wire scheme flips, so it's the breaking change.
- **#0115** is additive and can land any time after #0114.
- **#0110** stays in place until #0114 ships. It's the v1 bridge that keeps the viewer functional while the TLS work is in flight.
- **No dependency on #0093** (multicast entitlement). TLS work is orthogonal to Bonjour and can land in any order.

## Notes

- TLS doesn't replace bearer-token auth — it adds a *channel* guarantee on top of the *authentication* guarantee. A token-only design with TLS off is fine on Tailscale (tunnel encrypts); TLS makes the design fine on untrusted LANs too.
- This is the resolution of `RemoteAccess.md` line 70 ("TLS not strictly required for v1. … We can add TLS later (self-signed, pinned via Keychain) if v2 needs it."). v2 is now.
- Future v3 considerations not addressed here: cert revocation lists, cert lifetime renewal automation, optional client certs for mTLS. None needed for single-user two-Mac case.

## Scope (out, until v3 or later)

- Public CA / Let's Encrypt — requires a static hostname, defeats the design.
- Custom CA + leaf certs — overkill for one host, one user.
- mTLS — the bearer-token model is the auth path; adding client certs replaces tokens with another credential to manage.
- Automatic cert rotation on a schedule — manual rotation via #0115 is fine for v2.
