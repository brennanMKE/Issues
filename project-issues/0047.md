# 0047 — Drop deprecated `EventModifiers.function` from `KeyboardShortcutRecorder`

| | |
|---|---|

## Description

`Issues/Views/Settings/KeyboardShortcutRecorder.swift` references `SwiftUI.EventModifiers.function` at two sites, producing a deprecation warning at build time:

```
'function' was deprecated in macOS 12.0: Function modifier is reserved for system applications
```

Lines (current state):

```swift
// 185 — eventModifiers(from:) helper
if flags.contains(.function) { mods.insert(.function) }   // .insert is deprecated

// 197 — makeBinding(from:modifiers:) helper
let nonShiftMods: SwiftUI.EventModifiers = modifiers.subtracting([.shift, .function])
//                                                                       ^^^^^^^^^ deprecated
```

Note: the LHS `flags.contains(.function)` on line 185 is `NSEvent.ModifierFlags.function` (AppKit) and is not deprecated — only the RHS `mods.insert(.function)` (`SwiftUI.EventModifiers.function`) and the `.function` literal on line 197 are.

## Why the deprecation exists

Apple reserved the `Fn` (function / globe) key as a system-level modifier in macOS 12 — it's owned by the input-source switching layer (Spotlight, dictation, language input). Third-party apps shouldn't accept it as part of a user-defined keyboard shortcut, because the user can't reliably press it without triggering system behavior.

## Fix

Stop carrying the function modifier through our binding model entirely. Two surgical edits:

1. **`eventModifiers(from:)`** — drop the line that propagates Fn into the SwiftUI `EventModifiers` set:

   ```swift
   private func eventModifiers(from flags: NSEvent.ModifierFlags) -> SwiftUI.EventModifiers {
       var mods: SwiftUI.EventModifiers = []
       if flags.contains(.command)  { mods.insert(.command) }
       if flags.contains(.option)   { mods.insert(.option) }
       if flags.contains(.shift)    { mods.insert(.shift) }
       if flags.contains(.control)  { mods.insert(.control) }
       // (.function intentionally dropped — reserved for system use as of macOS 12)
       return mods
   }
   ```

2. **`makeBinding(from:modifiers:)`** — once Fn can never appear in `modifiers`, drop it from the subtraction set:

   ```swift
   // Plain character keys: require at least one non-shift modifier so the
   // user can't bind `t` and shadow text input.
   let nonShiftMods: SwiftUI.EventModifiers = modifiers.subtracting([.shift])
   ```

That's the entire change — two lines edited, zero new types, no behavior anyone will notice (the recorder simply won't accept Fn-modified bindings, which is what Apple already enforces at the system level).

## Related state to verify

- `ShortcutBinding.modifiers` is `Int` (the raw bitfield of `EventModifiers`). Existing persisted bindings that happened to include the Fn bit would still decode, but `keyboardShortcut` constructs a `KeyboardShortcut` from the raw `EventModifiers`, which currently still accepts the bit; the system will simply ignore it. No migration needed.
- The reserved-shortcut list in `ShortcutsStore.reservedShortcuts` doesn't reference `.function`, so no change there.
- No other file in the codebase references `EventModifiers.function`.

## Verification

- After the fix, `xcodebuild ... build` should complete without the `'function' was deprecated` warning. Grep the build log for `was deprecated in macOS 12.0` — expect zero hits.
- The Settings → Shortcuts recorder still records Cmd, Option, Shift, Control combinations as it does today.
- Pressing Fn alone or with another modifier in the recorder no longer changes the binding (matches system behavior — Fn opens Globe / dictation overlay rather than landing as a key event for us).

## Out of scope

- Migrating any user-facing string about supported modifier keys (there isn't one).
- Updating Help docs (`03-keyboard-shortcuts.md`) — that file doesn't mention Fn.
