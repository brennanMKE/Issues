## Description

Dragging the detail-panel resize handle (#0069) produces a heavy jitter in the rendered markdown body — text appears to flicker / re-flow on every frame of the drag rather than smoothly reflowing with the new width. The panel chrome itself moves smoothly; only the markdown content area jitters.

The likely cause is that the markdown rendering (Textual `MarkdownView` in `DetailPanelDescriptionView`) re-parses the source text on every body-rebuild as `panelWidth` changes during the drag. The content itself is not changing — only the available width — so the rendered output should be cached per `(issue.id, source-text)` and only re-laid-out on width change, not re-parsed.

## Steps to reproduce

1. Open Issues.app with a folder loaded.
2. Select an issue that has a meaningful description body (a few paragraphs is enough — the jitter is more visible with more text).
3. Grab the detail panel's leading-edge resize handle and drag it left/right continuously.
4. Watch the markdown body in the panel.

## Expected behavior

The markdown reflows smoothly as the panel width changes. Text wrapping updates per frame; glyphs do not flicker, jump, or re-position visibly within a line.

## Actual behavior

The text visibly jitters — characters/lines appear to redraw repeatedly, as if the markdown is being re-parsed and re-laid-out from scratch on every drag tick. The faster the drag, the more pronounced the flicker. Standalone evidence in the screen recording.

## Notes

- Suspected culprit: `DetailPanelDescriptionView` (and possibly `IssueMarkdownSheet` / `HelpView`, which also use Textual) building the `MarkdownView` inside `body` so SwiftUI tears it down and recreates it whenever any ancestor state changes — which now includes `panelWidth` from #0069.
- Cache strategy worth trying: hoist the parsed markdown (or the `MarkdownView` itself) into a memoized layer keyed by `(issue.id, descriptionText)`, so width changes only re-run layout against the already-parsed content. `Equatable` view conformance on the description subview, or `EquatableView { … }`, may be enough to short-circuit the rebuild.
- Another angle: confirm whether Textual's `MarkdownView` itself does its own caching of the parsed AST, and whether the resize is invalidating something it shouldn't (e.g. a `@State` inside the package). If so, the fix is on the consumer side regardless — keep the same `MarkdownView` instance across width changes.
- Pairs with #0069 (the resize handle that exposed this). Not strictly a regression — the panel was previously content-sized so its width never changed, so the path was never exercised. Now that resize is a routine action, the cost is visible.

## Root cause

`DetailPanelDescriptionView.body` was doing two expensive things on every recompute: reading the issue file from disk (`bodyMarkdown()` → `String(contentsOf:)`) and running `InlineImageMarkdown.split(IssueCrossRef.rewrite(body))` to produce the chunk list. The result was then handed to `StructuredText` (Textual) inside a brand-new `VStack`, which gave SwiftUI a structurally-fresh subtree on every parent body call. After #0069 introduced a draggable resize handle, `panelWidth` in `MainView` changed on every drag tick, which propagated down through `DetailPanelView` → `DetailPanelDescriptionView`, re-running the disk read, the split, and the Textual instantiation per frame. Visible result: text flickering / re-flowing instead of smoothly reflowing for the new width.

## Fix

- Hoisted the parse into a `private struct ParsedBody: Equatable` with `(issueID, modifiedAt, body, chunks)`. Stored as `@State` on `DetailPanelDescriptionView` and refreshed via `.task(id: parseCacheKey)` where the key is `"\(issue.id)|\(modifiedAt)"` — so the disk read + split happen once per `(identity, mtime)` rather than per body recompute. FSEvents-driven file edits still surface because `IssueStore` updates `modifiedAt` on reload, which retriggers `.task`.
- Extracted the rendered subtree into a `private struct BodyView: View, Equatable`. Equality is keyed on `(issueID, modifiedAt, bodyText, baseURL)` — deliberately omitting the closures and `panelWidth` (which the view doesn't observe directly anyway). Wrapping the call site in `EquatableView(content: BodyView(...))` lets SwiftUI short-circuit rebuilding the chunks `ForEach` and the `StructuredText` instances when only an upstream non-content state (like the resize-drag tick) changes.
- Preserved the existing `.id(issue.id)` outer marker so Textual's internal layout cache is still invalidated on issue switch. The `.sheet`/`.quickLookPreview` modifiers and the `attachmentURL`/`quickLookURL` `@State` plumbing for #0073 are unchanged; the `onOpen`/`onPlay` callbacks now flow through `BodyView` to `AttachmentThumbnailView` exactly as before.

## Files changed

- `Issues/Views/DetailPanelDescriptionView.swift` — added `ParsedBody` cache + `.task(id:)` refresh path; extracted `BodyView: View, Equatable` and wrapped its instantiation in `EquatableView`; renamed the body-text field to `bodyText` to avoid clashing with SwiftUI's `body` requirement.

## Gotchas

- Equality on `BodyView` excludes the callback closures by design — closures aren't `Equatable` and including them would defeat the short-circuit. The callbacks read `@State` on the parent which still works because SwiftUI reuses the same `BodyView` instance and its captured environment when equality holds.
- The `body` stored property name from a first attempt clashed with `View.body`; the field is now `bodyText`. The error was straightforward (`invalid redeclaration of 'body'`) but worth flagging for anyone copying this pattern.
- GUI verification of "no jitter" wasn't possible from this subagent context — only that the build is green and the structural change is correct (cache key invalidates on identity + mtime, equality excludes width). The acceptable bar per the task brief was structural soundness; user-side visual confirmation is still the gate before this lands as `closed`.
