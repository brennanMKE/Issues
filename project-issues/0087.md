## Description

Phase 1 of the remote-access plan (#0076). Before any viewer UI exists, we need a way to confirm the host is serving correctly from another Mac. A small standalone Swift Package executable that takes `host:port` and a bearer token, hits every v1 endpoint, and prints a pass/fail report makes that test loop tight and reproducible.

This is also useful long-term: when something breaks in the viewer, running the CLI against the host narrows down whether the bug is on the wire or in the Mac viewer code.

## Where to look

- **New: `Tools/IssuesRemoteSmoke/`** — a sibling Swift Package next to `Issues.xcodeproj`. Standalone `Package.swift` with a single `executableTarget`. Not added to the Xcode project; runs via `swift run` in its own directory.
- **Reuses `Issues/Remote/RemoteProtocol.swift`** (from #0080) — the package depends on a path-based local source so the wire types stay in lockstep without a duplicate definition.

## CLI shape

```
$ swift run issues-remote-smoke \
    --host 100.74.12.5:51823 \
    --token iat_K5p9Lq7N4mR2oB6tV1xZ3jA0wY8sH9cD1eF2gH3iJ4
```

Arguments:

- `--host <ip-or-name:port>` — required.
- `--token <iat_…>` — required. Read from `$ISSUES_REMOTE_TOKEN` if not passed, so the token doesn't end up in shell history.
- `--folder <id>` — optional. If omitted, the CLI picks the first folder from `/v1/folders`.
- `--issue <id>` — optional. If omitted, picks the first issue.
- `--verbose` — print full response bodies; otherwise print only pass/fail lines.

## What it checks

In order, with one line per check:

1. `GET /v1/host` → 200, body decodes to `HostInfo`. Prints `displayName` and `folderCount`.
2. `GET /v1/host` with no token → 401 `missing_token`.
3. `GET /v1/host` with `iat_garbage` → 401 `invalid_token`.
4. `GET /v1/folders` → 200, decodes to `[FolderInfo]`, count matches `HostInfo.folderCount`.
5. `GET /v1/folders/<id>` → 200, decodes to `FolderInfo`, id matches.
6. `GET /v1/folders/<id>/issues` → 200, decodes to `[IssueMetadata]`, count matches `FolderInfo.issueCount`.
7. `GET /v1/folders/<id>/issues/<iid>` → 200, decodes to `IssueDetail`, body is non-empty.
8. If the issue has attachments: `GET /v1/folders/<id>/issues/<iid>/attachments/<name>` → 200, `Content-Length` matches the bytes received, hash logged.
9. (When #0100 lands) Open `/v1/events` WebSocket, send `subscribe` for the chosen folder, receive within 1 sec, send `unsubscribe`, close cleanly.

Each check prints `ok` or `fail` + a brief message; the binary exits with status `0` if all checks pass, `1` otherwise.

## Verification

- Run against a host with hosting on and a valid token — every check prints `ok`, exit `0`.
- Run with a wrong token — checks 1 fails (401), 2 and 3 still pass (they're negative checks). Exit `1` with a clear summary line.
- Run against a host with hosting off — `connection refused` early, exit `1` with a one-line error.
- Run before #0100 lands — checks 1–8 pass, check 9 is skipped with `skip: ws not yet implemented`.

## Notes

- Use `URLSession` for HTTP and `URLSession.webSocketTask(with:)` for WS. No third-party deps.
- Keep the package dependency-free so it's trivial to clone onto another Mac.
- The CLI is **not** a replacement for unit tests on the host. It's the integration test from another box.
- Consider committing a tiny shell script `Tools/IssuesRemoteSmoke/smoke.sh` that wraps the common invocation (`ISSUES_REMOTE_TOKEN=… swift run …`).

## Scope (out)

- A test for the WS event delivery itself (sending an edit on the host and asserting the event arrives) — that's a follow-up under #0102 once both ends speak WS.
- Performance checks (latency, throughput). Not the right tool for that.
- Cross-platform Linux build of the CLI. macOS-only is fine — the host is macOS-only.

## Root cause

Greenfield. With every host-side endpoint shipped (#0079–#0081, #0085, #0086), the loop "edit, build, verify on a remote Mac" needed a one-shot tool. Curl works for spot checks but isn't a regression suite, and writing the viewer (#0094) before the host is verified would let bugs hide on either side.

## Fix

Standalone Swift Package at `Tools/IssuesRemoteSmoke/`:

- **`Package.swift`** — `executableTarget` `IssuesRemoteSmoke` plus a `IssuesRemoteProtocol` library target. macOS 14+. No external dependencies.
- **Wire-type sharing via symlink.** SPM forbids targets that point outside the package root, so the spec's "path-based local source" is implemented with a git-tracked symlink at `Sources/IssuesRemoteProtocol/RemoteProtocol.swift` → `../../../../Issues/Remote/RemoteProtocol.swift`. Both the host module and the CLI compile the exact same bytes. The shared types and `RemoteProtocol` enum became `public` (plus explicit memberwise inits) so they cross the module boundary; that's a benign change for the host since `public` is also accessible internally.
- **`Sources/IssuesRemoteSmoke/main.swift`** — `Reporter` collects pass/fail/skip counts; `runChecks(opts:)` walks the spec's checks 1–9 in order, threading `HostInfo` / `FolderInfo` / `IssueMetadata` / `IssueDetail` through each step so cross-check assertions ("`folders.count == hostInfo.folderCount`") fire naturally. Auth-failure cases (no token, garbage token) verify both the status code and the JSON `error` reason. Attachment check sniffs the first attachment's first 8 bytes via SHA-256 and confirms the response `Content-Length` header matches the body length. Step 9 (WebSocket) is unconditionally skipped with the spec's wording until #0100 lands.
- **`smoke.sh`** — zsh wrapper that defers to `swift run -c release issues-remote-smoke "$@"` and warns when `$ISSUES_REMOTE_TOKEN` isn't set so the bearer doesn't end up in shell history.

Verified `swift build` and `swift run IssuesRemoteSmoke --help` both succeed; the host project still builds clean and the full unit suite is green after the `public` modifier change.

## Files changed

- `Tools/IssuesRemoteSmoke/Package.swift` — new. Two targets, no deps.
- `Tools/IssuesRemoteSmoke/Sources/IssuesRemoteSmoke/main.swift` — new. CLI logic + reporter.
- `Tools/IssuesRemoteSmoke/Sources/IssuesRemoteProtocol/RemoteProtocol.swift` — git-tracked symlink to `../../../../Issues/Remote/RemoteProtocol.swift`.
- `Tools/IssuesRemoteSmoke/smoke.sh` — convenience wrapper.
- `Issues/Remote/RemoteProtocol.swift` — wire types + the `RemoteProtocol` enum became `public` so the smoke CLI module can import them.

## Gotchas

- **SPM doesn't allow target paths outside the package root.** The spec called for a "path-based local source" but `path: "../../Issues/Remote"` errors with "outside the package root." A git-tracked symlink under `Sources/` is the workaround that survives `git clone` (git stores the symlink, not the target's content) and keeps the wire types byte-identical.
- **`public` types need explicit `public init(...)`.** Synthesized memberwise inits are internal-by-default even on public types; without the explicit inits the smoke CLI can't construct them. Worth keeping in mind for any future shared module.
- **`@MainActor` on `runChecks` and `main`.** The top-level `let reporter` in a Swift 6 / strict-concurrency CLI is implicitly main-actor; calling it from a nonisolated async function emits warnings. Tagging the entry points as `@MainActor` keeps everything on one actor and silences the warnings without making `Reporter` `Sendable`.
- **Step 9 stays as `skip`, not `fail`.** The spec is explicit: WebSocket check is for #0100. A "fail" would be misleading until that endpoint exists; the skip line documents the gap.
