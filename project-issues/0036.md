## Description

Most views in the project break their bodies up using computed properties (`var header: some View`, `var description: some View`, …) and helper methods (`func row(for:) -> some View`, `func bar(...) -> some View`, …). The SwiftUI guidance in `swiftui-pro/references/views.md` is explicit (and repeated twice in that file for emphasis):

> Strongly prefer to avoid breaking up view bodies using computed properties or methods that return `some View`, even if `@ViewBuilder` is used. Extract them into separate `View` structs instead, placing each into its own file.

The performance reference adds: extracting real `View` structs preserves structural identity and lets SwiftUI diff branches cheaply; computed-property and `@ViewBuilder` returns recompose every body invocation.

## Affected files (largest first)

- `Issues/Views/TabBarView.swift` — `addButton`, `cell(for:)`, plus the inline `TabChipView` does this internally too.
- `Issues/Views/TimelineView.swift` — `tickHeader(geometry:)`, `moduleRow(...)`, `bar(...)`, `todayMarker(...)`, `weeklyTicks(...)`.
- `Issues/Views/DetailPanelView.swift` — `header`, `metadata`, `description`, `fileLink`, `label(_:)`.
- `Issues/Views/ListView.swift` — every `TableColumn` body is a closure over `Text` etc.; `displayDate(_:)`, `selectionBinding`, `sortedIssues`.
- `Issues/Views/IssueMarkdownSheet.swift` — `header`, `contentBody`.
- `Issues/Views/LintSheetView.swift` — `header`, `content`, `row(for:)`.
- `Issues/Views/RecentView.swift` — `row(for:)`, `emptyState`.
- `Issues/Views/SwimlaneView.swift` — `moduleSection(module:issues:)`, `groups`.
- `Issues/Views/HelpView.swift` — `sidebar`, `detail`, `sectionView(for:)`, `placeholder`.
- `Issues/Views/StatsBarView.swift` — `lintBanner`, `statRow(...)`, `viewModeSwitcher`.
- `Issues/Views/ToolbarView.swift` — `statusPills`, `statusPill(for:)`.
- `Issues/Views/FolderPickerView.swift` — `emptyState`, `rememberedList`, `folderRow(_:)`.
- `Issues/Views/MainView.swift` — `contentArea`, `errorBanner(_:)`, `placeholderIssue`.

## Suggested approach

Don't try to do this in one sweep — the refactor is mechanical but touches every view. A reasonable plan:

1. Pick one view per pass (e.g. `DetailPanelView` first — high reuse value, clearly bounded).
2. Extract each `var section: some View` into a top-level `private struct DetailHeader: View { ... }` (or non-private if reused) in its own file under `Issues/Views/`.
3. Pass dependencies in via `let` properties or `@Bindable` references.
4. Methods that take parameters (`row(for: issue)`) become structs that take those parameters as `let`s.
5. Delete the original computed property / method.

The `Issues/` source group uses `PBXFileSystemSynchronizedRootGroup` (per CLAUDE.md), so creating the new files on disk auto-registers them with the target.

## Notes

- This is a quality / maintainability issue, not a correctness bug — the app works today.
- Some helpers are pure computation that returns non-view data (`bodyMarkdown()`, `formatted(_:raw:)`, `displayDate(_:raw:)`, `sortedIssues`). Those are *not* affected — the rule is specifically about returning `some View`.
- After extraction, `previewProvider`s for the smaller views become possible, which the codebase currently doesn't have (only `SettingsView`, `GeneralSettingsView`, `ShortcutsSettingsView` ship with `#Preview`).
- Combine with #0037 — that issue covers the secondary problem of multiple structs sharing a single file, which this refactor naturally fixes if each extracted view lands in its own file.

### Intentional exceptions

- **`TabBarView.cell(for:)`** kept as a `@ViewBuilder` method, not extracted. It consumes six pieces of `TabBarView`'s drag-state `@State` (`draggingID`, `originalSlot`, `phantomSlot`, `draggedWidth`, `measuredWidths`, plus `tabs`); extraction would require five `@Binding`s and obfuscate rather than clarify the drag flow. The views.md rule says "strongly prefer" — this is the case where the trade-off favors the method. An explanatory comment is in place at the call site.
- **`DetailPanelMetadataView.label(_:)`** kept as a file-private one-line helper inside the metadata view — extracting a trivial `Text` styling helper into its own `View` struct would inflate file count without clarity gain.
- `EmptyMainView` and `FolderPickerSceneView` still share `MainView.swift`; `HelpLoadError` still lives at the bottom of `HelpView.swift`. These are **multi-type-file** concerns covered by #0037, not #0036's `some View` rule.
