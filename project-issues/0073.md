# 0073 — Video attachments should render as a poster thumbnail with in-app playback

| | |
|---|---|
| **Status** | open |
| **Module** | Views / Services |
| **Platform** | macOS |
| **First seen** | 2026-05-07 |

## Description

Video attachments (`.mov`, `.mp4`, etc.) currently render as "Missing attachment" in `DetailPanelView`, even when the file is present, because `AttachmentThumbnailView` loads attachments through `NSImage(contentsOf:)` — which returns `nil` for non-image formats. Filed as the labeling defect underlying the "missing" report on #0072's recording.

The IssuesSkill is being updated (separately, see `IssuesSkill-video-attachments.md` at the repo root) to emit a new markdown form for videos: an image-inside-a-link, where the image is an auto-generated poster frame and the link target is the video file:

```markdown
[![Sidebar resize jitter](0072/sidebar-resize-jitter.poster.png)](0072/sidebar-resize-jitter.mov)
```

Issues.app should:

1. Render the poster image as a normal inline thumbnail (existing path).
2. Detect that the surrounding link target is a video (UTI conforms to `.movie`) and overlay a play-button glyph on the thumbnail.
3. On click, present an `AVPlayerView` in a popover or sheet to play the video in-app instead of opening it in QuickTime / the default app.

Plain image attachments (no link wrapper) keep today's behavior unchanged.

## Where to look

- **`Issues/Services/InlineImageMarkdown.swift`** (or wherever `InlineImageMarkdown.split` lives — the AttachmentThumbnailView header references it). This is the parser that extracts `![alt](path)` from the markdown body. Extend it to also recognize the optional surrounding `[…](videoPath)` and pass the link target through alongside the image alt/path.
- **`Issues/Views/AttachmentThumbnailView.swift`** — accepts an optional `videoURL: URL?`. When set:
  - Add a centered play-button overlay (`Image(systemName: "play.circle.fill")`, ~44pt, `.regularMaterial` background or just `.white` with shadow).
  - Reroute the click: instead of `onOpen(resolvedURL)` (which opens the poster image), invoke a new `onPlay(videoURL)` callback.
- **`Issues/Views/DetailPanelDescriptionView.swift`** — owner of the description body. Already presents the attachment sheet via `onOpen`; add a parallel video-popover state that pushes an `AVPlayerView` when `onPlay` fires.

## Behavior details

- **UTI check** for the link target — `UTType(filenameExtension: url.pathExtension)?.conforms(to: .movie)`. If false, fall back to today's link-as-link behavior (or treat as plain image, since the surrounding link is then essentially decorative). Don't crash, don't show "Missing attachment."
- **Player UI** — `AVPlayerView` (AppKit) wrapped via `NSViewRepresentable`. Show standard controls (`.controlsStyle = .floating`). Default size: aspect-fit inside ~720×480, but cap at the window's content rect minus padding.
- **Presentation style** — sheet is the safest choice; a popover anchored to the thumbnail also works but can be awkward when the description scrolls. Pick sheet for v1, leave popover as a follow-up if it feels heavy.
- **Autoplay** — yes, on present. Pause on dismiss. Don't loop.
- **Keyboard** — Escape dismisses the sheet (default `.sheet` behavior). Space toggles play/pause (AVPlayerView default).
- **Missing video file** — same fallback as missing image: show the existing "Missing attachment" placeholder, but with the video's path so the user can tell which file is gone.
- **Non-video link target** — if `[![alt](image)](something.pdf)` ever appears, treat the surrounding link as a regular link (open in default app via `NSWorkspace`). Don't attempt to play it.

## Verification

- Open #0072. Description renders with a poster thumbnail (not "Missing attachment") and a play-button overlay.
- Click the thumbnail → a sheet opens with the video playing.
- Escape or click-away closes the sheet, video pauses.
- Open an issue with a plain `![alt](image.png)` attachment → renders unchanged, no play overlay, click still opens the existing attachment sheet.
- Delete the `.mov` from disk while the app is open → next render shows the missing-attachment placeholder with the `.mov` path.
- Drop a `.mp4` (different extension, still video UTI) → still recognized.

## Notes

- Pairs with the IssuesSkill update — once both ship, the round-trip is: skill emits the new markdown form when the user attaches a video, app renders it natively. The two changes are independent: the app upgrade also works on existing posters generated by hand, and the skill change degrades to today's broken-thumbnail behavior on a non-upgraded app.
- `AttachmentThumbnailView` already has an `onOpen` callback hoisted to the parent. Adding `onPlay` keeps the same shape and avoids per-thumbnail sheet state, which the existing comment explicitly calls out.
- Keep the `.movie` UTI check, not an extension allowlist — `UTType` resolves new container formats automatically as the system gains support.
- Consider caching the resolved video `AVAsset` per `(url, mtime)` to avoid re-creating it on every panel rebuild (same caching shape as #0072's markdown caching).
