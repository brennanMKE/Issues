# 0073 — Video attachments should render as a poster thumbnail with Quick Look playback

| | |
|---|---|
| **Status** | open |
| **Module** | Views / Services |
| **Platform** | macOS |
| **First seen** | 2026-05-07 |

## Description

Video attachments (`.mov`, `.mp4`, etc.) currently render as "Missing attachment" in `DetailPanelView`, even when the file is present, because `AttachmentThumbnailView` loads attachments through `NSImage(contentsOf:)` — which returns `nil` for non-image formats. Filed as the labeling defect underlying the "missing" report on #0072's recording.

The IssuesSkill is being updated (separately, see `IssuesSkill-video-attachments.md` at the repo root) to emit a new markdown form for videos: an image-inside-a-link, where the image is an auto-generated poster frame and the link target is the video file:

```markdown
[![Sidebar resize jitter](0072/sidebar-resize-jitter.poster.png)](0072/sidebar-resize-jitter.mov)
```

Issues.app should:

1. Render the poster image as a normal inline thumbnail (existing path).
2. Detect that the surrounding link target is a video (UTI conforms to `.movie`) and overlay a play-button glyph on the thumbnail.
3. On click, present the system **Quick Look** preview panel via SwiftUI's `.quickLookPreview()` modifier (macOS 13+). Quick Look renders video with full transport controls; we don't need to build or wrap a player view.

Plain image attachments (no link wrapper) keep today's behavior unchanged.

## Where to look

- **`Issues/Services/InlineImageMarkdown.swift`** (or wherever `InlineImageMarkdown.split` lives — the AttachmentThumbnailView header references it). This is the parser that extracts `![alt](path)` from the markdown body. Extend it to also recognize the optional surrounding `[…](videoPath)` and pass the link target through alongside the image alt/path.
- **`Issues/Views/AttachmentThumbnailView.swift`** — accepts an optional `videoURL: URL?`. When set:
  - Add a centered play-button overlay (`Image(systemName: "play.circle.fill")`, ~44 pt, white tint with a soft shadow so it reads on any poster).
  - Reroute the click: instead of `onOpen(resolvedURL)` (which opens the poster), invoke a new `onPlay(videoURL)` callback.
- **`Issues/Views/DetailPanelDescriptionView.swift`** — owner of the description body. Add a single `@State private var quickLookURL: URL?` and apply `.quickLookPreview($quickLookURL)` to the body. The `onPlay` callback simply assigns the resolved video URL to that state; SwiftUI presents the system Quick Look panel and clears the binding when dismissed.

## Behavior details

- **UTI check** for the link target — `UTType(filenameExtension: url.pathExtension)?.conforms(to: .movie)`. If false, treat the surrounding link as a regular link (open in default app via `NSWorkspace`); do not present Quick Look as a video.
- **Quick Look presentation** — bind a single optional `URL` to `.quickLookPreview()`. When the binding becomes non-nil, SwiftUI presents the system Quick Look window (the same one Finder shows on Space-bar). Setting it back to nil dismisses; SwiftUI also clears it when the user closes the panel. No custom sheet, no `NSViewRepresentable`, no `AVPlayerView`.
- **Playback** — Quick Look's video preview includes play/pause, scrub, volume, and fullscreen out of the box. Don't try to influence autoplay or loop behavior; let Quick Look handle it (matches what users expect from Finder's Space-bar preview).
- **Missing video file** — same fallback as missing image: show the existing "Missing attachment" placeholder, but with the video's path so the user can tell which file is gone. Don't pass a non-existent URL to `.quickLookPreview()`.
- **Non-video link target** — if `[![alt](image)](something.pdf)` ever appears, route through `NSWorkspace.shared.open(url)` rather than Quick Look. The play-button overlay should not appear in this case (no `.movie` conformance → don't decorate).

## Verification

- Open #0072. Description renders with a poster thumbnail (not "Missing attachment") and a play-button overlay.
- Click the thumbnail → the system Quick Look panel opens with the video and standard transport controls.
- Press Space-bar / Escape → Quick Look closes; the description view returns to normal.
- Open an issue with a plain `![alt](image.png)` attachment → renders unchanged, no play overlay, click still opens the existing attachment sheet.
- Delete the `.mov` from disk while the app is open → next render shows the missing-attachment placeholder with the `.mov` path; clicking does nothing harmful.
- Drop a `.mp4` (different extension, still video UTI) → still recognized and previewed by Quick Look.

## Notes

- Pairs with the IssuesSkill update — once both ship, the round-trip is: skill emits the new markdown form when the user attaches a video, app renders it natively. The two changes are independent: the app upgrade also works on existing posters generated by hand, and the skill change degrades to today's broken-thumbnail behavior on a non-upgraded app.
- `AttachmentThumbnailView` already has an `onOpen` callback hoisted to the parent. Adding `onPlay` keeps the same shape — both callbacks exist side-by-side; the parent decides whether to show the existing attachment sheet (for images) or Quick Look (for videos).
- Keep the `.movie` UTI check, not an extension allowlist — `UTType` resolves new container formats automatically as the system gains support.
- `.quickLookPreview()` was introduced in macOS 13; this project targets macOS 15+ so the API is unconditionally available. No availability annotation needed.
- Quick Look generalizes for free — if a future issue ever attaches a `.pdf`, `.wav`, or `.mp3`, the same `.quickLookPreview()` binding can preview it. This issue scopes only video, but the parser change (image-in-link) is the gateway, and audio/PDF support could follow with no additional plumbing.
