## Description

Phase 5 of the remote-access plan (#0076). When a remote tab can't reach its host, the user needs a visible cue plus a sensible recovery path. There are two distinct disconnect modes:

1. **Reconnecting** — the WS is down, but the token is still valid and the WS wrapper (#0102) is retrying on backoff. The user just waits.
2. **Token revoked / expired** — the host returned 401 (or closed with WS code 4001). Reconnect won't help; the user needs a new token.

Both modes get clear UI without falling back to a modal alert.

## Where to look

- **`Issues/Views/`** — wherever tabs render. Add a small status badge per tab.
- **`Issues/State/IssueStore.swift`** — exposes the source's connection state (`connected | reconnecting | disconnected | tokenInvalid | folderUnavailable`).
- **`Issues/Views/RemoteTokenPasteView.swift`** (from #0096) — reused as the recovery sheet when the user clicks "Paste new token" on the expired UI.

## Connection state

```swift
enum RemoteConnectionState: Equatable {
    case connected
    case reconnecting(since: Date)
    case disconnected(reason: String)
    case tokenInvalid
    case folderUnavailable
}
```

`RemoteHostIssueSource` (#0094) is the source of truth. `IssueStore` republishes it for the views to bind to.

## UI

### Tab badge

- `connected` — no badge (or a faint `network` glyph if the tab is showing the existing remote indicator from #0099).
- `reconnecting` — small spinner overlay on the network glyph. Tooltip: `Reconnecting to <host>… (since 12:34:56)`.
- `disconnected` — yellow exclamation icon. Tooltip: `Disconnected: <reason>`.
- `tokenInvalid` — red icon. Tooltip: `Token revoked or expired — click to paste a new token`.
- `folderUnavailable` — yellow icon. Tooltip: `This folder is no longer shared by the host`.

### Inline tab content

When the connection isn't `connected`, the issues list still shows the **last-known** content with a banner across the top:

```
┌─────────────────────────────────────────────────────────┐
│  ⓘ Disconnected — reconnecting to Brennan's MacBook Air │
│     Last updated 12:34:56                  [ Reload now ]│
└─────────────────────────────────────────────────────────┘
```

For `tokenInvalid`:

```
┌─────────────────────────────────────────────────────────┐
│  ⚠ Access token was revoked or has expired              │
│     Generate a new one in Issues.app on the host        │
│                                       [ Paste new token ]│
└─────────────────────────────────────────────────────────┘
```

Clicking "Paste new token" opens the same `RemoteTokenPasteView` from #0096 in a sheet. On success, the WS reconnects via #0102 with the new token.

For `folderUnavailable`:

```
┌─────────────────────────────────────────────────────────┐
│  ⚠ This folder is no longer shared by Brennan's MacBook │
│     Air. Choose another folder or close this tab.       │
│                                  [ Manage subscriptions ]│
└─────────────────────────────────────────────────────────┘
```

Clicking "Manage subscriptions" opens the sheet from #0098.

## Behavior

- All non-connected states show the last-known content beneath the banner. Don't blank the view — the user is reading; transient outages shouldn't erase their context.
- Banners are dismissible (a small `×`) per session. Dismissal lasts until the connection state changes; a reconnect followed by a re-disconnect re-shows the banner.
- "Reload now" forces a manual `reload()` on the source. Useful when the user thinks the WS missed an event (rare, but cheap to expose).

## Verification

- Toggle the host off → banner appears within ~10 seconds (after the WS ping/pong window). Tab badge updates accordingly.
- Toggle the host back on → banner disappears, content refreshes via the resub-driven reload.
- Revoke the token on the host → red banner with "Paste new token" appears.
- Click "Paste new token", paste a valid one → banner disappears and content reloads.
- Toggle a folder off in #0085 → `folderUnavailable` banner appears with "Manage subscriptions" CTA.
- Closing the tab cleanly removes the banner and the source.

## Notes

- The banner is the only place we surface networking errors to the user. Avoid alert dialogs — they steal focus and feel disproportionate for a transient disconnection.
- Don't accumulate multiple banners stacked on top of each other. The latest connection state replaces any previous banner.
- Resist the urge to show technical details in the banner ("WebSocket close 1006", "TCP RST", etc.). Log those to `os.Logger` for debugging, but the banner stays human-readable.
