## Description

Most views in the app currently have no `#Preview`. Only `SettingsView`, `GeneralSettingsView`, and `ShortcutsSettingsView` ship with one. Every other view has to be exercised by running the full app, which is slow and makes design iteration painful — especially across light/dark mode, which is now adaptive (#0020, #0030).

Add `#Preview` blocks to every view file under `Issues/Views/` and `Issues/Views/Settings/` so a developer can hot-reload any single view in Xcode's canvas, in either appearance, in isolation. #0036 is doing a parallel refactor that splits subviews into their own files — the preview work pairs cleanly with that, since each newly-extracted view becomes individually previewable.

## Preview template

For each view, ship three previews so the canvas can show one or both appearances at a glance. Pattern follows the user's reference:

```swift
#Preview("Light & Dark") {
    VStack(spacing: 0) {
        SomeView(/* sample inputs */)
            .frame(maxWidth: .infinity, maxHeight: .infinity)
            .background(.background)
            .environment(\.colorScheme, .light)

        SomeView(/* sample inputs */)
            .frame(maxWidth: .infinity, maxHeight: .infinity)
            .background(.background)
            .environment(\.colorScheme, .dark)
    }
    .ignoresSafeArea()
}

#Preview("Light") {
    SomeView(/* sample inputs */)
        .preferredColorScheme(.light)
}

#Preview("Dark") {
    SomeView(/* sample inputs */)
        .preferredColorScheme(.dark)
}
```

Why all three: "Light & Dark" gives an instant side-by-side comparison; the single-mode previews are easier to interact with (clicks, hover) when iterating on one appearance.

## Sample-data dependency

Many views need a sample `Issue`, `IssueStore`, `TabsModel`, `FolderBookmarkService`, `LintFinding`, or `RememberedFolder` to render. Create one shared helper file so every preview pulls from the same canonical fixtures rather than duplicating constructor calls:

**`Issues/Views/PreviewSamples.swift`** (new — `#if DEBUG` gated so it doesn't ship in release):

```swift
#if DEBUG
import Foundation

enum PreviewSamples {
    static let issue = Issue(
        id: "0042",
        title: "Reply button not functional on post cells",
        status: .inProgress,
        statusRaw: "in-progress",
        module: "Views",
        platform: "macOS",
        firstSeen: Date(timeIntervalSinceReferenceDate: 800_000_000),
        firstSeenRaw: "2026-05-01",
        closed: nil,
        closedRaw: "",
        description: "Tapping reply does nothing. The button highlights but no action fires.",
        fileURL: URL(fileURLWithPath: "/tmp/preview/0042.md"),
        modifiedAt: Date()
    )

    static let issues: [Issue] = [
        issue,
        // a few more across statuses for the swimlane / list / timeline previews
    ]

    static let lintFinding = LintFinding(
        fileURL: URL(fileURLWithPath: "/tmp/preview/0007.md"),
        kind: .missingAttachment(path: "screenshot.png"),
        summary: "0007.md references missing attachment screenshot.png"
    )

    @MainActor
    static func makeStore() -> IssueStore {
        let store = IssueStore(folderURL: URL(fileURLWithPath: "/tmp/preview"))
        // populate via reflection / a test-only seed method, or accept that
        // some store-driven previews will show empty state. See "Caveats" below.
        return store
    }
}
#endif
```

The `IssueStore` exposes `private(set) var issues` — populating it from outside requires either:
- Adding an `#if DEBUG` `seed(_:)` test seam to `IssueStore`, or
- Pointing the preview store at a real on-disk fixtures folder under the bundle (`Bundle.main.url(forResource: "PreviewIssues", withExtension: nil)`) and letting `start()` parse it.

Pick whichever feels less invasive when implementing. The seed-method route is cleaner for unit-test reuse.

## Affected files

Every file in `Issues/Views/` and `Issues/Views/Settings/`. After #0036's sweep finishes, the list will be ~30 files; with each post-#0036 view in its own file, every preview is a tight 30-line block.

Files that already have `#Preview` (from #0024) — leave as-is or upgrade to the three-preview pattern:
- `SettingsView.swift`
- `GeneralSettingsView.swift`
- `ShortcutsSettingsView.swift`

## Caveats

- `IssueMarkdownSheet`, `DetailPanelDescriptionView` read the issue's markdown file from disk via `String(contentsOf: issue.fileURL)`. For a preview, `issue.fileURL` points at `/tmp/preview/0042.md` which doesn't exist — the read fails and the failure pane renders. Either ship a real fixtures folder under the app bundle (recommended) or add an `#if DEBUG` injection seam that lets a preview pass a literal markdown string instead of a URL.
- `MainView`, `RootView`, `TabBarView` need `TabsModel` + `FolderBookmarkService`. Both are `@MainActor @Observable` classes safe to construct in a preview's scope. The bookmark restoration on init reads `UserDefaults`, which is fine in DEBUG.
- The `Window`-based scenes (`FolderPickerSceneView`, the dedicated Help window) don't need previews of their own — they're thin wrappers; preview the inner view (`FolderPickerView`, `HelpView`) instead.
- Views that depend on `AppCommandsController.shared` (e.g. `StatsBarSearchField`, `EmptyMainView`, `GeneralSettingsView`) work in previews because the singleton is reachable; the preview just won't see real data flow through it.

## Out of scope

- `RenderPreview` capture / golden image testing. Just ship the preview blocks; image testing can come later.
- Re-styling any view as part of this work. Behavior and appearance must not change — this is purely a preview-coverage pass.
- Localized variants — every preview uses the same English strings the views ship today.

## Verification

After the implementation lands:
- Open any view file in Xcode, hit ⌥⌘P → the three previews render in the canvas.
- The "Light & Dark" preview shows the same view stacked vertically, light on top, dark on bottom, with no chrome bleeding.
- Tweaking a color in `Assets.xcassets/` updates both halves of the side-by-side preview live.
- The single-mode previews can be interacted with (click, hover) without forcing a full-app rebuild.
