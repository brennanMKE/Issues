## Description

Phase 1 of the remote-access plan (#0076). The host needs a place to mint, store, look up, and revoke access tokens used by viewers. Modeled directly on GitHub Personal Access Tokens: the host generates a token, displays the plaintext to the user once, stores only its SHA-256 hash, and validates incoming bearer tokens by hashing on receipt.

This issue covers the host side only. The viewer side has its own (much simpler) Keychain helpers in #0095.

## Where to look

- **New: `Issues/Remote/AccessToken.swift`** — the model + Keychain helpers, gated `#if os(macOS)` for now (the iOS target won't host).
- **No existing file should change** for this ticket — it's a self-contained service that later issues will call into (#0079 for validation on incoming requests, #0084 for the settings UI).

## Token shape

- Format: `iat_<43-char base64url>` where the body is 32 random bytes (`SecRandomCopyBytes`). The `iat_` prefix makes leaked tokens easy to spot.
- 32 raw bytes → 43 base64url characters (no padding). Total token length is 47 characters.

## Persistence

- A **single Keychain item** holds the entire token database as a JSON-encoded blob. This avoids one Keychain entry per token while still letting us store metadata.
- Item attributes:
  - `kSecClass = kSecClassGenericPassword`
  - `kSecAttrService = "co.sstools.Issues.RemoteAccess.HostTokens"`
  - `kSecAttrAccount = "default"`
  - `kSecAttrSynchronizable = false` for v1 (no iCloud sync; the user transfers tokens manually).
- Blob shape:

  ```swift
  struct TokenRecord: Codable {
      var hash: Data            // 32 bytes — SHA-256 of plaintext token
      var name: String          // user label, e.g. "MacBook Air"
      var createdAt: Date
      var expiresAt: Date?
      var lastUsedAt: Date?
      var lastUsedFrom: String? // reverse-DNS name or IP, optional
  }

  struct TokenDatabase: Codable {
      var records: [TokenRecord]
  }
  ```

- Plaintext is **never** persisted. It is shown to the user once at generation time and discarded.

## API

```swift
struct AccessToken {
    static func generate(name: String, expiresAt: Date?) throws -> (plaintext: String, record: TokenRecord)
    static func revoke(hash: Data) throws
    static func list() throws -> [TokenRecord]
    static func validate(plaintext: String, from peer: String?) throws -> TokenRecord  // throws on miss / expired
    static func touch(hash: Data, from peer: String?) throws  // updates lastUsedAt / lastUsedFrom
}
```

- `validate` hashes the incoming plaintext, looks up by hash, checks `expiresAt`, and returns the record. It does **not** update `lastUsedAt` itself — that's a separate `touch` so the read path can be cheap and the bookkeeping write can happen out of band (or batched).
- Hash comparison is by-key lookup against a `[Data: TokenRecord]` index built once per database load; equal-time comparison isn't needed because the lookup is on the SHA-256 hash, not on the plaintext.

## Errors

- `AccessTokenError.notFound` — no record with this hash.
- `AccessTokenError.expired` — record exists, `expiresAt` is in the past.
- `AccessTokenError.keychain(OSStatus)` — pass through Keychain failures so the UI can surface them meaningfully.

## Verification

- New unit tests in `IssuesTests/`:
  - `generate` produces a 47-char plaintext starting with `iat_`.
  - `generate` then `validate(plaintext)` returns the record.
  - `generate` then `revoke` then `validate` throws `.notFound`.
  - Token with `expiresAt` in the past throws `.expired`.
  - `touch` updates `lastUsedAt` and `lastUsedFrom` and persists.
  - Generating a second token does not invalidate the first (DB grows, doesn't replace).
- After a test run, `security find-generic-password -s "co.sstools.Issues.RemoteAccess.HostTokens"` from a terminal shows the item exists. Tests should clean up after themselves, but a manual `security delete-generic-password -s "co.sstools.Issues.RemoteAccess.HostTokens"` recovers from a stuck state.

## Notes

- We're picking single-item-with-JSON-blob over one-Keychain-entry-per-token (one of the open questions in `RemoteAccess.md` §"What I'd like feedback on first" #2). The single-item shape is simpler and the blob is small (≪ Keychain's per-item limit even with hundreds of tokens). If we ever want per-token revocation auditing through the Keychain itself rather than through the app's logs, we revisit.
- iCloud sync (`kSecAttrSynchronizable = true`) is intentionally **off** for v1 per #0076. If a future issue wants to opt in, it's a single attribute flip.
- The `from peer` field is best-effort. We'll usually have an IPv4/IPv6 string from the connection; reverse-DNS is a follow-up nicety.

## Scope (out)

- No UI in this ticket. The settings sheet is #0084.
- No HTTP plumbing here. Auth middleware that calls `validate` lives in #0079.
- No iOS variant — the iOS target is viewer-only and uses the simpler #0095 helpers.

## Root cause

Greenfield — no prior implementation. The phase-1 plan called for a self-contained host-side token store as the foundation #0079 (auth middleware) and #0084 (settings UI) build on, so this issue creates it from scratch.

## Fix

New `Issues/Remote/AccessToken.swift` (gated `#if os(macOS)`) with the spec'd shape:

- `TokenRecord` and `TokenDatabase` Codable structs as defined in the issue.
- `AccessTokenError` enum with `.notFound`, `.expired`, `.malformed`, `.keychain(OSStatus)`. (`.malformed` was added so a corrupt blob has a distinct error rather than masquerading as a Keychain failure — only reachable through external tampering of the item.)
- `AccessToken` enum (namespace) with `generate`, `revoke`, `list`, `validate`, `touch`, `deleteAll`. Each method takes an optional `service:` parameter that defaults to the production service name; tests pass a unique value to isolate from the live database and call `deleteAll` in `defer` to clean up. Production callers don't see the parameter.
- `generate` produces `iat_<43-char base64url>` from 32 random bytes via `SecRandomCopyBytes`; the host stores only the SHA-256 hash. Length is fixed at 47 characters and exposed via `AccessToken.plaintextLength`.
- `validate` rehashes the incoming plaintext, looks up by hash via `Array.first(where:)` (sufficient for the small-N database the issue calls for), and checks `expiresAt`. It does **not** update `lastUsedAt`; that's `touch`'s job.
- Persistence is a single generic-password Keychain item under service `co.sstools.Issues.RemoteAccess.HostTokens`, account `default`, holding a JSON-encoded `TokenDatabase`. `save` tries `SecItemUpdate` first and falls back to `SecItemAdd` when the item doesn't exist yet. `kSecAttrSynchronizable = false` per #0076.
- Date encoding intentionally uses the default `.deferredToDate` strategy (Double seconds since reference) so `createdAt` / `lastUsedAt` round-trip exactly. The first cut tried `.iso8601`, which dropped fractional seconds and broke the generate→validate equality tests; the comment in the file calls this out.

## Files changed

- `Issues/Remote/AccessToken.swift` — new. The full type, Keychain plumbing, and the static helpers.
- `IssuesTests/AccessTokenTests.swift` — new. 11 tests covering format, generate→validate, unknown-plaintext, revoke→notFound, idempotent revoke, expired, future expiry, touch, touch-on-unknown, multi-record preservation, and load-from-Keychain on a populated database.

## Gotchas

- **Date precision through Codable.** `.iso8601` without `.withFractionalSeconds` rounds to the second and silently breaks any equality check that includes a `Date`. Default `.deferredToDate` round-trips exactly; use it for opaque/internal blobs.
- **Keychain test isolation.** Tests must pass a unique `service:` value and clean up via `deleteAll` — without that the production `co.sstools.Issues.RemoteAccess.HostTokens` item gets clobbered on every run. If a stuck state ever appears, `security delete-generic-password -s "co.sstools.Issues.RemoteAccess.HostTokens"` clears it.
- **`SecItemUpdate` then `SecItemAdd`.** The save path tries update first because that's the common case after generation #2. The order matters: `SecItemAdd` on an existing item returns `errSecDuplicateItem`, not a silent overwrite.
- **`peer` parameter on `validate` is currently unused** but kept in the signature so #0079's auth middleware wiring won't need a signature change. It threads through to `touch` instead, which is where the bookkeeping actually happens.
