# 0128 — Exact-issue-number lookup bypasses the closed-age filter

| | |
|---|---|
| **Status** | open |
| **Module** | Remote / Views |
| **Platform** | macOS |
| **First seen** | 2026-05-12 |

## Description

Pair to #0127. With closed issues older than 90 days filtered out of the live snapshot, the viewer can't see them in any list, swimlane, or timeline — by design. But the user occasionally needs to look one up: a closed issue from six months ago that just got cited in a code review, or a duplicate they want to link to a new ticket.

The escape hatch is **search by exact issue number**: type `0042` (or `42`, or `#42`), the viewer asks the host for that one issue regardless of status or age, and renders it. No filter. No bulk download. One lookup, one issue.

## Behavior

Viewer side:

- The search field accepts an exact id (zero-padded or not): `42`, `0042`, `#42` all resolve to issue `0042`.
- When the input matches the strict `^#?\d{1,4}$` shape, the field offers a "**Look up #0042**" action (Cmd+Return, or just hitting Return when it's the only token in the field).
- Triggering the action calls a new host endpoint: `GET /v1/folder/issue/NNNN/bypass`. The `bypass` suffix is a deliberate signal that the request is not subject to the snapshot's age filter.
- The host returns the raw markdown for that id, including a small `meta` blob with status + closed date so the viewer can label it. On 404 (no such issue), the viewer shows "Issue #NNNN not found."
- The returned issue renders in a dedicated **"Lookup result"** sheet or panel — *not* injected into the visible list, because (a) it'd jump back out on the next reload, and (b) it would lie about the live state.
- Closing the sheet drops the lookup; the user can re-trigger to bring it back.

Host side:

- A new handler for `GET /v1/folder/issue/NNNN/bypass`. The path's `bypass` token is the contract — the handler reads the file regardless of the snapshot filter from #0127.
- Bearer-token auth still required. Bypass is a search affordance, not a permission elevation.
- No WebSocket events for bypassed lookups. The viewer pulls once and that's the end of it; if the issue then changes, the viewer doesn't see the update unless they re-look-up. That's acceptable for an archive-style read.

## What this ticket does *not* do

- It does not add a "show all closed issues" toggle. The filter from #0127 stays absolute for the live view.
- It does not add fuzzy search, full-text search across descriptions, or matching on title keywords. That's a much larger #0007 surface and isn't scoped here.
- It does not write the bypassed issue into the temp-dir mirror. The lookup is a one-shot fetch and lives only in memory for the sheet's lifetime.
- It does not present the bypassed issue in swimlane / timeline / list views.

## Where to look

- `Issues/Remote/RemoteHandlers.swift` — host's request handlers. New endpoint sits next to the existing per-issue fetcher (the existing one would be affected by #0127's filter; this one isn't).
- `Issues/Remote/RemoteWebServerSource.swift` (lands in #0126) — the client adds a `lookup(issueId:)` async method. Doesn't mutate the temp dir or the inner `LocalFolderIssueSource`; returns a one-shot `Issue` for the caller.
- `Issues/Views/` — wherever the search field lives once #0007 ships, or a smaller dedicated entry point if this lands first. The lookup sheet is a new view (`IssueLookupSheet` or similar).
- `Issues/Models/Issue.swift` — possibly add a small "transient" marker on `Issue` so the sheet can render it without the watcher trying to reconcile it against the live list. May not be needed if the sheet just holds a copy locally.

## Acceptance criteria

- Typing `42` (or `0042` / `#42`) in the search field and triggering the lookup action returns the issue's markdown — even when the issue is `closed` and `Closed` is over 90 days old.
- The lookup result renders in a dedicated panel/sheet, never injected into the live list.
- Lookups against a non-existent id show a clear "Issue #NNNN not found" empty state.
- The local `LocalFolderIssueSource` (non-remote tabs) is also supported — for a local tab the lookup just reads the disk file directly, no endpoint involved.
- Build clean; one unit test exercises the host endpoint against a fixture folder containing a 6-month-old `closed` issue and confirms it returns.

## Notes

- The "exact id" shape is intentional: it's the one query where there's zero ambiguity about what the user wants. Anything broader (title fragments, date ranges, status filters) requires UI surface we haven't designed and is out of scope here.
- The `bypass` path component vs. a `?bypass=1` query string: path is louder in logs, easier to grep, easier to lock down later if the host wants a separate audit policy on archive lookups.
- For local-folder tabs the entire flow is offline — the file watcher and the on-disk markdown are right there. The same UX should still work, so the lookup logic lives at the `IssueSource` level and each conformer implements it (local reads disk; remote hits the bypass endpoint).

## Dependencies

- **Pairs with #0127.** Without the age filter, the bypass endpoint is just a duplicate of the regular per-issue fetcher. With it, the bypass is the *only* way to see filtered-out issues.
- **Lands after #0126** (the host's snapshot/issue endpoints exist).
- **Independent of #0007** (full search). This is a much narrower affordance.
