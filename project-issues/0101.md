## Description

Phase 4 of the remote-access plan (#0076). With the WebSocket endpoint in place (#0100), the host needs to wire its existing reload signal into the WS broadcast. Today `IssueStore` reloads in response to `FolderWatcher`'s debounced FSEvents; this ticket forwards each reload as a broadcast to every WS session subscribed to the relevant folder.

This is small but easy to get wrong (cross-actor sends, ordering, dropped events). Treating it as its own ticket so the wiring gets dedicated review.

## Where to look

- **`Issues/State/IssueStore.swift`** — `reload()` is the central choke point.
- **`Issues/State/MultiFolderStore.swift`** (from #0085) — owns the set of stores; gives `RemoteServer` a `folderId → IssueStore` map.
- **`Issues/Remote/RemoteServer.swift`** (from #0079, extended in #0100) — owns the WS sessions and the fan-out map.
- **New: a small `BroadcastBus` actor** (or a method on `RemoteServer`) that takes `(folderId, RemoteEvent)` and pushes to subscribed sessions.

## Behavior

- After an `IssueStore.reload()` completes, the store posts a `RemoteEvent.reload(folderId:)` to the broadcast bus. The bus iterates the fan-out map for that folder and writes the encoded JSON to each session's send queue.
- The bus is `@MainActor` for v1 — broadcast work is small (encode once, write to N sockets). If profiling shows a hot spot, move to a dedicated actor.
- Sessions that fail to write (TCP error, closed) are reaped from the fan-out map and the session is closed.
- The store's existing 150 ms debounce already coalesces save bursts into one `reload()`. The bus emits one event per debounced reload, never per file write — so the broadcast frequency is naturally bounded.
- When `IssueStore.reload()` is triggered by a remote source (#0094), do **not** rebroadcast — only local-source-driven reloads broadcast. Otherwise a viewer loop becomes possible (host A reloads → broadcasts to viewer B → viewer B's source reloads → … on a host-and-viewer instance).

## Per-issue events (optional v1.0, full in v1.1)

The store knows which `id` changed when only one file in a debounce window was touched. For v1 it's fine to always emit `reload` and let the viewer rescan; per-issue `update` / `delete` events are a small follow-on optimization (less data on the wire, smoother viewer UX).

If we land per-issue events in this ticket:

- `update(folderId, id)` when a file matching `^\d{4}\.md$` was created or modified.
- `delete(folderId, id)` when one was removed.
- Fall back to `reload(folderId)` when more than one file changed in a debounce window, or on any non-`.md` change inside the folder root.

## Verification

- Edit a file on the host while a viewer is subscribed → viewer receives `reload` (and/or `update`) within the existing 150 ms debounce + WS write latency.
- Subscribe two viewers to the same folder, edit a file → both receive the event.
- Subscribe to folder A on viewer 1 and folder B on viewer 2 → editing in A reaches only viewer 1; editing in B reaches only viewer 2.
- Disconnect viewer 1 (close WS) — the host stops trying to send to it on the next event; no spinning, no log spam.
- Edit during a host-and-viewer instance (Issues.app hosting and viewing in different tabs) — the local viewer side does not receive duplicates from the broadcast loop.

## Notes

- The fan-out path runs whether or not anyone is subscribed. That's fine — the iteration is over an empty list when no one's listening, and we don't pay for the encode unless we have a recipient.
- Don't reach across `IssueStore`s to compute diffs; if the per-issue events get tricky, fall back to `reload`.

## Scope (out)

- WS endpoint and message decoding — #0100.
- Viewer-side handling and reconnect — #0102.
- Backpressure for slow viewers — out of v1; if a viewer can't keep up, dropping the connection is acceptable.
