# 0105 — Network-change pause behavior on the host via NWPathMonitor

| | |
|---|---|
| **Status** | open |
| **Module** | Services |
| **Platform** | macOS |
| **First seen** | 2026-05-08 |

## Description

Phase 5 of the remote-access plan (#0076). Today (after #0083) the host listens on every non-loopback interface for as long as hosting is enabled. If the user toggles hosting on at home and later joins a coworking Wi-Fi, the service is still publishing on whatever the new network is — which is almost certainly not what they want.

The fix is `NWPathMonitor`-driven: when the network path changes in a way that suggests the user has moved networks, **pause** publishing and broadcasting and surface a clear "paused on this network — re-enable to publish" prompt. The user re-enables in one click; the toggle preserves the previous-on state.

## Where to look

- **`Issues/Remote/RemoteServer.swift`** (from #0079) — owns the `NWListener` and the `RemoteServer.enabled` state. Add an `enabled` vs. `paused` distinction.
- **`Issues/Views/RemoteHostSettingsView.swift`** (from #0083) — surface the paused state as a banner with a re-enable CTA.

## Behavior

### What counts as a "network change"

`NWPathMonitor` exposes a `NWPath` with `availableInterfaces`, `gateways`, and a status. A change worth pausing on is:

- The set of `NWInterface.InterfaceType` values changes (e.g. `wifi` → `wifi + wiredEthernet` is fine; `wifi` → `wifi + cellular` is suspicious if cellular is actively used as the gateway).
- The `gateways` set changes by adding or removing an SSID-comparable path. We can't reliably read SSIDs from a sandboxed app without prompts; instead, hash the gateway set (IP + interface type) and treat any change as "moved networks."

When this happens **while hosting is on**, transition to `paused`.

### What `paused` means

- `NWListener` is stopped — no incoming connections accepted, no Bonjour publishing.
- The `RemoteServer.enabled` *intent* stays `true` so the user's previous choice isn't lost.
- All connected viewers receive a WS close (#0100) with a non-error code (1001 going away). They handle reconnect on their side; if the host re-enables, they reconnect via #0102.
- The settings banner (described below) tells the user what happened.

### Re-enable

From the banner, one click re-enables hosting on the new network. Internally this is a fresh `NWListener.start()` plus Bonjour re-publish.

Optionally, remember "this network is OK" so the next time the user joins it we don't pause again. Hash the `(gateway-set, interface-types)` tuple and store an allowlist in `UserDefaults` under `RemoteServer.allowedNetworks`. On `NWPath` change, check the hash:

- Hash matches an allowlisted network → don't pause.
- Hash is new → pause and prompt.

The first time the user toggles hosting on, that network's hash is added to the allowlist automatically. So "join home Wi-Fi, toggle on, leave for coffee shop" pauses; "rejoin home Wi-Fi" doesn't.

## UI

In the host settings sheet (#0083), add a dismissible banner above the toggle when paused:

```
┌─────────────────────────────────────────────────────────┐
│  ⏸ Hosting paused — your network changed                │
│     Joined a new Wi-Fi network. Re-enable to publish    │
│     issues on this network.                             │
│                                          [ Re-enable ]  │
└─────────────────────────────────────────────────────────┘
```

The main toggle continues to show on/off matching `RemoteServer.enabled`'s *intent*; the badge state on the toggle row indicates whether it's actively running or paused.

## Verification

- Toggle hosting on at home (Wi-Fi A). It runs.
- Switch to a different Wi-Fi (Wi-Fi B) → host pauses within a couple of seconds. Banner appears.
- Click Re-enable → hosting starts on Wi-Fi B. The current network is added to the allowlist.
- Toggle back to Wi-Fi A → no pause (it's allowlisted).
- Toggle to Wi-Fi C (new) → pauses, prompts. Click Re-enable → C is allowlisted.
- Pause via this mechanism, then quit the app → on relaunch with the same network, host runs (because that network is allowlisted from the most recent re-enable). On a different network, host is paused with banner.

## Notes

- The viewer side doesn't need its own `NWPathMonitor`-driven pause; the WS reconnect logic in #0102 handles all the disconnect/reconnect mechanics naturally.
- This is about *publishing surface area*, not about security. The auth layer already protects the data; pause-on-network-change is a sane default that prevents the user from realizing two days later that their personal Mac was advertising on a coworking Wi-Fi all along.
- Don't try to detect specific SSIDs without `CoreLocation`-style permission — too privacy-heavy for this feature.

## Scope (out)

- A "list of allowlisted networks" management UI. Probably one row in settings: "Forget allowlisted networks". Don't grow it into a network manager.
- VPN-aware logic ("only host when Tailscale is up"). Out of v1; if it becomes a real need, a follow-up issue.
- Viewer-side pause behavior — not needed.
