## Description

The codebase has `IssuesTests/MarkdownIssueParserTests.swift` and a stub `IssuesTests/IssuesTests.swift`, plus the UI tests, but the three pieces of pure logic most likely to regress as features land are uncovered:

1. **`IssueStore.apply(_:)`** — restores a `TabPersistedState` and silently drops invalid status filters / unknown enum raw values / stale module/platform/issue-id references. Per-tab state restore (#0009) depends on this being exact.
2. **`LintRunner.run(folderURL:parsedIssues:)`** — produces five kinds of `LintFinding`. Each has subtle conditions (parse failure reasons, code-span stripping for image refs, orphan folders matching `^\d{4}$`, duplicate-id detection).
3. **`TabsModel.diff(baseline:new:issues:)`** — produces additions / removals / status changes between snapshots; drives notification firing (#0013). Body-only edits intentionally don't surface — that's a contract worth pinning down with a test.

## What to test

### `IssueStoreTests`

- Round-trip: persist a state, apply it back to a freshly-populated store, assert every field matches.
- Filter validation:
  - Unknown status raw values are dropped silently.
  - Stale `moduleFilter` (no current issue with that module) → cleared to nil.
  - Stale `platformFilter` → cleared to nil.
  - Stale `selectedIssueID` (issue no longer in list) → cleared to nil.
- Enum default fallbacks: a corrupt `viewMode` raw value falls through to existing default rather than crashing.
- `selectNext()` / `selectPrevious()` wrap-around at both ends; empty `visibleIssueOrder` is a no-op.

### `LintRunnerTests`

- Parse-failure finding for each of the three `MarkdownIssueParseError` cases (`.filenameMismatch`, `.missingTitle`, `.missingStatus`).
- Orphan-folder finding fires when a `0042/` directory has no matching `0042.md`, doesn't fire when `0042.md` exists.
- Missing-attachment finding fires for a `![cap](missing.png)` reference, doesn't fire for `https://...` external URLs, doesn't fire for matches inside fenced code blocks (`stripCodeSpans` coverage).
- Unknown-status finding for `in_progress` (underscore) but not for the canonical `in-progress`.
- Duplicate-id finding produced once per *extra* duplicate (not for the primary).

### `TabsModelTests`

- `diff` returns empty when baseline and new are identical.
- New id in `new` not present in baseline → addition.
- Id in baseline gone from `new` → removal (placeholder Issue with last-known status).
- Same id, status changed → exactly one statusChange entry, never an addition/removal.
- Body-only edit (same id, same status, different `modifiedAt`) → no entries surface.
- Multiple kinds of changes simultaneously → all classified correctly.

## Implementation notes

- All three targets are pure value computations or read-only file IO; no UI, no SwiftUI, no `@Observable` state mutation needed in tests beyond the store init for `IssueStore.apply`.
- `LintRunnerTests` needs a temp-directory fixture: write a small set of `NNNN.md` and `NNNN/` folders to a unique subdir in `FileManager.default.temporaryDirectory`, run `LintRunner.run(...)`, tear down. Existing parser tests already use this pattern via `String`-based parse, so a shared helper might be worthwhile.
- Use Swift Testing (`@Test`, `#expect`, `@Suite`) — modern syntax, doesn't drag XCTest assertions into new code.

## Out of scope

- Snapshot tests of view rendering.
- UI tests for keyboard nav (#0049 verification was manual; that's intentional for now).
- Integration tests that spin up an actual `FolderWatcher` — too dependent on FSEvents timing.

## Root cause

Three pieces of pure value computation — `IssueStore.apply(_:)`, `LintRunner.run(folderURL:parsedIssues:)`, and `TabsModel.diff(baseline:new:issues:)` — drive user-visible behavior (per-tab restore #0009, lint banner #0019, change notifications #0013) and had no test coverage. Each has subtle branches (silent drops of stale references, code-span stripping, body-only edit suppression) that are easy to break inadvertently as features land.

## Fix

Added three new Swift Testing test files to `IssuesTests/`:

- `IssueStoreTests.swift` — 13 tests covering `apply(_:)` round-trip, filter validation (status / module / platform / selectedIssueID), enum fallback for corrupt `viewMode` / `sortColumn`, and `selectNext()` / `selectPrevious()` wrap-around (including the empty-list no-op).
- `LintRunnerTests.swift` — 11 tests covering parse-failure findings (`.missingTitle`, `.missingStatus`; `.filenameMismatch` is unreachable post-gate, so the gate is exercised instead), orphan-folder pos/neg, missing-attachment pos/neg/external-URL/code-span, unknown-status for `in_progress` underscore vs. canonical `in-progress`, and duplicate-id firing once per *extra* duplicate.
- `TabsModelTests.swift` — 6 tests covering `diff` identity, addition, placeholder removal with last-known status, status-change classification, body-only edit suppression, and a mixed-changes scenario.

`TabsModel.diff` was lifted from `private func` to `static func` so the suite can call it directly. The function never read instance state, so the visibility change is contained and the call site in `handleReload(_:)` keeps working via `Self.diff(...)`.

All 30 new tests pass alongside the existing parser suite.

## Gotchas

- Swift Testing exposes its own `Issue` type, which collides with the app's `Issue` model. Tests qualify the model as `Issues.Issue` everywhere a return type or parameter type would otherwise be ambiguous.
- `TabsModel` is `@MainActor`-isolated, so `static` methods on it inherit main-actor isolation. `TabsModelTests` is therefore `@MainActor`-annotated.
- `IssueStore.setIssuesForPreview(_:)` is `#if DEBUG`-gated; tests run in Debug, so it's available as the seam for populating the in-memory issue list without touching disk.
- `LintRunner.run(...)` accepts `parsedIssues` as a parameter, which lets the duplicate-id test synthesize same-id collisions even though the on-disk filename gate makes this impossible in practice.
