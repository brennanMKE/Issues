## Description

The Timeline view shows per-issue bars across an absolute date range, which is great for "where did issue 0042 sit in the calendar" but nothing for "what's the project's activity rhythm?" Add a small sparkline-style chart above the timeline header that shows opened-vs-closed counts per day across roughly the last 30 days. It turns the dashboard into a glance-able activity gauge in one extra row.

## Visual shape

- A horizontal strip ~36pt tall, full-width.
- Two stacked tiny bar series per day:
  - Opened (status went `open` / first seen on this date) → `Color.statusOpen`.
  - Closed (status went `resolved` / `closed` on this date) → `Color.statusResolved`.
- Y-axis is the count for that day (no labels needed; relative bar height conveys the picture).
- A 1pt baseline at the bottom edge in `Color.appBorder.opacity(0.4)`.
- Hover a bar → tooltip showing the date + the two counts.

## Implementation sketch

Use SwiftUI Charts (`import Charts`):

```swift
struct TimelineActivitySparkline: View {
    let issues: [Issue]
    let dayCount: Int = 30

    var body: some View {
        Chart(buckets) { bucket in
            BarMark(
                x: .value("Day", bucket.day, unit: .day),
                y: .value("Opened", bucket.opened),
                stacking: .standard
            )
            .foregroundStyle(Color.statusOpen)
            BarMark(
                x: .value("Day", bucket.day, unit: .day),
                y: .value("Closed", bucket.closed),
                stacking: .standard
            )
            .foregroundStyle(Color.statusResolved)
        }
        .chartXAxis(.hidden)
        .chartYAxis(.hidden)
        .frame(height: 36)
        .padding(.horizontal, 16)
    }

    private var buckets: [(id: Date, day: Date, opened: Int, closed: Int)] {

    }
}
```

Drop into `TimelineView.body` above the `TimelineTickHeaderView`.

## Edge cases

- New folder with only a few issues → chart is mostly empty bars + the baseline. Acceptable; the sparse picture is itself informative.
- Very large folder (hundreds of opens/closes per day): cap bar height visually but still let the user see the relative shape. Stacking + auto-scale handles this for free.
- `accessibilityDifferentiateWithoutColor`: differentiate the two series with a 1pt-tall pattern stroke or a label legend, not just hue.
- Reduce Motion: don't animate bars on first appear; render statically.

## Out of scope

- Click a bar to filter the list to that date range — could come later.
- Pinch-zoom the time range — out of scope; the sparkline is intentionally fixed-width.
- Showing in non-Timeline views.

## Root cause

Not a bug — feature gap. The Timeline view rendered per-issue bars but had no aggregate, time-aware view of activity. There was no place in the dashboard that answered "how busy has the project been recently?".

## Fix

Added `TimelineActivitySparkline`, a Charts-based stacked bar chart that buckets the supplied issues into the last 30 days using `firstSeen` (opened) and `closed` (closed). Dropped it into `TimelineView.body` immediately above `TimelineTickHeaderView`, fed from the same `filteredIssues` slice so it reflects the active status / module / platform filters (v1 default per the spec).

Implementation details:
- 30 daily buckets ending today; days with no activity render as empty space, with a 1pt baseline at the bottom edge in `Color.appBorder.opacity(0.4)` so the strip remains visible.
- Two `BarMark` series per day, stacked: opened in `Color.statusOpen`, closed in `Color.statusResolved`.
- Hidden X/Y axes and legend — the sparkline conveys shape, not exact counts.
- `accessibilityDifferentiateWithoutColor` dims the closed bars slightly so the two series don't rely on hue alone.
- `accessibilityReduceMotion` disables the bar-update animation.
- A `.help(...)` summary tooltip shows the 30-day totals.

## Gotchas

- `Issue` already exposes `firstSeen` and `closed` as parsed `Date?`, so no schema change was needed. Issues with malformed dates simply don't appear in any bucket — fine for v1.
- Bucketing uses a UTC calendar (matching `TimelineTickHeaderView`'s tick alignment) so the sparkline's day boundaries line up with the rest of Timeline view's date math, not the user's local timezone. If we ever want timezone-correct "today" highlighting we'd need to revisit both call sites together.
- `firstSeen` isn't a true "opened-on-this-date" event — it's the date the issue was first reported. For this project that's effectively the same thing; for projects that backdate their `First seen`, the sparkline will show the backdate, not the file-creation date.
- Per-status-change history isn't tracked in the markdown, so "closed on day N" relies on the `**Closed**` row being filled in. Issues that were resolved/closed without a `**Closed**` row simply won't appear in the closed series — the issue guide already requires that row, so this is consistent with the rest of the app.
