## Description

The main window's title is hardcoded to "Issues" via `Window("Issues", id: "main")` in `IssuesApp.swift`. With multi-tab support (#0029), the user often has 2–3 tabs open across different projects; the title bar tells them nothing about which one is active. Surface the active tab's repo name plus a "filtered / total" count so the title bar carries useful context.

Target appearance:

- **Title:** active tab's `repoName` (e.g. `BucketKit`)
- **Subtitle:** `"\(filteredIssues.count) of \(issues.count) issues"`, or just `"\(count) issues"` when no filter is active.

When no tab is active (post-launch with nothing restored, or all tabs closed), fall back to `"Issues"` and no subtitle.

## Implementation sketch

`Window` doesn't take a binding for its title argument, so the title needs to be driven from inside `RootView` or `MainView` via `.navigationTitle(_:)` and `.navigationSubtitle(_:)`. On macOS, both modifiers update the window chrome.

```swift
// MainView.body
VStack(spacing: 0) { ... }
    .navigationTitle(store.repoName)
    .navigationSubtitle(subtitle)

private var subtitle: String {
    let filtered = store.filteredIssues.count
    let total = store.issues.count
    if filtered == total {
        return total == 1 ? "1 issue" : "\(total) issues"
    }
    return "\(filtered) of \(total)"
}
```

For the no-tab case, `EmptyMainView` sets `.navigationTitle("Issues")` and no subtitle.

## Edge cases

- Switching tabs: `.navigationTitle` is keyed off the view that's mounted. Since `RootView` swaps the active store via `MainView(store: active, ...)` the title updates as a side effect of tab changes. ✓
- Localization (#0061-#0063 will follow): wrap the subtitle string keys in `LocalizedStringKey` once xcstrings is set up.
- Inflection: prefer `Text("^[\(count) issue](inflect: true)")` per modern SwiftUI guidance. `.navigationSubtitle` wants `String`, so format outside.

## Verification

- Launch with one tab restored → title shows the repo name, subtitle shows the count.
- Switch tabs → title + subtitle update immediately.
- Apply a status filter → subtitle becomes "X of Y".
- Clear all filters → subtitle becomes "Y issues".
- Close all tabs → title falls back to "Issues", subtitle clears.

## Root cause

The main window's title was hardcoded via `Window("Issues", id: "main")` in `IssuesApp.swift`, so the title bar carried no per-tab context. With the multi-tab work from #0029, the user could have several different repos open with no way to tell from window chrome which tab is active.

## Fix

Drive the window title from inside the mounted view. `MainView` now sets `.navigationTitle(store.repoName)` and `.navigationSubtitle(windowSubtitle)`, where `windowSubtitle` returns `"\(filtered) of \(total)"` when filters/search narrow the list, or `"\(total) issues"` (singular `"1 issue"`) when nothing is filtered. `EmptyMainView` sets `.navigationTitle("Issues")` with an empty subtitle so the no-tab state falls back to the original chrome.

Switching tabs reuses the same `MainView` with a different `store`, so SwiftUI re-evaluates the modifiers and the title/subtitle update as a side effect of the tab change. No extra wiring needed.

## Files changed

- `Issues/Views/MainView.swift` — adds `.navigationTitle` / `.navigationSubtitle` and the `windowSubtitle` computed property.
- `Issues/Views/EmptyMainView.swift` — pins the no-tab title to `"Issues"` with no subtitle.

## Gotchas

- `Window(_:id:)` doesn't take a binding for its title argument; the title has to be driven from inside the mounted view via `.navigationTitle`. The `"Issues"` literal in `IssuesApp.swift` is now just the fallback used when no view sets a title.
- `.navigationSubtitle` wants a plain `String`, not `LocalizedStringKey`, so the formatted count is built outside the modifier. When localization lands (#0061-#0063) the format will move into xcstrings.
