# 0015 — Change an issue's status from a right-click menu

## Description

Right-clicking an issue (in any view) shows a context menu with **Set Status →** submenu containing Open / In Progress / Resolved / Closed / Won't Fix. Picking one rewrites the markdown file's `**Status**` row in place. Adds a `**Closed**` row with today's date when transitioning to `resolved` or `closed`.

## Notes

- App is currently read-only. This is the first write path — keep the surface tiny.
- Editing the status row needs a small `MarkdownIssueWriter` service: read file, regex-replace the Status field row preserving everything else, write back atomically (`Data.write(to:options:.atomic)`).
- Sandbox: needs `com.apple.security.files.user-selected.read-write` (currently read-only). Update entitlements; this is a deliberate scope expansion.
- Honor the IssuesSkill rule: only the *user* moves an issue to `closed`. The submenu is fine — the user chose it manually.
- After write, the `FolderWatcher` will pick up the change and the `IssueStore` reloads automatically.
- ListView / Swimlane / Timeline / Recent each need `.contextMenu` on the row.

## Resolution — won't fix

Issues.app is intentionally a **read-only** viewer of the markdown files. Only AI agents (via the IssuesSkill) write or edit issue files; the app is the live dashboard. Adding a write path — status edits, drag-and-drop attachments, anything that mutates the markdown — would expand the sandbox entitlements and split the source of truth between the agent and the GUI. Closing as `wontfix`. Same reasoning applies to #0016 (drag-and-drop attachments).
