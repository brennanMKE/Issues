## Description

Today every user-facing string is an inline `Text("...")` literal in English. That's fine for a personal-use app but blocks any future translation work. Add modern Apple localization scaffolding via `Localizable.xcstrings` (the JSON-backed string catalog introduced in Xcode 15) so subsequent issues (#0060–#0062) can drop in language-specific translations without further app-side edits.

## What this issue covers

1. Create `Issues/Localizable.xcstrings` (single file at the source-folder root) with `extractionState: "manual"` so Xcode doesn't auto-extract every literal — we want curated symbol keys, not a wall of automatically-extracted English strings.
2. Set base language to `en` and add `es`, `fr`, `de` as additional supported locales (the actual translations land in #0060–#0062, but the slots need to exist now so each language issue is purely a content drop).
3. Audit every user-facing English literal in the app and convert them to localized lookups via either:
   - `Text("symbolKey")` with `LocalizedStringKey` interpolation, or
   - The generated symbol API: `Text(.symbolKey)` once the file is wired up.
4. Pick the symbol-key style and document it in CLAUDE.md so subagents stay consistent.

## Out of scope here

- Translations themselves (covered by #0060 Spanish, #0061 French, #0062 German).
- Localizing the issue markdown bodies (those are user content, not app chrome).
- Localizing the `IssuesSkill` markdown-format spec — that's documentation, separate effort.

## Caveats

- `Text("^[\(count) issue](inflect: true)")` is the right shape for plural-aware counts — ship that pattern wherever the UI shows "N issues".
- `LocalizedStringKey` interpolation: prefer `Text("Status: \(status.displayName)")` over manual concatenation so the catalog can position interpolations correctly per language.
- `String(localized:)` for cases where we need a `String`, not a `Text` (e.g. inside `NSPasteboard`, log strings — though logs should stay English for debuggability).

## Verification

- `Localizable.xcstrings` exists in `Issues/` with `en`, `es`, `fr`, `de` slots.
- App builds cleanly; English strings render unchanged because the catalog's `en` entries match the literals.
- Switching the system language to a not-yet-translated locale falls back to English (Apple's default behavior). Once #0060–#0062 land, those locales render in their language.

## Root cause

Not a bug — feature work. Every user-facing literal was inline English with no catalog to back it, so any future translation work would require editing every view. Apple's modern path is the JSON-backed `Localizable.xcstrings` catalog (Xcode 15+), which is what this issue scaffolds.

## Fix

Created `Issues/Localizable.xcstrings` with `sourceLanguage: "en"`, all entries marked `extractionState: "manual"` (curated keys only — no auto-extraction wall), and `en`/`es`/`fr`/`de` localizations populated in the same pass so #0060–#0062 ship together. Style decision: keys are the English source string (Apple's recommended default for `LocalizedStringKey`), so `Text("Open Folder")` resolves through the catalog automatically without changing any view code. Three enum `displayName: String` accessors (`IssueStatus`, `IssueStore.ViewMode`, `IssueStore.AttachmentFilter`) needed explicit `String(localized:)` wrappers because their callers consume `String`, not `Text`. Added `es`, `fr`, `de` to `knownRegions` in `project.pbxproj` so Xcode recognises the locales the catalog declares.

Two `GenerateStringSymbols` collisions surfaced on the first build: "Close tab" vs "Close Tab" (case collapse), and "Rotate certificate?" vs "Rotate certificate…" (punctuation collapse). Normalised the former to "Close Tab" project-wide and renamed the confirmation-dialog title to "Confirm certificate rotation".

## Files changed

- `Issues/Localizable.xcstrings` (new) — curated string catalog with ~70 keys × 4 locales.
- `Issues.xcodeproj/project.pbxproj` — added `es`, `fr`, `de` to `knownRegions`.
- `Issues/Models/IssueStatus.swift` — wrapped each `displayName` case in `String(localized:)`.
- `Issues/State/IssueStore.swift` — wrapped `ViewMode.displayName` and `AttachmentFilter.displayName` in `String(localized:)`.
- `Issues/Views/ServerCertificateSection.swift` — renamed confirmation-dialog title to dodge symbol collision.
- `Issues/Views/TabChipView.swift`, `Issues/Views/ManageSubscriptionsSheet.swift` — normalised "Close tab" → "Close Tab".

## Gotchas

- The `xcstrings` symbol generator collapses case differences and trailing `?`/`…` into the same Swift symbol — two keys that differ only by those will fail the build with "would generate the same symbol as …". The fix is either renaming a key or using a per-key `shouldGenerateSymbol: false` override; we chose renames for clarity. If a future contribution adds another colliding pair, the build will catch it.
- `String(localized:)` resolves at runtime against the *bundle that owns the calling code* — for a single-target app this is fine, but if any of these displayName accessors ever move into a Swift package, you'll need `String(localized:, bundle: .module)`.
- The catalog stores English as both the key *and* the `en` localization value. This is redundant but Apple's recommended pattern — without an explicit `en` entry, `String(localized:)` falls back to the key, which is brittle if a key ever gets renamed.
