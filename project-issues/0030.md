# 0030 — Move Theme palette into the Asset catalog

## Description

The palette in `Issues/Theme/Theme.swift` is currently defined inline — `Color(hex:)` for status colors, `Color(light:dark:)` for the `app*` background/text/border/muted constants, plus `Color.accentColor` for the accent. Move every named color into `Assets.xcassets/` as a `Color Set` and reference each via `Color("name", bundle: .main)`. That way the user can tweak any palette value directly from Xcode's color picker without editing Swift.

## What changes

`Theme.swift` shrinks to one-liners:

```swift
extension Color {
    static let appBackground       = Color("appBackground")
    static let appBackgroundCard   = Color("appBackgroundCard")
    static let appBackgroundHover  = Color("appBackgroundHover")
    static let appBorder           = Color("appBorder")
    static let appText             = Color("appText")
    static let appMuted            = Color("appMuted")
    static let appAccent           = Color.accentColor              // already in asset catalog
    static let appAccentDim        = Color.accentColor.opacity(0.6) // derived from accent

    static let statusOpen          = Color("statusOpen")
    static let statusInProgress    = Color("statusInProgress")
    static let statusResolved      = Color("statusResolved")
    static let statusClosed        = Color("statusClosed")
    static let statusWontfix       = Color("statusWontfix")
}
```

The existing `Color(hex:)` initializer can stay — it's a useful utility — but it's no longer load-bearing for the theme.

## Asset catalog entries

Each color becomes an `Assets.xcassets/<name>.colorset/Contents.json` file. Color sets support light/dark variants natively via `appearances: [{ appearance: "luminosity", value: "dark" }]` entries.

### Adaptive `app*` colors (Any + Dark appearance)

| Name | Light | Dark |
|---|---|---|
| `appBackground` | `#FFFFFF` | `#0D1117` |
| `appBackgroundCard` | `#F5F5F7` | `#161B22` |
| `appBackgroundHover` | `#EBEBED` | `#1C2230` |
| `appBorder` | `#D2D2D4` | `#30363D` |
| `appText` | `#0A0A0F` | `#E6EDF3` |
| `appMuted` | `#6B7280` | `#8B949E` |

### Status colors (single value, used unchanged in both light + dark)

| Name | Value |
|---|---|
| `statusOpen` | `#F59E0B` |
| `statusInProgress` | `#3B82F6` |
| `statusResolved` | `#10B981` |
| `statusClosed` | `#6B7280` |
| `statusWontfix` | `#EF4444` |

If any status color reads poorly in light mode in practice, you can add a dark-appearance variant later — same Asset catalog UI, no code changes.

## Implementation notes

- The colorset directories can be created on disk; Xcode's file-system-synchronized group picks them up automatically (same way it handles new `.swift` files). No `project.pbxproj` edits needed.
- The `Contents.json` schema for a single-value color uses `idiom: universal` with no `appearances` field. For light/dark, two entries — the second carries `appearances: [{ appearance: "luminosity", value: "dark" }]`.
- `AccentColor.colorset` already exists in the catalog (set to purple per #0023). Don't touch it.
- `Color("appBackground")` looks up the named asset in the main bundle by default; explicit `bundle: .main` is unnecessary.
- After the migration, the `Color(light:dark:)` extension function and the `Color(hex:)` initializer are still defined in `Theme.swift` but unused at the theme level. Keep `Color(hex:)` (cheap utility, may be used elsewhere); remove the `Color(light:dark:)` helper since the asset catalog covers that case now and there's no other caller.

## Verification

After the migration, the app should look identical in both light and dark mode. Open `Assets.xcassets` in Xcode → confirm each new color set has the right values (and the right appearance variants). Pick a color, change it via Xcode's color picker, rebuild → the new value appears in the running app without a code edit.

## Out of scope

- Adding more colors to the palette (e.g. semantic "warning" / "danger" beyond the existing status hues).
- Per-platform variants (iOS / iPadOS) — the app is macOS only.
- High-contrast appearance variants. Defer until a real accessibility need surfaces.
