# 0027 — Lint pane flags inline-code examples as missing attachments

## Description

The lint runner from #0019 reported `#0016` as having a missing attachment, even though the only `![…](…)` reference in that file was an illustrative example wrapped in inline backticks (`` `![filename](NNNN/filename.png)` ``). The image-ref regex in `extractImagePaths(from:)` was scanning the raw markdown source and matching that example as a real attachment reference, then flagging it because `NNNN/filename.png` doesn't exist on disk.

## Steps to reproduce

1. Author an issue file whose prose includes an example of attachment syntax inside backticks, e.g. `` `![cap](0042/foo.png)` ``.
2. Reload the folder.
3. Lint pane shows a "missing attachment" finding for that example, even though Textual renders the example as inline code (not an image).

## Expected behavior

The linter should treat inline-code spans the same way the markdown renderer does — as text, not as live markdown — and skip them when scanning for image references. Same for fenced code blocks (``` … ```).

## Actual behavior

`LintRunner.extractImagePaths(from:)` ran the image-ref regex over the raw file contents, matching anywhere including inside backticks. False positives surfaced in the lint pane on every reload.

## Fix

Strip code spans before regexing. Pre-process `text` with two passes:

```swift
private static func stripCodeSpans(_ text: String) -> String {
    var stripped = text
    stripped = stripped.replacingOccurrences(
        of: #"```[\s\S]*?```"#,
        with: "",
        options: .regularExpression
    )
    stripped = stripped.replacingOccurrences(
        of: #"`[^`\n]*`"#,
        with: "",
        options: .regularExpression
    )
    return stripped
}
```

Then `extractImagePaths(from:)` regexes the stripped string. Fenced blocks are handled first so a ``` … ``` containing a single backtick doesn't get mangled by the inline pass. Inline spans are constrained to a single line so an unterminated stray backtick doesn't swallow the rest of the file.

The fix is positionally lossy (the stripped string has different ranges than the original) but `extractImagePaths` only returns the captured paths — no source-range consumers — so this is fine.

## Notes

- The renderer already treats backticks as code spans, so the lint behavior now matches what users see in the Textual sheet.
- A more robust path would be a real CommonMark parser, but the regex-based linter is intentionally simple and the strip-then-regex approach catches the realistic cases without adding a dependency.
- No changes to issue files were needed — the example in #0016 is correctly authored.
