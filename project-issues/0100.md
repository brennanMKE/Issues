## Description

Phase 4 of the remote-access plan (#0076). Until now the viewer can refresh on demand but has no live updates — edits on the host show up only on a manual reload. This ticket adds a WebSocket endpoint that the viewer connects once and stays subscribed to, with per-folder subscribe/unsubscribe messages so a single connection covers any subset of the host's folders.

## Where to look

- **`Issues/Remote/RemoteServer.swift`** (from #0079) — adds a route handler that performs the HTTP→WS upgrade.
- **New: `Issues/Remote/WebSocketSession.swift`** — server-side per-connection state: peer info, subscribed `folderIds`, send queue.
- **`Issues/Services/FolderWatcher.swift`** + **`Issues/State/IssueStore.swift`** — already debounced; on reload, broadcast events to all WS sessions subscribed to the relevant folder.

## Wire shape

### Upgrade

`GET /v1/events` with the standard WebSocket upgrade headers and `Authorization: Bearer iat_<…>`. The same auth middleware as everything else (#0079); a 401 is returned **before** the upgrade succeeds.

### Client → Server

```json
{ "type": "subscribe",   "folderIds": ["a3f1…", "97c0…"] }
{ "type": "unsubscribe", "folderIds": ["a3f1…"] }
{ "type": "ping" }
```

- Subscribe replaces the current subscription set is **additive** — to drop a folder, send `unsubscribe` for it explicitly.
- Pings are optional. The server doesn't require them; it sends its own pings to keep NATs alive (see below).

### Server → Client

```json
{ "type": "hello",    "displayName": "…", "version": 1 }
{ "type": "reload",   "folderId": "a3f1…" }
{ "type": "update",   "folderId": "a3f1…", "id": "0042" }
{ "type": "delete",   "folderId": "a3f1…", "id": "0042" }
{ "type": "unsubscribed", "folderId": "a3f1…", "reason": "host_unshared" }
{ "type": "pong" }
```

- `hello` is sent immediately after the upgrade, so the viewer knows the host's display name and protocol version.
- `reload` is the catch-all "rescan this folder" event, fired when `IssueStore.reload()` runs and the diff is non-trivial.
- `update` and `delete` are emitted when the host has the per-issue diff cheap (small folder reload, single-file change). For the v1 implementation it's fine to always send `reload` and skip the per-issue events; the viewer-side handler in #0102 should accept either pattern.
- `unsubscribed` fires when the host removes a folder from the share list (#0085) or the folder bookmark goes stale.

### Keepalives

- Server sends a WS ping every 30 seconds. If no pong returns within 10 seconds, close the connection. The viewer reconnects via #0102.

## Behavior

- Multiple viewers can be connected at once. Each `WebSocketSession` keeps its own `subscribedFolderIds: Set<String>`.
- The host maintains a small fan-out map: `folderId → [WebSocketSession]`. Reload events iterate the relevant entry; for ~10 viewers and ~10 folders this is trivial.
- The existing 150 ms FSEvents debounce in `FolderWatcher` means a save burst becomes one `reload()` call which becomes one broadcast — no flooding.
- An unauthenticated WS upgrade attempt → 401 before upgrade. Connections that lose their token via host-side revocation (#0084) are closed by the server with status code 4001 ("token revoked"). The viewer surfaces the expired-token UI (#0104).

## Verification

- Smoke-test CLI (#0087) opens `/v1/events`, sends `subscribe`, edits a file on the host, receives `reload` within ~200 ms.
- Open two viewers (CLI + Mac app) subscribed to different folder sets — only the relevant viewer gets the event.
- Subscribe, unsubscribe, subscribe again — events flow accordingly.
- Revoke the token on the host — the WS closes with code 4001 within a couple of seconds.
- Toggle a folder off in #0085 → all subscribers to that folder receive `{"type":"unsubscribed", …, "reason":"host_unshared"}`.

## Notes

- The viewer-side fanout is asymmetric: subscribe is additive, unsubscribe is explicit. Replace-style subscribe was considered and rejected because it forces the client to know the full set when adding one folder.
- Keep the message types small and JSON-encoded. `Codable` synthesis on `RemoteEvent` (defined in `RemoteProtocol.swift` per #0080) gives parity with the REST shape.
- This is the only place the host writes to viewers. All other endpoints are pull. Stay disciplined here — adding push for non-event data (e.g. issue body bodies) tempts complexity.

## Scope (out)

- Viewer-side WS handling and reconnect — #0102.
- Per-issue diff computation. The `reload` event is the floor; finer events are nice-to-have.
- Heartbeat configuration UI. Hardcoded constants are fine.

## Root cause

Greenfield. The REST surface (#0080 / #0081) was complete but the viewer had no push channel — every refresh required a manual reload. The wire shape was specified; the implementation needed an HTTP/1.1 → WS upgrade on top of the existing `NWListener`.

## Fix

Implemented in `worktree-agent-A` (`ce233f2`), merged onto `main`:

- **`Issues/Remote/WebSocketSession.swift`** (new) — hand-rolled RFC 6455. `WebSocketHandshake.accept(forKey:)` computes `Sec-WebSocket-Accept` (base64 SHA-1 of `key + "258EAFA5-E914-47DA-95CA-C5AB0DC85B11"`); `WebSocketFraming` parses + builds text / close / ping / pong frames including client-side masking; `@MainActor` `WebSocketSession` holds the connection, the FIFO send queue, the 30 s ping watchdog + 10 s pong deadline, the `subscribedFolderIds: Set<String>`, and a `WebSocketCloseCode` enum that adds `.tokenRevoked = 4001`.
- **`RemoteServer.swift`** — `handleRequest` special-cases `/v1/events` after auth, performs the 101 Switching Protocols handshake, owns `sessions: [ObjectIdentifier: WebSocketSession]` and `subscribers: [String: Set<ObjectIdentifier>]`, exposes `broadcast(_:toFolderId:)`, `unshareFolder(folderId:reason:)`, and `closeSessions(matching:code:)`. Auth runs before the upgrade — 401 still returns plain HTTP per spec.
- **`RemoteProtocol.swift`** — wire types `RemoteEvent` (server→client) and `RemoteCommand` (client→server) with a custom `encode(to:)` that skips nil fields so a `hello` event doesn't carry empty `folderId`/`id`.

The server-side fanout for #0101 is wired in `RemoteHostController` (see #0101's resolution). The `RemoteHostIssueSource` (#0094) defines its own `RemoteIssueSourceEvent` for the viewer's source-level events; the name was clarified post-merge to avoid collision with the wire `RemoteEvent`.

## Gotchas

- **Existing `HTTPRequestParser` rejects body bytes after `\r\n\r\n`** with `.bodyOnGet`. Standard WS clients don't pipeline before the 101 so this isn't observed in practice, but worth knowing if a quirky client appears.
- **`RemoteEvent` (wire) vs `RemoteIssueSourceEvent` (source)** — kept distinct on purpose. The wire type is for the server↔client protocol; the source-side enum is the viewer's local lifecycle channel.
- **v1 always emits `reload(folderId:)`**, never `update`/`delete`. The Codable shape supports both per spec, but the per-issue diff isn't produced by the store.
- **`_applyCommandForTest` / `_testRecorder`** are deliberate underscore-prefixed test seams on `WebSocketSession` so the unit tests don't have to stand up real sockets.
