## Description

Ship a small `issues` CLI binary that the user can run from any directory:

```sh
issues                                # opens the current dir if it looks like an issues folder
issues ~/Developer/foo/project-issues # opens a specific folder
```

Behavior:

- If **Issues.app isn't running**, launch it and open the requested folder as the active tab.
- If Issues.app is running and the **folder is already open as a tab**, switch to that tab and bring the window to the front.
- If Issues.app is running but the **folder isn't open yet**, create a new tab for it and bring the window to the front.

Modeled directly on [GitUp's CLI pattern](https://github.com/git-up/GitUp) (`/Users/brennan/Developer/brennanMKE/GitUp/GitUp/Tool/`), which uses LaunchServices to enforce single-instance + a Mach-port-style channel for the actual command. For our sandboxed setup, the right transport is **custom URL scheme** rather than `CFMessagePort` — explained in "Transport choice" below.

## Where to look

- **`Issues.xcodeproj/project.pbxproj`** — new second target for the CLI binary; new `PBXCopyFilesBuildPhase` on the Issues target copying the binary + a tiny `install.sh` into `Contents/SharedSupport/`.
- **`Issues/IssuesApp.swift`** — already has `Window("Issues", id: "main")` and the commands block. Add `.handlesExternalEvents(matching:)` (or `onOpenURL` on the scene's `RootView`) for the URL scheme.
- **`Issues/Info.plist`** — once #0110 has flipped the target to an explicit Info.plist, declare `CFBundleURLTypes` with `URLSchemes = ["issues"]`. Until then, `INFOPLIST_KEY_CFBundleURLSchemes` won't quite do it (URL schemes need the wrapping dictionary), so this ticket is naturally **gated on #0110**.
- **`Issues/State/TabsModel.swift`** (or the tabs container) — needs an `openOrActivate(folderPath:)` entry point that checks the persisted remembered-folders list (or active tabs) before opening.
- **`Issues/Services/FolderBookmarkService.swift`** — the security-scoped bookmark lifecycle; a CLI-supplied path can match an existing remembered bookmark (silent reuse) or be a new path (needs user grant via `NSOpenPanel`).

## Transport choice: custom URL scheme, not CFMessagePort

GitUp uses `CFMessagePort` (named Mach port) because it's *not* sandboxed (Developer ID distribution, no sandbox entitlement). Issues.app **is** sandboxed (`com.apple.security.app-sandbox = true` in `Issues.entitlements`), and that changes the calculus:

- **Sandboxed apps can register Mach service names** only via `com.apple.security.mach-register.global-name` entitlements. Workable, but adds entitlement surface.
- **A non-sandboxed CLI binary calling a sandboxed app's Mach port** requires the app's name to be visible to bootstrap lookup — also workable but fiddly.
- **`/usr/bin/open issues://…`** sidesteps all of it. LaunchServices already handles single-instance (same as `LSOpenFromURLSpec` in GitUp) and routes the URL to the running instance via `application(_:open:)` / SwiftUI's `onOpenURL`. No Mach ports, no polling, no extra entitlements, no archived `NSDictionary`.

So the design is:

1. **Issues.app declares a custom URL scheme `issues://`** via Info.plist.
2. **SwiftUI's `onOpenURL` modifier** on the main scene's root view parses the URL and routes to `TabsModel.openOrActivate(folderPath:)`.
3. **The CLI binary builds the URL and calls `/usr/bin/open`** (or invokes `LSOpenFromURLSpec` directly for cleaner error reporting). LaunchServices either activates the running instance or launches a fresh one, then dispatches the URL.

The CLI binary itself stays small — argument parsing + URL construction + `LSOpenFromURLSpec`. Cross-target dependency on Foundation only; no libgit2 (unlike GitUp), no Sparkle, no `IssuesKit`.

### URL shape

```
issues:///open?path=<percent-encoded-absolute-path>
issues:///                                            # no args → focus the app, no-op for tabs
```

- Scheme: `issues`
- Authority: empty (always `///` to signal "no host").
- Path: `/open` (one verb for v1; extensible: `/reveal`, `/refresh`, etc.).
- Query: `path=<absolute-fs-path>`, percent-encoded.

Absolute paths only. The CLI resolves relative paths (and `$PWD` for the no-arg form) before constructing the URL — the app side never has to interpret a relative path against an unknowable working directory.

## CLI behavior (`issues`)

- **`issues`** (no args) — resolve `$PWD` to an absolute path. If `$PWD/project-issues` or `$PWD/issues` exists, prefer that. Otherwise treat `$PWD` itself as the target. Build the URL and fire it.
- **`issues <path>`** — resolve `<path>` (absolute, relative, or `~`-prefixed) to an absolute path. Fail with `error: not a directory: <path>` if it isn't one.
- **`issues -h` / `issues --help`** — print usage to stdout, exit 0.
- **`issues --version`** — print the embedded version string. The version embeds at build time; pulled from `MARKETING_VERSION`.
- **Exit codes**: `0` on successful URL dispatch (does **not** wait for the app to ack — fire-and-forget is fine for v1). `1` on argument / filesystem errors. `2` on LaunchServices errors (e.g. Issues.app not installed at all).

Stays under ~150 lines of Swift / Objective-C in one file. No Package.swift sibling, no library — single source file built as an executable target.

## App-side behavior

### URL handling

`RootView` (or `IssuesApp.body`) attaches:

```swift
.onOpenURL { url in
    guard url.scheme == "issues" else { return }
    let comps = URLComponents(url: url, resolvingAgainstBaseURL: false)
    guard comps?.path == "/open" else { return }
    if let path = comps?.queryItems?.first(where: { $0.name == "path" })?.value {
        AppCommandsController.shared.tabsModel?.openOrActivate(folderPath: path)
    }
}
```

`AppCommandsController.shared` is the existing singleton; the new `tabsModel?` reference mirrors `hostController?` from the remote-access work.

### `openOrActivate(folderPath:)` semantics

In `TabsModel` (or wherever the tab-bar state lives):

1. **Already-open match**: iterate the current tabs; if any tab's resolved bookmark path equals the requested path (after `standardizingPath`), `activeTabID = thatTab.id` and bring the window to the front. Done.
2. **Remembered-but-not-open match**: check the persisted `rememberedFolders` list (CLAUDE.md mentions the `UserDefaults` key). If a bookmark exists and resolves cleanly to the requested path, open a new tab from that bookmark — no permission prompt, sandbox already trusts it.
3. **Brand-new path**: open the existing folder picker scene (`id: "folderPicker"` in `IssuesApp.swift`) **pre-targeted at the requested path**, so the user's `NSOpenPanel` shows the right directory and the sandbox grant happens via the standard flow. On confirm, the new tab opens and the bookmark is persisted via `FolderBookmarkService`. On cancel, nothing happens.

Path 3 is the load-bearing one: the sandbox requires explicit user grant for any new folder, and the CLI can't bypass that — but pre-targeting the panel makes it one click.

### Window-front behavior

The existing `Window("Issues", id: "main")` is already single-instance per #0026, so URL dispatch into an existing instance just activates it; the app becomes frontmost via the standard LaunchServices activation that's already happening.

## Install: sandboxed-app friendly

GitUp's installer (`AuthorizationExecuteWithPrivileges` → `ln -sf` into `/usr/local/bin`) doesn't compose with the App Sandbox. Three viable replacements:

1. **`NSSavePanel`-driven install**. Menu item under **Issues → Install Command Line Tool…** opens a save panel pre-targeted at `/usr/local/bin/issues` (or `~/.local/bin/issues` as fallback for non-admin Macs). User confirms the location → sandbox grants write to *that one URL only* via the user-selected-file flow → the app writes a symlink from there back to `Contents/SharedSupport/issues`. No `sudo`, no AuthorizationServices, no privileged helper.
2. **Document a one-liner in Help (`Issues/Help/`)**. The user runs `ln -sf "/Applications/Issues.app/Contents/SharedSupport/issues" /usr/local/bin/issues` themselves in Terminal. Zero app-side install code; the Issues→Install menu item just shows the command to copy.
3. **Privileged helper via SMJobBless / SMAppService**. Correct, robust, and total overkill for a one-binary developer tool.

**Recommendation: option 1.** It mirrors the GitUp flow (menu item, one click), works inside the sandbox, and handles the common case of `/usr/local/bin` requiring admin without forcing the user into Terminal. Option 2 stays documented as a fallback for users who want to drop the symlink elsewhere (e.g. `~/bin`).

### Packaging at build time

`PBXCopyFilesBuildPhase` named "Copy CLI" on the Issues target, `dstSubfolderSpec = 12` (SharedSupport), with the `issues` binary set to `CodeSignOnCopy`. After build:

```
Issues.app/
└── Contents/
    ├── MacOS/Issues
    └── SharedSupport/
        └── issues          ← the CLI binary, signed
```

The symlink at `/usr/local/bin/issues` resolves into the bundle; app updates instantly update the CLI behind the symlink, no reinstall needed.

## Verification

### CLI

- `issues --help` prints usage, exits 0.
- `issues --version` prints the embedded version string.
- `issues /tmp/nope` exits 1, stderr says "not a directory".
- `issues ~/Developer/brennanMKE/Issues/project-issues` exits 0, Issues.app foregrounds (or launches) and shows that folder.

### Activate-vs-create

- Pre-condition: Issues.app open with two folders as tabs (e.g. `~/Developer/foo`, `~/Developer/bar`). `bar` is the active tab.
- `issues ~/Developer/foo` — `foo` becomes active. No new tab. Window foregrounds.
- `issues ~/Developer/baz` (not previously opened, not remembered) — folder picker opens pre-targeted at `~/Developer/baz`. User confirms. New tab opens, becomes active.
- `issues ~/Developer/baz` again — already open, switches to it. No picker.
- Quit Issues.app. `issues ~/Developer/foo` — app launches, opens with `foo` as the only tab.

### Install

- Menu **Issues → Install Command Line Tool…** opens a save panel at `/usr/local/bin/issues`.
- Confirm → symlink lands at the chosen path. `which issues` finds it. `issues --version` works.
- Cancel → no change.

### Sandbox

- `Info.plist` declares the URL scheme; `LSOpenFromURLSpec` from the CLI works.
- No new entitlement on Issues.app required for the URL handling itself. The `NSSavePanel`-driven install relies on the existing user-selected-files entitlement.

## Dependencies

- **Gated on #0110** (explicit Info.plist). `CFBundleURLTypes` is a dictionary-of-arrays-of-dictionaries; not expressible via `INFOPLIST_KEY_*` build settings. Once #0110 lands and `Issues/Info.plist` exists, the URL-scheme declaration is a one-block addition.

## Notes

- We're not adding subcommands beyond `open` in v1. Future verbs (`reveal`, `refresh`, etc.) are extensible via the URL path; document the wire format in a header comment near the URL handler.
- The CLI is intentionally fire-and-forget — it doesn't wait for the app to ack the open. That keeps the CLI simple and matches the user's mental model ("I asked for the app, it'll come"). If the app fails to open the folder for some reason, the user sees it in the app UI, not on stderr.
- Path canonicalization on both sides uses `URL(fileURLWithPath:).standardized` so trailing slashes / `..` / `~` resolve identically.
- The CLI doesn't need libgit2 or any third-party deps. Foundation + AppKit's `NSWorkspace` (for `LSOpenFromURLSpec`) is enough.
- Reference implementation to draw from: `/Users/brennan/Developer/brennanMKE/GitUp/GitUp/Tool/main.m` + `GitUp/Application/AppDelegate.m` (the install action). Same shape, different transport (URL scheme not Mach port) and different installer (NSSavePanel not AuthorizationServices).

## Scope (out)

- A library or `IssuesKit`-style package. The CLI is one file, no library needed.
- Subcommands beyond `open`. Extensible via the URL path; add when an actual need shows up.
- Reading from stdin or processing multiple paths in one invocation. One path per call is fine; users can shell-loop.
- Uninstall menu item. The user can `rm /usr/local/bin/issues` themselves; the symlink is unambiguous.
- Updating `PATH` automatically. If `/usr/local/bin` isn't on the user's PATH, the install panel's location choice handles it.
- A separate iOS-side equivalent. iPhone doesn't have a CLI; out of scope, paired with #0108.

## Scope (in but flagged)

- **The Issues→Install menu item alters a file outside the sandbox.** The `NSSavePanel` flow is the standard way to do this and is approved for sandboxed apps, but it's worth code-review attention when the implementation PR lands — the write must use the URL the user picked (carrying its `NSURL`-bookmark grant), not a separately-constructed `URL(fileURLWithPath:)` that would be denied.

## Root cause

Not a bug — new feature. v1 ships the CLI as a standalone SwiftPM binary rather than an in-bundle helper because the spec's full packaging path (new Xcode target + Copy Files build phase + in-app NSSavePanel installer) requires structural project.pbxproj surgery the user wanted to avoid in an unattended session. The end-user experience is the same — `issues path` opens or focuses the folder — but the install step is currently a manual `ln -s`.

## Fix

### App side

- `Info.plist` declares `CFBundleURLTypes` with `URLSchemes = ["issues"]` and `CFBundleTypeRole = Viewer`. The dependency on the explicit Info.plist migration (#0110) was already satisfied; this was a one-block addition.
- `RootView.onOpenURL` parses inbound URLs. Only `/open` with a `path=` query is acted on; everything else just brings the app forward via `NSApp.activate`. Non-`issues` schemes are logged and ignored.
- `TabsModel.openOrActivate(folderPath:bookmarks:)` is the routing entry point. Three branches per the spec:
  - **Already-open**: any tab whose `folderURL.standardizedFileURL.path` matches the requested path becomes active. No new tab.
  - **Remembered**: `FolderBookmarkService.remembered` has a matching entry — resolve the bookmark, open a new tab. The sandbox already trusts the bookmark; no prompt.
  - **Brand-new**: returns `.needsGrant`. The caller (`RootView`) opens the existing `folderPicker` scene, where the user picks the folder via `NSOpenPanel` and the sandbox grant is collected through the standard flow.

### CLI

- `Tools/IssuesCLI/` ships as a SwiftPM executable mirroring the existing `Tools/IssuesRemoteSmoke` precedent. Single `main.swift` (~150 lines), depends only on Foundation + AppKit, no third-party packages.
- `issues` with no args resolves `$PWD` — preferring `$PWD/project-issues` then `$PWD/issues` if either exists, otherwise `$PWD` itself.
- `issues <path>` accepts absolute, relative, or `~`-prefixed paths; relative paths resolve against `$PWD` before standardising. Filesystem-existence + is-directory check enforces the "not a directory" failure path the spec asked for.
- `--help` / `-h` and `--version` exit 0. Argument or filesystem errors exit 1 with stderr message. LaunchServices errors (e.g. Issues.app not installed) exit 2.
- Dispatch builds `issues:///open?path=<percent-encoded-absolute-path>` and calls `NSWorkspace.shared.open(_:configuration:completionHandler:)`. The callback is awaited up to 5 seconds so the binary can return a real exit code if Issues.app isn't installed; otherwise it's effectively fire-and-forget.

## Gotchas

- **Sandbox / bookmark interaction**: a remembered folder resolves via `FolderBookmarkService.resolve(_:)` which already pairs `start/stopAccessingSecurityScopedResource`. The new path through `openOrActivate` keeps that contract — it calls `resolve` and hands the resulting URL to `openTab`, which mints its own bookmark for the tab's lifetime. Don't be tempted to short-circuit by reading `displayPath` directly; the sandbox would deny the file-system access.
- **Path standardisation**: comparisons on both sides go through `standardizedFileURL.path`, not raw input. `~/foo/`, `~/foo`, and `/Users/me/foo/` all match the same tab. If a future contributor adds a comparison elsewhere, mirror the same standardisation or false misses will surface.
- **LaunchServices ack timeout**: the 5-second wait is a compromise — long enough to surface "app not installed" reliably, short enough to feel fire-and-forget to the user. If the user's Mac is under heavy load and the open genuinely takes longer than 5s, the CLI exits 0 even if the open eventually fails. Acceptable for v1; revisit if it bites.
- **`onOpenURL` fires *before* `RootView.onAppear` completes on cold launch** in some Apple Silicon scheduling timings — meaning `AppCommandsController.shared.tabs` may not yet be wired. `RootView.onOpenURL` uses the local `tabs` `@State` directly (not the controller), so this race is dodged. Don't refactor to go through the controller singleton — the URL handler must be able to act before the controller setup runs.
