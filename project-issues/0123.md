## Description

Move the Issues target's build settings from the Xcode-managed `project.pbxproj` blob into three `Configuration/*.xcconfig` files, then bump the project from `SWIFT_VERSION = 5.0` to `SWIFT_VERSION = 6.0` and adopt the explicit concurrency annotations Swift 6 requires.

Two motivations:

1. **Build-setting readability.** The `project.pbxproj` had `SWIFT_*`, `INFOPLIST_KEY_*`, signing, and platform settings scattered across four config blocks (Debug/Release × app/test). xcconfig files put each setting on one human-readable line and let the project, app, and CLI targets share a common base. Diffs become reviewable.
2. **Swift 6 language mode.** Swift 5 + Approachable Concurrency was emitting warnings for several concurrency edges (cross-actor reads, non-Sendable existentials, implicit MainActor inference on nonisolated initializers). Swift 6 promotes those warnings to errors. Catching them at the compiler is the right place; bumping the mode forces us to make the annotations explicit.

A new scaffolded `IssuesCommand` executable target also lands as part of this commit — it's a placeholder for #0119 (the `issues` CLI) and a sanity check that the per-target `Command.xcconfig` plumbing works. The actual CLI behavior is the subject of #0119.

## Acceptance criteria

- `Configuration/Build.xcconfig`, `Configuration/App.xcconfig`, `Configuration/Command.xcconfig` exist and are referenced from the matching `XCBuildConfiguration` blocks in `project.pbxproj`.
- `SWIFT_VERSION = 6.0` and `SWIFT_DEFAULT_ACTOR_ISOLATION = MainActor` set at the project level.
- Build clean against macOS with no concurrency warnings or errors.
- Unit tests pass.
- The `IssuesCommand` target builds an executable `issues` binary that prints `Arguments: [...]` (placeholder).

## Why these specific changes

### `nonisolated` on protocol requirement and conformers (`folderURL`)

`IssueSource` is intentionally not `@MainActor`-annotated (per the existing comment on the protocol). Under Swift 5 mode, that intent was honored by leniency in the warning path. Under Swift 6, the conforming types' default isolation (MainActor, per the project default) makes their `folderURL` implementations implicitly MainActor-isolated — which means cross-actor callers (the `AttachmentLoader` actor) can't read them without an `await` *and* an existential `Sendable` boundary cross.

Marking the requirement `nonisolated` says "every conforming type must expose this property in a non-actor context." Both `LocalFolderIssueSource` (where it's a stored `let`) and `RemoteHostIssueSource` (computed from a stored `let`) satisfy that without further restructuring. The `AttachmentLoader`'s disk-read path now reads `source.folderURL` in-isolation without any cross-actor boundary.

### `nonisolated struct` on data types (`Issue`, `TokenRecord`, `TokenDatabase`)

Value types are concurrency-safe by construction: each call gets a copy, so there's no shared mutable state for the actor system to protect. The project's MainActor-default-isolation setting still annotates them as MainActor-isolated by default, which makes their *KeyPaths* non-Sendable (because reading a property requires hopping to MainActor). `KeyPathComparator(\Issue.id, order: .forward)` in `ListView` then fails because Swift 6 strictly requires the KeyPath to be `Sendable`.

Marking the type `nonisolated` opts it out of the project default; its members become nonisolated, KeyPaths to them are Sendable, the existing `Sendable` conformance now means what it always meant to mean.

### `nonisolated` on Keychain helpers (`AccessToken.touch / load / save`) and pure-data statics

`RemoteServer`'s bookkeeping path uses `Task.detached { try AccessToken.touch(...) }` to update `lastUsedAt` after a request without blocking the response. Under Swift 6, `Task.detached` runs nonisolated, so calling a MainActor-isolated `touch` from inside it fails. Keychain I/O doesn't touch MainActor state, so the right fix is to mark the helpers nonisolated. The static lets they reference (`defaultService`, `account`, JSON coders) also have to be nonisolated for the same reason — a nonisolated function can't read a MainActor-isolated default value or static property.

The synchronous `touch / load / save` API is preserved; `#0122` covers the follow-on migration to an `@concurrent` async overload.

### Explicit `self.pongDeadline` in `RemoteWebSocket`

Swift 6 requires explicit `self.` in escaping closures so capture semantics are unambiguous. The shorthand `if let pongDeadline` works at function scope but reads as a stored-property access inside a `@MainActor` closure inside a `Task`. Writing `if let pongDeadline = self.pongDeadline` makes the capture explicit and the dead `_ = pongDeadline` shim that was there to silence a warning gets removed.

### `completionHandler()` moved out of the MainActor Task (`NotificationService`)

The `UNUserNotificationCenter` delegate method's `completionHandler: @escaping () -> Void` is not `@Sendable`. Capturing it inside a `Task { @MainActor in … }` is a Swift 6 sending-data-race error: the MainActor task can't call a non-Sendable closure that was received from a different isolation domain.

Moving the call to the synchronous tail of the nonisolated delegate method — after the Task is dispatched but before the method returns — is the standard Swift 6 idiom for `UN…` delegate handlers. Apple's contract is "execute completionHandler as soon as possible after processing"; dispatching the work qualifies. The behavior change (completion fires before routing finishes) is followed up in #0121 with manual verification steps.

### `SDKROOT = macosx` (dropped the `[sdk=macos*]` condition)

`SDKROOT[sdk=macos*] = macosx` is self-referential — `SDKROOT` is what *selects* the SDK in the first place — so Xcode was ignoring it with a warning. Dropping the condition silences the warning without changing the resolved value.

## Notes

- The biggest readability win is in `project.pbxproj`: the four config blocks per target now contain only the build-specific settings that *can't* live in xcconfig (file references, IDs). Diffs of the pbxproj for future settings changes are now meaningful.
- The `nonisolated` annotations follow `swift-guidance/references/actor-isolation.md` recommendations almost verbatim — value types, protocol requirements, pure-data statics, file-scoped loggers all get nonisolated. The `NotificationService` change isn't covered by the guide but matches Apple's own Swift 6 notification-handler examples.
- This work intentionally does **not** adopt `@concurrent` anywhere. That migration is #0122 — a separate, independent step. Under Swift 6, `Task.detached` and `nonisolated` together still produce correct behavior; `@concurrent` is the more declarative form.
- The `Logging.subsystem` change is small but fixes four otherwise-baffling errors: every `nonisolated private let logger = Logger(subsystem: Logging.subsystem, …)` was hitting "main actor-isolated default value in a nonisolated context" because `Logging.subsystem` was implicitly MainActor under the project default.

## Root cause

Greenfield migration. The previous `project.pbxproj` carried `SWIFT_VERSION = 5.0` plus Approachable Concurrency in warning mode; the actor-isolation checks that would have caught these concurrency edges existed but weren't fatal. Bumping to Swift 6 promoted those checks to errors, and at the same time moving to xcconfig made the build configuration grep-able and reviewable.

## Fix

Two layers landed in commit `a63316f`:

**Configuration migration.** Three new `Configuration/*.xcconfig` files house the build settings; the `project.pbxproj` blocks now carry only the per-target reference + the residual settings xcconfig can't express. The IssuesCommand executable target is scaffolded at the same time so the per-target overlay (`Command.xcconfig`) gets exercised.

**Concurrency annotations** (one per source-file diff):
- `Issues/Logging.swift` — `nonisolated static let subsystem`. Clears the four "main actor-isolated default value in a nonisolated context" errors that file-scoped `Logger` initializers were tripping.
- `Issues/Models/Issue.swift` — `nonisolated struct Issue`. Makes KeyPaths to its properties Sendable, which `KeyPathComparator(\Issue.id, …)` in `ListView` requires.
- `Issues/Remote/AccessToken.swift` — `nonisolated struct TokenRecord` / `TokenDatabase`, plus `nonisolated` on `defaultService`, `account`, `plaintextPrefix`, `jsonEncoder`, `jsonDecoder`, `touch`, `load`, `save`. Lets `RemoteServer`'s `Task.detached { try AccessToken.touch(...) }` bookkeeping path compile.
- `Issues/Remote/RemoteHostIssueSource.swift` — `nonisolated var folderURL: URL { syntheticFolderURL }`.
- `Issues/Remote/RemoteWebSocket.swift` — explicit `self.pongDeadline` in the `@MainActor` closure (Swift 6 requires it for capture clarity).
- `Issues/Services/IssueSource.swift` — `nonisolated var folderURL: URL { get }` on the protocol requirement, with doc-comment explaining the cross-actor read pattern.
- `Issues/Services/LocalFolderIssueSource.swift` — `nonisolated let folderURL: URL`.

## Gotchas

- **`SWIFT_VERSION` deliberately not duplicated per-target.** Setting it at `Build.xcconfig` once is enough — every target inherits it. Verified with `xcodebuild -target <X> -showBuildSettings | grep SWIFT_VERSION` for all four targets. Adding it again at the target level would invite drift the next time someone edits a single target's settings.
- **`nonisolated` is not the same as `Sendable`.** `nonisolated` opts a declaration out of the project's MainActor default; `Sendable` is a separate conformance. Several types here got both because they need to be both — readable from any context AND safe to share across contexts. The guide in `swift-guidance/references/actor-isolation.md` documents the distinction.
- **The `NotificationService` completion-handler reorder is the one observable behavior change.** Apple's contract is satisfied (completion runs "as soon as possible after processing"; dispatching the work qualifies), but cold-launch deep-link routing is worth a manual verification before declaring victory. #0121 owns that follow-up.
- **`@concurrent` is intentionally absent here.** Swift 6 allows it; the call sites that would benefit (`AttachmentLoader.loadRaw`, `AccessToken.touch` via `Task.detached`) are documented in #0122. Splitting the migration from the `@concurrent` adoption keeps each commit's blast radius small.
- **Test targets don't have a `baseConfigurationReference` to an xcconfig.** They inherit project-level Build.xcconfig directly. If a future change needs test-specific build settings, give them their own `Tests.xcconfig` then; for now the existing target-level `buildSettings` blocks plus project inheritance are sufficient.
