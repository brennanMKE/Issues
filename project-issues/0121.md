| | |
|---|---|

## Description

The Swift 6 migration required moving the `UNUserNotificationCenter` completion-handler call out of the `Task { @MainActor in … }` closure in `NotificationService.userNotificationCenter(_:didReceive:withCompletionHandler:)`. Under Swift 6, an `@escaping () -> Void` is not `Sendable`, so the MainActor task can't capture and invoke it without a data-race diagnostic.

The fix shipped:

```swift
nonisolated func userNotificationCenter(
    _ center: UNUserNotificationCenter,
    didReceive response: UNNotificationResponse,
    withCompletionHandler completionHandler: @escaping () -> Void
) {
    // …extract userInfo synchronously…

    Task { @MainActor in
        // …routing, app activation, window-front, deep-link drain…
    }

    completionHandler()      // ← moved out; now fires before the Task runs
}
```

This is the standard Swift 6 idiom for `UN…` delegate methods (the protocol signature isn't marked `@Sendable` so Apple's contract has to be honored at the call site). Apple's docs say "execute the completionHandler block as soon as possible after processing," and dispatching the work counts as "processing." But it is a **behavior change** worth a manual verification pass before the migration commits to `main`.

## What changed in observable behavior

- **Before**: completion handler fired *after* the MainActor routing work finished (tab activation, deep-link consume, window-front).
- **After**: completion handler fires *immediately* after dispatching the MainActor task; the routing work runs asynchronously.

In practice the routing path is already idempotent — `pendingDeepLink` is stashed first thing inside the Task, and `RootView.onAppear` drains it on a cold launch — so the change should be invisible. The risk is in cold-launch timing: if the OS suspends the process between the synchronous `completionHandler()` and the unawaited Task firing, the routing might silently drop.

## Steps to verify

Each step needs an actual notification interaction; the changes can't be exercised by unit tests alone.

### Warm path (app already running)

1. Run the app, open at least two folders so there are multiple tabs.
2. Trigger a change to a different tab's folder so a rollup notification fires.
3. Click the notification.
4. **Expect**: the originating tab activates, the issue (if named in `userInfo`) selects, the window comes forward.

### Cold-launch path

1. Quit the app entirely.
2. From another shell, write a fake change to a watched folder so a notification is delivered (or wait for the next legitimate notification).
3. After the notification appears, **with the app still terminated**, click the notification.
4. **Expect**: the app launches; the originating tab restores; the issue (if any) selects; the deep-link banner state (if any) is consistent. No silent drop where the app launches but the tab activation never happens.

### Stress-test path

1. With the app running, generate three or four rollup notifications in quick succession by editing different folders' files.
2. Click each notification in turn.
3. **Expect**: each click activates the right tab. No missed routing, no stuck states.

## Where to look

- **`Issues/Services/NotificationService.swift:264–325`** — the delegate method. The `completionHandler()` invocation is now at line ~325, outside the `Task { @MainActor in … }` block. `pendingDeepLink` stashing happens inside the Task (the first action) so cold-launch reads from `RootView.onAppear` still work.
- **`Issues/Views/RootView.swift`** — `onAppear` calls `AppCommandsController.shared.consumePendingDeepLinkIfPossible()`. This is the cold-launch backstop.
- **`Issues/State/AppCommandsController.swift`** — owns `pendingDeepLink` and the `consumePendingDeepLinkIfPossible()` flow.

## If a regression appears

Two recovery options, listed in order of preference:

1. **Wrap the delegate in a small synchronous bridge that signals completion via a continuation.** The delegate would await the routing work in its own scope:

   ```swift
   nonisolated func userNotificationCenter(_:didReceive response:withCompletionHandler completionHandler:) {
       // …extract userInfo…
       let work = Task { @MainActor in
           // …routing…
       }
       Task {
           _ = await work.value      // wait without capturing completionHandler in MainActor
           completionHandler()
       }
   }
   ```

   The outer `Task` is nonisolated, so it can capture and call `completionHandler` directly. The MainActor work still runs in its own isolation domain.

2. **Force the MainActor work to run synchronously via `MainActor.assumeIsolated` from a dispatched helper.** More invasive; only if option 1 doesn't compose with whatever the regression turns out to be.

## Verification

- All three test paths above pass on at least one Debug build installed locally.
- `log stream --predicate 'subsystem == "co.sstools.Issues"'` shows the expected `DID_RECEIVE` sequence: identifier, action, kind, then the post-Task activation entries. No "missing pendingDeepLink" warning during cold-launch routing.
- Quit + relaunch a few times between tests to make sure the cold-launch path is fresh, not warm-started.

## Notes

- This is a verification-only ticket; no code change is expected unless the manual tests fail. If they pass, set status to `resolved` with a one-line "verified on YYYY-MM-DD on macOS X.Y" Notes entry.
- The Swift 6 reason the previous code worked is that Swift 5 mode silently allowed the cross-isolation capture (warning at most). Swift 6 promotes it to an error. There's no plausibly-cleaner pattern given Apple's protocol signature; the choice is between this and the continuation bridge above.
- If we ever see real cold-launch regressions, the right fix is the continuation bridge (option 1). It's intentionally not the v1 choice because the current pattern is simpler and is what Apple's own sample code uses in their Swift 6 notification examples.

## Scope (out)

- Changing the protocol signature. Apple's `UNUserNotificationCenterDelegate` is fixed.
- Replacing the notification system with something else. Out of scope.
- Background-thread analysis of `UNUserNotificationCenter` delivery. The system handles dispatch; we just need to honor the contract.
