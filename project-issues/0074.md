# 0074 — Closing the main window should terminate the app

| | |
|---|---|
| **Status** | open |
| **Module** | App |
| **Platform** | macOS |
| **First seen** | 2026-05-07 |

## Description

Issues.app should be a **single-main-window** application. When the user closes the main window, the app should terminate cleanly with a success exit status — the same way utilities like System Settings or Calculator behave when their window is closed (they exit on close rather than lingering with only a menu bar).

Today the macOS default applies: closing the main window leaves the app running with just its menu bar, requiring the user to ⌘Q (or right-click the dock icon → Quit) to actually exit. That's the right default for document-based apps that may have many windows, but it's the wrong fit here — Issues.app is a viewer for one folder at a time, all functionality lives in the single main window, and there's no reason to keep the process around after that window goes away.

## Expected behavior

- One window only — the main `Issues` window. No additional ad-hoc windows for normal flows. Sheets, popovers, and Quick Look panels are fine; they're not new top-level windows.
- Closing the window (red traffic light, ⌘W, or `File → Close`) terminates the app.
- Process exit status is `0` (success).

## Actual behavior

Closing the window leaves the app running with only its menu bar, until the user explicitly chooses Quit.

## Where to look

- **`Issues/App/`** (or wherever the `@main` `App` struct lives — `IssuesApp.swift` likely). The top-level scene declares the window. If it's currently using `WindowGroup`, that should change to `Window` (macOS 13+) for a true single-instance window.
- The single-window-on-close termination behavior is set via an `NSApplicationDelegateAdaptor` whose delegate returns `true` from `applicationShouldTerminateAfterLastWindowClosed(_:)`. SwiftUI on macOS does not expose this through a Scene modifier; it has to come from the AppKit delegate.

## Behavior details

- **Single-window enforcement.** With `Window { … }` instead of `WindowGroup { … }`, SwiftUI prevents users from spawning additional copies via `File → New Window`. If `New Window` should be hidden from the menu entirely, suppress it with `.commands { CommandGroup(replacing: .newItem) {} }`.
- **Termination after close.** Add an `NSApplicationDelegateAdaptor`:
  ```swift
  final class AppDelegate: NSObject, NSApplicationDelegate {
      func applicationShouldTerminateAfterLastWindowClosed(_ sender: NSApplication) -> Bool {
          true
      }
  }
  ```
  And wire it into the `App`:
  ```swift
  @NSApplicationDelegateAdaptor(AppDelegate.self) private var appDelegate
  ```
- **Re-open via Dock.** When the app's process is alive but window is hidden (e.g. user clicks the close button but hasn't killed the process yet on a slow termination), clicking the Dock icon should restore the window. With `Window`, SwiftUI handles this automatically — no extra delegate work needed.
- **Clean exit.** No teardown beyond what AppKit already does. The `IssueStore`'s active folder access (`startAccessingSecurityScopedResource`) is paired correctly today via `deinit`/`stop()`, so `applicationWillTerminate` doesn't need to do anything special.
- **Notifications.** Notification taps that route into the app (#0070) should still wake the process and bring the window to the front. If the process has already exited, macOS will relaunch the app from scratch — that's fine.

## Verification

- Launch Issues.app. Close the main window. Process exits within ~1 second; `ps -ef | grep Issues.app` shows nothing afterward.
- Echo the exit status from a terminal-launched run:
  ```sh
  open -W -a Issues.app; echo $?    # 0
  ```
- Hit ⌘N — either there's no second window (because we replaced `newItem`) or it brings the existing window to the front rather than spawning a duplicate.
- Click Dock icon while window is showing → it focuses (no change). Hide the window with ⌘H, click the Dock icon → window returns. Close the window → app terminates.
- Tap a notification (per #0070) when the app has been quit → app relaunches and brings up the window.

## Notes

- This is a behavioral fit, not a bug fix — the macOS default is correct for document apps but wrong for single-window utilities. Treating Issues.app as the latter aligns with how the user actually uses it.
- If the project ever adds a Settings window, that's a separate Scene (`Settings { … }`), not a second main window — `applicationShouldTerminateAfterLastWindowClosed` should still return `true` since Settings isn't usually open at the moment the user closes the main window.
- This is not the same as `applicationShouldHandleReopen(_:hasVisibleWindows:)` (which is about Dock-icon reopen behavior). That delegate method can stay at its default; SwiftUI's `Window` already handles reopen.
