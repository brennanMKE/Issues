## Description

Refactor the project so that `IssuesKit/` (the local SwiftPM package just added at the repo root) owns nearly all source code and every external SwiftPM dependency. The `Issues.xcodeproj` becomes a thin shell whose targets are little more than `@main` entry points, `Info.plist`, entitlements, and the asset catalog — every piece of business logic, every view, every service, every parser lives inside `IssuesKit`.

Motivation (verbatim from the user): "Xcode handles packages very badly, so I would rather just have 1 local package which handles all dependencies in Package.swift." That's the design constraint. The Xcode project keeps doing what Xcode is uniquely good at (`.app` bundling, code signing, sandbox entitlements, App Store / Sparkle distribution). Everything else moves into SwiftPM where dependency declarations live in one editable `Package.swift` and resolution is deterministic from the CLI.

This is structural; it touches every file in `Issues/`, every file in `IssuesCommand/`, every file in `DashboardCommand/`, and every file in `IssuesTests/`. Plan accordingly.

## Goals

1. `IssuesKit/Package.swift` declares **all** external SwiftPM dependencies: `Watcher`, `Textual`, `Sparkle`, `SwiftTUI`. The four `XCRemoteSwiftPackageReference` entries in `Issues.xcodeproj/project.pbxproj` are gone.
2. ~95 % of the Swift source under `Issues/`, `IssuesCommand/`, `DashboardCommand/`, and `IssuesTests/` lives in `IssuesKit/Sources/...` and `IssuesKit/Tests/...`.
3. Each Xcode target's source folder shrinks to a single-file `@main` shim that imports an `IssuesKit` product and calls into it. Specifically:
   - `Issues/IssuesApp.swift` becomes the only `.swift` file in the app target: imports `IssuesAppKit`, declares the `@main` `App`, wires Sparkle, owns the entitlements.
   - `IssuesCommand/IssuesCommand.swift` becomes ~5 lines: import + `@main` + delegate to `IssuesCLIRunner.main()`.
   - `DashboardCommand/DashboardCommand.swift` same shape for the dashboard.
4. Tests run via both `swift test` (from inside `IssuesKit/`) and `xcodebuild test` on the `Issues` scheme. UI tests stay in the Xcode project (they need the `.app` bundle).
5. The `Issues.app`, `issues`, and `issues-dashboard` binaries continue to build, sign, and launch exactly as they do today. End-user behaviour is unchanged.

## Non-goals

- Re-architecting the app's internals (no module boundary changes beyond the package-vs-app split).
- Splitting `IssuesKit` into many tiny packages or repos. One local package, multiple targets inside it if useful.
- Moving the CLIs out of the Xcode project entirely (they could live as SwiftPM executables, but then they lose Xcode's automatic code-sign-on-build flow, which #0119's still-pending Copy Files build phase needs).
- Changing the deployment target, language mode, or any Swift settings (`SWIFT_VERSION = 6.0`, `SWIFT_DEFAULT_ACTOR_ISOLATION = MainActor`, `MACOSX_DEPLOYMENT_TARGET = 15.0` all stay).
- Removing duplicates introduced by #0133 (DashboardCommand's local `Issue.swift` / `IssueStatus.swift` / `MarkdownIssueParser.swift`) **before** this refactor — they go away naturally as part of step 5 below.

## Where to look

Inputs:

Resources: `Assets.xcassets/`, `Localizable.xcstrings`, `Help/*.md`, `Issues.entitlements`. Total ~18,000 lines.
- **`IssuesCommand/IssuesCommand.swift`** — 121 lines, the `issues` CLI body (#0132).
- **`DashboardCommand/`** — 8 `.swift` files, ~900 lines total. Three of them (`Issue.swift`, `IssueStatus.swift`, `MarkdownIssueParser.swift`) are duplicates of `Issues/Models/*` and `Issues/Services/MarkdownIssueParser.swift`, kept locally per #0133's "option 2: duplicate" choice. Step 5 below dissolves the duplication.
- **`IssuesTests/`** — 23 `.swift` files testing the parser, stores, remote handlers, web socket fanout, etc.
- **`IssuesUITests/`** — stays in the Xcode project (needs the `.app`).
- **`Configuration/*.xcconfig`** — `App.xcconfig`, `Build.xcconfig`, `IssuesCommand.xcconfig`, `DashboardCommand.xcconfig`. All stay.
- **`Configuration/Info.plist`** — the app's `Info.plist`. Stays.
- **`Issues.xcodeproj/project.pbxproj`** — four `XCRemoteSwiftPackageReference` entries (`Watcher`, `textual`, `Sparkle`, `SwiftTUI`) and the four corresponding `XCSwiftPackageProductDependency` entries; one product dependency on each of the three native targets that consume the packages. All of this gets pruned and replaced with a single `XCLocalSwiftPackageReference` pointing at `IssuesKit/`, plus three `XCSwiftPackageProductDependency` entries (one per consuming Xcode target) referencing the local package's library products.

Reference patterns for the local-package wiring:

- Apple's own [SwiftPM-in-Xcode docs](https://developer.apple.com/documentation/xcode/creating-a-standalone-swift-package-with-xcode) (specifically the "Use a local package" section) — the canonical mechanics for declaring an `XCLocalSwiftPackageReference` and consuming its products from a native target.

### `Package.swift` shape

```swift
// swift-tools-version: 6.3
import PackageDescription

// Shared across every library / source target so the package matches the
// app's `SWIFT_DEFAULT_ACTOR_ISOLATION = MainActor` build setting. The
// test target intentionally omits this — test code wants the toolchain
// default (nonisolated) so tests can run async helpers without colliding
// with the main actor.
let swiftSettings: [SwiftSetting]? = [.defaultIsolation(MainActor.self)]

let package = Package(
    name: "IssuesKit",
    defaultLocalization: "en",
    platforms: [.macOS(.v15)],
    products: [
        .library(name: "IssuesCore", targets: ["IssuesCore"]),
        .library(name: "IssuesAppKit", targets: ["IssuesAppKit"]),
        .library(name: "IssuesCLI", targets: ["IssuesCLI"]),
        .library(name: "IssuesDashboardCLI", targets: ["IssuesDashboardCLI"]),
    ],
    dependencies: [
        .package(url: "git@github.com:brennanMKE/Watcher.git", branch: "main"),
        .package(url: "https://github.com/gonzalezreal/textual.git", branch: "main"),
        .package(url: "https://github.com/sparkle-project/Sparkle.git", from: "2.6.0"),
        .package(url: "https://github.com/rensbreur/SwiftTUI.git", branch: "main"),
    ],
    targets: [
        .target(
            name: "IssuesCore",
            dependencies: [
                .product(name: "Watcher", package: "Watcher"),
            ],
            swiftSettings: swiftSettings
        ),
        .target(
            name: "IssuesAppKit",
            dependencies: [
                "IssuesCore",
                .product(name: "Textual", package: "textual"),
                .product(name: "Sparkle", package: "Sparkle"),
            ],
            resources: [
                // .process("Resources") if any package-side resources end up here
            ],
            swiftSettings: swiftSettings
        ),
        .target(
            name: "IssuesCLI",
            dependencies: ["IssuesCore"],
            swiftSettings: swiftSettings
        ),
        .target(
            name: "IssuesDashboardCLI",
            dependencies: [
                "IssuesCore",
                .product(name: "SwiftTUI", package: "SwiftTUI"),
            ],
            swiftSettings: swiftSettings
        ),
        .testTarget(
            name: "IssuesKitTests",
            dependencies: ["IssuesCore", "IssuesAppKit"],
            resources: [
                .process("Fixtures")    // if any test fixtures need to ship
            ]
            // No swiftSettings — tests stay nonisolated by default.
        ),
    ],
    swiftLanguageModes: [.v6]
)
```

Notes on the shape:

- `IssuesCLI` and `IssuesDashboardCLI` are `.target(...)` not `.executableTarget(...)`. The actual `@main` lives in the Xcode target's shim file; the package side just exposes a `public enum IssuesCLIRunner { public static func main() async { … } }`. This keeps Xcode in charge of code signing, entitlements, and Copy Files distribution (the #0119 follow-up still works without modification).
- `IssuesAppKit` is a separate library from `IssuesCore` so the CLI targets don't have to drag Sparkle and Textual into their dependency graph. The CLI binaries stay small.
- `Watcher` is referenced by SSH URL today (`git@github.com:brennanMKE/Watcher.git`). Keep that — if a future CI host can't authenticate, switch to HTTPS in one place rather than per-target.
- The `defaultLocalization: "en"` line is required as soon as the package ships any localized resources; declare it pre-emptively to match the app's existing `Localizable.xcstrings`.

## What stays in the Xcode project

- `Issues.xcodeproj/project.pbxproj` — pruned but not replaced.
- `Configuration/*.xcconfig` and `Configuration/Info.plist` — untouched.
- `Issues/IssuesApp.swift` — reduced to ~5 lines as shown above.
- `Issues/Issues.entitlements` — entitlements are app-bundle-level, can't move to SwiftPM.
- `Issues/Assets.xcassets/` — **stays in the app target.** SwiftPM resource bundles can ship asset catalogs, but `AppIcon.appiconset` specifically must live in the *app target's* asset catalog so `Info.plist`'s `CFBundleIcons` / `CFBundleIconName` resolves. Other color sets *could* move to a package resource, but moving some-but-not-all is a maintenance pothole. Keep the whole asset catalog in the app target; views in `IssuesAppKit` access colours via `Color("appBackground")` against `Bundle.main`, which still works because the consumer (the app) supplies the bundle.
- `Issues/Localizable.xcstrings` — **stays in the app target.** Same reasoning: localized strings looked up via `String(localized:)` resolve against `Bundle.main` by default. If we move the catalog into `IssuesAppKit`'s resources, every callsite needs `String(localized: "...", bundle: .module)`, which is mechanically possible but a large mechanical edit. Easier to leave the catalog in the app target and have package-side views reference Bundle.main lookups (Xcode's macro for `String(localized:)` resolves at compile time via `.xcstrings`).
- `Issues/Help/*.md` — stays in the app target. These are bundled resources read at runtime by `HelpCatalog`. Move alongside `HelpCatalog.swift` only if Bundle.main lookup is refactored to `Bundle.module`; otherwise leave in place.
- `IssuesUITests/` — depends on the `.app` bundle; can't run from `swift test`.

## Migration phases

Do these in order. Each phase should leave the project building and tests passing — that's the regression net.

### Phase 1 — Rewrite `Package.swift` and prove resolution

Replace the scaffold `Package.swift` with the shape above (all four products, all four dependencies, four targets with empty source folders). Run `swift package resolve` from inside `IssuesKit/`. The build will fail (empty source folders) but resolution must succeed. Validates the dependency declarations.

### Phase 2 — Wire the local package into the Xcode project, no consumption yet

Add an `XCLocalSwiftPackageReference` to `Issues.xcodeproj/project.pbxproj` pointing at `IssuesKit/` (relative path). Don't add any product dependencies to the native targets yet. Build all four Xcode schemes; everything must still compile against the existing remote packages.

### Phase 3 — Move `IssuesCore` content

Move models, services, state, remote, logging into `Sources/IssuesCore/`. Mark every type/function used outside the file `public`. Inside the Xcode project, swap the `Issues` target's `Watcher` product dependency for the local package's `IssuesCore` product dependency. Build the Issues scheme; fix the inevitable visibility errors. Tests at this point are still in the Xcode project's `IssuesTests` target — temporarily add `import IssuesCore` to test files until they move too.

### Phase 4 — Move `IssuesAppKit` content (theme, views, app scene)

Move `Theme/`, `Views/`, `HelpMenuButton.swift`, `UpdaterController.swift` into `Sources/IssuesAppKit/`. Extract the scene/command bodies from `IssuesApp.swift` into `IssuesRootScene`. Swap the Issues target's `Sparkle` and `Textual` product dependencies for the local package's `IssuesAppKit` product. The Xcode-side `Issues/IssuesApp.swift` is now ~5 lines.

### Phase 5 — Move CLI targets

`IssuesCommand/IssuesCommand.swift` body → `Sources/IssuesCLI/IssuesCLIRunner.swift` (rename, mark `public`). Same for `DashboardCommand/*` → `Sources/IssuesDashboardCLI/`. Delete the three duplicated files (`Issue.swift`, `IssueStatus.swift`, `MarkdownIssueParser.swift` under `DashboardCommand/`). Xcode shims:

```swift
// IssuesCommand/IssuesCommand.swift
import IssuesCLI

@main
struct IssuesCommand {
    static func main() async { await IssuesCLIRunner.main() }
}
```

```swift
// DashboardCommand/DashboardCommand.swift
import IssuesDashboardCLI

@main
struct DashboardCommand {
    static func main() async { await IssuesDashboardCLIRunner.main() }
}
```

Swap each target's `SwiftTUI` / nothing dependency for the matching local-package product. Build both schemes; smoke-test `issues --help`, `issues --version`, `issues-dashboard project-issues`.

### Phase 6 — Move tests

Move every `IssuesTests/*.swift` into `Tests/IssuesKitTests/`. Run `swift test` from inside `IssuesKit/`. Fix any tests that relied on `@testable import Issues` — they switch to `@testable import IssuesCore` (or `IssuesAppKit`, depending on what they exercise). Decision: keep the Xcode `IssuesTests` target as a thin re-exporter, or delete it. **Recommendation:** delete it. `xcodebuild test -scheme Issues` will still run UI tests; package tests run via `swift test`. CI runs both.

### Phase 7 — Remove the remote `XCRemoteSwiftPackageReference` entries

Once nothing in the Xcode project consumes `Watcher`, `Textual`, `Sparkle`, or `SwiftTUI` directly, delete:
- The four `XCRemoteSwiftPackageReference` blocks in `project.pbxproj`.
- The four corresponding `XCSwiftPackageProductDependency` blocks.
- The four entries from the project-level `packageReferences` array.
- Each native target's now-empty `packageProductDependencies` entries that referenced them.

Build all schemes one more time. Confirm `swift package show-dependencies` from inside `IssuesKit/` lists Watcher, Textual, Sparkle, SwiftTUI as the only direct dependencies.

### Phase 8 — `swift package describe`-level sanity

From `IssuesKit/`:
```
swift package describe
swift build
swift test
```
All three must succeed. The package is now self-contained and the Xcode project depends on it.

## Resources — the hairy bits

1. **`Localizable.xcstrings`** — Apple's String Catalog. Stays in the app target. `String(localized: "key")` calls inside `IssuesAppKit` resolve against `Bundle.main` by default *unless* the macro/initializer is given `bundle: .module`. The `String(localized:)` macro form (the literal-init one) uses `Bundle.main` by default; the runtime initializer `String(localized: LocalizationValue, bundle: Bundle)` requires an explicit bundle. Since the app target *is* `Bundle.main`, package-side views can call `String(localized:)` and the catalog in the app target services the lookups. **No mechanical rewrite needed.**

2. **`Assets.xcassets`** — same logic. `Color("appBackground")` and `Image("foo")` use `Bundle.main` unless told otherwise. Asset catalog stays in the app target; package views resolve via `Bundle.main`. AppIcon must stay in the app target's catalog regardless.

3. **`Help/*.md`** — bundled markdown. `HelpCatalog` currently locates them via `Bundle.main.url(forResource:)`. If the catalog stays in the app target, no change needed. If it moves to the package, the catalog code switches to `Bundle.module` and the markdown files move with it. **Recommendation:** leave Help content in the app target for v1 (one less moving piece). Revisit only if it bites.

4. **Test fixtures** — anything `IssuesTests/` currently loads from disk needs to live alongside the test target as a `resources: [.process("Fixtures")]` or `.copy("Fixtures")` block. Grep `IssuesTests/` for `Bundle(for:)`, `URL(fileURLWithPath:)`, and `#filePath` to find them.

## Gotchas

- **Public access modifiers everywhere.** Inside a single Xcode target, default `internal` access works for cross-file references. Across SwiftPM module boundaries, you need `public`. Every type, every method, every property, every initializer consumed across `IssuesCore` → `IssuesAppKit`, or either of those → the CLI targets, needs `public`. The compiler will tell you which ones, but expect a long tail of these fixes during Phase 3.
- **Tests use `@testable import`.** `@testable` works across SwiftPM modules but requires the target to be built with testing enabled (the default in `swift test`). Per-symbol `internal` access works under `@testable`; no need to publicize internals just to test them.
- **MainActor default isolation is locked in via a shared `swiftSettings` constant.** Package.swift already declares:
  ```swift
  let swiftSettings: [SwiftSetting]? = [.defaultIsolation(MainActor.self)]
  ```
  Every library/source target (`IssuesCore`, `IssuesAppKit`, `IssuesCLI`, `IssuesDashboardCLI`) must pass `swiftSettings: swiftSettings` so moved code keeps the same default isolation as the Xcode-side `SWIFT_DEFAULT_ACTOR_ISOLATION = MainActor` it had before. **The test target intentionally omits it** — tests run nonisolated-by-default so async helpers and detached parsing tests don't collide with `@MainActor`. If any test ends up needing MainActor isolation, decorate that specific test method/class with `@MainActor` rather than flipping the whole target. A missing `swiftSettings:` argument on a library target is the single most likely cause of post-Phase-3 threading bugs; treat it as a checklist item before each phase.
- **Sparkle initialization.** `UpdaterController` reads `SU_FEED_URL` and `SU_PUBLIC_ED_KEY` from the app's `Info.plist`. After moving to the package, the controller still reads `Bundle.main.infoDictionary` — the app target's Info.plist remains the source. Don't accidentally read from `Bundle.module` (which would be the package's resource bundle, which has no Info.plist).
- **Sandbox + security-scoped bookmarks.** `FolderBookmarkService`'s `startAccessingSecurityScopedResource()` works the same from the package — the sandbox grant is on the *process*, not the framework. No change needed.
- **`os.log` subsystem.** `Logging.subsystem` is currently `"co.sstools.Issues"`. After the move, it stays the same — the subsystem string isn't tied to bundle ID, it's free-form. The DashboardCommand variant (`"co.sstools.IssuesDashboard"`) gets removed along with the duplicate parser.
- **Code signing.** SwiftPM library products built as part of an Xcode target's dependency graph inherit the app's code signing automatically. No additional plumbing.
- **Macros.** SwiftPM packages can use macro packages (e.g. `@Observable` from Observation). `Observation` is part of Swift itself; no extra dep declaration needed.
- **Build performance.** First clean build after the refactor will be slow (the local package compiles four targets where previously there was one). Subsequent builds use Xcode's package cache — comparable to the current setup.
- **CI parity.** If any CI lane runs `xcodebuild test`, it'll keep working as long as the Xcode `IssuesTests` target either stays as a re-exporter or is replaced by `swift test`. Audit `.github/workflows/` (if any) before deleting the Xcode test target.
- **`swift package resolve` resolution conflicts.** `swiftui-math` is a transitive dep of `textual`; `swift-concurrency-extras` is a transitive dep of `Watcher`. Both currently resolve cleanly in `Package.resolved`. Carry the existing `Package.resolved` over to the package directory (the SwiftPM `Package.resolved` lives at `IssuesKit/Package.resolved` once Xcode forgets about the remote refs).
- **The local-package reference in `project.pbxproj` is a relative path.** Use `relativePath = "IssuesKit"` not an absolute one — keeps the project portable across checkouts.
- **`PBXFileSystemSynchronizedRootGroup`** stops being relevant once `Issues/`, `IssuesCommand/`, and `DashboardCommand/` are mostly empty. The sync groups can stay (they're harmless when their directories contain just a shim) or be replaced with explicit per-file membership. Either works.

## Risks

1. **Big-bang merge conflict surface.** Every Swift file in `Issues/` moves. Any in-flight branch touching app code will conflict. Mitigation: land this on a quiet day, communicate the cutover, freeze unrelated PRs for the duration.
2. **Default-actor-isolation flip.** Real semantic risk, but mitigated by the shared `swiftSettings` constant already declared in `Package.swift`. The mitigation only works if every library target passes `swiftSettings: swiftSettings` — forgetting it on even one target re-introduces the threading bug surface. Audit each `.target(...)` declaration before each phase build.
3. **Resource lookup regressions.** A view that calls `String(localized: "Open")` and resolves it via `Bundle.main` today might silently resolve via `Bundle.module` after the move if the call shape changes. Smoke-test localization in `es`, `fr`, `de`, `zh-Hans`, `ko` (the five locales currently in `Localizable.xcstrings`) after Phase 4.
4. **Sparkle bundle resolution.** `UpdaterController` reads `Info.plist` keys via `Bundle.main`. If the controller accidentally moves to `Bundle.module`, Sparkle won't find the feed URL or public key and updates silently fail. Smoke-test "Check for Updates…" after Phase 4.
5. **CI lane breaks.** If a CI workflow runs `xcodebuild test` against the deleted `IssuesTests` Xcode target, the build goes red. Audit before Phase 6.
6. **App icon resolution.** If anything moves the asset catalog accidentally, `AppIcon` doesn't resolve and the app ships iconless. Keep the asset catalog in the app target; smoke-test the Dock icon after Phase 4.

## Rollback

The refactor lands as a sequence of commits on a feature branch (`refactor/issueskit-package` or similar). If something blocks on, say, Phase 6 and can't be unblocked, revert the branch back to a known-good phase commit. The phases are designed so each is individually safe: Phase 3 alone, with Watcher proxied through `IssuesCore`, ships a working app. Any later phase can be deferred without leaving the project in an unbuildable state.

## Out of scope

- Splitting `IssuesKit` into multiple packages or moving any of it to a separate git repo.
- Renaming `IssuesKit` to something else.
- Restructuring the existing source tree inside `Issues/` (no folder renames, no architecture changes; just verbatim moves).
- Moving `Issues.entitlements`, `Configuration/Info.plist`, or any xcconfig into the package.
- Touching `IssuesUITests` (it stays in the Xcode project; UI tests need the `.app`).
- The #0119 follow-ups (CLI Copy Files build phase, NSSavePanel install menu item, folder-picker pre-targeting, MARKETING_VERSION injection) — those still ride on top of `IssuesCommand` / `DashboardCommand` as Xcode targets and are unaffected by this refactor.
- Modernising any of the moved code as it moves. No drive-by API changes; the diff is "wherever it was, it's now here, with `public` where needed."

## Notes

The user's stated motivation is the source of truth: Xcode handles SwiftPM packages poorly. Symptoms that bite today include slow package resolution on `git pull`, occasional spurious "missing package product" errors that require resetting the package cache, and the inability to edit dependencies without round-tripping through Xcode's UI. Moving everything into a `Package.swift` editable in any text editor makes dependency changes a one-line diff reviewable in a normal PR — which is the win that justifies the disruption.
