## Description

Issue files commonly reference screenshots via `![cap](NNNN/file.png)`. Textual's `StructuredText` resolves these via the `baseURL` we already pass and renders them inline at full image resolution — which is too big. Screenshots from a 14"+ macOS display can be 3000+ pixels wide, dwarfing the body text and forcing horizontal layout shifts.

Render attachment images at a sensible inline size (e.g. max width 480pt, scaled-to-fit, preserving aspect ratio) so they live alongside body text without dominating it. Provide a click affordance that opens the image at full resolution.

## Behavior

- Inline: image sized to `min(intrinsicWidth, 480pt)` width-wise, scaled proportionally for height. A subtle border + corner-radius matches the rest of the chrome.
- Click an inline image → present a full-size viewer in a new sheet (or popover) with:
  - Original-resolution image inside a scrollable / zoomable frame.
  - Esc / spacebar / click-outside dismisses.
  - Optionally a "Reveal in Finder" button to open the file location.
- If the image fails to load (file missing on disk — already a `LintFinding.missingAttachment`), show a small "missing attachment" placeholder rather than blank space.

## Implementation sketch

Two paths depending on Textual's customization surface:

1. **Override the image rendering inside Textual** if it exposes a hook. Cleanest if available — we'd inject our own thumbnail+click view for `<img>` nodes.
2. **Pre-process the markdown** before passing to `StructuredText`: extract every `![…](path)` reference, replace with a placeholder, render the body text as one chunk and the images as separate `AttachmentThumbnailView` instances laid out in order. More work but doesn't depend on Textual exposing a delegate.

Path 1 is preferable; investigate Textual's API first.

For the full-size viewer, a new `AttachmentSheet` view wraps the `Image` in a `ScrollView` with `.magnificationGesture()` for pinch-zoom (or just `.frame(maxWidth: .infinity)` for v1 — zooming can come later).

## Caveats

- The detail panel (`DetailPanelDescriptionView`) and the preview sheet (`IssueMarkdownSheetContentView`) both render via `DetailPanelDescriptionView` post-#0048, so the change lands once and applies to both.
- Animated GIFs / oversized PNGs from screenshot bursts: cap by file size as a guard? Not for v1; only address if it surfaces as a real perf issue.
- Image accessibility: the markdown alt text should become the SwiftUI `Image`'s accessibility label so VoiceOver reads "screenshot of broken reply button" instead of "image".

## Verification

- Open an issue whose body has a `![cap](NNNN/screenshot.png)` reference. The image renders at ≤480pt wide, not full screen.
- Click the image → opens at full resolution in a sheet.
- Esc / click-outside dismisses cleanly.
- Issues without attachments render unchanged.
- Issues with broken attachment paths (covered by `LintFinding.missingAttachment`) show a small placeholder rather than crashing or blank space.

## Root cause

The `DetailPanelDescriptionView` rendered the entire body, including image references, through `StructuredText`. Textual's attachment loader honored `baseURL`, so `NNNN/file.png` resolved correctly — but it inlined the image at the file's intrinsic pixel size. Retina-era screenshots routinely run 2400+ pixels wide, so the body laid out 2–4× wider than the description column and forced horizontal overflow.

## Fix

Pre-process the body markdown before handing it to `StructuredText` and lift `![alt](path)` references out into sibling SwiftUI views the description column lays out itself:

- `InlineImageMarkdown.split` walks the body once, splits around image refs, and skips refs inside fenced/inline code spans (mirrors `IssueCrossRef` and `LintRunner.stripCodeSpans`). External (`http://`, `https://`, `data:`) refs are passed through to Textual unchanged.
- `AttachmentThumbnailView` renders a width-capped (`min(intrinsic, 480pt)`), bordered, clickable thumbnail. Missing files fall back to a "Missing attachment" placeholder so blank space doesn't appear (the lint sheet keeps the actionable channel via `LintFinding.missingAttachment`).
- `AttachmentSheet` presents the original-resolution image in a scrollable frame with a Reveal-in-Finder action and Esc/space dismissal that matches the existing preview-sheet key vocabulary.
- `DetailPanelDescriptionView` now renders an ordered `VStack` of prose chunks (via `StructuredText`) and image chunks (via `AttachmentThumbnailView`), and owns the sheet-presentation `@State` so multiple thumbnails share one sheet.

Both the detail panel and the preview sheet (`IssueMarkdownSheetContentView`) get the new behavior automatically because both go through `DetailPanelDescriptionView`.

## Gotchas

- **Path 1 (Textual hook) was investigated and ruled out.** Textual exposes `imageAttachmentLoader` via `.textual.imageAttachmentLoader(_:)`, which lets you return a custom `Attachment` whose `body` is any SwiftUI view. The catch is in `Internal/Attachment/AttachmentView.swift`: attachment bodies are drawn through `Canvas.resolveSymbol` + `context.draw(symbol, in:)`, and `Canvas` does not hit-test child views. A `.onTapGesture` on the attachment body silently never fires, so we couldn't drive the click affordance from inside the loader. Path 2 (pre-process) was therefore the only viable option.
- The `Image` type Textual hands the loader closure (`URLAttachmentLoader<Content>`) is internal to the package, so even if Canvas hit-testing weren't a concern we couldn't reuse `URLAttachmentLoader` directly with our own thumbnail content type.
- Pre-processing happens on the cross-ref-rewritten body, so `#NNNN` mentions inside image alt text continue to work correctly (they're handled by `IssueCrossRef.rewrite` before `InlineImageMarkdown.split`).
- Image refs inside fenced or inline code blocks are intentionally left as literal text in the prose chunk — same exclusion the linter and cross-ref pre-processor already use.

## Round 2 (2026-05-05)

Round 1 (commit `c19baa4`) wired up the splitter, thumbnail, and sheet but the two user-visible behaviors weren't actually finished. After running the rebuilt app the user reported:

1. Inline thumbnails still rendered at full image dimensions — the cap wasn't taking effect for tall captures.
2. The full-size viewer opened with scrollbars and no zoom controls; pinch-to-zoom and panning were missing.

### Round 2 root cause

- `AttachmentThumbnailView` capped only width at 480pt and computed height proportionally from the intrinsic aspect ratio. For a 1200×2400 portrait screenshot that's 480×960 — narrow enough to fit the column, but tall enough to dominate the description block. There was no max-height clamp.
- `AttachmentSheet` wrapped the image in a `ScrollView([.horizontal, .vertical])` and pinned its frame to native pixel dimensions. Comment in the file explicitly deferred zoom ("Pinch-zoom is intentionally out for v1"). Result: opening a 3024-pixel-wide capture in a 900pt sheet showed scrollbars and a zoomed-in fragment, with no way to zoom out to fit.

### Round 2 fix

`AttachmentThumbnailView`:
- Drop the cap to **360×240pt** and clamp **both** axes. New `displaySize` picks the limiting ratio (`min(maxWidth/intrinsicWidth, maxHeight/intrinsicHeight)`) and never upscales (clamps the scale at 1).
- Image keeps `.resizable() + .scaledToFit()` and is now framed at the computed fit size, so portrait and landscape captures both behave like thumbnails.

`AttachmentSheet`:
- Replace the `ScrollView` with a `GeometryReader` that computes a fit-scale from the available container size. The image renders at `image.size * fitScale * zoom`, so `zoom == 1` means "fit," not "1:1."
- Add `@State zoom: CGFloat = 1` driven by `MagnificationGesture` (clamped `[0.5, 8]`, with a `zoomAnchor` snapshot so successive pinches compose).
- Add `@State offset: CGSize` + `DragGesture` that only pans when `zoom > 1` (panning at fit-scale is meaningless).
- Double-click toggles between fit and 2×.
- Header gains a zoom-controls row: minus / percent label / plus / fit, wired with Cmd-`-`, Cmd-`=`, and Cmd-`0` keyboard shortcuts.
- Reveal-in-Finder, Esc, and space-to-dismiss all unchanged.
