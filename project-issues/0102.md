## Description

Phase 4 of the remote-access plan (#0076). The viewer keeps a single WebSocket open per host (#0100). On the wire, networks drop, hosts restart, the viewer's app sleeps, etc. ‚Äî the WS needs to reconnect cleanly without the views noticing more than a brief "reconnecting‚Ä¶" indicator.

## Where to look

- **`Issues/Remote/RemoteHostIssueSource.swift`** (from #0094) ‚Äî currently has a `// TODO(#0102)` for the WS wiring. Replace with the real implementation.
- **New: `Issues/Remote/RemoteWebSocket.swift`** ‚Äî small wrapper around `URLSessionWebSocketTask` with reconnect, ping/pong, and a published `RemoteEvent` stream.

## Behavior

### Connection lifecycle

1. On `RemoteHostIssueSource.start()`, after the initial `/v1/folders/{id}` and `/v1/folders/{id}/issues` fetches succeed, open `wss-or-ws://<host>:<port>/v1/events` with the bearer token in the `Authorization` header.

   - `URLSessionWebSocketTask` accepts the header on the request used to create the task. (We're plain `ws://` for v1 ‚Äî no TLS yet, per `RemoteAccess.md`.)

2. Send `{ "type": "subscribe", "folderIds": ["<this folderId>"] }` immediately on open.

3. While open, dispatch incoming events into the source's change stream:
   - `reload` ‚Üí call `reload()` on this source (re-fetches the issues list).
   - `update` ‚Üí re-fetch just that issue's body, replace in the in-memory list.
   - `delete` ‚Üí drop the matching issue.
   - `unsubscribed` ‚Üí emit `.folderUnavailable` for the affected folder.
   - `hello` ‚Üí ignored (host-info already known).
   - `pong` ‚Üí ignored (we keep our own ping schedule).

### Reconnect

Reconnect on:

- TCP / WS error.
- WS close with any code other than 1000 (normal closure) or 4001 (token revoked ‚Äî see below).
- No `pong` within 10 sec of a sent ping.
- `NWPathMonitor` reports the path went unsatisfied and is now satisfied again (paired with #0105 for the host side).

Reconnect strategy:

- Backoff: 1s, 2s, 4s, 8s, 16s ‚Äî capped at 30s. Jitter ¬±25%.
- Reset backoff on a successful open.
- Surface `.disconnected` to the source on the first failure; `.reconnected` (or `.reloaded`, since reload happens on resub) on success. The viewer's tab indicator (#0104) listens for these.

Stop reconnecting on:

- WS close code 4001 ‚Üí token was revoked. Emit `.tokenInvalid`, don't retry. (#0104 handles UX.)
- Source `stop()` ‚Üí user closed the tab.

### Resubscribe

After reconnect, immediately send `subscribe` for the folder this source watches. The host treats this as additive (#0100) ‚Äî if the previous session's subscription is still being torn down on the host side, the new subscribe just re-asserts.

### Pings

Send a `{ "type": "ping" }` message every 25 seconds. The host responds with `pong`. If no `pong` within 10 sec ‚Üí close + reconnect.

The host already pings every 30 sec (#0100). Two-sided pings are belt-and-suspenders but cheap.

## Verification

- Cut the network on the viewer (turn Wi-Fi off, leave Tailscale): tab shows "reconnecting‚Ä¶" within 10 sec. Turn Wi-Fi back on ‚Üí tab reconnects within ~10 sec, content refreshes via the resubscribe.
- Restart the host (toggle hosting off and back on): viewer reconnects within ~30 sec, no manual refresh needed.
- Revoke the token on the host: viewer surfaces "token revoked" UI within seconds; no retry storm in logs.
- Quit the host: viewer surfaces "disconnected" with retries continuing on the backoff schedule indefinitely until stopped.
- Long-running session (overnight): viewer stays connected; no leak in `URLSessionWebSocketTask` retain count.

## Notes

- Don't try to re-deliver missed events. The contract is "you'll get a `reload` after reconnect"; the viewer reloads on resub regardless.
- Don't pile up multiple reconnect timers if `start()` is called repeatedly. The wrapper should be re-entrant: a fresh `start()` cancels any in-flight reconnect.
- Backoff state is per-source; don't share across sources. A flapping host shouldn't slow down reconnect for a healthy host.

## Scope (out)

- TLS ‚Äî out of v1.
- Resumable subscriptions with sequence numbers ‚Äî overkill for the file-watcher model.
- The "reconnecting‚Ä¶" tab indicator UI ‚Äî #0104.
- NWPathMonitor wiring on the host ‚Äî #0105.

## Root cause

#0094 left a `// TODO(#0102)` marker in `RemoteHostIssueSource.start()`; reload was the only refresh mechanism. With #0100's WS endpoint now live the viewer can finally subscribe and react to host-side events in real time.

## Fix

Two pieces:

- **`Issues/Remote/RemoteWebSocket.swift`** (new, `@MainActor`) wraps `URLSessionWebSocketTask`. Bearer auth via `Authorization` header on the upgrade request. Sends `subscribe(folderIds:)` immediately on open. The receive loop decodes `RemoteEvent` frames via `RemoteProtocol.decoder`; pong frames clear the pong deadline silently, the rest yield on `events: AsyncStream<RemoteWebSocketEvent>`. Application-level ping every 25 s with a 10 s pong deadline ‚Üí forced reconnect on miss. Reconnect backoff is `[1, 2, 4, 8, 16]` s with ¬±25% jitter, capped at 30 s; the first successful frame after open resets the counter. Close code `4001` (token revoked) yields `.tokenInvalid` and stops without retrying. `stop()` is idempotent and finishes the stream.
- **`RemoteHostIssueSource.start()`** now opens the WS once per source lifetime via `startWebSocketIfNeeded()` after the initial HTTP fetches succeed. `handleWebSocketEvent(_:)` translates incoming `RemoteWebSocketEvent` into the source's existing `RemoteIssueSourceEvent` channel:
  - `reload(folderId == self.folderId)` ‚Üí `reload()` on the source.
  - `update(folderId, id)` ‚Üí drop the body cache for `id`, re-fetch metadata, yield `.issueChanged(id)`.
  - `delete(folderId, id)` ‚Üí drop the row, `onUpdate?(self)`, yield `.issueChanged(id)`.
  - `unsubscribed(folderId)` ‚Üí `folderInvalidated = true`, yield `.folderUnavailable`.
  - `disconnected` ‚Üí `loadError = "Connection to host lost ‚Äî reconnecting‚Ä¶"`, yield `.disconnected`. The socket schedules its own reconnect.
  - `tokenInvalid` ‚Üí `loadError = "Token rejected by host."`, yield `.tokenInvalid`, tear the socket down (no further retries).
  - `hello` and `pong` are ignored.

`stop()` tears down both the in-flight reload Task and the WS listener Task plus the socket itself.

`RemoteCommand` grew the static factories the new client uses: `.subscribe(folderIds:)`, `.unsubscribe(folderIds:)`, `.ping` ‚Äî matching `RemoteEvent`'s existing pattern.

## Gotchas

- **`URLSessionWebSocketTask.closeCode` may report `.invalid` after a transport failure** before the server's close frame fully arrives. We probe it on the failure path so a graceful 4001 from the host still triggers `.tokenInvalid`; non-4001 paths fall through to reconnect.
- **The receive loop is a recursive `task.receive { ‚Ä¶ }`** ‚Äî `URLSessionWebSocketTask` only delivers one message per call. Forgetting to schedule the next receive after a successful frame silently stalls the connection.
- **Subscribe is additive on the host** (#0100). After reconnect we re-send `subscribe` for our `folderId`; if the host hasn't fully torn down the prior session yet, that's harmless ‚Äî the new subscribe just re-asserts.
- **Don't share backoff state across sources.** Each `RemoteHostIssueSource` owns its own `RemoteWebSocket` with its own counter; one flapping host doesn't slow down a healthy one.
- **Visual verification still needed** for the "reconnecting‚Ä¶" UX surface ‚Äî that's #0104's territory. This issue lands the transport; the user-visible chip indicator is the next ticket.
