## Description

Phase 4 of the remote-access plan (#0076). The viewer keeps a single WebSocket open per host (#0100). On the wire, networks drop, hosts restart, the viewer's app sleeps, etc. — the WS needs to reconnect cleanly without the views noticing more than a brief "reconnecting…" indicator.

## Where to look

- **`Issues/Remote/RemoteHostIssueSource.swift`** (from #0094) — currently has a `// TODO(#0102)` for the WS wiring. Replace with the real implementation.
- **New: `Issues/Remote/RemoteWebSocket.swift`** — small wrapper around `URLSessionWebSocketTask` with reconnect, ping/pong, and a published `RemoteEvent` stream.

## Behavior

### Connection lifecycle

1. On `RemoteHostIssueSource.start()`, after the initial `/v1/folders/{id}` and `/v1/folders/{id}/issues` fetches succeed, open `wss-or-ws://<host>:<port>/v1/events` with the bearer token in the `Authorization` header.

   - `URLSessionWebSocketTask` accepts the header on the request used to create the task. (We're plain `ws://` for v1 — no TLS yet, per `RemoteAccess.md`.)

2. Send `{ "type": "subscribe", "folderIds": ["<this folderId>"] }` immediately on open.

3. While open, dispatch incoming events into the source's change stream:
   - `reload` → call `reload()` on this source (re-fetches the issues list).
   - `update` → re-fetch just that issue's body, replace in the in-memory list.
   - `delete` → drop the matching issue.
   - `unsubscribed` → emit `.folderUnavailable` for the affected folder.
   - `hello` → ignored (host-info already known).
   - `pong` → ignored (we keep our own ping schedule).

### Reconnect

Reconnect on:

- TCP / WS error.
- WS close with any code other than 1000 (normal closure) or 4001 (token revoked — see below).
- No `pong` within 10 sec of a sent ping.
- `NWPathMonitor` reports the path went unsatisfied and is now satisfied again (paired with #0105 for the host side).

Reconnect strategy:

- Backoff: 1s, 2s, 4s, 8s, 16s — capped at 30s. Jitter ±25%.
- Reset backoff on a successful open.
- Surface `.disconnected` to the source on the first failure; `.reconnected` (or `.reloaded`, since reload happens on resub) on success. The viewer's tab indicator (#0104) listens for these.

Stop reconnecting on:

- WS close code 4001 → token was revoked. Emit `.tokenInvalid`, don't retry. (#0104 handles UX.)
- Source `stop()` → user closed the tab.

### Resubscribe

After reconnect, immediately send `subscribe` for the folder this source watches. The host treats this as additive (#0100) — if the previous session's subscription is still being torn down on the host side, the new subscribe just re-asserts.

### Pings

Send a `{ "type": "ping" }` message every 25 seconds. The host responds with `pong`. If no `pong` within 10 sec → close + reconnect.

The host already pings every 30 sec (#0100). Two-sided pings are belt-and-suspenders but cheap.

## Verification

- Cut the network on the viewer (turn Wi-Fi off, leave Tailscale): tab shows "reconnecting…" within 10 sec. Turn Wi-Fi back on → tab reconnects within ~10 sec, content refreshes via the resubscribe.
- Restart the host (toggle hosting off and back on): viewer reconnects within ~30 sec, no manual refresh needed.
- Revoke the token on the host: viewer surfaces "token revoked" UI within seconds; no retry storm in logs.
- Quit the host: viewer surfaces "disconnected" with retries continuing on the backoff schedule indefinitely until stopped.
- Long-running session (overnight): viewer stays connected; no leak in `URLSessionWebSocketTask` retain count.

## Notes

- Don't try to re-deliver missed events. The contract is "you'll get a `reload` after reconnect"; the viewer reloads on resub regardless.
- Don't pile up multiple reconnect timers if `start()` is called repeatedly. The wrapper should be re-entrant: a fresh `start()` cancels any in-flight reconnect.
- Backoff state is per-source; don't share across sources. A flapping host shouldn't slow down reconnect for a healthy host.

## Scope (out)

- TLS — out of v1.
- Resumable subscriptions with sequence numbers — overkill for the file-watcher model.
- The "reconnecting…" tab indicator UI — #0104.
- NWPathMonitor wiring on the host — #0105.
