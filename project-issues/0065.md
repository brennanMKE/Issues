## Description

Follow-up to #0064. The text-only markdown report is useful but a single visualization carries more at-a-glance information than a paragraph of counts. Render a PNG chart alongside each generated report and reference it inline so the markdown viewer (Issues.app, GitHub, any standard renderer) shows the chart with the prose.

## What to render

The strongest single-image candidates, by usefulness:

1. **Status donut** ŌĆö a donut chart with one slice per `IssueStatus`, sized by count, colored by `status.foreground`. Dense and instantly readable.
2. **Activity sparkline** ŌĆö the same 30-day opens/closes bar chart from #0057. Shows project rhythm.
3. **Module distribution** ŌĆö horizontal bar chart of issues per primary module. Useful but redundant with the markdown table.

Ship the **status donut** for v1. The activity sparkline can come as a sibling image or a second pass.

## Output

Save the PNG next to the markdown:


Reference from the markdown via a relative path so the image renders inline:


Because the path is relative, GitHub renders it on the issue-folder repo's web view, Issues.app's own preview sheet renders it (`baseURL` is the file's directory, see #0048's `DetailPanelDescriptionView`), and any third-party markdown viewer the user pipes the file through resolves it correctly.

## Implementation sketch

Use SwiftUI's `ImageRenderer` (per the project's swift.md guidance ŌĆö don't reach for `UIGraphicsImageRenderer`). Build a small `StatusDonutView`:

```swift
struct StatusDonutView: View {
    let counts: [IssueStatus: Int]

    var body: some View {
        Chart(slices, id: \.status) { slice in
            SectorMark(
                angle: .value("Count", slice.count),
                innerRadius: .ratio(0.55),
                angularInset: 1.5
            )
            .foregroundStyle(slice.status.foreground)
            .annotation(position: .overlay) {
                Text("\(slice.count)")
                    .font(.system(size: 11, weight: .bold))
                    .foregroundStyle(.white)
            }
        }
        .frame(width: 360, height: 240)
        .padding(20)
        .background(Color.appBackground)
    }

    private var slices: [(status: IssueStatus, count: Int)] {
        IssueStatus.displayOrder.compactMap {
            guard let n = counts[$0], n > 0 else { return nil }
            return ($0, n)
        }
    }
}
```

Render via `ImageRenderer`:

```swift
@MainActor
static func writeStatusDonutPNG(counts: [IssueStatus: Int], to url: URL) throws {
    let renderer = ImageRenderer(content: StatusDonutView(counts: counts))
    renderer.scale = 2  // retina-friendly export
    guard let nsImage = renderer.nsImage,
          let tiff = nsImage.tiffRepresentation,
          let bitmap = NSBitmapImageRep(data: tiff),
          let png = bitmap.representation(using: .png, properties: [:]) else {
        throw ReportImageError.renderFailed
    }
    try png.write(to: url)
}
```

`ReportGenerator.generate(for:)` (from #0064) calls this before writing the markdown, so by the time the `.md` references the `.png` the file is already on disk.

## Color scheme

The donut should be readable on both white-paper printing and screen viewing. Render in **light scheme** explicitly:

```swift
StatusDonutView(counts: counts)
    .environment(\.colorScheme, .light)
```

The slice fills are the `Color.statusOpen` / `.statusInProgress` / etc. from the asset catalog, which already work as opaque foregrounds. Use `Color.appBackground` resolved against light mode for the background ŌĆö gives a clean white card that prints cleanly and reads correctly inline on dark-themed markdown viewers (GitHub dark mode shows white panels around images uniformly).

## Edge cases

- Zero issues: donut would have no slices. Skip the image entirely; the report's existing "No issues yet." prose carries the message.
- One status with all the issues: a full-disc donut with one slice. Still informative; render it.
- New status added in a future build that the asset catalog doesn't have a color for: `status.foreground` falls through to the existing `IssueStatus` extension and yields one of the five known colors. No crash.
- Sandbox write: same constraint as #0064 ŌĆö writes are scoped strictly to `<folder>/reports/`. The PNG never touches the issue markdown files. Same `assert(url.path.contains("/reports/"))` guard the report writer uses applies here.

## Out of scope

- The activity sparkline as a second image. Could be a follow-up; not blocking.
- Per-module pie / bar charts. Defer; the donut is the highest-density single image.
- SVG export. PNG covers every viewer; SVG is unnecessary complexity.

## Root cause

#0064 shipped a text-only report. This issue adds the visualization the spec called for so the report carries its summary at a glance.

## Fix

Two pieces:

- **`Issues/Views/StatusDonutView.swift`** (new) ŌĆö `Charts`-based `SectorMark` donut, one slice per `IssueStatus`, sized by count, colored by `status.foreground`. The view is forced into `.environment(\.colorScheme, .light)` with a white background so the export reads identically on any markdown viewer (Issues.app's preview sheet, GitHub light/dark, third-party tools).
- **`Issues/Services/ReportGenerator.writeStatusDonutPNG(counts:to:expectedDirectory:)`** (new static) ŌĆö `ImageRenderer(content: StatusDonutView(...))` at `scale = 2`, exports via `nsImage.tiffRepresentation` ŌåÆ `NSBitmapImageRep` ŌåÆ `.png`. Same write-scope guard as the markdown writer (resolved target dir must equal `reports/`).

`ReportGenerator.generate(for:)` now writes the PNG first (with the same filename stem as the markdown ŌĆö `report-<date>.png`), then composes the body. If the PNG writes successfully and the store has any issues, the body inserts an `![Status snapshot](<filename>)` line right after the timestamp header. Same-day re-runs land at `report-<date>-2.md` / `report-<date>-2.png` thanks to the existing `nextAvailableURL` plumbing.

Edge cases per spec:
- **Zero issues** ŌĆö PNG is skipped (the renderer returns an error on no slices, which `try?` swallows); the markdown stays donut-free and the existing "No issues yet." prose carries the message.
- **One status with all issues** ŌĆö single full-disc slice renders correctly.
- **Future statuses without an explicit color** ŌĆö `IssueStatus.foreground` falls through to one of the five known colors via the existing extension. No crash.

## Gotchas

- **`ImageRenderer.nsImage` is `@MainActor`** ŌĆö `ReportGenerator` is already `@MainActor`, so the call lands on the right actor. If a future refactor moves the writer off main, the renderer call has to come back through `MainActor.run`.
- **The PNG signature check** (`89 50 4E 47 0D 0A 1A 0A`) is the cheap way to confirm the export actually produced a PNG and not some other format that NSBitmapImageRep happened to encode.
- **Image embedding is relative**, not absolute. Don't be tempted to fully qualify the path ŌĆö that would break the GitHub-renders-it use case and stops Issues.app's preview from resolving against the file's directory.
- **Write-scope guard is intentional belt-and-suspenders.** The markdown writer already has one; the PNG writer has its own. If either ever tries to escape `reports/`, `assertionFailure` fires loud in Debug and `.writeScopeViolation` is thrown.
- **Visual verification still needed** for the donut's actual look in light/dark markdown viewers, slice ordering, and the count annotations being legible against the slice colors.
