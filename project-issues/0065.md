# 0065 — Embed an inline status-snapshot PNG in the generated report

| | |
|---|---|
| **Status** | open |
| **Module** | Services / Views |
| **Platform** | macOS |
| **First seen** | 2026-05-05 |

## Description

Follow-up to #0064. The text-only markdown report is useful but a single visualization carries more at-a-glance information than a paragraph of counts. Render a PNG chart alongside each generated report and reference it inline so the markdown viewer (Issues.app, GitHub, any standard renderer) shows the chart with the prose.

## What to render

The strongest single-image candidates, by usefulness:

1. **Status donut** — a donut chart with one slice per `IssueStatus`, sized by count, colored by `status.foreground`. Dense and instantly readable.
2. **Activity sparkline** — the same 30-day opens/closes bar chart from #0057. Shows project rhythm.
3. **Module distribution** — horizontal bar chart of issues per primary module. Useful but redundant with the markdown table.

Ship the **status donut** for v1. The activity sparkline can come as a sibling image or a second pass.

## Output

Save the PNG next to the markdown:

```
<watched-folder>/reports/
├── report-2026-05-05.md
└── report-2026-05-05.png
```

Reference from the markdown via a relative path so the image renders inline:

```markdown
## Summary

![Status snapshot](report-2026-05-05.png)

- **Total issues:** N
…
```

Because the path is relative, GitHub renders it on the issue-folder repo's web view, Issues.app's own preview sheet renders it (`baseURL` is the file's directory, see #0048's `DetailPanelDescriptionView`), and any third-party markdown viewer the user pipes the file through resolves it correctly.

## Implementation sketch

Use SwiftUI's `ImageRenderer` (per the project's swift.md guidance — don't reach for `UIGraphicsImageRenderer`). Build a small `StatusDonutView`:

```swift
struct StatusDonutView: View {
    let counts: [IssueStatus: Int]

    var body: some View {
        Chart(slices, id: \.status) { slice in
            SectorMark(
                angle: .value("Count", slice.count),
                innerRadius: .ratio(0.55),
                angularInset: 1.5
            )
            .foregroundStyle(slice.status.foreground)
            .annotation(position: .overlay) {
                Text("\(slice.count)")
                    .font(.system(size: 11, weight: .bold))
                    .foregroundStyle(.white)
            }
        }
        .frame(width: 360, height: 240)
        .padding(20)
        .background(Color.appBackground)
    }

    private var slices: [(status: IssueStatus, count: Int)] {
        IssueStatus.displayOrder.compactMap {
            guard let n = counts[$0], n > 0 else { return nil }
            return ($0, n)
        }
    }
}
```

Render via `ImageRenderer`:

```swift
@MainActor
static func writeStatusDonutPNG(counts: [IssueStatus: Int], to url: URL) throws {
    let renderer = ImageRenderer(content: StatusDonutView(counts: counts))
    renderer.scale = 2  // retina-friendly export
    guard let nsImage = renderer.nsImage,
          let tiff = nsImage.tiffRepresentation,
          let bitmap = NSBitmapImageRep(data: tiff),
          let png = bitmap.representation(using: .png, properties: [:]) else {
        throw ReportImageError.renderFailed
    }
    try png.write(to: url)
}
```

`ReportGenerator.generate(for:)` (from #0064) calls this before writing the markdown, so by the time the `.md` references the `.png` the file is already on disk.

## Color scheme

The donut should be readable on both white-paper printing and screen viewing. Render in **light scheme** explicitly:

```swift
StatusDonutView(counts: counts)
    .environment(\.colorScheme, .light)
```

The slice fills are the `Color.statusOpen` / `.statusInProgress` / etc. from the asset catalog, which already work as opaque foregrounds. Use `Color.appBackground` resolved against light mode for the background — gives a clean white card that prints cleanly and reads correctly inline on dark-themed markdown viewers (GitHub dark mode shows white panels around images uniformly).

## Edge cases

- Zero issues: donut would have no slices. Skip the image entirely; the report's existing "No issues yet." prose carries the message.
- One status with all the issues: a full-disc donut with one slice. Still informative; render it.
- New status added in a future build that the asset catalog doesn't have a color for: `status.foreground` falls through to the existing `IssueStatus` extension and yields one of the five known colors. No crash.
- Sandbox write: same constraint as #0064 — writes are scoped strictly to `<folder>/reports/`. The PNG never touches the issue markdown files. Same `assert(url.path.contains("/reports/"))` guard the report writer uses applies here.

## Verification

- Generate a report → both `report-<date>.md` and `report-<date>.png` appear in `reports/`.
- Open the `.md` in Issues.app's preview sheet → the donut renders inline above the summary list.
- Open the `.md` on GitHub (after committing) → the donut renders inline there too.
- The donut visually matches the in-app stats bar (same colors, same proportions).
- Light mode forced even if the user's system is dark — the chart is legible regardless of viewer appearance.

## Out of scope

- The activity sparkline as a second image. Could be a follow-up; not blocking.
- Per-module pie / bar charts. Defer; the donut is the highest-density single image.
- SVG export. PNG covers every viewer; SVG is unnecessary complexity.
