# 0069 — Detail panel width jumps with content and cannot be resized

| | |
|---|---|
| **Status** | open |
| **Module** | Views |
| **Platform** | macOS |
| **First seen** | 2026-05-06 |

## Description

The right sidebar that renders the selected issue's markdown (`DetailPanelView`) sizes itself to the intrinsic width of its content. Switching from a short issue to a long one — or one with a wider attachment thumbnail — causes the panel to grow or shrink, which in turn pushes the swimlane / list / timeline content area around. The panel also has no drag handle, so the user cannot pin it to a width they prefer.

Two coupled problems:

1. **Stability**: the panel's width should NOT depend on the currently-selected issue. Once the panel is open, its width stays constant as the user navigates between issues.
2. **User control**: the user should be able to resize the panel by dragging its leading edge. Allow widths from a sensible minimum (e.g. 280pt) up to **one-third of the window's current width**. The chosen width persists per-window for the session (and ideally across launches via `UserDefaults`).

## Where to look

- `Issues/Views/DetailPanelView.swift` — the panel itself.
- `Issues/Views/MainView.swift` — wraps the content area + detail panel in an `HStack`. The width logic likely lives here.

Today the panel is probably rendering with no `.frame(width:)` (so it grows to its content's intrinsic width). The fix needs:

- A `@State` (or `@AppStorage`) value for the panel width, initialized to a default like 360pt.
- `.frame(width: panelWidth)` on the panel.
- A draggable resize handle on the panel's leading edge — typically a 6pt-wide invisible spacer with `.onHover { NSCursor.resizeLeftRight.push() / .pop() }` and a `DragGesture` that updates `panelWidth`, clamped to `[280, mainWindowWidth / 3]`.

`GeometryReader` on `MainView` (or just reading the window size from a `@Environment` value if available) gives the current main width for the upper-bound clamp.

## Behavior details

- Min width: ~280pt (enough for the metadata grid + a 240-wide thumbnail with breathing room).
- Max width: `windowWidth / 3` — recompute live as the user resizes the window. If the window shrinks below the panel's current width × 3, the panel shrinks to fit.
- Resize handle: invisible 6pt strip on the panel's leading edge. Cursor flips to horizontal-resize on hover. Drag updates width with snapping to no specific increment.
- Persistence: `@AppStorage("detailPanelWidth")` so the user's preference survives launches.
- Hidden state (no issue selected → panel collapsed) is unaffected; this only governs width when the panel is visible.

## Verification

- Open an issue with a short body, then switch to one with a long body and a wide thumbnail. The panel should not change width.
- Drag the panel's leading edge inward and outward; cursor should switch to a horizontal-resize cursor on hover.
- Resize past 1/3 of the window — width clamps. Shrink the window — panel shrinks if needed.
- Quit and relaunch — chosen width persists.

## Notes

Related to #0056's inline thumbnails (the wide attachments are part of why content-driven width feels worst right now). This issue isn't a regression from #0056 — the underlying width-from-content layout pre-dates it — but the symptom became more visible after thumbnails landed.
