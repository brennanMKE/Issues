## Description

Tracking issue for a structural simplification of the remote-access design. Rather than the original "any Mac can host, viewers discover hosts via Bonjour" model, the canonical setup is:

- **One always-on Mac mini** runs Issues.app as the host. Its existing `RemoteServer` (`#0079`) + `/v1/events` WebSocket endpoint (`#0100`) already serve everything a remote viewer needs.
- **The MacBook (and eventually iPhone via `#0108`)** is the viewer. It connects to the Mac mini directly using a known address — a Tailscale MagicDNS name (`mac-mini`), a `.local` mDNS-resolved name, or a typed-in IP / hostname.
- **TLS is required** for the connection. ATS demands `wss://` for the WebSocket and rejects plain `http://` after `#0110` is rolled back. There's no CA-issued certificate available for the Mac mini (no public DNS name, no Let's Encrypt path inside the user's own network), so the cert is **self-signed** and the viewer pins its fingerprint. That's exactly what `#0111`–`#0115` already plans.

The original Bonjour cluster (`#0088` / `#0089` / `#0090` / `#0093`) becomes **wontfix**: discovery isn't needed when there's one known host, and Apple's multicast entitlement (request `P8372QL98M`) is no longer blocking the design.

## Why this is a clarification, not a new architecture

What I almost filed (incorrect): a Cloudflare-hosted WebSocket relay that brokers connections between arbitrary Issues.app instances. That would have meant rewriting the host as an outbound WSS client, deleting most of `#0079`, and putting user data through a third-party endpoint.

What the user actually wants (correct): the Mac mini *is* the web server. The MacBook connects to it directly. This matches the original `RemoteAccess.md` design with two refinements:

1. **Bonjour is replaced by direct addressing.** Tailscale MagicDNS handles "find my Mac mini from anywhere," which is the only discovery the user needs. The "browse all hosts on the LAN" Bonjour feature doesn't apply when there's exactly one host the user cares about.
2. **TLS is treated as mandatory** rather than v1-optional. ATS won't let the viewer connect over plain `http://` without an entitlement (`#0110` was the v1 stop-gap), and a self-signed cert + viewer pinning is the path that scales to the iPhone case too.

## What about ATS and the self-signed cert

This is the load-bearing constraint that drives the TLS work:

- macOS App Transport Security blocks plain `http://` and `ws://` connections by default. `#0110` shipped a global `NSAllowsArbitraryLoads = YES` exception as the v1 stop-gap.
- Under TLS, ATS still rejects self-signed certs by default — but `URLSession`'s server-trust delegate can override the cert-chain check. `#0114` implements exactly that: viewer pins the host's SHA-256 fingerprint at token-paste time, the delegate accepts the cert iff the fingerprint matches.
- Net result: ATS stays **on** for the app. The remote-access path works without the broad exception. The `#0110` exception is removed when `#0114` ships.

The Mac mini generates the self-signed cert on first hosting-enable (`#0112`); the fingerprint is bundled with the bearer token at issue time (`#0113`); the viewer pins it (`#0114`); rotation + mismatch UX is `#0115`. All four were already designed exactly for this case — the only thing this issue changes is the urgency (was "v2 nice-to-have," is now "v1 required").

## Notes

- Tailscale's MagicDNS is what makes "type `mac-mini` and it works from anywhere" practical. Without Tailscale, the user falls back to typing an IP or `.local` name. The app doesn't require Tailscale; it just composes with it.
- The Mac mini's web server is the existing `Issues/Remote/RemoteServer.swift` (per `#0079`). Nothing new to build server-side.
- This design is intentionally **single-host-centric**. Multi-host (viewer connects to several different Mac minis) still works mechanically — each one is just a separate token paste — but isn't the optimization target. The picker's "Recent" list (`#0091`) covers the rare multi-host case.
- Cellular reach for the iPhone works the same way: the iPhone joins Tailscale, then `mac-mini` resolves to the host's Tailscale IP regardless of which network the phone is on. No relay needed.

## Scope (out)

- A cloud-hosted relay. Considered and rejected during this evaluation — adds operational surface (running a Cloudflare Worker / Fly app), routes user data through a third party, and isn't necessary now that "Mac mini as canonical host" is the model.
- Removing `RemoteHostController.pauseForNetworkChange()` (`#0105`). The pause logic is still useful as a "I left home Wi-Fi, pause the listener" affordance, just less load-bearing under the direct-addressing model. Leave it; revisit only if it gets in the way.
- Federating across multiple Mac minis. Single-host model for v1.
- Anything about iCloud-synced tokens. Same as before — out of v1.
