# 0002 — Add a tab interface to monitor multiple project folders simultaneously

## Description

The app currently watches one issues folder at a time — switching folders tears down the old `IssueStore` and replaces it. Add a tab interface so multiple projects can be monitored side-by-side, with each tab bound to its own folder path and its own live `IssueStore`. Switching tabs should be instant (no reload), and each tab's filters, sort, and selection should persist independently.

## Expected behavior

- A tab bar appears below the header (or replaces the single "Switch folder…" button) listing each open project by repo name.
- New tab action ("+") opens the folder picker; the chosen folder becomes a new tab and the active one. The remembered-folders list still works for the picker.
- Each tab owns its own `IssueStore`, `FolderWatcher`, and security-scoped resource lifetime. Background tabs keep watching their folders so file edits stay live when the user switches back.
- Closing a tab tears down its store and watcher, releases the security scope, and removes it from the open set. If it was active, fall back to the next tab (or `FolderPickerView` if none remain).
- Tab order is user-controllable (drag to reorder); the open set + order is persisted across launches.
- Each tab independently remembers selected issue, filters, sort column, and view mode.

## Actual behavior

`RootView` holds a single `@State var store: IssueStore?` (`Issues/Views/RootView.swift:5`). Picking a new folder replaces the store entirely, killing watcher + selection + filters for the previous folder. Only one folder can be observed at a time.

## Notes

- Lift the single optional store to a small `TabsModel` (`@Observable @MainActor`) holding `[IssueStore]` plus an `activeTabID`. `RootView` reads from it; `FolderPickerView` is shown when the array is empty.
- Each `IssueStore` already encapsulates folder URL, watcher, filters, and selection — multi-instance support should mostly work, but verify nothing in the store assumes singleton behavior (e.g. shared `UserDefaults` keys, logger labels).
- `FolderBookmarkService` resolves bookmarks at folder pick time; check that resolving + `startAccessingSecurityScopedResource()` works concurrently for several URLs. The sandbox allows multiple concurrent scopes, but each must be balanced with `stop…` on tab close.
- Persistence: extend the existing `rememberedFolders` UserDefaults entry (or add a sibling `openTabs` entry) storing the ordered list of bookmarks for tabs that were open at quit. On launch, restore tabs in order; if a bookmark is stale, leave the tab present but in an error state with "Locate folder…".
- Tab UI: SwiftUI's `TabView` with `.tabViewStyle(.automatic)` is the obvious starting point but doesn't give a Safari-style closable/draggable bar — likely need a custom horizontal `LazyHStack` of tab chips above the content area. Consider macOS 14+ `TabView(selection:)` with `.tabViewStyle(.sidebarAdaptable)` only if it fits.
- Header title currently derives `<RepoName>` from `folderURL.deletingLastPathComponent().lastPathComponent`. With tabs, the header can drop the repo name (it's on the tab) or show the active tab's name.
- Logging: include the repo name (or a short tab id) in `os.Logger` messages so multi-folder log streams stay readable.
